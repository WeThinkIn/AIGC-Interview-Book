<h1 id="4.4.1_长视频生成的核心难点：长程依赖、漂移与上下文爆炸">4.4.1 长视频生成的核心难点：长程依赖、漂移与上下文爆炸</h1>

> 本节回答长视频生成为什么难。核心不在于帧数增加，而在于模型必须在更长时间尺度上保持主体、场景、状态、动作和事件逻辑一致。

<h2 id="4.4.1.1_面试问题：为什么长视频生成不能简单理解为“短视频生成跑更久”？">4.4.1.1 面试问题：为什么长视频生成不能简单理解为“短视频生成跑更久”？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

长视频生成不是短视频生成的时间外推。短视频通常只需要在几秒内保证局部运动自然、相邻帧连续和 prompt 大致对齐；长视频还要求模型在更长时间范围内保持主体身份、场景布局、物体状态、动作阶段和事件因果关系。

从条件生成角度，视频生成可以写成：

```math
p(x_{1:T} \mid c)
```

当 $T$ 较小时，模型主要处理局部时空一致性；当 $T$ 增大时，模型需要处理更长历史对当前内容的约束：

```math
p(x_{1:T} \mid c)=\prod_{t=1}^{T}p(x_t \mid x_{<t}, c)
```

这个式子不要求所有模型都逐帧自回归生成，而是强调一个事实：后续内容必须与历史内容保持一致。例如角色在第 5 秒和第 40 秒再次出现时，身份、服饰和场景关系不能无故变化。

因此，长视频生成新增了四类核心问题：长程依赖、漂移累积、上下文爆炸和叙事结构。面试中可以概括为：**短视频主要考察局部时空质量，长视频考察长期稳定生成能力。**

<h2 id="4.4.1.2_面试问题：长程依赖和短时一致性有什么区别？">4.4.1.2 面试问题：长程依赖和短时一致性有什么区别？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

短时一致性关注相邻帧或短窗口内是否连续，例如纹理是否闪烁、动作是否跳变、局部光流是否平滑。长程依赖关注远距离片段之间是否保持同一主体、同一场景、同一状态和同一事件逻辑。

短时一致性可以近似理解为局部窗口依赖：

```math
x_t \sim p(x_t \mid x_{t-k:t-1}, c)
```

其中 $k$ 是较短时间窗口。长程依赖则要求模型保留更早历史中的关键信息：

```math
x_t \sim p(x_t \mid \mathcal{M}_{<t}, c)
```

其中 $\mathcal{M}_{<t}$ 表示压缩后的历史记忆，不一定是完整历史帧。

典型长程依赖包括：人物身份是否稳定、空间布局是否延续、物体状态是否被记住、前面发生的事件是否影响后续画面。面试中可以这样区分：**短时一致性解决“这一小段不要抖”，长程依赖解决“整段视频不要忘”。**

<h2 id="4.4.1.3_面试问题：长视频生成中的漂移问题本质是什么？">4.4.1.3 面试问题：长视频生成中的漂移问题本质是什么？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

漂移是指视频随着时间推进逐渐偏离原始条件、早期画面或应有状态。它通常不是某一帧突然错误，而是局部误差不断累积后的结果。

可以抽象为：

```math
\hat{x}_t=f(\hat{x}_{<t}, c)+\epsilon_t
```

如果后续生成继续依赖已经带有误差的历史结果，误差会传播并放大：

```math
E_T=\sum_{t=1}^{T}\epsilon_t
```

这里的误差不只指像素误差，也包括身份、场景、动作方向和语义目标上的偏差。

常见漂移包括：身份漂移、场景漂移、运动漂移和语义漂移。根因通常是条件约束衰减、历史记忆不足、片段边界传播错误，以及训练长度和推理长度不匹配。面试中可以概括为：**短视频更怕 flicker，长视频更怕 drift；前者是局部抖动，后者是长期跑偏。**

<h2 id="4.4.1.4_面试问题：为什么上下文爆炸会成为长视频生成的核心瓶颈？">4.4.1.4 面试问题：为什么上下文爆炸会成为长视频生成的核心瓶颈？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

视频 token 数量同时受时间长度、空间分辨率和 latent 压缩率影响。若 latent token 数近似为：

```math
N=T \times H' \times W'
```

全局 self-attention 的计算复杂度通常近似为：

```math
\mathcal{O}(N^2)=\mathcal{O}\left((T H' W')^2\right)
```

这意味着视频变长以后，成本不只是线性增加。更长的时间轴会放大 attention 计算、显存占用、中间激活、KV cache 和采样延迟。

上下文爆炸还会带来信息选择问题。并不是所有历史帧都值得保留，但主体身份、场景布局、关键事件和当前状态必须被保留。因此长视频生成的核心不是“尽可能看完整历史”，而是**在有限计算预算下保留真正有用的长期信息**。

<h1 id="4.4.2_长视频扩展的技术路径：Causal、Chunk、Sliding-Window-与-Hierarchical">4.4.2 长视频扩展的技术路径：Causal、Chunk、Sliding Window 与 Hierarchical</h1>

> 本节关注如何把视频模型扩展到更长时间。Causal、Chunk、Sliding Window 和 Hierarchical 不是互斥方法，而是从生成方向、片段拆分、局部上下文和全局规划四个角度解决长视频问题。

<h2 id="4.4.2.1_面试问题：长视频生成常见技术路径有哪些？它们分别解决什么问题？">4.4.2.1 面试问题：长视频生成常见技术路径有哪些？它们分别解决什么问题？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

长视频生成常见路径可以概括为四类：

| 路径 | 核心思想 | 解决问题 | 主要风险 |
|---|---|---|---|
| Causal | 当前片段只依赖历史 | 流式生成、在线扩展 | 全局规划较弱 |
| Chunk | 将长视频切成多个片段 | 显存可控、复用短视频模型 | 片段边界不连续 |
| Sliding Window | 用局部窗口滚动生成或去噪 | 平衡上下文和成本 | 窗口外信息丢失 |
| Hierarchical | 先规划全局，再生成局部 | 叙事结构和长期一致性 | 上下层对齐困难 |

实际系统通常组合使用。例如先做 storyboard 或 keyframe 规划，再按 chunk 生成，每个 chunk 内部用 sliding window 保持局部连续，并通过 memory 或 reference bank 传递长期状态。

面试回答应强调：**这些方法不是模型名称，而是长视频生成的通用组织方式。**

<h2 id="4.4.2.2_面试问题：Causal-Video-Generation-为什么适合流式生成？">4.4.2.2 面试问题：Causal Video Generation 为什么适合流式生成？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

Causal Video Generation 的核心是当前时刻只访问历史信息，不依赖未来帧：

```math
p(x_{1:T} \mid c)=\prod_{t=1}^{T}p(x_t \mid x_{<t}, c)
```

在视频扩散或视频 Transformer 中，它不一定表现为逐帧生成，也可以体现为 causal attention mask、causal VAE、历史 latent cache、memory token 或分段递推。

它适合长视频和流式生成，原因有三点：第一，不需要一次性处理完整视频；第二，可以边生成边输出；第三，历史状态可以被压缩和复用。

代价是模型只能基于过去推断未来，缺乏完整未来上下文，容易出现长期规划不足、后段重复、语义逐渐跑偏等问题。因此 causal 路线通常需要配合高层规划、参考记忆或后续一致性校正。

面试中可以总结为：**Causal 方法增强了时间扩展性，但牺牲了一部分全局一致性，需要额外机制补足长期规划。**

<h2 id="4.4.2.3_面试问题：Chunk-与-Sliding-Window-的核心区别是什么？">4.4.2.3 面试问题：Chunk 与 Sliding Window 的核心区别是什么？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

Chunk-based Generation 是按时间把长视频切成多个片段分别生成：

```math
x_{1:T}=\{x^{(1)},x^{(2)},\ldots,x^{(K)}\}
```

生成第 $i$ 个片段时，通常依赖文本、上一片段尾帧、历史摘要或参考库：

```math
p(x^{(i)} \mid x^{(<i)}, c)
```

Sliding Window 则强调用固定长度窗口滚动处理长序列：

```math
\mathcal{W}_i=x_{is:is+W}
```

相邻窗口通常包含重叠区域，用于传递局部上下文：

```math
\mathcal{W}_i \cap \mathcal{W}_{i+1}\neq \emptyset
```

二者的区别在于：Chunk 更强调“分段组织”，Sliding Window 更强调“局部上下文滚动”。Chunk 的难点是片段边界；Sliding Window 的难点是窗口外长期信息不可见。实际系统常将二者结合：按 chunk 组织长视频，在 chunk 内或 chunk 间使用重叠窗口增强连续性。

<h2 id="4.4.2.4_面试问题：Hierarchical-Generation-如何兼顾全局规划和局部质量？">4.4.2.4 面试问题：Hierarchical Generation 如何兼顾全局规划和局部质量？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

Hierarchical Generation 的核心是先生成高层规划，再生成低层视频细节：

```math
p(x_{1:T}\mid c)=p(z\mid c)\,p(x_{1:T}\mid z,c)
```

其中 $z$ 可以是脚本、分镜、关键帧、运动轨迹、场景状态或粗粒度 latent。高层规划回答“生成什么、按什么顺序生成”，低层生成回答“每一段具体如何呈现”。

它适合长视频，是因为长视频不仅需要画面质量，还需要事件顺序、镜头结构和角色状态稳定。直接生成完整长视频容易失控；先规划再细化，可以把问题拆成可控的子任务。

难点在于上下层对齐。高层规划如果错误，低层很难修正；低层如果不遵守规划，整体叙事仍会断裂。面试中可以概括为：**Hierarchical 方法用结构化规划换取长期一致性，但系统复杂度和对齐难度更高。**

<h1 id="4.4.3_跨片段一致性：状态记忆、参考库与边界对齐">4.4.3 跨片段一致性：状态记忆、参考库与边界对齐</h1>

> 本节是长视频生成的核心工程问题。生成更长的视频并不难，难的是让不同片段之间保持同一主体、同一场景、同一状态和同一运动过程。

<h2 id="4.4.3.1_面试问题：长视频生成中到底需要记住什么？">4.4.3.1 面试问题：长视频生成中到底需要记住什么？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

长视频生成不需要记住所有历史帧，而需要记住对后续生成有约束作用的状态。主要包括四类：

1. **主体状态**：人物身份、服饰、发型、配饰、物体外观。
2. **场景状态**：空间布局、背景物体、光照、镜头方向。
3. **动作状态**：动作阶段、速度、方向、姿态延续。
4. **事件状态**：哪些事情已经发生，哪些物体已经被移动或改变。

可以把历史信息压缩为状态记忆：

```math
\mathcal{M}_t=g(x_{1:t}, c)
```

生成后续片段时，不一定访问完整历史，而是访问记忆：

```math
x_{t+1:t+L}\sim p(x_{t+1:t+L}\mid \mathcal{M}_t,c)
```

面试回答应强调：**长视频一致性依赖状态记忆，而不是无限上下文。真正关键的是记住稳定身份、场景结构和事件状态。**

<h2 id="4.4.3.2_面试问题：Reference-Bank、Memory-Token-和-State-Propagation-分别解决什么问题？">4.4.3.2 面试问题：Reference Bank、Memory Token 和 State Propagation 分别解决什么问题？</h2>

**难度评分：⭐⭐⭐⭐⭐ (5/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

这三类机制都用于跨片段一致性，但关注点不同。

| 机制 | 核心作用 | 典型信息 |
|---|---|---|
| Reference Bank | 保存可检索视觉参考 | 人脸、服饰、场景、关键帧 |
| Memory Token | 用紧凑 token 表示历史状态 | 主体状态、场景摘要、动作阶段 |
| State Propagation | 将上一片段状态传给下一片段 | 尾帧、latent、pose、trajectory |

Reference Bank 更像“外部参考库”，适合跨镜头主体保持和场景复用。Memory Token 更像“压缩历史”，用于减少完整历史上下文成本。State Propagation 更像“递推状态”，常用于 chunk 或 causal 生成中衔接前后片段。

抽象地说，生成第 $i$ 个片段时可以写成：

```math
x^{(i)}\sim p(x^{(i)}\mid c_i,\mathrm{Retrieve}(\mathcal{R},c_i),m_{i-1},s_{i-1})
```

其中 $\mathcal{R}$ 是参考库，$m_{i-1}$ 是历史 memory，$s_{i-1}$ 是上一片段状态。面试中可以概括为：**Reference Bank 负责查，Memory Token 负责压缩，State Propagation 负责传递。**

<h2 id="4.4.3.3_面试问题：Chunk-边界为什么容易出问题？如何缓解边界不连续？">4.4.3.3 面试问题：Chunk 边界为什么容易出问题？如何缓解边界不连续？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

Chunk 边界容易出问题，是因为相邻片段往往在不同去噪过程、不同随机噪声或不同局部上下文下生成。即使每个片段单独质量较好，拼接处也可能出现动作跳变、身份变化、光照变化或语义断裂。

常见缓解方法包括：

1. **Overlap generation**：相邻 chunk 保留重叠帧，让模型在交界处共享上下文。
2. **Boundary conditioning**：用上一段尾帧、尾部 latent 或 motion state 作为下一段条件。
3. **Keyframe anchoring**：为每个片段提供首尾关键帧，约束片段起止状态。
4. **Feature / latent consistency loss**：训练阶段约束边界附近的特征连续。
5. **Post blending**：推理后对重叠区域做融合，但它只能缓解视觉跳变，不能解决语义断裂。

可以把边界一致性理解为约束：

```math
d(\phi(x^{(i)}_{\text{tail}}),\phi(x^{(i+1)}_{\text{head}})) < \tau
```

其中 $\phi$ 可以是视觉特征、身份特征、pose、motion 或语义特征。面试中可以总结为：**chunk 边界不是剪辑问题，而是状态连续性问题。**

<h2 id="4.4.3.4_面试问题：为什么长视频不能简单保存所有历史帧？">4.4.3.4 面试问题：为什么长视频不能简单保存所有历史帧？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

保存所有历史帧看似可以解决遗忘问题，但在长视频中通常不可行，原因有三点。

第一，计算和显存成本过高。历史帧越多，token 数和 attention 成本越大。第二，历史信息存在冗余，大量相邻帧并不会提供新的长期约束。第三，历史中可能包含错误帧，若全部保留，错误会被后续生成反复引用并放大。

更合理的方式是做历史压缩和检索：

```math
\mathcal{M}_t=\mathrm{Compress}(x_{1:t})
```

再根据当前生成目标检索相关记忆：

```math
r_t=\mathrm{Retrieve}(\mathcal{M}_t,c_t)
```

面试中可以概括为：**长视频需要的是选择性记忆，而不是完整录像回放。关键是保留身份、场景、状态和事件这些长期约束信息。**

<h1 id="4.4.4_多镜头与分镜叙事生成：Storyboard、Keyframe-与-Shot-level-Planning">4.4.4 多镜头与分镜叙事生成：Storyboard、Keyframe 与 Shot-level Planning</h1>

> 本节关注多镜头视频。它比单镜头长视频更难，因为镜头之间允许视角、景别和时间跳跃，但不允许角色、场景和事件逻辑无故断裂。

<h2 id="4.4.4.1_面试问题：Multi-shot-Video-Generation-和单镜头长视频有什么本质区别？">4.4.4.1 面试问题：Multi-shot Video Generation 和单镜头长视频有什么本质区别？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

单镜头长视频强调一个连续时间轴上的持续生成：

```math
x_{1:T}
```

Multi-shot Video Generation 则生成多个镜头片段：

```math
X=\{S_1,S_2,\ldots,S_K\}
```

每个 $S_i$ 可以有不同视角、景别、构图和动作阶段，但它们共同服务于同一个故事或表达目标。

二者的本质区别在于：单镜头长视频主要解决连续时间演化，多镜头生成还要解决镜头组织、跨镜头主体一致性、场景连续性和事件因果。单个 shot 质量高，并不意味着整个多镜头视频成立。

面试中可以概括为：**单镜头长视频考察时间连续性，多镜头生成考察镜头级组织和故事级一致性。**

<h2 id="4.4.4.2_面试问题：Storyboard、Keyframe-和-Shot-Video-分别承担什么角色？">4.4.4.2 面试问题：Storyboard、Keyframe 和 Shot Video 分别承担什么角色？</h2>

**难度评分：⭐⭐⭐ (3/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

分镜叙事生成通常采用分层流程：

```text
Script -> Storyboard -> Keyframe -> Shot Video -> Final Video
```

各层职责不同：

| 层级 | 作用 | 解决问题 |
|---|---|---|
| Script | 描述故事、角色、事件顺序 | 语义规划 |
| Storyboard | 拆分镜头、确定景别和镜头顺序 | 镜头规划 |
| Keyframe | 提供首帧、尾帧或关键视觉状态 | 视觉锚定 |
| Shot Video | 生成每个镜头的动态片段 | 局部视频生成 |

这种分层的核心价值是降低问题难度。模型不需要一次性从长文本生成完整视频，而是先得到结构化计划，再逐镜头生成。

需要注意的是，分层并不自动保证正确。Script 到 Storyboard 可能丢失语义，Storyboard 到 Keyframe 可能构图错误，Keyframe 到 Shot Video 可能动作不连续。因此分镜生成的关键是上下层对齐。

<h2 id="4.4.4.3_面试问题：Story-level-Coherence-为什么比单个-shot-质量更难？">4.4.4.3 面试问题：Story-level Coherence 为什么比单个 shot 质量更难？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

Story-level Coherence 指多个镜头组合后，在角色、场景、事件、镜头语言和视觉风格上形成一致整体。它比单个 shot 质量更难，因为它评价的是镜头之间的关系。

如果独立生成每个镜头，可以写成：

```math
\prod_{i=1}^{K}p(S_i\mid c_i)
```

这种方式容易忽略跨镜头依赖。更合理的形式应考虑故事上下文和历史镜头：

```math
p(S_i\mid S_{<i},c_i,c_{\text{story}})
```

Story-level Coherence 至少包括四类一致性：角色一致、场景一致、事件因果一致、视觉风格一致。面试中可以总结为：**shot-level quality 关注单个镜头是否好看，story-level coherence 关注所有镜头合起来是否像同一个故事。**

<h2 id="4.4.4.4_面试问题：多镜头生成中的镜头切换为什么难？">4.4.4.4 面试问题：多镜头生成中的镜头切换为什么难？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

镜头切换难，是因为它既要允许视角、景别或时间发生变化，又不能破坏主体、空间和事件连续性。真实影视中镜头切换遵循空间轴线、动作连续、视线匹配和叙事节奏；生成模型如果缺乏这些约束，就容易像随机片段拼接。

常见问题包括：同一人物跨镜头换脸、同一场景空间方向反转、动作阶段倒退、物体状态被重置、事件缺少承接。

可以把镜头生成写成：

```math
S_i \sim p(S_i \mid S_{i-1}, c_i, c_{\text{global}})
```

其中 $c_{\text{global}}$ 包含角色、场景、事件和风格约束。面试中可以概括为：**镜头切换不是简单拼接片段，而是在允许变化的同时维持可解释的连续性。**

<h1 id="4.4.5_长视频可控生成：Camera、Motion、Subject、Structure-的统一条件传播">4.4.5 长视频可控生成：Camera、Motion、Subject、Structure 的统一条件传播</h1>

> 本节关注可控生成。长视频可控性的核心不是“加更多条件”，而是让条件在时间维度上持续生效，并在多条件之间保持一致。

<h2 id="4.4.5.1_面试问题：长视频可控生成需要控制哪些核心变量？">4.4.5.1 面试问题：长视频可控生成需要控制哪些核心变量？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

长视频可控生成通常需要控制四类变量：

| 控制类型 | 控制对象 | 典型条件 |
|---|---|---|
| Camera | 观察方式 | pan、tilt、zoom、dolly、camera pose |
| Motion | 主体或区域如何运动 | trajectory、pose、flow、motion strength |
| Subject | 生成谁 | reference image、identity embedding、subject token |
| Structure | 画面结构 | layout、depth、mask、bbox、segmentation |

可以写成统一条件生成形式：

```math
p(x_{1:T}\mid c_{\text{text}},c_{\text{camera}},c_{\text{motion}},c_{\text{subject}},c_{\text{structure}})
```

长视频中的难点不是条件种类多，而是条件必须持续约束整段视频。例如主体身份不能只在首帧正确，相机轨迹不能只在开头生效，结构条件不能中途漂移。

面试中可以总结为：**长视频可控生成的关键是条件的时间传播和跨条件协调。**

<h2 id="4.4.5.2_面试问题：Camera-Control-和-Motion-Control-的区别是什么？">4.4.5.2 面试问题：Camera Control 和 Motion Control 的区别是什么？</h2>

**难度评分：⭐⭐⭐ (3/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

Camera Control 控制“怎么看”，Motion Control 控制“怎么动”。二者都会造成画面运动，但物理含义不同。

Camera Control 约束相机视角和镜头运动，例如推近、拉远、平移、旋转、环绕。主体不动但相机移动，画面仍会产生整体运动。

Motion Control 约束主体、局部区域或场景元素的运动，例如人体姿态序列、主体轨迹、光流方向或动作强度。相机不动但主体移动，也会产生运动。

区别可以概括为：

| 类型 | 关注对象 | 常见条件 |
|---|---|---|
| Camera Control | 观察坐标系变化 | camera pose、trajectory、zoom |
| Motion Control | 被观察对象变化 | pose、flow、trajectory、action reference |

长视频中二者必须协调。若相机向左移动、主体向右移动，最终光流可能相互抵消或叠加。模型需要区分 camera-induced motion 和 object-induced motion，否则容易出现背景扭曲、主体漂移或空间不稳定。

<h2 id="4.4.5.3_面试问题：Subject-Control-和-Structure-Control-如何跨时间持续生效？">4.4.5.3 面试问题：Subject Control 和 Structure Control 如何跨时间持续生效？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

Subject Control 约束“是谁”，Structure Control 约束“画面结构是什么”。二者都需要跨时间持续生效。

Subject Control 常依赖参考图、身份 embedding、reference attention 或 subject token。其目标是维持人物、商品、动物或角色的身份和关键外观。Structure Control 常依赖 layout、depth、pose、mask、trajectory 等条件，用于约束空间布局、几何关系和姿态结构。

二者可以抽象为：

```math
p(x_{1:T}\mid c_{\text{text}},c_{\text{subject}},c_{\text{structure}})
```

跨时间生效的关键包括：参考特征持续注入、关键帧锚定、历史状态检索、结构条件序列化，以及对生成结果进行一致性约束。

面试中应避免只说“加参考图”或“加 ControlNet”。更准确的回答是：**Subject Control 需要保持身份记忆，Structure Control 需要保持几何和布局约束；长视频场景下，二者都必须在时间维度上传播。**

<h2 id="4.4.5.4_面试问题：多条件控制发生冲突时，模型应如何处理？">4.4.5.4 面试问题：多条件控制发生冲突时，模型应如何处理？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

长视频生成中，多条件冲突很常见。例如文本要求角色转身，参考图只提供正脸；相机轨迹要求左移，主体轨迹要求右移；pose 条件和身份参考在局部区域不一致。

处理思路通常包括：

1. **条件分层**：高层语义、主体身份、几何结构、局部运动分别进入不同条件通道。
2. **条件加权**：根据任务目标设置 guidance scale 或动态权重。
3. **冲突检测**：识别明显不一致的参考或控制信号并降权。
4. **时间调度**：不同阶段使用不同条件强度，例如首帧更强身份约束，后续更强运动约束。
5. **局部化控制**：通过 mask 或区域条件限制控制影响范围。

可抽象为多条件融合：

```math
h=\sum_i \alpha_i h_i,\quad \sum_i \alpha_i=1
```

其中 $h_i$ 是不同条件特征，$\alpha_i$ 是条件权重。面试中可以概括为：**多条件控制的关键不是把条件全部塞进模型，而是解决条件优先级、作用范围和时间调度。**

<h1 id="4.4.6_长视频生成评估：长时一致性、叙事一致性与条件遵循">4.4.6 长视频生成评估：长时一致性、叙事一致性与条件遵循</h1>

> 本节关注评估。长视频质量是多维问题，不能被单帧画质或短时运动指标概括。

<h2 id="4.4.6.1_面试问题：为什么长视频不能只沿用短视频评估指标？">4.4.6.1 面试问题：为什么长视频不能只沿用短视频评估指标？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

短视频指标主要关注单帧质量、短时运动自然度、文本对齐和局部 flicker。长视频还需要评估长期主体一致性、场景稳定性、事件因果、镜头组织和条件持续遵循。

可以把长视频质量写成多维函数：

```math
Q=f(Q_{\text{visual}},Q_{\text{motion}},Q_{\text{consistency}},Q_{\text{story}},Q_{\text{control}})
```

其中视觉质量只是其中一部分。一个视频可以单帧很好，但角色中途变脸；也可以 prompt 对齐很好，但镜头之间缺少叙事关系。因此长视频评估必须从局部质量扩展到全局一致性。

面试中可以概括为：**短视频评估看一段是否像视频，长视频评估看整段是否像同一个作品。**

<h2 id="4.4.6.2_面试问题：长时一致性应该如何评估？">4.4.6.2 面试问题：长时一致性应该如何评估？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

长时一致性评估关注远距离时间点之间是否仍保持同一主体、同一场景、同一状态和同一语义目标。它不是只看相邻帧平滑。

可以用特征相似度描述：

```math
\mathrm{Consistency}(t_i,t_j)=\mathrm{sim}(\phi(x_{t_i}),\phi(x_{t_j}))
```

其中 $\phi$ 可以是身份特征、场景特征、CLIP / VLM 特征、pose 特征或 motion 特征。

具体评估维度包括：

1. **身份一致性**：人物、动物或商品是否漂移。
2. **场景一致性**：布局、背景物体和光照是否稳定。
3. **运动一致性**：动作方向、速度和阶段是否连续。
4. **语义一致性**：是否长期符合 prompt、script 或 storyboard。
5. **状态一致性**：物体状态和事件结果是否被保留。

面试中可以总结为：**长时一致性评估的核心是跨时间比较，而不是只检查相邻帧。**

<h2 id="4.4.6.3_面试问题：自动指标和人工评测应该如何结合？">4.4.6.3 面试问题：自动指标和人工评测应该如何结合？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

长视频生成需要自动指标和人工评测结合。自动指标可规模化、可复现，适合模型迭代；人工评测更擅长判断叙事连贯、可观看性和高层语义错误。

自动指标可覆盖：视觉质量、运动平滑、文本对齐、身份相似度、轨迹误差、pose error、storyboard 对齐、音画同步等。人工评测应覆盖：主体是否稳定、故事是否连贯、镜头是否自然、控制是否持续生效、整体是否可用。

更合理的评估形式不是单一总分，而是多维 profile：

```math
Q=\{Q_{\text{visual}},Q_{\text{motion}},Q_{\text{text}},Q_{\text{identity}},Q_{\text{story}},Q_{\text{control}}\}
```

单一分数不可靠，因为不同维度可能互相冲突。例如画质高但主体漂移，或文本对齐好但动作不自然。面试中可以概括为：**自动指标负责规模化筛选，人工评测负责高层一致性和可用性判断。**
