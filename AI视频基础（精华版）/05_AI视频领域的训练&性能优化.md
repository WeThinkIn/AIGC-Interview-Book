<h1 id="4.5.1_视频数据管线：数据质量、解码成本与-caption-对齐">4.5.1 视频数据管线：数据质量、解码成本与 caption 对齐</h1>

> 视频生成训练首先是数据工程。很多训练问题表面看是模型不收敛、运动差或文本不跟随，底层往往是数据解码、切片、过滤、caption 或分布采样出了问题。

<h2 id="4.5.1.1_面试问题：训练视频生成模型时，数据管线最容易拖垮哪些环节？">4.5.1.1 面试问题：训练视频生成模型时，数据管线最容易拖垮哪些环节？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

视频数据管线的瓶颈通常不在“有没有数据”，而在数据是否能以稳定、干净、足够快的方式进入训练。一个典型视频训练数据链路可以写成：

```text
raw video -> decode -> shot/clip sampling -> filtering -> captioning
          -> resize/crop/bucket -> VAE encode -> training batch
```

最常见的工程痛点包括：

| 环节 | 常见问题 | 训练影响 |
|---|---|---|
| Decode | CPU 解码慢、随机 seek 慢、坏视频多 | GPU 等数据，吞吐低 |
| Clip Sampling | 采到静止段、切到镜头边界、动作不完整 | 运动先验差 |
| Filtering | 水印、字幕、低码率、抖动未过滤 | 画质和稳定性下降 |
| Captioning | 只描述静态内容、动作缺失、时序错误 | T2V 文本跟随差 |
| Bucketing | padding 过多、规格混乱 | 有效 token 利用率低 |
| VAE Encode | 在线编码太慢、latent 缓存不一致 | 训练吞吐不稳定 |

工程上需要重点监控两个指标：**GPU utilization** 和 **effective tokens/sec**。如果 GPU 利用率低，往往不是模型太小，而是数据解码、网络存储、CPU augment、VAE 预处理或 batch 组织拖慢了训练。

面试中可以总结为：**视频训练的数据管线不仅决定数据质量，也决定训练吞吐；数据工程做不好，模型结构再强也难以稳定训练。**

<h2 id="4.5.1.2_面试问题：视频切片、去重和镜头切分为什么会直接影响训练效果？">4.5.1.2 面试问题：视频切片、去重和镜头切分为什么会直接影响训练效果？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

视频生成模型训练通常不会直接使用完整长视频，而是从原始视频中采样 clip。clip 的边界、时长和内容分布会直接决定模型学到的运动模式。

如果一个训练 clip 跨越了无关镜头切换，模型会把“硬切镜”误认为一种可学习的时序变化；如果采样到大量静止片段，模型会倾向生成低运动视频；如果重复视频过多，模型会过拟合常见模板，降低多样性。

因此视频切片需要处理三类问题：

1. **镜头一致性**：clip 内尽量保持同一 shot，避免无语义切镜破坏时间建模。
2. **运动覆盖**：采样策略要覆盖静态、低速、中速、高速等不同 motion bucket。
3. **去重与近重复**：不仅要去完全重复，还要去相似片段、模板视频和重复转码视频。

可以用运动强度做简单分桶：

```math
m=\frac{1}{T-1}\sum_{t=1}^{T-1}d(x_t,x_{t+1})
```

其中 $d(\cdot)$ 可以是帧差、光流强度或特征距离。训练采样时需要避免 motion distribution 过度偏向静态或异常高速片段。

面试中可以概括为：**clip sampling 不是随机截一段视频，而是在同一镜头、合理运动和数据多样性之间做分布控制。**

<h2 id="4.5.1.3_面试问题：高质量-video-caption-应该如何服务于-T2V-训练？">4.5.1.3 面试问题：高质量 video caption 应该如何服务于 T2V 训练？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

T2V 学习的是：

```math
p(x_{1:T}\mid c_{\text{text}})
```

因此 caption 不是附属信息，而是条件分布的训练入口。低质量 caption 会直接导致文本遵循差、动作理解弱、镜头控制差。

高质量 video caption 应至少覆盖四层信息：

1. **静态语义**：主体、场景、属性、风格。
2. **动态语义**：动作、运动方向、状态变化、事件过程。
3. **镜头语义**：景别、相机运动、视角变化。
4. **时间顺序**：先发生什么、后发生什么，是否存在转场或阶段变化。

一个常见工程做法是多阶段 caption：

```text
frame caption -> clip caption -> motion caption -> rewritten training prompt
```

其中 frame caption 补充主体和场景，clip caption 描述整体事件，motion caption 强调动作和镜头，最后再重写成训练 prompt。对于 T2V，caption 过短会导致动作缺失，caption 过长且不准确会引入噪声。

面试中可以总结为：**视频 caption 的关键不是描述某一帧，而是描述时间过程；caption 质量决定模型能否把文本动作和视频运动正确对齐。**

<h1 id="4.5.2_Token-Budget-与训练规格：Video-VAE、帧数、分辨率和-bucketing">4.5.2 Token Budget 与训练规格：Video VAE、帧数、分辨率和 bucketing</h1>

> 视频生成训练的第一性约束是 token budget。帧数、分辨率、VAE 压缩率和 batch bucketing 本质上都在决定模型每一步需要处理多少时空 token。

<h2 id="4.5.2.1_面试问题：为什么-token-budget-是视频生成训练的核心约束？">4.5.2.1 面试问题：为什么 token budget 是视频生成训练的核心约束？</h2>

**难度评分：⭐⭐⭐⭐⭐ (5/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

视频训练成本首先由时空 token 数决定。若视频经过 VAE 压缩后的 latent 尺寸为 $T' \times H' \times W'$，则 token 数为：

```math
N=T'\times H'\times W'
```

对于 Transformer / DiT 主干，全局 attention 成本近似为：

```math
\mathcal{O}(N^2)
```

这意味着帧数、分辨率、压缩率的变化会被放大到显存和计算成本上。训练中常见的“显存突然爆”“吞吐突然下降”“某些 batch 特别慢”，很多都来自 token budget 不稳定。

工程上通常要控制的是每个 batch 的总 token：

```math
N_{\text{batch}}=\sum_{i=1}^{B}T'_iH'_iW'_i
```

一个实用原则是：**batch size 不一定固定，但 batch token budget 应尽量稳定。** 这也是多分辨率、多帧数训练必须配合 bucketing 或 dynamic batching 的原因。

面试中可以概括为：**视频训练不是按样本数算成本，而是按时空 token 数算成本。**

<h2 id="4.5.2.2_面试问题：Video-VAE-的压缩率会如何影响训练和生成质量？">4.5.2.2 面试问题：Video VAE 的压缩率会如何影响训练和生成质量？</h2>

**难度评分：⭐⭐⭐⭐⭐ (5/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

Video VAE 决定像素视频如何被压缩到 latent space。压缩率越高，生成主干面对的 token 越少，训练和推理越省；但压缩过强会损失纹理、身份、细小动作和时间连续性。

如果空间压缩率为 $r_h,r_w$，时间压缩率为 $r_t$，则 latent token 数近似为：

```math
N_{\text{latent}}\approx \frac{T}{r_t}\times \frac{H}{r_h}\times \frac{W}{r_w}
```

压缩率影响三类能力：

1. **视觉重建上限**：VAE 本身如果重建不出细节，生成主干很难补回来。
2. **运动保真度**：时间压缩过强可能损失快速动作和短时事件。
3. **训练效率**：latent token 少，DiT / U-Net 训练成本显著降低。

工程上需要同时看 VAE reconstruction quality 和 downstream generation quality。只看重建指标不够，因为 VAE latent 还必须适合扩散或 flow 模型学习。

面试中可以总结为：**Video VAE 是视频生成系统的计算闸门；它在压缩效率和可生成信息保留之间做核心权衡。**

<h2 id="4.5.2.3_面试问题：训练中如何设计帧数、分辨率、帧率和长宽比的-bucketing？">4.5.2.3 面试问题：训练中如何设计帧数、分辨率、帧率和长宽比的 bucketing？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

视频数据天然规格不统一：横屏、竖屏、不同分辨率、不同帧率、不同 clip length 都会出现。如果直接混在一个 batch 中，padding 会浪费大量 token，吞吐也会不稳定。

bucketing 的核心是把相近规格的样本放到同一 batch：

```math
\text{bucket}=(T,H,W,\text{fps},\text{aspect ratio})
```

实际设计时要考虑：

1. **控制总 token**：不同 bucket 的 batch size 可以不同，但总 token 应尽量接近。
2. **覆盖产品规格**：训练 bucket 要覆盖未来推理的横屏、竖屏、方图、短视频和长视频规格。
3. **控制运动采样**：同一帧数下，不同 fps 对应不同真实时间长度，不能混淆。
4. **避免稀疏 bucket**：过细分桶会导致数据不足和采样不均。

一个常见策略是固定 token budget，动态调整 batch size：

```math
B_k \approx \frac{N_{\max}}{T'_kH'_kW'_k}
```

其中 $B_k$ 是第 $k$ 个 bucket 的 batch size。面试中可以概括为：**bucketing 的目标不是格式整齐，而是减少 padding、稳定 token budget，并覆盖真实生成规格。**

<h1 id="4.5.3_训练-Recipe：图像-视频混合、课程学习与阶段化训练">4.5.3 训练 Recipe：图像-视频混合、课程学习与阶段化训练</h1>

> 视频生成模型通常不是一步训成的。更常见的 recipe 是：图像预训练建立空间先验，视频预训练建立时间先验，高质量视频微调提升可用性，后训练进一步优化指令遵循和偏好。

<h2 id="4.5.3.1_面试问题：图像-视频联合训练的混合比例应该如何理解？">4.5.3.1 面试问题：图像-视频联合训练的混合比例应该如何理解？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

图像-视频联合训练不是简单“数据越多越好”，而是在空间质量和时间能力之间做权衡。图像数据帮助模型学习物体、构图、纹理、风格和文本-图像语义；视频数据帮助模型学习运动、时序一致性和镜头变化。

混合训练目标可写成：

```math
\mathcal{L}=\lambda_{\text{img}}\mathcal{L}_{\text{img}}+\lambda_{\text{vid}}\mathcal{L}_{\text{vid}}
```

关键不只是 $\lambda_{\text{img}}$ 和 $\lambda_{\text{vid}}$ 的数值，还包括采样频率、token 数、batch 组织和训练阶段。图像样本 token 少、质量高，过多会让模型偏静态；视频样本 token 多、噪声大，过多可能拖慢收敛并降低单帧质量。

工程上常见策略是前期提高图像比例稳定空间生成，中后期提高视频比例强化运动和一致性。面试中可以概括为：**图像补空间，视频补时间；混合比例决定模型更偏画质、运动还是综合能力。**

<h2 id="4.5.3.2_面试问题：为什么视频生成训练通常需要-progressive-training？">4.5.3.2 面试问题：为什么视频生成训练通常需要 progressive training？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

progressive training 的核心是逐步增加任务难度，避免一开始就在高分辨率、长序列、多条件、强噪声数据上训练导致不稳定。

常见阶段包括：

```text
image pretraining
-> low-res short-video pretraining
-> higher-res / longer-clip training
-> high-quality video finetuning
-> instruction / preference post-training
```

每个阶段解决的问题不同：图像阶段建立空间先验，短视频阶段学习局部运动，高分辨率阶段补视觉细节，长视频阶段增强状态保持，高质量微调阶段提升审美和可用性。

如果直接训练最终规格，常见问题是 loss 抖动大、运动学不稳、显存成本过高、数据噪声放大。progressive training 可以降低优化难度，并让模型逐步适应更高 token budget。

面试中可以总结为：**视频生成训练通常是分阶段构建能力，而不是一次性把所有难题同时交给模型。**

<h2 id="4.5.3.3_面试问题：高质量微调和偏好后训练主要解决什么问题？">4.5.3.3 面试问题：高质量微调和偏好后训练主要解决什么问题？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

大规模预训练解决“模型是否具备基础分布能力”，高质量微调和偏好后训练解决“模型是否可用、好用、符合用户偏好”。

高质量微调通常使用更干净、更高分辨率、更准确 caption 的精选数据，主要改善：

1. 单帧视觉质量和审美。
2. 动作自然度和运动幅度。
3. 文本遵循和镜头语言。
4. 人物、手部、文字、复杂物体等难点区域。

偏好后训练则进一步对齐人类偏好，例如减少伪影、增强主体一致性、提升 prompt following、改善构图和镜头节奏。它可以采用人工偏好、VLM 评估、奖励模型或 DPO/RL 类方法。

面试中可以概括为：**预训练决定能力边界，精选微调提升质量分布，偏好后训练让模型更接近真实产品需求。**

<h1 id="4.5.4_训练稳定性：噪声调度、条件-dropout、loss-weighting-与异常排查">4.5.4 训练稳定性：噪声调度、条件 dropout、loss weighting 与异常排查</h1>

> 视频训练中的“不稳定”很少只有一个原因。它可能来自数据噪声、caption 错误、token budget 变化、VAE latent 分布、噪声调度、学习率、混合精度或分布式通信。

<h2 id="4.5.4.1_面试问题：视频扩散或-Flow-Matching-训练中常见不稳定现象有哪些？">4.5.4.1 面试问题：视频扩散或 Flow Matching 训练中常见不稳定现象有哪些？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

视频生成训练的不稳定通常先表现在样本质量上，而不一定只体现在 loss 上。常见现象包括：

1. **loss 正常但样本差**：模型可能学到平均化结果，运动弱或细节糊。
2. **单帧质量好但视频闪烁**：空间先验强，时间建模不足。
3. **prompt 跟随差**：caption 噪声、CFG dropout 不合理或文本编码弱。
4. **运动塌缩**：训练数据过静、motion bucket 失衡或条件过强。
5. **高频伪影**：VAE 重建不足、低码率数据污染或 loss 权重不合理。
6. **训练中后期退化**：学习率、EMA、数据阶段切换或混合比例变化不当。

工程排查时，不能只看整体 loss。更实用的是固定一组 validation prompts，周期性生成样本，观察画质、运动、时序一致性、文本遵循和失败类型。

面试中可以总结为：**视频训练稳定性要看 loss、样本、数据分布和系统吞吐的联合信号；loss 下降不等于视频生成质量提升。**

<h2 id="4.5.4.2_面试问题：Diffusion-与-Flow-Matching-在工程训练上各有什么关注点？">4.5.4.2 面试问题：Diffusion 与 Flow Matching 在工程训练上各有什么关注点？</h2>

**难度评分：⭐⭐⭐⭐⭐ (5/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

Diffusion 通常学习去噪预测，典型目标为：

```math
\mathcal{L}_{\text{diff}}=
\mathbb{E}_{x_0,\epsilon,t}
\left[
\lVert \epsilon-\epsilon_\theta(x_t,t,c)\rVert^2
\right]
```

Flow Matching 学习从噪声到数据的连续速度场，可抽象为：

```math
\mathcal{L}_{\text{fm}}=
\mathbb{E}_{x_t,t}
\left[
\lVert v_\theta(x_t,t,c)-u_t(x_t)\rVert^2
\right]
```

工程上，Diffusion 更关注 noise schedule、prediction type、SNR weighting、sampler 和 CFG 配合；Flow Matching 更关注路径设计、时间采样、速度场尺度、ODE/SDE sampler 以及少步采样稳定性。

两者在视频场景下共同面对的问题是：高维时空 token、数据噪声、运动建模和长序列成本。不能简单说 Flow Matching 一定更好，或 Diffusion 一定过时。更严谨的说法是：**Diffusion 和 Flow Matching 是不同训练目标与采样路径设计，工程优劣取决于模型结构、数据规模、采样器和部署约束。**

<h2 id="4.5.4.3_面试问题：CFG-dropout、EMA、timestep-sampling-和-loss-weighting-分别解决什么问题？">4.5.4.3 面试问题：CFG dropout、EMA、timestep sampling 和 loss weighting 分别解决什么问题？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

这些机制分别控制条件能力、参数稳定性、训练时间步分布和不同噪声区间的优化平衡。

**CFG dropout** 让模型在训练中同时学习有条件和无条件预测，推理时可以做 classifier-free guidance：

```math
\hat{\epsilon}
=\epsilon_\theta(x_t,t,\varnothing)
+s\left(\epsilon_\theta(x_t,t,c)-\epsilon_\theta(x_t,t,\varnothing)\right)
```

dropout 太低，无条件分支学不好；dropout 太高，条件跟随可能变弱。

**EMA** 用滑动平均降低参数震荡：

```math
\theta_{\text{ema}}\leftarrow \beta\theta_{\text{ema}}+(1-\beta)\theta
```

**timestep sampling** 决定模型更频繁学习哪些噪声阶段。不同时间步对应不同难度：高噪声阶段更偏全局语义，低噪声阶段更偏细节和纹理。

**loss weighting** 用于平衡不同 SNR、不同区域、不同帧或不同任务的梯度贡献，避免训练被某一类样本或某一噪声区间主导。

面试中可以概括为：**CFG dropout 控条件，EMA 稳参数，timestep sampling 控训练分布，loss weighting 控梯度贡献。**

<h1 id="4.5.5_分布式训练效率：数据吞吐、显存优化与并行策略">4.5.5 分布式训练效率：数据吞吐、显存优化与并行策略</h1>

> 视频训练的效率瓶颈通常是系统级的。只讨论模型 FLOPs 不够，还要看数据读取、VAE 编码、token padding、activation memory、通信开销和 GPU 利用率。

<h2 id="4.5.5.1_面试问题：视频训练吞吐低，应该从哪些环节排查？">4.5.5.1 面试问题：视频训练吞吐低，应该从哪些环节排查？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

视频训练吞吐低时，不应直接归因于模型太大。排查应按链路进行：

```text
storage -> video decode -> sampling/augment -> VAE encode/cache
        -> batch bucketing -> GPU forward/backward -> communication
```

常见瓶颈包括：

1. **存储与网络 IO**：远程读取视频导致 dataloader 等待。
2. **CPU 解码**：视频随机 seek 和解码速度跟不上 GPU。
3. **在线 VAE 编码**：VAE encode 成为训练前置瓶颈。
4. **padding 浪费**：bucket 设计差，实际有效 token 占比低。
5. **通信开销**：多机训练中梯度同步或参数切分通信过重。
6. **负载不均衡**：不同 batch token 数差异大，step time 抖动。

工程上常用指标包括：samples/sec、tokens/sec、GPU utilization、dataloader wait time、step time p50/p95、padding ratio。对于视频训练，**effective tokens/sec** 往往比 samples/sec 更有解释力。

面试中可以总结为：**视频训练吞吐优化要按端到端链路排查，很多时候瓶颈在数据和 batch 组织，而不在模型计算本身。**

<h2 id="4.5.5.2_面试问题：视频模型训练中如何做显存优化，而不是简单减-batch？">4.5.5.2 面试问题：视频模型训练中如何做显存优化，而不是简单减 batch？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

简单减 batch 会降低吞吐和训练稳定性，不是最优解。视频模型显存主要来自参数、优化器状态、梯度、激活、attention 中间量和输入 latent。

常见优化方式包括：

1. **activation checkpointing**：用重算换显存。
2. **mixed precision / FP8**：降低参数和激活显存。
3. **memory-efficient attention**：减少 attention 中间矩阵占用。
4. **FSDP / ZeRO**：切分参数、梯度和优化器状态。
5. **sequence parallel**：沿 token 维度拆分长序列。
6. **bucket token budget**：避免某些 batch 因 token 过多 OOM。
7. **VAE latent precompute**：减少训练时 VAE activation 占用。

需要注意，显存优化往往改变计算和通信成本。例如 checkpointing 会降低吞吐，FSDP 会增加通信，sequence parallel 对实现要求更高。

面试中可以概括为：**显存优化不是简单缩小 batch，而是在激活、参数、序列长度、通信和重算之间做系统权衡。**

<h2 id="4.5.5.3_面试问题：数据并行、张量并行、序列并行和-FSDP-如何组合？">4.5.5.3 面试问题：数据并行、张量并行、序列并行和 FSDP 如何组合？</h2>

**难度评分：⭐⭐⭐⭐⭐ (5/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

不同并行方式解决不同瓶颈：

| 并行方式 | 拆分对象 | 主要解决 |
|---|---|---|
| Data Parallel | batch | 数据规模和吞吐 |
| Tensor Parallel | 层内矩阵 | 单层计算和参数过大 |
| Sequence Parallel | token 序列 | 长视频上下文过长 |
| FSDP / ZeRO | 参数、梯度、优化器状态 | 模型状态显存 |
| Pipeline Parallel | 模型层 | 超深模型跨设备放置 |

视频 DiT 训练中，瓶颈经常来自两个方向：模型参数大和时空 token 长。只用 data parallel 不能解决单卡显存；只用 FSDP 也不能完全解决长序列 attention。实际训练常需要组合：

```text
data parallel + FSDP + sequence parallel + activation checkpointing
```

若模型单层非常大，还可能加入 tensor parallel。选择并行策略时要看瓶颈是 parameter memory、activation memory、attention memory 还是 communication。

面试中可以总结为：**视频模型分布式训练不是套一个并行策略，而是根据参数规模和时空 token 规模组合多种并行方式。**

<h1 id="4.5.6_推理部署优化：采样加速、VAE-解码、服务调度与成本控制">4.5.6 推理部署优化：采样加速、VAE 解码、服务调度与成本控制</h1>

> 视频模型能训练出来，不代表能在线服务。线上推理要同时优化采样步数、VAE 解码、显存峰值、队列调度、失败重试和单位生成成本。

<h2 id="4.5.6.1_面试问题：视频生成线上推理为什么常常慢在不止一个地方？">4.5.6.1 面试问题：视频生成线上推理为什么常常慢在不止一个地方？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

视频推理延迟来自整条链路，而不只是去噪主干：

```text
prompt processing -> text / image encoding -> denoising steps
-> VAE decode -> post-processing -> safety check -> video encoding
```

粗略成本可写成：

```math
C_{\text{infer}}\approx
S\cdot C_{\text{denoise}}(T,H,W)
+C_{\text{vae-decode}}
+C_{\text{condition}}
+C_{\text{post}}
```

其中 $S$ 是采样步数。对于高分辨率或长视频，VAE decode、视频编码、上传下载和安全审核也可能成为显著瓶颈。

工程排查时需要分别记录：首帧时间、总生成时间、去噪时间、VAE 解码时间、排队时间、显存峰值和失败率。只优化主干模型，不一定能显著降低端到端延迟。

面试中可以概括为：**视频生成慢是端到端链路慢，不能只看 diffusion step 的耗时。**

<h2 id="4.5.6.2_面试问题：视频生成推理加速有哪些工程路线？各自代价是什么？">4.5.6.2 面试问题：视频生成推理加速有哪些工程路线？各自代价是什么？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

常见推理加速路线包括：

| 路线 | 作用 | 代价 |
|---|---|---|
| 减少采样步数 | 降低 denoising 时间 | 细节和稳定性可能下降 |
| 蒸馏 / consistency | 少步生成 | 多样性和可控性可能下降 |
| 低精度推理 | 降低显存和计算 | 数值稳定性风险 |
| KV / condition cache | 复用条件特征 | 适用场景有限 |
| VAE tiling / slicing | 降低解码显存 | 可能引入边界伪影 |
| dynamic batching | 提升吞吐 | 增加排队延迟 |
| 多档模型 | 区分预览和高质量生成 | 系统复杂度增加 |

实际部署中，加速通常不是单点方案，而是组合：少步采样 + 半精度 + VAE 优化 + 动态 batch + 队列调度。

面试中应强调代价：**推理加速一定要同时评估质量、稳定性、显存、吞吐和失败率。**

<h2 id="4.5.6.3_面试问题：工业部署中如何设计质量、延迟、吞吐和成本的分层策略？">4.5.6.3 面试问题：工业部署中如何设计质量、延迟、吞吐和成本的分层策略？</h2>

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

工业部署不是追求每次生成都用最高质量配置，而是在不同场景下提供不同质量和成本档位。可以抽象为：

```math
\max Q_{\text{video}}
\quad \text{s.t.}\quad
L\leq L_{\max},\;
M\leq M_{\max},\;
C\leq C_{\max}
```

常见分层策略包括：

1. **预览模式**：低分辨率、短帧数、少步采样，用于快速反馈。
2. **标准模式**：平衡质量和速度，作为默认线上配置。
3. **高质量模式**：更多采样步数、更高分辨率、更强后处理，适合离线任务。
4. **异步队列**：长视频和高质量任务进入后台队列，避免阻塞交互。
5. **失败回退**：OOM、超时或安全失败时自动降级规格或重试。

还需要按业务监控单位生成成本，例如每秒视频成本、GPU 分钟成本、失败重试成本和队列等待时间。

面试中可以总结为：**视频生成部署的核心不是单次效果最强，而是在可控成本下稳定提供分层质量服务。**
