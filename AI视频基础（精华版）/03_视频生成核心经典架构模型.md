<h2 id="4.3.1.1_Text-to-Video：从文本到视频的基本生成范式">4.3.1.1 Text-to-Video：从文本到视频的基本生成范式</h2>

### 面试问题1：什么是 Text-to-Video 任务？其核心目标是什么？

**难度评分：⭐⭐⭐ (3/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

Text-to-Video（T2V）任务指的是：**给定文本条件，让模型生成一段与文本语义一致、视觉上合理、时间上连贯的视频。** 从概率建模角度看，它本质上是在学习条件分布：

```math
p(x_{1:T} \mid c)
```

其中 $x_{1:T}$ 表示视频序列，$c$ 表示文本条件。

它的核心目标通常包括三层：

1. **语义对齐**：生成内容要准确响应文本中的主体、场景、动作、属性和风格描述。
2. **时序连贯**：相邻帧之间要保持自然过渡，不能出现明显闪烁、跳变或身份漂移。
3. **视觉质量**：单帧要清晰、结构合理，整段视频要具有可观看性和真实感。

站在 AI 视频任务的视角，T2V 的难点不只是“把文字变成画面”，而是要把**静态语义**扩展成**动态过程**。也就是说，模型不仅要理解“画面里有什么”，还要决定“这些对象如何随时间变化、变化是否符合文本描述和物理常识”。

因此，T2V 可以理解为：**文本条件驱动下的时空联合生成任务**。它既要求跨模态语义理解，也要求视频级运动建模和一致性控制。

### 面试问题2：Text-to-Video 与 Text-to-Image 任务的本质区别是什么？

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

Text-to-Image（T2I）和 Text-to-Video（T2V）都属于文本条件生成任务，但两者的本质区别在于：**T2I 只需要生成单帧空间内容，而 T2V 需要生成完整的时空演化过程。**

如果用条件分布表示：

```math
\text{T2I: } p(x \mid c), \qquad
\text{T2V: } p(x_{1:T} \mid c)
```

T2V 相比 T2I 多出的不是简单的帧数，而是整个**时间维度建模问题**。这会带来几个关键差异：

1. **多了运动建模**：T2I 只需决定“这一帧长什么样”，T2V 还要决定“主体怎么动、镜头怎么动、动作如何展开”。
2. **多了一致性约束**：T2I 只关注单张图质量，T2V 还必须保证人物身份、背景结构、光照纹理和动作轨迹在多帧中稳定。
3. **多了长程依赖**：视频前后相隔较远的帧仍可能存在强关联，例如角色再次出现时要保持身份一致。
4. **计算与数据复杂度更高**：视频输入输出维度更大，训练、采样、评估和标注都比图像更复杂。

所以更准确地说，T2V 不是“T2I 的多帧版本”，而是**在图像生成基础上叠加了时序建模、运动建模和视频一致性建模**的更难任务。

### 面试问题3：Text-to-Video 模型的基本架构组成有哪些？

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

一个典型的 Text-to-Video 模型通常由以下几个核心模块组成：

1. **文本编码器**  
   用于把自然语言 prompt 编码成条件特征，常见如 T5、CLIP Text Encoder、LLaMA 系列文本模块等。

2. **视频潜空间编码器 / 解码器**  
   通常是 VAE 或 Video VAE。它负责把高维像素视频压缩到 latent space，再把生成后的 latent 解码回视频：

```math
z = E(x_{1:T}), \qquad \hat{x}_{1:T} = D(z)
```

3. **生成主干网络**  
   这是模型的核心，负责在 latent space 中完成生成。主流做法包括 3D U-Net、Video U-Net、DiT、Video Transformer 等。

4. **条件注入模块**  
   把文本条件注入主干网络，常见方式包括 Cross-Attention、AdaLN、FiLM、ControlNet 类结构等。

5. **噪声调度器与采样器**  
   在扩散框架中负责定义前向加噪和反向采样策略，例如 DDPM、DDIM、Euler、DPM-Solver 等。

如果用一条简化流水线表示，通常可以写成：

```text
Text Prompt -> Text Encoder -> Conditional Features
Noise / Initial Latent -> Video Backbone -> Denoised Video Latent
Video Latent -> Video VAE Decoder -> Output Video
```

从工程视角看，T2V 模型的系统设计重点主要落在三件事上：

1. **条件是否读得懂**：文本编码能力和条件注入方式是否足够强。
2. **时空是否建得稳**：主干网络能否有效处理时间维度和一致性问题。
3. **潜空间是否压得好**：视频 VAE 是否在压缩效率和重建质量之间取得平衡。

因此，Text-to-Video 架构本质上是**文本条件模块 + 视频潜空间模块 + 时空生成主干**三者的组合。

### 面试问题4：Text-to-Video 生成中的时序一致性问题如何解决？

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

时序一致性问题指的是：**模型虽然能生成单帧合理的图像，但前后帧之间可能出现闪烁、身份漂移、动作不连续或背景跳变。** 这是 Text-to-Video 中最核心的技术难点之一。

解决思路通常分为三类：

1. **在架构层显式建模时间维度**  
   例如引入 3D 卷积、Temporal Attention、Space-Time Attention、Motion Module、3D RoPE 等，让模型在生成时同时看到多帧关系，而不是逐帧独立生成。

2. **在训练目标中加入一致性约束**  
   常见做法是让相邻帧或对齐后的特征尽量保持一致。例如基于运动补偿的时序一致性损失可写成：

   ```math
   \mathcal{L}_{tc}
   = \sum_t
   \left\|
   \phi(x_{t+1}) - \mathcal{W}(\phi(x_t), F_{t \rightarrow t+1})
   \right\|_1
   ```

   其中 $\phi(\cdot)$ 是特征提取器，$\mathcal{W}$ 表示 warp 操作，$F_{t \rightarrow t+1}$ 是帧间运动场。

3. **在采样与推理阶段增强跨帧稳定性**  
   例如共享部分噪声初始化、引入关键帧约束、使用首帧条件、做滑动窗口生成、加入后处理去闪烁模块等。

更具体地说，时序一致性通常要同时覆盖以下几个层面：

1. **内容一致性**：场景、物体、背景结构不能无故突变。
2. **身份一致性**：人物或主体外观在连续帧中要稳定。
3. **运动一致性**：动作轨迹和速度变化要自然。
4. **视觉一致性**：亮度、颜色、纹理、边缘不能随机抖动。

所以从 AI 视频任务视角看，时序一致性不是一个单独的小优化项，而是**决定 T2V 是否真正从“图像生成”升级为“视频生成”的核心能力**。

### 面试问题5：Text-to-Video 技术的主要应用场景有哪些？

**难度评分：⭐⭐ (2/5)  |  考察频率：⭐⭐⭐ (3/5)**

Text-to-Video 的应用场景可以概括为：**用自然语言直接生成动态视觉内容，降低视频制作门槛，并提升内容生成效率。**

主要场景包括：

1. **内容创作与短视频生产**  
   用于生成广告片段、剧情分镜、视觉概念短片、社媒内容和创意素材。

2. **影视预演与分镜可视化**  
   导演、编剧和美术可以通过文本快速验证镜头语言、场景氛围和动作设计。

3. **游戏与虚拟世界内容生成**  
   用于生成过场动画、角色演示、世界观概念视频和 NPC 行为示意。

4. **教育与知识表达**  
   把抽象概念、历史事件、科学过程或工业流程转成动态演示视频。

5. **个性化营销与品牌传播**  
   基于不同用户群体或商品描述快速生成定制化视频素材。

6. **多模态智能体与人机交互**  
   在更长期的方向上，T2V 不只是内容生成工具，还可能成为世界模拟、决策预测和交互式视频代理的一部分。

不过需要注意，当前 T2V 在真实落地中仍受限于几个因素：

1. **长视频稳定性不足**  
2. **细粒度动作和复杂物理过程控制仍然较弱**  
3. **算力成本较高，生成延迟明显**  
4. **版权、安全与内容治理问题仍需工程配套**

因此，现阶段最成熟的应用更多集中在**创意视频生成、概念可视化和辅助内容生产**；而高精度、长时长、强可控的工业级视频生成，仍然是持续演进中的方向。

<h1 id="4.3.2_视频生成的核心技术实现路径">4.3.2 视频生成的核心技术实现路径</h1>

<h2 id="4.3.2.1_潜空间视频生成路线：Video-VAE-与-Latent-Video-Diffusion">4.3.2.1 潜空间视频生成路线：Video VAE 与 Latent Video Diffusion</h2>

视频生成之所以几乎都走 **Video VAE + Latent Video Diffusion** 路线，核心原因不是“理论上更优雅”，而是**像素空间直接生成在视频场景下代价过高**。视频比图像多一个时间维度，输入规模通常可写成：

```math
x \in \mathbb{R}^{T \times C \times H \times W}
```

当 $T$、$H$、$W$ 同时增大时，训练显存、注意力开销、采样时延都会迅速放大。因此主流系统通常先把视频压缩到潜空间，再在 latent space 中完成去噪或生成：

```math
z = E(x_{1:T}), \qquad \hat{x}_{1:T} = D(\hat{z}_{1:T})
```

这里真正重要的不是“用了 VAE”这件事本身，而是**通过压缩把生成问题从像素域转成更低维、更适合建模时序结构的表示域**。这也是 Latent Diffusion 从图像成功迁移到视频的根本原因。

站在视频生成任务视角，Video VAE 相比图像 VAE 的要求更高。图像压缩器只要保住空间结构即可，而视频压缩器除了要保住单帧内容，还要尽量保住：

1. **时间连续性**：压缩后相邻帧之间不能被打散，否则后续生成容易出现 flicker。
2. **运动信息**：物体位移、局部形变、镜头运动不能在编码阶段被过度抹平。
3. **长视频可扩展性**：压缩器本身要能支持更长序列，很多系统因此会强调 temporal downsampling、causal VAE 或分块编码设计。

因此，视频生成里真正的瓶颈常常不是“生成主干不够强”，而是**视频压缩器是否为后续建模保留了足够有用的时空结构**。Open-Sora、Open-Sora-Plan、HunyuanVideo 等路线都反复强调 video tokenizer / video VAE 的重要性，本质上就是这个原因。

Latent Video Diffusion 成为主流工程路线，是因为它在三件事之间提供了比较均衡的折中：

1. **质量**：潜空间仍能保留相对稳定的语义和结构信息，便于高质量生成。
2. **成本**：相比像素空间，latent token 数量显著下降，训练和推理更可行。
3. **可扩展性**：更容易叠加 temporal attention、3D 模块、reference control、audio condition 等视频特有能力。

这一点可以从代表工作中看得很清楚。`Latent Video Diffusion Models`、`Stable Video Diffusion` 强调的是“在 latent space 中复用成熟扩散框架”；Open-Sora 路线进一步强调 causal VAE、视频压缩比与长视频训练的关系；而更大规模的系统则会把 latent route 看成整个视频生成流水线的入口，而不是一个单独组件。

面试中可以概括为：**潜空间视频生成路线之所以成为主流，是因为它把视频生成从难以承受的像素空间问题，转化为一个在质量、效率和可扩展性之间更平衡的时空生成问题；而 Video VAE 的质量往往决定了整条视频生成链路的上限。**

<h2 id="4.3.2.2_视频扩散的时序建模：Temporal-Layer-/-Temporal-Attention-/-3D-Conv">4.3.2.2 视频扩散的时序建模：Temporal Layer / Temporal Attention / 3D Conv</h2>

视频扩散之所以必须引入专门的时序建模路径，核心原因是：**单帧生成质量并不等于视频生成质量**。如果模型只是把图像扩散逐帧应用到视频上，那么每一帧可能都“看起来还可以”，但整段视频往往会出现身份跳变、背景抖动、动作不连续和局部闪烁。因此，视频模型真正要解决的是：

```math
p(x_{1:T} \mid c)
```

而不是一组互相独立的 $p(x_t \mid c)$。也就是说，模型必须显式建模帧间依赖。

从工程上看，`Temporal Layer`、`Temporal Attention` 和 `3D Conv` 代表了三种很典型的时序建模思路，它们的差别不只是“模块长得不一样”，而是各自适合解决不同层次的问题。

1. **3D Conv** 更像局部时空耦合建模。  
它天然适合捕捉短时间窗口内的运动模式，比如局部位移、速度变化、短程动态纹理变化。优点是归纳偏置强、局部建模稳定，缺点是长程依赖能力有限，扩展到长视频时通常不够灵活。

2. **Temporal Attention** 更像显式跨帧对齐机制。  
它允许当前帧特征直接与其他时间位置交互，例如：

```math
\mathrm{Attn}(Q_t, K_{1:T}, V_{1:T})
=
\mathrm{softmax}\left(\frac{Q_t K_{1:T}^{\top}}{\sqrt{d}}\right)V_{1:T}
```

它更适合处理跨帧一致性、主体稳定、远距离依赖等问题，因此在中大规模视频扩散模型里非常常见。代价是计算和显存开销通常高于简单的局部卷积。

3. **Temporal Layer / Motion Module** 更像一种轻量注入式改造。  
这类方法常见于“从图像扩散迁移到视频扩散”的路线中，例如在原有 2D backbone 中插入时间模块，只在有限位置建模时间维度。它的优点是迁移成本低、复用图像先验强，AnimateDiff 就是很典型的代表；缺点是上限往往受原始 2D 主干约束，更适合短视频和中等复杂度运动。

所以，时序建模方案的选择，本质上是在做三方面权衡：

1. **运动表达能力**：能不能刻画复杂动作、镜头运动和长程依赖。
2. **一致性稳定性**：能不能降低 flicker、drift 和 identity collapse。
3. **训练与推理成本**：显存、吞吐、token 数量和可扩展性是否可接受。

从这个角度看，后续不同模型路线的差异就更容易理解了。U-Net / Video U-Net 路线更常见局部时间模块与 factorized attention；AnimateDiff 强调在图像模型上增量加入 motion module；Sora / Open-Sora / DiT 类路线则更倾向于把视频直接表示成时空 token，再在统一 backbone 里处理时间依赖。

面试里比较稳妥的回答是：**视频扩散的时序建模不是单纯“多加一个时间维度”，而是在局部运动、长程依赖和计算成本之间做结构化取舍。3D Conv、Temporal Attention 和 Temporal Layer 分别代表了局部耦合、跨帧交互和轻量迁移三类主流路径。**

<h2 id="4.3.2.3_条件控制与一致性建模：Cross-Attention、Reference、Audio、Identity">4.3.2.3 条件控制与一致性建模：Cross-Attention、Reference、Audio、Identity</h2>

在视频生成里，条件控制和一致性建模几乎不能分开讨论。原因很直接：**图像生成里的条件只需要影响一张图，而视频生成里的条件需要在整段时间轴上持续生效。** 因此，视频中的很多一致性问题，本质上都可以理解为“条件没有被稳定地传递到所有帧”。

如果从条件分布角度写，视频生成往往不是单一条件，而是多条件联合约束：

```math
p(x_{1:T} \mid c_{\text{text}}, c_{\text{ref}}, c_{\text{audio}}, c_{\text{id}}, \ldots)
```

这里真正困难的地方，不是条件数量变多，而是这些条件在时间维度上会相互耦合。比如文本条件决定整体语义，首帧或参考图决定外观锚点，identity condition 决定主体是否稳定，audio condition 又要求动作节奏与语音或音乐同步。它们都不是单帧约束，而是**跨帧持续约束**。

从任务适配关系看，不同条件类型各有典型使用场景：

1. **文本条件** 主要对应 T2V，负责高层语义、动作描述和风格约束。
2. **图像 / 首帧 / reference 条件** 常见于 I2V、V2V、reference-driven generation，重点是锁定外观、构图或初始状态。
3. **identity / subject 条件** 常见于 Subject-to-Video、Character-to-Video，重点是跨帧人物或主体一致性。
4. **音频条件** 常见于 talking head、music-driven video、audio-driven motion，重点是声画同步与节奏控制。

因此，条件控制在视频里不是“多加几个 condition token”这么简单，而是要解决两个更深的问题：

1. **条件是否能跨时间稳定传播**：同一个条件能不能持续作用到整段视频，而不是只在局部帧生效。
2. **多个条件之间是否协调**：例如文本想让角色转身，reference 又要求外观稳定，音频还要求口型同步，这些约束不能相互打架。

这也是为什么很多视频模型会把一致性建模和控制接口绑在一起设计。reference attention、identity feature injection、首帧锚定、motion guidance、audio alignment 等做法，本质上都是在提升“条件跨帧稳定传递”的能力。AnimateDiff 的 reference control、基于 IP-Adapter / ReferenceNet 的 I2V 路线、以及更大系统中的多条件联合控制，都可以放在这个框架下理解。

如果一定要一句话总结，这一节最核心的观点是：**视频生成中的一致性，很多时候不是一个独立后处理问题，而是条件控制问题在时间维度上的延伸。谁能把条件稳定、持续、协调地注入整段视频，谁就更容易做出真正可控的视频生成系统。**

<h2 id="4.3.2.4_长视频与高效生成：Causal、Chunk、Sliding-Window-与采样优化">4.3.2.4 长视频与高效生成：Causal、Chunk、Sliding-Window 与采样优化</h2>

长视频生成不是把短视频模型“直接跑更久”这么简单。视频一旦变长，系统会同时遇到四类问题：**上下文长度爆炸、误差累积、时序漂移和推理成本失控。** 这也是为什么很多短视频模型在 4 秒、8 秒内效果不错，但一旦拉长到十几秒甚至更长，就会明显暴露出身份不稳、镜头逻辑断裂、局部重复或动作崩坏。

从建模角度看，长视频常常需要一种带历史依赖的生成方式，最常见的抽象可以写成：

```math
p(x_{1:T} \mid c)
=
\prod_{t=1}^{T} p(x_t \mid x_{<t}, c)
```

虽然很多扩散模型并不是严格的自回归模型，但这个式子很好地表达了长视频的本质难点：**后面的帧必须依赖前面的上下文，而上下文一长，建模和计算都会迅速变难。**

在这个问题上，`Causal`、`Chunk`、`Sliding-Window` 代表了三种很实用的系统策略：

1. **Causal**  
当前时刻主要依赖历史信息，不强依赖未来帧。这种设计更适合流式生成、长视频扩展和在线推理，也更容易和 causal VAE、causal attention 结合。代价是全局一致性有时不如 full-context 方案。

2. **Chunk**  
把长视频拆成多个片段，在片段级别训练或生成，再通过边界对齐或条件传递拼接起来。它的优势是工程上容易扩展，显存压力可控；问题是片段边界最容易出现风格跳变和动作不连续。

3. **Sliding-Window**  
让模型始终只看一个有限长度的局部时间窗口，窗口随着时间向前滑动。这种方式能在一定程度上兼顾上下文利用和计算成本，因此在长视频生成中很常见，但它需要额外处理窗口交界处的一致性。

除了时序切分策略，视频里的**采样优化**也比图像更重要。原因在于视频推理不是多一张图，而是多一整段时空结构；扩散步数、latent 尺度、解码策略的任何一点冗余，都会被视频长度成倍放大。因此很多系统会特别强调：

1. **更高效的 sampler 或更少的步数**，降低整段视频生成时延。
2. **分块解码与分阶段生成**，避免一次性在高分辨率全时长上完成所有计算。
3. **稀疏注意力、窗口注意力或结构化 token 策略**，控制长序列复杂度。
4. **蒸馏、级联或 coarse-to-fine 路线**，在质量和吞吐之间做更稳妥的平衡。

因此，这一节和后续 Open-Sora、HunyuanVideo、Kling、Seedance 等系统章节是直接相连的。后面很多模型强调 causal VAE、sparse attention、分阶段采样、长视频推理策略，本质上都不是“零碎工程技巧”，而是在解决同一个核心问题：**如何把本来只适合短片段的生成模型，扩展成一个真正可训练、可推理、可部署的长视频系统。**

面试里可以收束成一句话：**长视频与高效生成的本质，不是单点模块优化，而是围绕上下文长度、误差累积和推理成本做系统级设计；Causal、Chunk、Sliding-Window 和采样优化只是这套系统设计中最典型的四类工具。**
