<h1 id="4.3.1_视频生成主流范式">4.3.1 视频生成主流范式</h1>

<h2 id="4.3.1.1_Text-to-Video：从文本到视频的基本生成范式">4.3.1.1 Text-to-Video：从文本到视频的基本生成范式</h2>

### 面试问题1：什么是 Text-to-Video 任务？其核心目标是什么？

**难度评分：⭐⭐⭐ (3/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

Text-to-Video（T2V）任务指的是：**给定文本条件，让模型生成一段与文本语义一致、视觉上合理、时间上连贯的视频。** 从概率建模角度看，它本质上是在学习条件分布：

```math
p(x_{1:T} \mid c)
```

其中 $x_{1:T}$ 表示视频序列，$c$ 表示文本条件。

它的核心目标通常包括三层：

1. **语义对齐**：生成内容要准确响应文本中的主体、场景、动作、属性和风格描述。
2. **时序连贯**：相邻帧之间要保持自然过渡，不能出现明显闪烁、跳变或身份漂移。
3. **视觉质量**：单帧要清晰、结构合理，整段视频要具有可观看性和真实感。

站在 AI 视频任务的视角，T2V 的难点不只是“把文字变成画面”，而是要把**静态语义**扩展成**动态过程**。也就是说，模型不仅要理解“画面里有什么”，还要决定“这些对象如何随时间变化、变化是否符合文本描述和物理常识”。

因此，T2V 可以理解为：**文本条件驱动下的时空联合生成任务**。它既要求跨模态语义理解，也要求视频级运动建模和一致性控制。

### 面试问题2：Text-to-Video 与 Text-to-Image 任务的本质区别是什么？

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

Text-to-Image（T2I）和 Text-to-Video（T2V）都属于文本条件生成任务，但两者的本质区别在于：**T2I 只需要生成单帧空间内容，而 T2V 需要生成完整的时空演化过程。**

如果用条件分布表示：

```math
\text{T2I: } p(x \mid c), \qquad
\text{T2V: } p(x_{1:T} \mid c)
```

T2V 相比 T2I 多出的不是简单的帧数，而是整个**时间维度建模问题**。这会带来几个关键差异：

1. **多了运动建模**：T2I 只需决定“这一帧长什么样”，T2V 还要决定“主体怎么动、镜头怎么动、动作如何展开”。
2. **多了一致性约束**：T2I 只关注单张图质量，T2V 还必须保证人物身份、背景结构、光照纹理和动作轨迹在多帧中稳定。
3. **多了长程依赖**：视频前后相隔较远的帧仍可能存在强关联，例如角色再次出现时要保持身份一致。
4. **计算与数据复杂度更高**：视频输入输出维度更大，训练、采样、评估和标注都比图像更复杂。

所以更准确地说，T2V 不是“T2I 的多帧版本”，而是**在图像生成基础上叠加了时序建模、运动建模和视频一致性建模**的更难任务。

### 面试问题3：Text-to-Video 模型的基本架构组成有哪些？

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

一个典型的 Text-to-Video 模型通常由以下几个核心模块组成：

1. **文本编码器**  
   用于把自然语言 prompt 编码成条件特征，常见如 T5、CLIP Text Encoder、LLaMA 系列文本模块等。

2. **视频潜空间编码器 / 解码器**  
   通常是 VAE 或 Video VAE。它负责把高维像素视频压缩到 latent space，再把生成后的 latent 解码回视频：

```math
z = E(x_{1:T}), \qquad \hat{x}_{1:T} = D(z)
```

3. **生成主干网络**  
   这是模型的核心，负责在 latent space 中完成生成。主流做法包括 3D U-Net、Video U-Net、DiT、Video Transformer 等。

4. **条件注入模块**  
   把文本条件注入主干网络，常见方式包括 Cross-Attention、AdaLN、FiLM、ControlNet 类结构等。

5. **噪声调度器与采样器**  
   在扩散框架中负责定义前向加噪和反向采样策略，例如 DDPM、DDIM、Euler、DPM-Solver 等。

如果用一条简化流水线表示，通常可以写成：

```text
Text Prompt -> Text Encoder -> Conditional Features
Noise / Initial Latent -> Video Backbone -> Denoised Video Latent
Video Latent -> Video VAE Decoder -> Output Video
```

从工程视角看，T2V 模型的系统设计重点主要落在三件事上：

1. **条件是否读得懂**：文本编码能力和条件注入方式是否足够强。
2. **时空是否建得稳**：主干网络能否有效处理时间维度和一致性问题。
3. **潜空间是否压得好**：视频 VAE 是否在压缩效率和重建质量之间取得平衡。

因此，Text-to-Video 架构本质上是**文本条件模块 + 视频潜空间模块 + 时空生成主干**三者的组合。

### 面试问题4：Text-to-Video 生成中的时序一致性问题如何解决？

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

时序一致性问题指的是：**模型虽然能生成单帧合理的图像，但前后帧之间可能出现闪烁、身份漂移、动作不连续或背景跳变。** 这是 Text-to-Video 中最核心的技术难点之一。

解决思路通常分为三类：

1. **在架构层显式建模时间维度**  
   例如引入 3D 卷积、Temporal Attention、Space-Time Attention、Motion Module、3D RoPE 等，让模型在生成时同时看到多帧关系，而不是逐帧独立生成。

2. **在训练目标中加入一致性约束**  
   常见做法是让相邻帧或对齐后的特征尽量保持一致。例如基于运动补偿的时序一致性损失可写成：

   ```math
   \mathcal{L}_{tc}
   = \sum_t
   \left\|
   \phi(x_{t+1}) - \mathcal{W}(\phi(x_t), F_{t \rightarrow t+1})
   \right\|_1
   ```

   其中 $\phi(\cdot)$ 是特征提取器，$\mathcal{W}$ 表示 warp 操作，$F_{t \rightarrow t+1}$ 是帧间运动场。

3. **在采样与推理阶段增强跨帧稳定性**  
   例如共享部分噪声初始化、引入关键帧约束、使用首帧条件、做滑动窗口生成、加入后处理去闪烁模块等。

更具体地说，时序一致性通常要同时覆盖以下几个层面：

1. **内容一致性**：场景、物体、背景结构不能无故突变。
2. **身份一致性**：人物或主体外观在连续帧中要稳定。
3. **运动一致性**：动作轨迹和速度变化要自然。
4. **视觉一致性**：亮度、颜色、纹理、边缘不能随机抖动。

所以从 AI 视频任务视角看，时序一致性不是一个单独的小优化项，而是**决定 T2V 是否真正从“图像生成”升级为“视频生成”的核心能力**。

### 面试问题5：Text-to-Video 技术的主要应用场景有哪些？

**难度评分：⭐⭐ (2/5)  |  考察频率：⭐⭐⭐ (3/5)**

Text-to-Video 的应用场景可以概括为：**用自然语言直接生成动态视觉内容，降低视频制作门槛，并提升内容生成效率。**

主要场景包括：

1. **内容创作与短视频生产**  
   用于生成广告片段、剧情分镜、视觉概念短片、社媒内容和创意素材。

2. **影视预演与分镜可视化**  
   导演、编剧和美术可以通过文本快速验证镜头语言、场景氛围和动作设计。

3. **游戏与虚拟世界内容生成**  
   用于生成过场动画、角色演示、世界观概念视频和 NPC 行为示意。

4. **教育与知识表达**  
   把抽象概念、历史事件、科学过程或工业流程转成动态演示视频。

5. **个性化营销与品牌传播**  
   基于不同用户群体或商品描述快速生成定制化视频素材。

6. **多模态智能体与人机交互**  
   在更长期的方向上，T2V 不只是内容生成工具，还可能成为世界模拟、决策预测和交互式视频代理的一部分。

不过需要注意，当前 T2V 在真实落地中仍受限于几个因素：

1. **长视频稳定性不足**  
2. **细粒度动作和复杂物理过程控制仍然较弱**  
3. **算力成本较高，生成延迟明显**  
4. **版权、安全与内容治理问题仍需工程配套**

因此，现阶段最成熟的应用更多集中在**创意视频生成、概念可视化和辅助内容生产**；而高精度、长时长、强可控的工业级视频生成，仍然是持续演进中的方向。

<h2 id="4.3.1.2_Image-to-Video：从静态图到动态视频的运动补全">4.3.1.2 Image-to-Video：从静态图到动态视频的运动补全</h2>

### 面试问题1：什么是 Image-to-Video 任务？它的核心目标是什么？

**难度评分：⭐⭐⭐ (3/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

Image-to-Video（I2V）任务指的是：**给定一张或少量参考图像，让模型生成一段在内容上与参考图保持一致、在时间上具有合理运动和连续性的动态视频。**  

如果从条件生成角度表达，它可以写成：

```math
p(x_{1:T} \mid I, c)
```

其中 $I$ 表示输入图像或参考图集合，$c$ 可以表示可选的辅助条件，例如文本、动作信号、相机轨迹或风格控制。

I2V 的核心目标通常包括三层：

1. **内容保持**：输入图像中的主体外观、场景布局、风格特征不能在生成过程中被随意改写。
2. **运动补全**：模型要在静态图像没有显式时间信息的前提下，推断出合理的动作、镜头变化和动态过程。
3. **时序稳定**：生成后的视频不能只有“动起来”，还必须在身份、背景、结构和纹理上保持连续稳定。

因此，I2V 不是简单把一张图“做成会动的 GIF”，而是**以图像为内容锚点，对缺失的时间维度进行补全**。它比 T2V 多了一层强内容约束，也比纯图像动画更强调视频级一致性与物理合理性。

### 面试问题2：Image-to-Video 与 Text-to-Video 的本质区别是什么？

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

Image-to-Video（I2V）与 Text-to-Video（T2V）都属于视频生成任务，但两者的本质区别在于：**T2V 是从抽象语义出发生成完整视频，而 I2V 是在已有视觉内容的基础上补全时间维度。**

如果用条件分布表示：

```math
\text{T2V: } p(x_{1:T} \mid c), \qquad
\text{I2V: } p(x_{1:T} \mid I, c)
```

二者最关键的差异体现在以下几个方面：

1. **条件约束强度不同**  
   T2V 主要依赖文本描述，条件较抽象；I2V 直接给出图像内容，条件更具体、更强。

2. **生成自由度不同**  
   T2V 可以自由决定主体外观、构图和场景细节；I2V 必须围绕输入图像展开，不能随意改写主体和背景。

3. **任务难点不同**  
   T2V 更难在“从无到有”的语义生成和时空建模；I2V 更难在“保持内容不变”的同时推断出合理运动。

4. **评价重点不同**  
   T2V 更强调语义对齐、创意性和整体视频质量；I2V 更强调参考一致性、首帧锁定、局部运动自然性和跨帧稳定性。

所以可以把两者理解为：
- T2V 的重点是“**根据文本生成什么**”
- I2V 的重点是“**在已知内容下让它如何自然地动起来**”

这也是为什么很多 I2V 模型在结构上会额外强调参考图注入、首帧约束和主体一致性控制。

### 面试问题3：为什么说 I2V 的本质是“内容已知、运动待补全”？

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

之所以说 I2V 的本质是“内容已知、运动待补全”，是因为输入图像已经给出了大量静态信息，但**没有给出明确的时间演化过程**。模型真正需要推断的，不再是“画面里有什么”，而是“这些内容接下来如何变化”。

一张输入图像通常已经包含：

1. **主体外观**：人物长相、服饰、物体形态、材质纹理。
2. **空间布局**：前景背景关系、构图方式、场景结构。
3. **视觉风格**：光照、颜色、镜头质感、艺术风格。

但它缺失的是：

1. **动作方向**：主体接下来往哪里动、怎么动。
2. **运动速度**：动作是平缓展开还是快速变化。
3. **镜头语言**：镜头是推进、拉远、平移还是保持静止。
4. **物理连续性**：局部形变、遮挡变化、动态细节是否自然。

因此，I2V 的核心生成任务并不是重建静态内容，而是学习一种**从单帧状态推断时序演化轨迹**的能力。  
从更抽象的角度看，它接近于在已知初始状态 $x_1$ 的情况下预测一个合理的视频轨迹：

```math
p(x_{2:T} \mid x_1, c)
```

这里的难点在于，这个问题天然具有 **one-to-many** 属性。也就是说，同一张图像可能对应很多合理的视频展开方式。例如同一个人物可以抬头、转身、微笑，也可以镜头缓慢推近。  

所以 I2V 的核心不是“把图像变成视频”这句表面描述，而是：**在固定内容约束下，找到一个既合理又稳定的动态补全过程。**

### 面试问题4：Image-to-Video 模型的常见输入条件与基本架构组成有哪些？

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

一个典型的 I2V 模型，通常不是只输入“一张图”，而是会结合参考图、文本、运动提示或控制信号共同生成。常见输入条件包括：

1. **单张参考图 / 首帧图像**  
   这是最基础的 I2V 条件，用来锁定主体外观、构图和场景初始状态。

2. **文本描述**  
   用于补充图像里没有明确给出的动态意图，例如“人物转头微笑”或“镜头缓慢推进”。

3. **运动或结构控制信号**  
   例如姿态序列、光流先验、深度图、轨迹、相机控制参数等，用于增强动态可控性。

4. **多参考图**  
   用于补充不同视角、不同姿态或更多主体细节，常见于更复杂的角色一致性生成。

从系统结构上看，I2V 模型与 T2V 有相似之处，但会额外强调“参考图条件如何稳定注入”。典型结构通常包括：

1. **参考图编码器**  
   把输入图像编码成空间特征、identity feature 或 latent feature。

2. **视频潜空间模块**  
   使用 Video VAE 或 latent tokenizer，把视频生成过程放到 latent space 中进行。

3. **时空生成主干**  
   常见为 Video U-Net、DiT、Video Transformer 等，用于建模帧间演化过程。

4. **条件注入模块**  
   通过 Cross-Attention、Reference Attention、Adapter、首帧锚定或特征拼接等方式，将参考图和其他条件注入主干。

5. **采样与解码模块**  
   在扩散框架中完成去噪采样，并最终解码为视频。

如果简化成一条流程，可以写成：

```text
Reference Image / First Frame -> Image Encoder -> Reference Features
Optional Text / Motion Condition -> Condition Encoder
Noise / Video Latent -> Video Backbone -> Generated Video Latent
Generated Video Latent -> Video Decoder -> Output Video
```

因此，I2V 架构的关键不只是“怎么生成视频”，而是**怎么让参考图条件在整个时间轴上持续生效，同时又给模型保留足够的运动自由度。**

### 面试问题5：Image-to-Video 的核心难点和典型应用边界分别是什么？

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

Image-to-Video 的核心难点，可以概括为一句话：**既要让输入图像稳定地保留下来，又要让视频自然地动起来。** 这两件事本身是相互拉扯的，因此 I2V 往往比表面看起来更难。

主要技术难点包括：

1. **首帧锁定与内容保持**  
   输入图像里的主体、背景和结构必须被稳定保留，否则模型容易“越生成越不像原图”。

2. **合理运动补全**  
   单张图像没有真实时间信息，模型需要从数据先验中推断动作，这天然存在不确定性和多解性。

3. **局部漂移与细节崩坏**  
   人脸、手部、头发、衣物边缘等细节区域最容易出现闪烁、变形或身份漂移。

4. **相机运动与主体运动耦合**  
   很多场景里，镜头变化和主体动作会同时发生，模型需要分清是“相机在动”还是“物体在动”。

5. **可控性与自由度的平衡**  
   条件约束太强，视频可能僵硬、不自然；约束太弱，又可能偏离参考图。

从应用边界看，I2V 目前最适合的场景通常是：

1. **角色照片或插画动画化**  
   例如头像微动、人物转头、情绪表达、小幅动作生成。

2. **商品图 / 场景图动态演示**  
   让静态广告图或产品图生成轻量动态展示视频。

3. **概念图到镜头预演**  
   把静态分镜、设定图或 keyframe 快速扩展成短时动态片段。

4. **社媒与创意内容生产**  
   用于将已有图像资产快速转成可传播的视频素材。

但它当前也有比较清晰的边界：

1. **复杂长动作仍不稳定**  
   长时间连续动作或剧烈姿态变化仍然容易崩。
2. **复杂三维视角切换能力有限**  
   单张图像本身缺少完整 3D 信息，大幅视角变化通常比较困难。
3. **细粒度可控性仍不足**  
   很多 I2V 系统能“动起来”，但不一定能精确控制“怎么动”。

因此，现阶段 I2V 最成熟的定位通常不是“完全替代视频制作”，而是**在已有视觉锚点的前提下，高效生成短时、可观看、风格统一的动态视频片段。**

<h2 id="4.3.1.3_Video-to-Video：基于源视频的风格迁移、重绘与重生成">4.3.1.3 Video-to-Video：基于源视频的风格迁移、重绘与重生成</h2>

### 面试问题1：什么是 Video-to-Video 任务？它与 Text-to-Video、Image-to-Video 的区别是什么？

**难度评分：⭐⭐⭐ (3/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

Video-to-Video（V2V）任务指的是：**给定一段源视频，让模型在保留其时空结构、运动轨迹或关键内容约束的前提下，对视频进行风格变换、局部重绘、语义编辑或整体重生成。**

如果用条件生成视角来写，它可以近似表示为：

```math
p(y_{1:T} \mid x_{1:T}, c)
```

其中 $x_{1:T}$ 是输入源视频，$y_{1:T}$ 是输出目标视频，$c$ 表示可选条件，例如文本指令、风格条件、参考图或控制信号。

它和 T2V、I2V 的本质区别在于：

1. **T2V 是从抽象语义出发从零生成视频**  
   条件主要是文本，模型需要同时决定“生成什么内容”和“内容怎么运动”。

2. **I2V 是在单张图像基础上补全时间维度**  
   输入图像提供内容锚点，但时间演化仍然需要模型推断。

3. **V2V 是在已有完整视频基础上做受控变换**  
   输入里已经存在真实的时序结构、动作节奏、镜头运动和场景布局，模型不需要从零发明这些内容，而是要在已有结构上进行改写。

所以，V2V 的核心不是“生成一段新视频”，而是**基于源视频做结构保留型变换**。  
这也是为什么 V2V 在很多场景下比 T2V 更可控，因为它天然继承了源视频的时序与构图先验。

### 面试问题2：Video-to-Video 常见可以分成哪些子任务，例如风格迁移、视频重绘、视频重生成？

**难度评分：⭐⭐⭐ (3/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

Video-to-Video 并不是单一任务，而是一类“以源视频为条件进行变换”的任务集合。常见子任务大致可以分为以下几类：

1. **视频风格迁移（Video Style Transfer）**  
   保留原视频的动作、布局和主要结构，把视觉风格改成油画、动漫、电影质感或特定美术风格。  
   核心是“时序不变，外观风格变化”。

2. **视频重绘 / 重渲染（Video Repainting / Re-rendering）**  
   保留视频中的时序和大体结构，但对纹理、材质、光照或局部区域进行重新生成。  
   例如把真实视频改成二次元风格，或将场景重新渲染为高质量 CG 质感。

3. **视频编辑（Instruction-based Video Editing）**  
   给定文本指令，对源视频中的特定内容进行修改，例如更换服装、替换背景、调整天气、改变物体属性等。  
   这类任务通常强调“局部编辑 + 全局时序稳定”。

4. **视频重生成（Video Regeneration / Video Rediffusion）**  
   用源视频提供时序和结构约束，再在更大自由度下重新生成目标视频。  
   它比简单风格迁移更激进，往往允许更大范围的外观和语义变化。

5. **视频修复与增强（Restoration / Enhancement）**  
   例如去噪、超分、去模糊、补帧、补洞、老片修复等。  
   这类任务也可以归入广义 V2V，因为它同样是“输入视频 -> 输出视频”的映射。

这些子任务虽然目标不同，但有一个共同点：**它们都不是从无到有地生成时空结构，而是在源视频提供的时间轴上做受控变换。**

### 面试问题3：为什么说 Video-to-Video 的核心不是“从零生成”，而是“在保留源视频结构的前提下做受控变换”？

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

这是理解 V2V 最关键的一点。源视频本身已经给出了大量高价值信息，包括：

1. **时序结构**：动作节奏、镜头切换、运动轨迹、物体交互顺序。
2. **空间结构**：构图、场景布局、前景背景关系、透视与几何关系。
3. **动态先验**：遮挡变化、局部形变、相机运动等真实视频中的复杂时空模式。

因此，V2V 模型不需要像 T2V 一样去决定“这段视频从头该怎么演化”，而是要回答另一个问题：**在不破坏已有时空骨架的情况下，哪些内容应该保留，哪些内容允许重写。**

从这个角度看，V2V 更像一个**受约束的视频映射问题**，而不是纯生成问题。  
它的重点是把输入视频中的一些不变因素当作锚点，例如：

- 时间轴顺序不变
- 主体运动大致不变
- 镜头轨迹大致不变
- 空间几何关系尽量不变

然后只在允许的维度上做变换，例如：

- 风格变化
- 局部区域替换
- 背景重绘
- 语义属性修改

这也是为什么 V2V 的模型设计通常会特别强调 source video conditioning、结构保留、一致性编辑和局部可控性。  
如果把源视频结构破坏掉，V2V 就会退化成“重新生成一段新视频”，那样不仅可控性更差，任务定义本身也会发生变化。

所以，V2V 的本质可以概括为：**源视频提供时空骨架，模型在这个骨架上做受控重写。**

### 面试问题4：Video-to-Video 模型通常需要保留源视频的哪些信息？哪些信息又可以被替换或重写？

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

V2V 的难点就在于“保留”和“改写”的边界必须清楚。一般来说，模型通常需要优先保留以下信息：

1. **时间顺序与动作节奏**  
   例如人物先抬手再转身的顺序、车辆的运动方向、镜头推进节奏等。

2. **空间布局与几何结构**  
   包括主体位置、场景透视、前景背景关系、相对尺度等。

3. **主体身份或关键对象轨迹**  
   特别是在人物视频、角色视频或需要连续编辑的场景里，主体不能在不同帧中“换人”。

4. **局部运动连续性**  
   例如手部动作、嘴型、头部姿态、衣物摆动等，不能因为编辑而断裂。

而通常允许被重写的部分包括：

1. **视觉风格**  
   颜色风格、材质表现、光照氛围、渲染质感。

2. **局部语义内容**  
   例如替换衣服、修改发色、改变天气、替换背景元素。

3. **画面细节与纹理**  
   在不破坏结构的前提下，细节可以被重新生成得更干净或更符合目标风格。

4. **部分场景属性**  
   例如从白天改成夜晚、从晴天改成下雨，但通常仍要维持原视频的整体时空逻辑。

因此，V2V 不是“全都保留”也不是“全都重写”，而是做一种**结构性保留 + 局部受控替换**。  
如果保留太多，模型只能做非常弱的视觉增强；如果改写太多，视频又容易偏离源视频，失去 V2V 的核心价值。

从面试角度，一个比较好的总结是：**V2V 模型通常保留源视频的时空骨架、主体轨迹和几何结构，而把风格、局部语义、纹理细节和部分场景属性作为可重写维度。**

### 面试问题5：Video-to-Video 的核心技术难点和典型应用场景分别是什么？

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

Video-to-Video 的核心技术难点，可以概括为：**在大幅修改视频内容时，仍然保持原视频的时序稳定、结构合理和编辑可控。**

主要难点包括：

1. **编辑一致性难**  
   一旦对外观、风格或局部语义做修改，这些修改必须稳定传播到所有相关帧，不能出现一帧改了、一帧又恢复原样的情况。

2. **局部编辑与全局稳定的冲突**  
   例如只想改人物衣服，但背景、手部、边缘区域常常会被连带破坏。

3. **遮挡与运动区域处理困难**  
   在主体快速运动、被遮挡或局部形变明显时，模型更容易出现边缘撕裂、结构漂移和纹理闪烁。

4. **结构保持与编辑强度的平衡**  
   编辑太弱，效果不明显；编辑太强，又容易破坏源视频的时间连续性与空间结构。

5. **长视频编辑更难稳定**  
   一旦视频变长，误差累积和 drift 会更明显，跨镜头编辑的一致性也会变得更难。

从应用场景看，V2V 很适合以下几类任务：

1. **视频风格化与重渲染**  
   把真实视频改成动画、电影或特定美术风格。

2. **广告与营销素材编辑**  
   在已有视频素材基础上快速替换背景、服饰、商品属性或整体氛围。

3. **影视后期与预演**  
   用于镜头风格测试、特效预演、粗剪视频重绘等。

4. **内容安全与修复**  
   包括局部去除、替换、补洞、修复、增强等任务。

5. **多模态指令编辑**  
   根据文本或参考条件对已有视频做可控改写，这是当前很多视频编辑系统的重要方向。

因此，V2V 的价值并不在于“从零创造一个世界”，而在于**把已有视频素材变成可编辑、可迁移、可重绘、可再利用的动态资产**。这也是它在工业落地中经常比纯 T2V 更容易形成稳定工作流的原因。

<h2 id="4.3.1.4_Subject-to-Video-/-Character-to-Video：主体一致性驱动的视频生成">4.3.1.4 Subject-to-Video / Character-to-Video：主体一致性驱动的视频生成</h2>

### 面试问题1：什么是 Subject-to-Video / Character-to-Video？它与普通 Text-to-Video、Image-to-Video 有什么区别？

**难度评分：⭐⭐⭐ (3/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

Subject-to-Video / Character-to-Video 任务指的是：**给定某个特定主体的参考信息，让模型生成一段主体身份稳定、外观一致、动作合理的视频。** 这里的主体可以是人物、角色、IP 形象、动物，甚至某类具有固定身份特征的物体。

如果从条件生成角度表示，它通常可以写成：

```math
p(x_{1:T} \mid s, c)
```

其中 $s$ 表示主体条件，例如参考图、主体特征 embedding、身份 token 或多视角主体信息，$c$ 表示其他辅助条件，例如文本、动作、场景或相机控制。

它与普通 T2V、I2V 的核心区别在于：

1. **T2V 更强调从语义生成内容**  
   给定文本，模型主要解决“视频里是什么”和“视频怎么动”。

2. **I2V 更强调从图像补全时间维度**  
   给定一张图，模型主要解决“这张图里的内容如何动起来”。

3. **Subject-to-Video 更强调跨帧主体身份稳定**  
   模型不仅要生成视频，还必须在所有帧里持续保持“这是同一个主体”。

因此，这类任务最关键的约束不是“能不能生成动态画面”，而是**能不能在动态变化中维持主体身份不漂移**。  
这也是为什么 Character-to-Video 往往被单独拿出来讨论，因为它比普通 I2V 或 T2V 多了一层非常强的 identity consistency 要求。

### 面试问题2：为什么 Subject-to-Video 的核心问题不是“能不能生成视频”，而是“能不能稳定生成同一个主体”？

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

因为视频生成本身的“动起来”已经不是唯一门槛，真正困难的是：**一旦人物开始转头、移动、遮挡、换姿态，模型还能不能让观众持续相信这是同一个人或同一个角色。**

对于 Subject-to-Video 来说，主体一致性通常比单帧质量更重要。原因在于：

1. **主体是整个视频的核心锚点**  
   如果角色外观不稳定，即使动作流畅、画质不错，视频也会立即显得不可信。

2. **视频中的主体变化是连续发生的**  
   人脸角度变化、手部动作、服饰摆动、光照变化、局部遮挡都会不断挑战模型的身份保持能力。

3. **身份漂移通常是渐进发生的**  
   不是某一帧突然完全变掉，而是随着时间推移逐步出现五官变化、发型偏移、服装纹理漂移、脸型变形等问题。

4. **生成自由度和身份约束天然冲突**  
   模型如果自由度太大，容易“越生成越不像”；如果约束太强，又可能让动作僵硬或视频缺少自然变化。

因此，Subject-to-Video 的本质不是单纯的视频合成，而是一个**受强 identity constraint 约束的动态生成问题**。  
换句话说，普通视频生成更关心“有没有生成出合理视频”，而 Subject-to-Video 更关心“这个合理视频里的主角是不是始终同一个主体”。

### 面试问题3：Subject-to-Video 中通常有哪些条件形式，例如单张参考图、多视角图、主体 embedding、身份 token？

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

在 Subject-to-Video 中，主体条件的形式非常关键，因为它直接决定模型能否稳定提取和传播 identity 信息。常见条件形式主要有以下几类：

1. **单张参考图（single reference image）**  
   这是最常见的形式。优点是使用简单、成本低；缺点是信息有限，尤其在大幅姿态变化和复杂视角切换时容易失真。

2. **多张参考图 / 多视角图（multi-reference / multi-view images）**  
   通过提供主体在不同角度、不同表情或不同姿态下的图像，帮助模型建立更完整的主体表示。  
   这类方式通常对复杂角色生成更有效。

3. **主体 embedding / identity feature**  
   先用图像编码器、人脸识别模型或专门的 identity encoder 提取主体特征，再将其注入视频主干网络。  
   它的优点是更适合跨帧传递统一主体特征。

4. **身份 token / subject token**  
   用离散 token 或可学习 token 表示某个主体身份，再和文本或其他条件一起输入模型。  
   这种方式常见于 personalization、LoRA、DreamBooth 类扩展路线。

5. **局部结构条件**  
   比如关键点、姿态、分割 mask、面部区域特征等，用来进一步加强主体某些关键部位的一致性。

实际系统里，这些条件往往不会单独使用，而是组合出现。例如：

- 参考图 + 文本
- 多参考图 + identity embedding
- 主体 token + 动作控制
- 参考图 + 首帧锚定 + 局部面部特征

因此，Subject-to-Video 的条件设计核心不是“输入一种什么格式”，而是**如何把主体身份信息稳定、充分且可跨帧地注入视频生成主干。**

### 面试问题4：主体一致性在视频生成里通常体现在哪些层面？为什么它比图像生成更难？

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

主体一致性在视频生成里通常至少体现在以下几个层面：

1. **身份一致性（identity consistency）**  
   例如人物五官、脸型、发型、肤色、服饰、体型等核心身份特征在连续帧中保持稳定。

2. **结构一致性（structural consistency）**  
   包括头部比例、身体轮廓、四肢长度、角色局部结构关系等不能随机变化。

3. **纹理一致性（texture consistency）**  
   例如服装图案、头发纹理、饰品细节、材质表现等在不同帧中不能频繁跳变。

4. **语义一致性（semantic consistency）**  
   例如“这个角色是同一个人”“这只猫还是同一只猫”“这个 IP 角色没有被改成另一种形象”。

它比图像生成更难，根本原因在于：**图像只需单帧看起来像，视频则要求在时间轴上持续像。**

具体来说，难度主要来自：

1. **时间维度带来的误差累积**  
   一帧里的微小偏差，经过几十帧传播后就可能演化成明显身份漂移。

2. **姿态与视角持续变化**  
   人物转头、仰视、低头、远近变化都会导致可见特征改变，模型必须学会“变化中保持同一身份”。

3. **遮挡与形变更复杂**  
   手遮脸、头发飘动、衣物摆动、肢体交叉等情况都会让 identity 特征提取和保持变得更难。

4. **一致性和自然度有冲突**  
   如果一味强压 identity，视频可能会变得僵硬；如果放松约束，又容易漂移。

所以，视频里的主体一致性不是“图像一致性多做几帧”这么简单，而是一个**跨时间传播 identity 的动态建模问题**。

### 面试问题5：Subject-to-Video / Character-to-Video 的核心技术难点和典型应用场景分别是什么？

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

Subject-to-Video / Character-to-Video 的核心技术难点，主要集中在“**身份稳定**”与“**动态自然**”这两件事如何同时成立。

主要难点包括：

1. **主体漂移（identity drift）**  
   视频越长、动作越复杂，主体越容易逐步偏离参考形象。

2. **复杂姿态和视角变化**  
   角色一旦大幅转身、俯仰、侧脸或远近变化，模型更容易出现脸部失真、体型变化或服装错误。

3. **局部高频区域特别脆弱**  
   人脸、眼睛、嘴部、手部、头发、饰品等区域最容易成为一致性崩坏的来源。

4. **多主体场景更难**  
   一旦视频里不止一个人物，模型不仅要保持单个主体一致，还要避免主体之间特征串扰。

5. **可控动作与主体保持的平衡**  
   动作强度越大、镜头变化越复杂，对主体一致性的冲击越大。

从应用场景看，这类任务非常重要，主要包括：

1. **数字人 / 虚拟主播 / AI Avatar**  
   需要长期稳定生成同一个数字人物形象。

2. **角色动画与 IP 内容生产**  
   包括动漫角色、游戏角色、品牌吉祥物等，需要在不同视频中持续保持同一角色形象。

3. **个性化视频生成**  
   例如给定用户照片生成个性化动态短视频、写真视频或广告素材。

4. **影视与游戏预演**  
   用固定角色设定快速生成多个动作片段、镜头预演和叙事段落。

5. **多模态角色智能体**  
   在更长期的方向上，稳定主体生成会成为可交互数字人和视频 Agent 的关键基础能力。

因此，Subject-to-Video 的工业价值非常高，因为它把“生成一个视频”推进到了“生成一个**属于某个固定主体**的视频”。对很多商业场景来说，这比单纯生成高质量短视频更重要。 

<h2 id="4.3.1.5_视听生成：Audio-to-Video、Audio-Driven-Video-与-Video-to-Audio">4.3.1.5 视听生成：Audio-to-Video、Audio-Driven-Video 与 Video-to-Audio</h2>

### 面试问题1：什么是视听生成？它主要包括哪些典型任务形式，例如 Audio-to-Video、Audio-Driven Video、Video-to-Audio？

**难度评分：⭐⭐⭐ (3/5)  |  考察频率：⭐⭐⭐ (3/5)**

视听生成（audio-visual generation）指的是：**在视频与音频两个模态之间进行联合生成、条件驱动生成或跨模态转换生成的任务集合。**  
它不再只关心“视频生成得像不像”，还关心**声音和画面之间是否语义一致、时间同步、事件对应**。

如果从条件生成视角写，常见形式可以概括为：

```math
p(x_{1:T} \mid a, c), \qquad
p(a \mid x_{1:T}, c), \qquad
p(x_{1:T}, a \mid c)
```

其中：
- $x_{1:T}$ 表示视频序列
- $a$ 表示音频信号
- $c$ 表示其他辅助条件，例如文本、主体、场景或控制信号

典型任务形式主要包括三类：

1. **Audio-to-Video（A2V）**  
   给定音频，生成与音频语义、节奏或事件相匹配的视频内容。  
   例如根据音乐生成节奏感视频，根据环境音生成对应动态场景。

2. **Audio-Driven Video**  
   给定音频和某个视觉锚点，驱动已有主体或场景产生对应运动。  
   例如给一张人物头像配语音，让口型、表情和头部动作随音频变化。

3. **Video-to-Audio（V2A）**  
   给定视频生成匹配的声音，例如环境音、动作音效、说话声或 Foley。  
   例如根据人物走路、开门、击打动作生成对应音效。

因此，视听生成不是单一任务，而是一个**围绕声音与画面相互生成、相互约束、相互对齐**的任务家族。

### 面试问题2：Audio-to-Video 与 Audio-Driven Video 有什么区别？为什么两者不能简单混为一类？

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

这两个概念很容易被混用，但它们的任务约束明显不同。

**Audio-to-Video（A2V）** 更强调：**根据音频本身生成整段视频内容。**  
也就是说，音频是主要条件，模型要同时决定：

- 视频里是什么主体
- 场景怎么呈现
- 运动如何展开
- 节奏如何跟随音频

而 **Audio-Driven Video** 更强调：**在已有视觉锚点的前提下，用音频去驱动动态变化。**  
这里通常已经给定了人物图像、角色形象、场景片段或视频模板，模型不需要从零决定“视频内容是什么”，而是要回答“这个已有主体应该如何随着音频动起来”。

两者的差异可以总结为：

1. **条件结构不同**  
   A2V 更接近 `p(x_{1:T} \mid a, c)`，音频是主条件；  
   Audio-Driven 更接近 `p(x_{1:T} \mid I, a, c)` 或 `p(x_{1:T} \mid s, a, c)`，其中图像或主体条件同样重要。

2. **生成自由度不同**  
   A2V 的自由度更高，模型可以决定视频内容本身；  
   Audio-Driven 的自由度更低，重点是驱动既有主体或结构。

3. **评价重点不同**  
   A2V 更强调音频语义与视频内容是否匹配；  
   Audio-Driven 更强调嘴型同步、节奏同步、表情一致性与主体稳定。

4. **典型应用不同**  
   A2V 常用于音乐可视化、事件视频生成、声音驱动场景生成；  
   Audio-Driven 常用于数字人、talking head、配音角色动画。

所以，两者不能简单混为一类。前者更像“音频条件下的视频生成”，后者更像“音频条件下的视频驱动与动画化”。

### 面试问题3：为什么视听生成比纯视频生成多了一层“跨模态时间对齐”难题？

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

因为纯视频生成只需要保证视频内部的时序一致，而视听生成还必须保证：**视频时间轴和音频时间轴之间是对得上的。**

这意味着模型面对的不再只是：

```math
p(x_{1:T} \mid c)
```

而是要同时处理视频内部时序关系与跨模态关系，例如：

```math
p(x_{1:T}, a \mid c)
```

或在条件生成场景下处理：

```math
p(x_{1:T} \mid a, c)
```

这里新增的困难主要在于：

1. **音频与视频是不同时间分辨率的信号**  
   视频按帧采样，音频按更高频率的波形或频谱采样，两者时间尺度天然不一致。

2. **一个模态中的局部变化未必在另一个模态中一一显式对应**  
   例如一段音乐节奏变化可能对应镜头切换、动作加速或情绪变化，但不是简单的“每个音符对应一帧动作”。

3. **对齐错误比单模态错误更容易被感知**  
   单帧画质轻微偏差有时用户能容忍，但口型对不上、击打声晚半拍、背景事件声画不同步会非常明显。

4. **跨模态对齐往往和语义理解绑定**  
   模型不仅要知道“什么时候发生”，还要知道“发生的是什么”，例如鼓点、脚步、说话、开门、爆炸等事件。

因此，视听生成的难点不是“多一个模态”这么简单，而是多了一个**跨模态、跨时间尺度、强感知敏感**的对齐问题。

### 面试问题4：视听生成中通常需要对齐哪些层面的信息，例如语义、节奏、事件、口型与情绪？

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

视听生成里的“对齐”不是单层概念，通常至少包括以下几个层面：

1. **语义对齐（semantic alignment）**  
   声音和画面表达的是同一类内容。  
   例如看到下雨，应该听到雨声；看到狗叫，声音语义也应匹配。

2. **节奏对齐（rhythm alignment）**  
   适用于音乐驱动或节奏明显的场景。  
   例如鼓点增强时镜头节奏、动作频率或视觉强度也相应变化。

3. **事件对齐（event alignment）**  
   指某个具体事件在时间上对应。  
   例如开门瞬间有门响，击掌瞬间有拍击声，脚落地时出现脚步声。

4. **口型对齐（lip-sync alignment）**  
   在 talking head、数字人、角色配音场景中尤其关键。  
   声音中的音素变化要和嘴型、下颌运动、表情节奏匹配。

5. **情绪对齐（emotion / affect alignment）**  
   声音中的情绪强弱、语气变化、音乐氛围，需要和角色表情、动作力度、画面情绪相协调。

这几个层面并不是独立存在的，很多高质量系统都需要同时满足。  
例如一个数字人视频至少要满足：
- 语音内容与角色说话场景一致
- 口型和音素同步
- 表情与语气匹配
- 头部动作与语音节奏协调

所以，视听生成里的对齐，本质上是一个**从低层时间同步到高层语义情绪一致**的多层次问题。

### 面试问题5：视听生成的核心技术难点和典型应用场景分别是什么？

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐ (3/5)**

视听生成的核心技术难点，主要集中在三个方面：**跨模态理解、时间同步和联合生成稳定性。**

具体来说，常见难点包括：

1. **跨模态表示难统一**  
   音频和视频的数据结构差异很大，模型需要建立共享语义空间或有效的对齐机制。

2. **时间同步要求高**  
   尤其是嘴型、打击声、事件触发等任务，对细粒度时间误差非常敏感。

3. **一对多关系明显**  
   同一段音频可能对应多种合理画面，同一段视频也可能对应多种合理配音或环境音。

4. **局部错误感知非常强**  
   口型不准、声画错位、事件声延迟等问题非常容易被用户察觉。

5. **联合生成链路更长**  
   一旦同时生成音频和视频，训练稳定性、推理成本和评估难度都会明显增加。

从应用场景看，视听生成目前非常重要，主要包括：

1. **数字人 / Talking Head / AI Avatar**  
   让角色根据语音做出嘴型、表情和头部动作。

2. **音乐驱动视频生成**  
   根据音乐生成节奏感画面、MV 风格内容或舞蹈/镜头变化。

3. **Video-to-Audio / Foley 生成**  
   给视频自动补环境音、动作音效或同步音轨。

4. **影视与短视频内容生产**  
   通过统一视听生成提升内容制作效率，减少手工配音与后期对齐成本。

5. **多模态交互系统**  
   在长期方向上，视听生成会成为交互式数字人、视频智能体和世界模拟系统的重要组成部分。

因此，视听生成的意义不只是“让视频多一个声音”，而是**让视频系统真正具备跨模态生成与交互能力**。这也是它在未来统一视频智能框架里越来越重要的原因。

<h2 id="4.3.1.6_多模态联合生成与统一条件建模：文本、图像、视频、音频">4.3.1.6 多模态联合生成与统一条件建模：文本、图像、视频、音频</h2>

### 面试问题1：什么是多模态联合生成？它与单一条件的视频生成任务有什么本质区别？

**难度评分：⭐⭐⭐ (3/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

### 面试问题2：为什么视频生成系统会逐渐从“单条件模型”演化到“统一条件建模”？

**难度评分：⭐⭐⭐⭐ (4/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

### 面试问题3：统一条件建模里常见会联合哪些模态，例如文本、图像、视频、音频、身份与控制信号？

**难度评分：⭐⭐⭐ (3/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

<h1 id="4.3.2_视频生成的核心技术实现路径">4.3.2 视频生成的核心技术实现路径</h1>

<h2 id="4.3.2.1_潜空间视频生成路线：Video-VAE-与-Latent-Video-Diffusion">4.3.2.1 潜空间视频生成路线：Video VAE 与 Latent Video Diffusion</h2>

视频生成之所以几乎都走 **Video VAE + Latent Video Diffusion** 路线，核心原因不是“理论上更优雅”，而是**像素空间直接生成在视频场景下代价过高**。视频比图像多一个时间维度，输入规模通常可写成：

```math
x \in \mathbb{R}^{T \times C \times H \times W}
```

当 $T$、$H$、$W$ 同时增大时，训练显存、注意力开销、采样时延都会迅速放大。因此主流系统通常先把视频压缩到潜空间，再在 latent space 中完成去噪或生成：

```math
z = E(x_{1:T}), \qquad \hat{x}_{1:T} = D(\hat{z}_{1:T})
```

这里真正重要的不是“用了 VAE”这件事本身，而是**通过压缩把生成问题从像素域转成更低维、更适合建模时序结构的表示域**。这也是 Latent Diffusion 从图像成功迁移到视频的根本原因。

站在视频生成任务视角，Video VAE 相比图像 VAE 的要求更高。图像压缩器只要保住空间结构即可，而视频压缩器除了要保住单帧内容，还要尽量保住：

1. **时间连续性**：压缩后相邻帧之间不能被打散，否则后续生成容易出现 flicker。
2. **运动信息**：物体位移、局部形变、镜头运动不能在编码阶段被过度抹平。
3. **长视频可扩展性**：压缩器本身要能支持更长序列，很多系统因此会强调 temporal downsampling、causal VAE 或分块编码设计。

因此，视频生成里真正的瓶颈常常不是“生成主干不够强”，而是**视频压缩器是否为后续建模保留了足够有用的时空结构**。Open-Sora、Open-Sora-Plan、HunyuanVideo 等路线都反复强调 video tokenizer / video VAE 的重要性，本质上就是这个原因。

Latent Video Diffusion 成为主流工程路线，是因为它在三件事之间提供了比较均衡的折中：

1. **质量**：潜空间仍能保留相对稳定的语义和结构信息，便于高质量生成。
2. **成本**：相比像素空间，latent token 数量显著下降，训练和推理更可行。
3. **可扩展性**：更容易叠加 temporal attention、3D 模块、reference control、audio condition 等视频特有能力。

这一点可以从代表工作中看得很清楚。`Latent Video Diffusion Models`、`Stable Video Diffusion` 强调的是“在 latent space 中复用成熟扩散框架”；Open-Sora 路线进一步强调 causal VAE、视频压缩比与长视频训练的关系；而更大规模的系统则会把 latent route 看成整个视频生成流水线的入口，而不是一个单独组件。

面试中可以概括为：**潜空间视频生成路线之所以成为主流，是因为它把视频生成从难以承受的像素空间问题，转化为一个在质量、效率和可扩展性之间更平衡的时空生成问题；而 Video VAE 的质量往往决定了整条视频生成链路的上限。**

<h2 id="4.3.2.2_视频扩散的时序建模：Temporal-Layer-/-Temporal-Attention-/-3D-Conv">4.3.2.2 视频扩散的时序建模：Temporal Layer / Temporal Attention / 3D Conv</h2>

视频扩散之所以必须引入专门的时序建模路径，核心原因是：**单帧生成质量并不等于视频生成质量**。如果模型只是把图像扩散逐帧应用到视频上，那么每一帧可能都“看起来还可以”，但整段视频往往会出现身份跳变、背景抖动、动作不连续和局部闪烁。因此，视频模型真正要解决的是：

```math
p(x_{1:T} \mid c)
```

而不是一组互相独立的 $p(x_t \mid c)$。也就是说，模型必须显式建模帧间依赖。

从工程上看，`Temporal Layer`、`Temporal Attention` 和 `3D Conv` 代表了三种很典型的时序建模思路，它们的差别不只是“模块长得不一样”，而是各自适合解决不同层次的问题。

1. **3D Conv** 更像局部时空耦合建模。  
它天然适合捕捉短时间窗口内的运动模式，比如局部位移、速度变化、短程动态纹理变化。优点是归纳偏置强、局部建模稳定，缺点是长程依赖能力有限，扩展到长视频时通常不够灵活。

2. **Temporal Attention** 更像显式跨帧对齐机制。  
它允许当前帧特征直接与其他时间位置交互，例如：

```math
\mathrm{Attn}(Q_t, K_{1:T}, V_{1:T})
=
\mathrm{softmax}\left(\frac{Q_t K_{1:T}^{\top}}{\sqrt{d}}\right)V_{1:T}
```

它更适合处理跨帧一致性、主体稳定、远距离依赖等问题，因此在中大规模视频扩散模型里非常常见。代价是计算和显存开销通常高于简单的局部卷积。

3. **Temporal Layer / Motion Module** 更像一种轻量注入式改造。  
这类方法常见于“从图像扩散迁移到视频扩散”的路线中，例如在原有 2D backbone 中插入时间模块，只在有限位置建模时间维度。它的优点是迁移成本低、复用图像先验强，AnimateDiff 就是很典型的代表；缺点是上限往往受原始 2D 主干约束，更适合短视频和中等复杂度运动。

所以，时序建模方案的选择，本质上是在做三方面权衡：

1. **运动表达能力**：能不能刻画复杂动作、镜头运动和长程依赖。
2. **一致性稳定性**：能不能降低 flicker、drift 和 identity collapse。
3. **训练与推理成本**：显存、吞吐、token 数量和可扩展性是否可接受。

从这个角度看，后续不同模型路线的差异就更容易理解了。U-Net / Video U-Net 路线更常见局部时间模块与 factorized attention；AnimateDiff 强调在图像模型上增量加入 motion module；Sora / Open-Sora / DiT 类路线则更倾向于把视频直接表示成时空 token，再在统一 backbone 里处理时间依赖。

面试里比较稳妥的回答是：**视频扩散的时序建模不是单纯“多加一个时间维度”，而是在局部运动、长程依赖和计算成本之间做结构化取舍。3D Conv、Temporal Attention 和 Temporal Layer 分别代表了局部耦合、跨帧交互和轻量迁移三类主流路径。**

<h2 id="4.3.2.3_条件控制与一致性建模：Cross-Attention、Reference、Audio、Identity">4.3.2.3 条件控制与一致性建模：Cross-Attention、Reference、Audio、Identity</h2>

在视频生成里，条件控制和一致性建模几乎不能分开讨论。原因很直接：**图像生成里的条件只需要影响一张图，而视频生成里的条件需要在整段时间轴上持续生效。** 因此，视频中的很多一致性问题，本质上都可以理解为“条件没有被稳定地传递到所有帧”。

如果从条件分布角度写，视频生成往往不是单一条件，而是多条件联合约束：

```math
p(x_{1:T} \mid c_{\text{text}}, c_{\text{ref}}, c_{\text{audio}}, c_{\text{id}}, \ldots)
```

这里真正困难的地方，不是条件数量变多，而是这些条件在时间维度上会相互耦合。比如文本条件决定整体语义，首帧或参考图决定外观锚点，identity condition 决定主体是否稳定，audio condition 又要求动作节奏与语音或音乐同步。它们都不是单帧约束，而是**跨帧持续约束**。

从任务适配关系看，不同条件类型各有典型使用场景：

1. **文本条件** 主要对应 T2V，负责高层语义、动作描述和风格约束。
2. **图像 / 首帧 / reference 条件** 常见于 I2V、V2V、reference-driven generation，重点是锁定外观、构图或初始状态。
3. **identity / subject 条件** 常见于 Subject-to-Video、Character-to-Video，重点是跨帧人物或主体一致性。
4. **音频条件** 常见于 talking head、music-driven video、audio-driven motion，重点是声画同步与节奏控制。

因此，条件控制在视频里不是“多加几个 condition token”这么简单，而是要解决两个更深的问题：

1. **条件是否能跨时间稳定传播**：同一个条件能不能持续作用到整段视频，而不是只在局部帧生效。
2. **多个条件之间是否协调**：例如文本想让角色转身，reference 又要求外观稳定，音频还要求口型同步，这些约束不能相互打架。

这也是为什么很多视频模型会把一致性建模和控制接口绑在一起设计。reference attention、identity feature injection、首帧锚定、motion guidance、audio alignment 等做法，本质上都是在提升“条件跨帧稳定传递”的能力。AnimateDiff 的 reference control、基于 IP-Adapter / ReferenceNet 的 I2V 路线、以及更大系统中的多条件联合控制，都可以放在这个框架下理解。

如果一定要一句话总结，这一节最核心的观点是：**视频生成中的一致性，很多时候不是一个独立后处理问题，而是条件控制问题在时间维度上的延伸。谁能把条件稳定、持续、协调地注入整段视频，谁就更容易做出真正可控的视频生成系统。**

<h2 id="4.3.2.4_长视频与高效生成：Causal、Chunk、Sliding-Window-与采样优化">4.3.2.4 长视频与高效生成：Causal、Chunk、Sliding-Window 与采样优化</h2>

长视频生成不是把短视频模型“直接跑更久”这么简单。视频一旦变长，系统会同时遇到四类问题：**上下文长度爆炸、误差累积、时序漂移和推理成本失控。** 这也是为什么很多短视频模型在 4 秒、8 秒内效果不错，但一旦拉长到十几秒甚至更长，就会明显暴露出身份不稳、镜头逻辑断裂、局部重复或动作崩坏。

从建模角度看，长视频常常需要一种带历史依赖的生成方式，最常见的抽象可以写成：

```math
p(x_{1:T} \mid c)
=
\prod_{t=1}^{T} p(x_t \mid x_{<t}, c)
```

虽然很多扩散模型并不是严格的自回归模型，但这个式子很好地表达了长视频的本质难点：**后面的帧必须依赖前面的上下文，而上下文一长，建模和计算都会迅速变难。**

在这个问题上，`Causal`、`Chunk`、`Sliding-Window` 代表了三种很实用的系统策略：

1. **Causal**  
当前时刻主要依赖历史信息，不强依赖未来帧。这种设计更适合流式生成、长视频扩展和在线推理，也更容易和 causal VAE、causal attention 结合。代价是全局一致性有时不如 full-context 方案。

2. **Chunk**  
把长视频拆成多个片段，在片段级别训练或生成，再通过边界对齐或条件传递拼接起来。它的优势是工程上容易扩展，显存压力可控；问题是片段边界最容易出现风格跳变和动作不连续。

3. **Sliding-Window**  
让模型始终只看一个有限长度的局部时间窗口，窗口随着时间向前滑动。这种方式能在一定程度上兼顾上下文利用和计算成本，因此在长视频生成中很常见，但它需要额外处理窗口交界处的一致性。

除了时序切分策略，视频里的**采样优化**也比图像更重要。原因在于视频推理不是多一张图，而是多一整段时空结构；扩散步数、latent 尺度、解码策略的任何一点冗余，都会被视频长度成倍放大。因此很多系统会特别强调：

1. **更高效的 sampler 或更少的步数**，降低整段视频生成时延。
2. **分块解码与分阶段生成**，避免一次性在高分辨率全时长上完成所有计算。
3. **稀疏注意力、窗口注意力或结构化 token 策略**，控制长序列复杂度。
4. **蒸馏、级联或 coarse-to-fine 路线**，在质量和吞吐之间做更稳妥的平衡。

因此，这一节和后续 Open-Sora、HunyuanVideo、Kling、Seedance 等系统章节是直接相连的。后面很多模型强调 causal VAE、sparse attention、分阶段采样、长视频推理策略，本质上都不是“零碎工程技巧”，而是在解决同一个核心问题：**如何把本来只适合短片段的生成模型，扩展成一个真正可训练、可推理、可部署的长视频系统。**

面试里可以收束成一句话：**长视频与高效生成的本质，不是单点模块优化，而是围绕上下文长度、误差累积和推理成本做系统级设计；Causal、Chunk、Sliding-Window 和采样优化只是这套系统设计中最典型的四类工具。**
