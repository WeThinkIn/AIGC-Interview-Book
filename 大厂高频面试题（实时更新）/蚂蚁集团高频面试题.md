工程侧可以强调稳定性和可控性：例如如何降低大模型幻觉、如何在工具调用失败时降级、如何记录 trace、如何做效果评估和灰度上线。后端基础上可以补充 Spring、MySQL、Redis、网络协议和并发控制。最后收束到岗位匹配：智能体与大模型应用不是简单调 API，而是把模型能力、检索系统、业务工具和工程可靠性组合成真正可用的系统。

<a id="ant-agent-20260520-q2"></a>
##### 如何解决大模型幻觉？

**回答：**

大模型幻觉本质上是模型在缺少可靠依据、上下文冲突、训练分布外问题或解码不确定时，仍然生成看似合理但事实错误的内容。治理幻觉不能只靠单一手段，要分层处理。

第一层是输入和输出约束。要明确模型的回答边界，例如要求只基于给定资料回答、不知道就拒答、引用证据段落、输出置信度或不确定原因。结构化任务可以用 JSON Schema、枚举值、正则约束或函数调用限制输出空间。

第二层是知识增强。对于企业知识、实时数据、私有文档这类模型参数中没有或可能过期的信息，应通过 RAG、数据库查询、搜索 API、业务系统工具调用来获取外部证据。RAG 的关键不只是召回文档，而是保证召回内容相关、完整、可追溯，并在生成阶段要求模型基于证据回答。

第三层是工具校验。对于可验证问题，不应让模型凭概率生成结果。例如订单状态、库存、账户余额、SQL 查询、数学计算、代码测试，都应调用确定性工具或业务接口，再由模型解释结果。Agent 系统里常见做法是把生成、执行和验证拆开，用工具执行、规则校验、单元测试、权限校验兜底。

第四层是训练和对齐。微调可以让模型适应领域表达、固定输出格式、学习拒答边界和业务流程，但不适合作为事实知识的唯一来源。知识会更新，微调也可能把错误样本固化进模型。更稳妥的做法是用 SFT 训练规范回答方式，用偏好优化训练拒答和引用习惯，再配合 RAG 与工具使用。

第五层是评估与线上治理。要建立 hallucination case 集，评估事实一致性、引用命中率、拒答准确率、工具调用成功率和人工标注通过率。线上还要做日志追踪、badcase 回流、敏感场景人工审核、灰度发布和回滚机制。

总结来说，幻觉治理是“证据增强 + 工具验证 + 输出约束 + 拒答机制 + 评估闭环”。RAG 解决知识来源问题，微调解决行为习惯问题，工具调用解决可验证执行问题，三者要组合使用。

<a id="ant-agent-20260520-q3"></a>
##### 问题2中说到了模型的微调，面试官问对模型微调那些算法了解吗？

**回答：**

大模型微调可以按目标分成三类：监督微调、参数高效微调和偏好对齐或强化学习。

监督微调 SFT 是最常见的方式，用高质量 instruction-response 数据继续训练模型，让模型学习特定领域的回答风格、任务格式和业务流程。SFT 适合让模型按要求做事，例如客服话术、结构化抽取、代码解释、领域问答格式。但它不适合频繁注入实时知识，因为参数更新成本高，也容易遗忘或过拟合。

全参数微调会更新模型全部参数，效果上限高，但显存和数据要求高，训练风险也更大，容易破坏原模型通用能力。企业落地更常用 PEFT 参数高效微调，例如 LoRA、QLoRA、Adapter、Prefix Tuning、P-Tuning。LoRA 的核心是冻结原模型权重，只在部分线性层旁加入低秩矩阵学习增量参数；QLoRA 在量化基础上训练 LoRA，显存更省，适合资源有限场景。

偏好优化用于让模型更符合人类偏好或业务标准。RLHF 通常包括奖励模型训练和 PPO 等强化学习优化；DPO 不显式训练奖励模型，而是直接用偏好对数据优化模型，使 chosen response 概率高于 rejected response。还有 IPO、KTO、ORPO、GRPO 等变体，目标是改善回答质量、拒答、安全性、推理风格或任务完成度。

在 Agent 或大模型应用场景中，微调还可以服务于工具调用和工作流。例如用 SFT 让模型学习什么时候调用工具、如何生成合法参数、如何在工具失败后重试；用偏好数据让模型更倾向于引用证据、避免编造、遵守业务边界。但工具调用是否正确不能只靠微调，还要有 schema 校验、权限控制、执行反馈和 trace 评估。

选型时可以这样回答：如果是领域表达和格式问题，优先 SFT 或 LoRA；如果是知识更新问题，优先 RAG，不优先微调；如果是回答偏好、安全边界、拒答问题，用 DPO/RLHF 类方法；如果资源受限，用 QLoRA；如果担心通用能力损伤，就控制学习率、训练轮数，并做回归评测。

<a id="ant-agent-20260520-q4"></a>
##### 项目的skills是怎么实现的？你是怎么写skills的？

**回答：**

Skill 不是单纯一段 prompt，而是一个可复用的能力单元，通常包含触发条件、任务说明、输入输出约定、可调用工具、执行脚本、示例和校验方式。它的目标是把一次成功完成任务的方法沉淀下来，让 Agent 后续遇到同类任务时稳定复用。

实现上可以分为五层。第一层是元信息和触发规则，包括 skill 名称、描述、适用场景、什么时候应该使用、什么时候不应该使用。这个部分决定 Agent 能否在正确任务上选中它。

第二层是操作规程，即进入这个 skill 后要怎么做：先读哪些上下文，如何规范输入，如何调用工具，如何处理中间产物，如何保存结果，如何验证输出。好的 skill 不是只写“认真完成”，而是把稳定流程写清楚。

第三层是工具和脚本。对于重复性强、容易出错的步骤，应写成脚本，例如抓取、解析、格式化、校验、导出。Agent 负责调度和判断，脚本负责确定性执行，这比每次让模型凭自然语言临时操作更稳定。

第四层是输出格式和质量标准。例如面经整理类 skill 要求问题原文不改、只补回答、输出 Markdown、保留面试轮次、不保存爬虫日志；代码类 skill 要求运行测试、检查 diff、不要改无关文件。

第五层是错误处理和边界说明。例如网页不可访问、字段缺失、解析不完整、登录受限、模型事实不确定时，要明确说明边界，而不是编造。

写 skill 时可以先从一个成功案例中抽象流程：哪些步骤每次都一样，哪些输入是可变的，哪些错误最常见，哪些输出必须校验。然后把稳定步骤写进说明文件，把机械处理写进脚本，把示例放进模板，最后用真实任务回归验证。

<a id="ant-agent-20260520-q5"></a>
##### Spring AI框架的skills调用更像Claude Code还是其他Harness agent？

**回答：**

如果这里的 skills 指的是模型可调用的工具或函数，那么 Spring AI 的调用机制更像通用 tool calling / function calling harness，而不是完整的 Claude Code 形态。

Spring AI 的核心抽象是把 Java 方法、函数或自定义回调包装成 tool。应用在请求里把工具定义提供给模型，工具定义通常包含名称、描述和参数 schema。模型不会直接拿到 Java 方法或系统 API 的执行权限，它只是返回“我要调用哪个工具、参数是什么”。真正的工具解析、执行、结果回传，是由应用侧的 ToolCallback、ToolCallbackResolver、ToolCallingManager 等组件完成的。

所以 Spring AI 更像后端应用里的工具调用框架：它提供工具注册、工具解析、参数绑定、执行管理、观测和与 Spring Bean 体系集成的能力，适合把数据库查询、业务服务、外部 API、工作流操作封装成模型可调用工具。

Claude Code 则更像面向代码仓库的端到端 coding agent。它运行在终端环境中，围绕代码理解、文件检索、编辑、执行命令、测试、提交等工作流设计，有更强的上下文管理、工具选择、权限确认和工程任务闭环能力。它不是只提供一个 tool calling API，而是把工具调用放进完整的软件开发循环里。

因此可以总结：Spring AI 默认更像“业务应用内的工具调用 harness”；Claude Code 更像“带文件系统和命令行能力的 coding agent”。如果在 Spring AI 上进一步接入 MCP、动态工具发现、规划器、执行器、记忆和评估模块，也可以搭出类似 Claude Code 的 agentic harness，但这需要应用层自己补齐编排能力。

<a id="ant-agent-20260520-q6"></a>
##### 你说看过Claude源码，Claude Code学习经验总结？

**回答：**

这个问题回答时要谨慎。Claude Code 本体并不是完整开源、可随便阅读核心实现的项目，所以更准确的说法应是：学习过 Claude Code 的公开文档、CLI 行为、工具设计和可观察的 agent 工作流，并结合类似 coding agent 的开源实现做过复盘。如果直接说“看过 Claude Code 源码”，面试官追问细节时容易变成风险点。

学习总结可以分成几条。第一，Claude Code 的核心不是“会写代码”，而是把代码任务拆成一个 agent loop：理解任务、读取仓库上下文、定位相关文件、制定修改方案、编辑文件、运行命令或测试、根据结果修正、最后总结变更。模型能力只是其中一部分，真正重要的是上下文获取和工具闭环。

第二，它非常重视代码仓库的实时上下文。代码任务和普通知识问答不同，答案必须和当前仓库文件一致。所以它会使用文件读取、目录遍历、grep 检索、编辑工具、bash 命令等方式直接观察当前工作区，而不是只依赖模型记忆。

第三，工具权限和可控性很关键。一个 coding agent 能改文件、跑命令，就必须有权限边界、确认机制、可审计日志和失败恢复。否则模型一次误操作就可能破坏仓库。好的 agent harness 会把高风险工具和低风险工具区分开。

第四，上下文压缩和任务记忆很重要。真实代码库很大，不可能把所有文件都塞进上下文。Agent 需要先用检索定位，再选择性读文件，还要把已经确认的事实保留下来，避免反复搜索和遗忘。

第五，Claude Code 的工程价值在于“从自然语言任务到可验证 diff”。也就是说，它最后不是给一段建议，而是完成可运行、可检查、可回滚的代码变更。这个思路也可以迁移到业务 Agent：不要只输出文本，而要形成可执行动作、可验证结果和 trace。

<a id="ant-agent-20260520-q7"></a>
##### Claude Code是给模型用grep检索，没有用RAG检索，这俩有什么区别？为什么没用rag？

**回答：**

grep 检索和 RAG 检索的核心区别在于：grep 是对当前文件系统做精确字符串或正则匹配，RAG 通常是把文档切块、向量化、建索引，再通过语义相似度召回相关 chunk。

grep 的优点是确定、实时、可解释。代码仓库里很多问题依赖精确符号，例如函数名、类名、配置项、错误日志、接口路径、变量名。grep 可以直接告诉你某个符号在哪个文件哪一行出现，而且检索的是当前工作区最新内容，不需要提前建索引，也不会因为 embedding 过期导致召回旧内容。

RAG 的优点是语义召回。用户不知道准确关键词时，RAG 可以通过语义相似度找到相关内容，适合文档问答、知识库问答、概念相近但措辞不同的场景。但在代码任务中，RAG 有几个问题：切块可能破坏函数或类的完整上下文；embedding 对精确符号、短变量名、路径名不一定敏感；索引可能落后于文件最新状态；top-k 召回可能漏掉关键定义；还会增加索引维护和延迟成本。

所以 Claude Code 这类 coding agent 更偏向使用 grep、文件读取、目录遍历等实时、确定、可定位的检索方式，是因为代码修改需要精确 grounding。它不是说 RAG 没用，而是代码仓库场景里第一优先级通常是找准当前代码里的真实引用和定义。

如果仓库非常大，或者要做跨项目语义理解，可以把 grep 和语义检索结合：先用语义检索找可能相关模块，再用 grep 精确定位符号；或者先 grep 找入口，再读文件构建上下文。比较好的 coding agent 不会迷信某一种检索，而是根据任务选择确定性检索、语义检索和人工结构信息。

<a id="ant-agent-20260520-q8"></a>
##### 讲讲RAG的混合召回

**回答：**

RAG 的混合召回是指同时使用多种召回方式，把它们的结果融合后再交给 reranker 或生成模型。目标是提升召回率和鲁棒性，避免单一向量检索或单一关键词检索的短板。

最常见的是稀疏召回加稠密召回。稀疏召回通常是 BM25、倒排索引或关键词匹配，擅长精确词、专有名词、编号、错误码、函数名、政策条款。稠密召回通常是 Embedding 向量检索，擅长语义相近但表述不同的问题，例如“怎么防止模型乱编”和“如何降低幻觉”。

标准流程是：用户 query 先做清洗和改写，例如补全领域词、生成多个子 query、抽取关键词；然后并行走 BM25 和向量检索，各自取 top-k；再做结果融合，常见方法有加权分数融合、归一化后求和、RRF；融合后去重，并按 metadata 过滤，例如权限、时间、文档类型、业务线；最后用 cross-encoder reranker 或 LLM reranker 重排，选出最适合放进 prompt 的证据。

混合召回还可以扩展到结构化数据库召回、知识图谱召回、FAQ 精确匹配、标题/摘要字段召回、历史对话召回、工具检索。企业场景中经常需要权限过滤和时效过滤，否则召回对了但用户无权访问，也不能放进上下文。

评价混合召回时，不只看最终回答好不好，还要看检索侧指标，例如 recall@k、MRR、nDCG、命中证据覆盖率、rerank 后正确证据排名、无答案问题的拒答率。常见坑包括 chunk 切得太碎、BM25 和向量分数直接相加、top-k 太大引入噪声、没有 rerank、没有权限过滤、没有评估集。

<a id="ant-agent-20260520-q9"></a>
##### url解析过程

**回答：**

URL 解析可以分成语法解析和网络访问两个阶段。

语法解析阶段，浏览器或客户端会把 URL 拆成几个部分：协议 scheme，例如 http 或 https；主机 host，例如域名或 IP；端口 port，如果没有显式端口，HTTP 默认 80，HTTPS 默认 443；路径 path；查询参数 query；片段 fragment。还会处理大小写、百分号编码、特殊字符、相对路径转绝对路径、国际化域名转换等。

例如 `https://example.com:443/a/b?x=1#top` 中，`https` 是协议，`example.com` 是主机，`443` 是端口，`/a/b` 是路径，`x=1` 是查询参数，`top` 是浏览器本地使用的片段，不会作为 HTTP 请求的一部分发送给服务器。

网络访问阶段，如果 host 是域名，需要先做 DNS 解析得到目标 IP。然后根据协议建立连接：HTTP/1.1 或 HTTP/2 over TLS 通常先 TCP 三次握手，再 TLS 握手；如果是 HTTP/3，则走 QUIC/UDP。连接建立后，客户端构造 HTTP 请求，包括 method、path、headers、body 等。服务器处理请求后返回状态码、响应头和响应体。浏览器再根据响应类型进行解析渲染，例如 HTML、CSS、JS、图片等，并可能继续发起子资源请求。

完整链路还包括缓存检查、HSTS 升级、代理设置、Cookie 携带、重定向处理、CORS、安全策略和连接复用。面试中回答到“URL 拆解 -> DNS -> 建连 -> TLS -> HTTP 请求响应 -> 浏览器渲染或客户端处理”基本就比较完整。

##### tcp三次握手、四次挥手

**回答：**

TCP 三次握手的目标是建立可靠连接，并让双方确认彼此的发送和接收能力，同时同步初始序列号。

第一次握手，客户端发送 SYN 报文，带上自己的初始序列号 `seq=x`，进入 SYN_SENT 状态。第二次握手，服务端收到后返回 SYN+ACK，带上自己的初始序列号 `seq=y`，并确认客户端序列号 `ack=x+1`，进入 SYN_RCVD 状态。第三次握手，客户端再发 ACK，确认服务端序列号 `ack=y+1`，双方进入 ESTABLISHED 状态。

为什么不是两次？因为两次只能让客户端确认服务端收到了自己的 SYN，并不能让服务端确认客户端收到了自己的 SYN+ACK。第三次 ACK 能证明双方收发能力都正常，也能避免历史延迟 SYN 导致服务端误建立连接。

TCP 四次挥手的目标是关闭双向字节流。因为 TCP 是全双工，客户端到服务端和服务端到客户端两个方向要分别关闭。

假设客户端主动关闭。第一次，客户端发送 FIN，表示自己不再发送数据，进入 FIN_WAIT_1。第二次，服务端回复 ACK，表示收到了关闭请求，客户端进入 FIN_WAIT_2；此时服务端可能还有数据要发。第三次，服务端数据发送完后发送 FIN，进入 LAST_ACK。第四次，客户端回复 ACK，进入 TIME_WAIT，服务端收到 ACK 后关闭，客户端等待 2MSL 后关闭。

TIME_WAIT 的作用主要有两个：确保最后一个 ACK 如果丢失，客户端还能重发；让网络中旧连接的残留报文自然过期，避免影响后续相同四元组的新连接。常见追问包括 SYN flood、半连接队列、TIME_WAIT 过多、CLOSE_WAIT 过多。CLOSE_WAIT 多通常说明应用程序没有正确 close socket；TIME_WAIT 多一般出现在主动关闭方，需要结合连接复用、keep-alive、端口范围和内核参数分析。

<a id="ant-agent-20260520-q11"></a>
##### mysql的数据结构，讲讲B+树

**回答：**

MySQL 要看具体存储引擎。以最常用的 InnoDB 为例，索引主要使用 B+ 树。B+ 树是一种多路平衡搜索树，特点是非叶子节点只存索引键和子节点指针，叶子节点存真正的数据或主键引用，叶子节点之间通过链表相连，适合磁盘页存储和范围查询。

InnoDB 的主键索引是聚簇索引，叶子节点存的是整行数据。因此根据主键查询非常快，找到叶子节点就拿到行。二级索引的叶子节点存的是二级索引键和对应主键值，如果查询字段不在二级索引里，就需要先通过二级索引找到主键，再回到主键索引查整行，这就是回表。

B+ 树适合数据库索引，原因有几个。第一，它是多路树，树高低，磁盘 IO 次数少。第二，非叶子节点不存整行数据，一个页能放更多 key，提高扇出。第三，所有数据在叶子节点，查询路径长度稳定。第四，叶子节点有序链表，范围查询和排序扫描效率高。

B 树和 B+ 树的区别是，B 树的非叶子节点也可以存数据，而 B+ 树数据只在叶子节点。数据库更偏向 B+ 树，因为它更适合范围扫描和页缓存。Hash 索引等值查询可能很快，但不支持范围查询和排序，也不适合作为通用索引结构。

索引设计还要注意：高选择性字段适合建索引；联合索引遵守最左前缀；避免在索引列上做函数或隐式类型转换；覆盖索引可以减少回表；范围查询后的列通常不能继续充分利用索引；索引不是越多越好，因为会增加写入维护成本和空间占用。

<a id="ant-agent-20260520-q12"></a>
##### Spring的控制反转、AOP

**回答：**

Spring 的控制反转 IoC 指的是对象创建和依赖管理的控制权从业务代码转移到 Spring 容器。传统写法是类自己 new 依赖对象，而 IoC 下，类只声明自己需要什么依赖，由容器负责创建 Bean、解析依赖、注入依赖、管理生命周期。依赖注入 DI 是 IoC 最常见的实现方式。

IoC 的价值是降低耦合、提升可测试性和可配置性。例如 Service 不需要关心 Repository 怎么创建，只依赖接口；测试时可以替换 mock 实现；不同环境可以注入不同配置。

Spring Bean 的大致生命周期包括：读取 BeanDefinition，实例化对象，属性注入，执行 aware 接口回调，执行 BeanPostProcessor 前置处理，初始化方法，执行后置处理，最后进入可使用状态；容器关闭时执行销毁回调。面试不一定要背所有细节，但要理解容器在对象创建和增强中起核心作用。

AOP 面向切面编程用于把横切逻辑从业务逻辑中抽离，例如事务、日志、权限、监控、缓存、限流。核心概念包括 Join Point、Pointcut、Advice、Aspect、Weaving。

Spring AOP 主要基于代理实现。如果目标类实现接口，默认可以用 JDK 动态代理；如果没有接口，通常用 CGLIB 生成子类代理。调用代理对象的方法时，代理会在目标方法前后执行切面逻辑。例如 `@Transactional` 本质上就是通过代理在方法执行前开启事务，成功后提交，异常时回滚。

常见坑是自调用失效：同一个类内部 `this.method()` 调用不会经过 Spring 代理，因此事务或切面可能不生效。另一个坑是方法可见性和 final 方法，CGLIB 对 final 方法无法覆盖。可以总结为：IoC 解决对象管理和依赖解耦，AOP 解决横切关注点复用，两者共同构成 Spring 的核心。

##### 你平时用什么做ai coding？

**回答：**

可以按任务类型选择工具，而不是只依赖一个工具。

如果是仓库级代码修改、重构、测试修复，更适合用 Claude Code、Codex CLI 这类终端 coding agent，因为它们可以直接读取项目文件、定位引用、生成 diff、运行测试，并围绕当前仓库做闭环。对工程任务来说，能否验证修改比单纯生成代码更重要。

如果是日常 IDE 内补全、局部函数实现、解释报错、快速生成样板代码，可以用 Cursor、Trae 或 IDE 插件类工具，因为它们和编辑器交互顺滑，适合小步快改。

如果是方案设计、接口设计、复杂 bug 分析，可以先让模型梳理假设和排查路径，再用日志、测试、断点或命令验证。AI Coding 不能替代工程判断，尤其涉及并发、事务、安全、线上数据、权限操作时，必须以测试和可观测证据为准。

使用原则可以总结为：不直接相信生成结果，必须 review diff；关键逻辑必须跑测试或构造 case；不让 AI 在没有上下文的情况下大改架构；复杂任务先让它读代码并给计划，再分步骤修改；沉淀常用 prompt、脚本和项目规范，让 AI 更符合团队工程习惯。AI 会降低样板代码和信息检索成本，但真正有价值的仍是问题定义、系统设计、边界判断、质量验证和线上负责能力。
