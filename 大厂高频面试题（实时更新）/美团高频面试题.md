###### 1. 已经构建推理类Reasoning训练数据，为什么还需要接入RL训练？仅依靠SFT能否满足业务需求？

**回答：**

SFT 能教会模型“模仿高质量推理轨迹”，但它本质仍是最大似然学习：给定 prompt 和标准答案，让模型提高参考回答中每个 token 的概率。它解决的是“像不像标注答案”的问题，不天然解决“在多条可能推理路径中哪条更有效、最终结果是否更优、遇到新题能否探索”的问题。

Reasoning 类任务的难点在于：正确性往往依赖长链路、多步搜索、自我校验和中间纠错。SFT 数据即使质量高，也只是覆盖了有限路径，模型容易学到表面格式，比如先写“我们一步步分析”，但真正遇到分布外问题时未必会探索正确路径。RL 的价值在于把优化目标从“模仿轨迹”进一步推进到“最大化结果或过程 reward”。例如数学、代码、字段抽取、结构化推理任务可以设计可验证 reward，让模型通过采样、比较、更新，学会更有收益的推理策略。

仅靠 SFT 在简单、分布稳定、答案形式固定的业务里可以满足初版需求，例如格式化抽取、标准客服话术、模板化摘要。但如果业务要求复杂推理、强泛化、低错误率，或者希望模型学会自检、反思、工具使用、拒答和多轮纠错，SFT 通常不够。更成熟的链路是：先用 SFT 建立基本指令遵循和输出格式，再用 DPO/RLHF/GRPO/RLVR 等后训练方法优化偏好、正确性和推理策略，最后用评测和线上反馈闭环。

<a id="meituan-llm-algo-20260706-q4"></a>
###### 2. SFT训练达到怎样的效果指标后，才适合接入RL优化阶段？

**回答：**

判断能否进入 RL，不是看 SFT loss 降到多低，而是看模型是否已经具备稳定的基本能力。RL 会放大模型已有行为，如果 SFT 阶段连格式、任务理解和基础正确率都不稳定，直接接 RL 很容易把噪声也放大，出现 reward hacking 或训练震荡。

可以从五类指标判断。第一是基础任务成功率：在固定验证集上，答案正确率、字段抽取 F1、代码通过率、数学结果准确率等要达到可用底线。第二是格式稳定性：结构化输出 JSON 可解析率、字段完整率、schema 合法率要足够高，否则 reward 无法可靠计算。第三是分布泛化：不能只在训练同源样本上好，要在保留集、困难集、长尾集上保持稳定。第四是安全和拒答能力：不能为了答题强行编造，要有基本边界。第五是采样多样性：同一 prompt 下多次采样要能产生有差异的候选，否则 GRPO/PPO 这类策略优化很难获得有效学习信号。

工程上常见门槛可以设成：格式合法率接近线上要求，核心任务指标明显高于基座模型，badcase 主要集中在推理策略或偏好排序，而不是基础理解错误；reward 函数可以稳定区分好坏样本；训练样本和评测集已经去重并分层。达到这些条件后，RL 才像是在“精修策略”，而不是拿强化学习给脏数据和弱模型兜底。

<a id="meituan-llm-algo-20260706-q5"></a>
###### 3. 在模型结构、训练策略层面，是否做过针对性优化改进？

**回答：**

这个问题要分清“结构改动”和“训练策略改动”。面试里不要为了显得高级就说自己改了 Transformer 架构，除非你真的做过消融。更可信的回答是：基座结构通常保持稳定，主要在数据、训练目标、参数高效微调、上下文组织和后训练策略上做针对性优化。

模型结构层面，可以考虑 LoRA/QLoRA 的 target module 选择，例如只调 attention 的 q/v，还是同时调 FFN；对于长上下文任务，可以关注 RoPE scaling、位置编码外推、滑动窗口注意力、KV Cache 管理；对于领域结构化任务，可以加入工具调用头、特殊标记、格式约束或分类/打分 head。但结构改动必须有收益证据，否则容易引入训练不稳定和部署成本。

训练策略层面更常见也更有效。比如数据上做难度分层、去重、污染检查、负样本构造、hard case 采样；SFT 上做样本权重、packing、loss mask，只对 assistant answer 或关键字段计算 loss；偏好优化上用 DPO/IPO/ORPO 或 RLVR/GRPO；稳定性上用 warmup、梯度裁剪、学习率分组、early stopping、混合精度和 checkpoint 回滚；评测上用固定 golden set + badcase set + 线上日志回流。

本质上，结构是能力上限的一部分，训练策略决定能力如何被激活。业务算法岗更看重你能不能用可验证实验证明：改动解决了什么问题、指标提升多少、代价是什么、有没有副作用。

<a id="meituan-llm-algo-20260706-q6"></a>
###### 4. Attention机制中为什么需要去除DK相关计算维度，核心作用是什么？

**回答：**

这里的“去除 DK 相关计算维度”更准确地说，应该是 scaled dot-product attention 中用 $\sqrt{d_k}$ 对 $QK^T$ 做缩放，而不是把 $d_k$ 维度删掉。公式是：

$$
\mathrm{Attention}(Q,K,V)=\mathrm{softmax}\left(\frac{QK^T}{\sqrt{d_k}}\right)V
$$

为什么要除以 $\sqrt{d_k}$？如果 query 和 key 的每个维度近似独立、均值为 0、方差为 1，那么点积 $q \cdot k$ 的方差会随着 $d_k$ 增大而增大。维度越高，$QK^T$ 的数值越容易变大，softmax 会进入饱和区：最大项接近 1，其余项接近 0，梯度变小，训练不稳定。

缩放的核心作用是保持 attention logits 的数值尺度稳定，让 softmax 不至于过早变得极端。它不是为了降低计算复杂度，因为 $QK^T$ 的矩阵乘法仍然存在；它是为了改善数值稳定性和梯度传播。面试时要特别纠正一个误区：$d_k$ 不是被去掉了，而是被作为归一化尺度使用。真正降低 attention 复杂度的技术是 sparse attention、linear attention、sliding window attention、MQA/GQA、KV cache 优化等，而不是这个缩放项。

<a id="meituan-llm-algo-20260706-q7"></a>
###### 5. 详细区分Workflow和Agent的核心差异与适用场景

**回答：**

Workflow 是预先定义好的流程编排，Agent 是具备一定自主决策能力的任务执行系统。两者都可能调用大模型和工具，但控制权不同：Workflow 的路径主要由工程师设计，Agent 的路径部分由模型根据状态动态决定。

Workflow 的典型特征是流程固定、状态可控、失败路径明确。例如“文档上传 -> OCR -> 分块 -> 向量化 -> 检索 -> 摘要 -> 入库”，每一步做什么基本确定。它适合合规要求高、稳定性要求高、任务边界清楚的业务，比如批处理、报表生成、标准化审核、字段抽取流水线。

Agent 的典型特征是目标驱动、路径开放、工具选择动态。例如用户说“帮我分析这个商家最近订单下降的原因”，系统可能需要查数据、对比指标、生成假设、调用 BI 工具、解释异常、再生成行动建议。Agent 适合开放式、多步骤、需要探索和环境反馈的任务，但风险也更高：可能循环、误用工具、上下文漂移、成本失控。

工程上不要迷信 Agent。可控任务优先 Workflow，开放任务才引入 Agent；成熟系统往往是二者结合：外层用 Workflow 控制预算、权限、审计和停止条件，内部某些节点用 Agent 做规划、检索、工具调用或异常修复。真正的判断标准不是“听起来智能”，而是任务路径是否固定、错误成本多高、是否需要动态决策。

<a id="meituan-llm-algo-20260706-q8"></a>
###### 6. 对比Memory机制与RAG检索增强的区别、优势和落地场景差异

**回答：**

Memory 和 RAG 都是在解决“模型上下文之外的信息如何进入推理”的问题，但信息来源和使用方式不同。

RAG 面向外部知识。它把文档、知识库、数据库、网页或业务资料切分、索引、召回、重排，再把相关证据放进上下文。优势是知识可更新、可引用、可审计，适合政策问答、企业知识库、商品知识、操作手册、字段定义等动态知识场景。它的核心指标是召回率、证据命中率、引用一致性、答案准确率和拒答能力。

Memory 面向用户、任务或系统的持续状态。短期记忆保存当前会话中的目标、约束、已完成步骤、工具返回和错误信息；长期记忆保存用户偏好、历史行为、常用模板、项目上下文和个性化信息。优势是能让系统跨轮次、跨任务保持连续性，适合 Agent、多轮协作、AI Coding、个人助手、商家运营助手等场景。

落地上，RAG 更像“查资料”，Memory 更像“记住我和任务”。两者都不能无脑塞 prompt。RAG 要做 chunk、embedding/BM25、rerank、引用和过期；Memory 要做写入策略、隐私权限、冲突消解、遗忘机制和重要性评分。一个成熟 Agent 通常同时需要 RAG 和 Memory：RAG 负责事实证据，Memory 负责任务连续性和个性化。

<a id="meituan-llm-algo-20260706-q9"></a>
###### 7. LLM输入维度为b、s的情况下，完整推理过程中Tensor维度的变化逻辑

**回答：**

假设输入 token ids 的形状是 `[b, s]`，其中 `b` 是 batch size，`s` 是序列长度。第一步经过 embedding 后变成 `[b, s, d_model]`，再加上位置编码或 RoPE 位置信息，仍然是 `[b, s, d_model]`。

进入每一层 Transformer 后，先通过线性层投影出 Q、K、V。若有 `h` 个注意力头，每个头维度是 `d_head = d_model / h`，则 Q/K/V 通常 reshape 为 `[b, h, s, d_head]`。计算 attention score 时，`Q @ K^T` 得到 `[b, h, s, s]`，表示每个 token 对所有历史 token 的注意力。经过 mask、缩放和 softmax 后仍是 `[b, h, s, s]`，再乘 V 得到 `[b, h, s, d_head]`。合并多头后回到 `[b, s, d_model]`。

随后经过输出投影、残差连接、LayerNorm，再进入 FFN。FFN 一般先升维到 `[b, s, d_ff]`，经过激活函数如 SwiGLU/GELU，再降回 `[b, s, d_model]`。堆叠 N 层后，最后通过 lm_head 投影到词表维度，得到 logits `[b, s, vocab_size]`。

自回归生成时，prefill 阶段会处理完整 `[b, s]` 并建立 KV cache；decode 阶段每次只输入新 token，形状近似 `[b, 1]`，新 Q 是 `[b, h, 1, d_head]`，K/V 会追加到 cache，attention score 变成 `[b, h, 1, past_len+1]`。所以推理优化的重点不只是矩阵形状，还包括 KV cache、batching、prefill/decode 分离和显存带宽。

<a id="meituan-llm-algo-20260706-q10"></a>
###### 8. 7B规模模型使用GRPO训练时的显存占用开销及优化方式

**回答：**

GRPO 的核心优势是相对 PPO 省掉独立 value/critic model，用同一 prompt 下的多条采样构成 group，通过组内 reward 标准化估计 advantage。因此它比 PPO 更省显存，但并不等于便宜。7B 模型做 GRPO 时，显存主要花在策略模型参数、梯度、优化器状态、reference model 或 KL 计算、rollout 生成的 KV cache、group samples 的 token 激活以及 reward 计算上。

粗略估算：7B 参数如果用 BF16，仅权重约 14GB；训练还需要梯度和优化器状态，Adam 全参训练会让显存显著上升，常规单卡很难承受。如果采用 LoRA/QLoRA，只训练低秩适配器，可以把可训练参数、梯度和优化器状态压下来。GRPO 还需要一次 prompt 采样多个回答，group size 越大，rollout token 和 KV cache 开销越高；如果回答很长，显存和吞吐压力会迅速增加。

优化方式包括：第一，用 LoRA/QLoRA、ZeRO/FSDP、gradient checkpointing、activation recomputation 降低训练显存；第二，用 vLLM 或高效推理引擎做 rollout，把生成和训练解耦；第三，控制 group size、最大生成长度和 prompt packing，避免无效长链路；第四，使用 flash attention、paged attention、GQA/MQA 和 KV cache 管理降低注意力开销；第五，reference model 可以共享权重、延迟计算或使用 KL 近似；第六，reward 尽量规则化和批处理，避免 reward model 成为瓶颈。

面试里要说清楚：GRPO 省的是 critic/value model 的训练成本，但 RL 训练的真实瓶颈往往在 rollout 生成、长序列、group 采样和 reward 评估。

<a id="meituan-llm-algo-20260706-q11"></a>
###### 9. 简述聚类算法的核心原理、主流实现方法，同时说明K-Means算法是否存在全局最优解

**回答：**

聚类是无监督学习，目标是在没有标签的情况下，把相似样本划到同一组，把差异大的样本分开。它本质是在定义“相似度”和“簇结构”：相似度可以是欧氏距离、余弦距离、密度可达性、概率分布相似性；簇结构可以是球状、层次、密度连通、图社区等。

主流方法可以分几类。第一类是划分式聚类，典型是 K-Means，通过最小化样本到所属簇中心的平方距离来迭代更新。第二类是层次聚类，自底向上或自顶向下构建树状结构，适合观察多粒度分组。第三类是密度聚类，如 DBSCAN/HDBSCAN，适合发现非球状簇和离群点。第四类是概率模型，如 GMM，用混合高斯和 EM 算法估计样本属于每个簇的概率。第五类是谱聚类或图聚类，把样本关系图转成图划分问题。

K-Means 的目标函数存在全局最优解，因为有限样本、有限 K 下所有划分的目标函数有最小值。但标准 Lloyd 迭代算法通常只能保证收敛到局部最优，不能保证找到全局最优。它对初始化、K 值、尺度、离群点和簇形状非常敏感。工程上常用 K-Means++ 初始化、多次重启、特征标准化、异常值处理、轮廓系数/肘部法选 K，以及在非球状数据上换用 DBSCAN、GMM 或谱聚类。

##### 三、手撕算法

<a id="meituan-llm-algo-20260706-q12"></a>
###### 基础算法题：两数之和

**回答：**

两数之和的本质是用空间换时间。暴力做法枚举两个数，时间复杂度 $O(n^2)$；更优做法是在遍历数组时维护哈希表，记录“值 -> 下标”。对每个元素 `x`，检查 `target - x` 是否已经出现，如果出现就返回两个下标；否则把当前值写入哈希表。

注意点有三个。第一，不能先把所有值放入哈希表后直接查，否则可能把同一个元素用两次。第二，如果有重复值，例如 `[3,3] target=6`，遍历时先查再写可以自然处理。第三，返回的是下标还是数值要看题目要求，LeetCode 原题返回下标。

```python
from typing import List


def twoSum(nums: List[int], target: int) -> List[int]:
    seen: dict[int, int] = {}
    for i, x in enumerate(nums):
        need = target - x
        if need in seen:
            return [seen[need], i]
        seen[x] = i
    return []
```

时间复杂度是 $O(n)$，空间复杂度是 $O(n)$。如果数组已排序，也可以用双指针做到 $O(n)$ 时间和 $O(1)$ 额外空间，但原题通常数组无序，所以哈希表是最直接的解法。

#### 二面（业务场景专项面，无手撕、无常规项目提问

##### 一、数据层相关问题

<a id="meituan-llm-algo-20260706-q13"></a>
###### 1. 实际业务中，如何量化、评估标注数据集的质量？

**回答：**

标注数据质量不能只看“标了多少条”，要看它能否稳定支撑模型学习和业务决策。可以从准确性、一致性、覆盖度、可学习性和业务价值五个层面量化。

准确性指标签是否符合真实标准。常见做法是抽样复审、专家仲裁、golden set 对照，计算字段级准确率、样本级准确率、错误类型分布。字段抽取任务尤其要分实体边界、字段类型、归一化值、缺失/多余字段来算。

一致性指不同标注员、不同批次、不同时间对同一标准是否一致。可以用一致率、Cohen's Kappa、Krippendorff's Alpha、冲突率等指标。大模型训练很怕“同一问题多套标准”，因为模型会学到混乱边界。

覆盖度指数据是否覆盖真实业务分布，包括高频场景、长尾场景、困难样本、异常样本、不同渠道、不同文本长度、不同格式噪声。可用标签分布、场景分布、难度分布、业务来源分布来衡量。

可学习性指数据是否能让模型形成稳定模式。重复样本、泄漏样本、模板化过强、答案不可验证、标注解释缺失，都会降低训练价值。业务价值则看这些数据是否对应高收益场景，例如高频工单、核心字段、关键决策链路。最终应该建立数据质量看板，而不是训练失败后才回头猜数据问题。

<a id="meituan-llm-algo-20260706-q14"></a>
###### 2. 定义标注数据的合格标准，从哪些维度判定标注有效？

**回答：**

标注数据合格标准要从“单条样本是否正确”和“整体数据集是否可训练”两层定义。单条样本合格不代表数据集合格，数据集规模大也不代表有效。

单条样本可以看六个维度。第一，输入有效：文本、图片、表格或业务记录本身不缺失、不乱码、不严重截断。第二，标签正确：标签符合标注规范和业务事实。第三，边界清晰：字段抽取要明确起止位置、实体类型、归一化规则。第四，解释充分：复杂推理或拒答样本最好有理由，方便训练和复审。第五，格式合法：JSON/schema/枚举值必须可解析。第六，可复现：给另一个标注员按同一规范应得到近似结果。

数据集层面还要看分布是否合理。正负样本比例、简单/困难样本比例、场景覆盖、长尾覆盖、噪声比例、重复率、训练集和测试集是否泄漏，都决定它是否适合训练。尤其是 SFT/RL 数据，不能只追求“答案看起来漂亮”，还要看 reward 是否可计算、评测是否可复现、线上分布是否匹配。

一个实用标准是：通过机器校验保证格式，通过人工复审保证语义，通过一致性指标保证规范，通过训练小实验验证可学习性。标注有效的最终证据不是数据表里有标签，而是模型能稳定从中学到业务需要的行为。

<a id="meituan-llm-algo-20260706-q15"></a>
###### 3. 多类型、多场景训练数据的混合策略与配比思路

**回答：**

多类型数据混合的核心不是简单拼接，而是控制模型学习到什么能力、以什么优先级学习、避免哪些能力互相干扰。常见数据类型包括通用指令、领域知识、字段抽取、复杂推理、拒答安全、工具调用、多轮对话、困难 badcase 等。

配比可以从三层设计。第一层按业务目标分配权重。若目标是字段抽取，核心 schema 样本和困难字段样本权重要高；若目标是推理能力，Reasoning、验证型任务和反例样本要提高；若目标是客服体验，则多轮对话、拒答和情绪安抚样本要占一定比例。

第二层按难度课程学习。训练早期放更多高质量、格式稳定、答案明确的样本，让模型先学会基本输出；中后期提高困难样本、长尾样本、对抗样本和真实 badcase 的比例。不要一开始就把脏难样本全部灌进去。

第三层按采样策略动态调整。可以用 temperature-based sampling 避免大类数据淹没小类数据，用 loss-aware 或 error-aware sampling 提高模型薄弱场景频率，用 replay 保留通用能力防止灾难性遗忘。配比最终要靠消融实验决定：每次只调整一类数据，看验证集、困难集和线上指标是否同步改善。数据混合的本质，是在模型能力、业务收益和副作用之间做权衡。

##### 二、SFT微调专项问题

<a id="meituan-llm-algo-20260706-q16"></a>
###### 1. SFT训练过程中，如何平衡简单任务与复杂任务的训练权重？

**回答：**

简单任务和复杂任务的矛盾在于：简单任务能稳定模型格式和基础能力，但占比过高会让模型变得模板化；复杂任务能提升推理和泛化，但太早或太多会造成训练不稳定。平衡策略应该是课程学习 + 动态采样 + 分层评测。

训练早期可以提高简单高质量样本比例，让模型先学会任务定义、输出格式、基本字段和安全边界。中期逐步增加复杂任务，例如多字段抽取、长文本推理、多约束生成、跨表推断、工具调用和拒答判断。后期重点加入真实 badcase、长尾场景和高难样本，修复线上薄弱点。

权重设计上，可以给不同样本打 difficulty score 和 business value score。高业务价值但困难的样本不一定一开始高权重，而是随训练阶段逐步提升。也可以对 loss 做样本级权重，对关键字段 token 做 loss mask 或加权，避免模型把大量无关解释学得很好、关键字段却不准。

评估时必须拆分简单集、困难集、长尾集和线上回放集。如果简单任务提升但复杂任务下降，说明模型可能过拟合模板；如果复杂任务提升但基础格式崩了，说明课程推进太快。真正的平衡不是平均混合，而是让模型先稳、再强、最后对业务高价值难点做定向优化。

<a id="meituan-llm-algo-20260706-q17"></a>
###### 2. 微调阶段防止模型过拟合的常用方案与实操技巧

**回答：**

大模型微调过拟合通常表现为训练 loss 继续下降，但验证集指标停滞或下降；输出变得模板化；对训练分布内问题很好，对真实线上问题泛化差；甚至记住训练样本中的格式噪声和错误标签。

数据层面，首先要去重和防泄漏，尤其是训练集与验证集、评测集之间的近重复。其次要保证数据多样性，覆盖不同表达、长度、场景和难度。再者要控制脏标签，低质量样本比小数据更危险。

训练层面，可以用较小学习率、warmup、weight decay、dropout、early stopping、梯度裁剪和更短训练轮次。LoRA 场景下要控制 rank、alpha、target modules 和 dropout；rank 太高、训练太久，都可能记忆训练集。QLoRA 还要注意量化误差和学习率稳定性。

评测层面，要建立 holdout set、hard set、线上回放集和人工抽检。不要只看训练 loss；要看任务指标、格式合法率、拒答准确率、幻觉率、长度分布和输出多样性。实操上最有效的一句话是：宁愿少训一点，也不要把模型训成“背答案机器”。过拟合不是训练技巧问题，本质是数据分布、模型容量和业务目标之间不匹配。

<a id="meituan-llm-algo-20260706-q18"></a>
###### 3. LLM SFT训练常用的Loss函数及适用场景

**回答：**

LLM SFT 最常用的是自回归语言模型的 token-level cross entropy。给定输入和目标输出，模型预测下一个 token，训练目标是最大化目标 token 的似然，等价于最小化负对数似然：

$$
\mathcal{L}_{CE}=-\sum_t \log p_\theta(y_t|x,y_{<t})
$$

在指令微调里，通常只对 assistant answer 部分计算 loss，对 user prompt、system prompt、工具返回等上下文做 mask。这样模型学习“如何回答”，而不是学习复述用户问题。多轮对话也要仔细处理 loss mask，否则模型可能学会生成用户消息。

结构化抽取任务可以继续用 cross entropy 训练 JSON 或 schema 输出，但要配合格式校验和字段级评估。如果某些字段更关键，可以做 token/field 加权 loss。分类任务可以接分类头用 cross entropy，也可以转成生成式标签。排序或偏好类任务更适合 DPO、pairwise ranking loss、contrastive loss，而不是普通 SFT。

对于 Thinking/Reasoning 数据，如果训练完整推理链，loss 会覆盖中间思考和最终答案；如果只希望模型输出短答案，可以对 reasoning 部分 mask 或蒸馏成简洁解释。选择 loss 的原则不是越复杂越好，而是训练目标要和线上输出、评测指标、可控性一致。

<a id="meituan-llm-algo-20260706-q19"></a>
###### 4. 模型开启/关闭Thinking推理机制，对微调训练效果的影响

**回答：**

Thinking 开关本质上是在控制模型是否显式生成中间推理过程。开启 Thinking 时，模型会分配更多 token 给问题分析、步骤拆解、自检和反思；关闭 Thinking 时，模型更倾向于直接给出结果或短解释。

对 SFT 的影响主要有三点。第一，训练目标不同。开启 Thinking 的样本如果包含高质量推理链，模型能学到中间策略，对数学、代码、复杂字段推断、长链路决策有帮助；但如果推理链质量差，模型会学习冗长、绕路甚至错误的思考模式。第二，成本和延迟不同。Thinking 会增加输出长度，训练和推理成本都上升。第三，评测口径不同。有些业务只需要最终结构化字段，显式思考反而可能泄露内部逻辑或影响格式稳定。

实操上要按任务分流。复杂推理、规则验证、疑难样本可以开启 Thinking 或训练 hidden reasoning；简单抽取、固定格式生成、低延迟线上接口更适合关闭 Thinking。训练数据也要区分：有 thinking 标签的样本、无 thinking 的直接答案样本、只对 final answer 计 loss 的样本不要混在一起乱训。

成熟方案不是永远开或永远关，而是让模型学会按任务选择推理预算：简单任务短答，复杂任务展开推理，高风险任务自检，最终对用户输出可控、简洁、可验证的结果。

<a id="meituan-llm-algo-20260706-q20"></a>
###### 5. SFT的终止判定条件，满足哪些标准可切入RL训练？

**回答：**

SFT 终止不能只看训练 loss。训练 loss 下降说明模型更像训练答案，但不代表业务效果继续提升。合理的终止条件应该同时看验证指标、格式稳定性、泛化能力、badcase 类型和进入 RL 的前置条件。

第一，验证集核心指标进入平台期。例如字段抽取 F1、准确率、代码 pass rate、数学正确率、拒答准确率不再明显提升。第二，格式合法率达到线上底线，例如 JSON 可解析率、字段完整率、schema 合法率稳定。第三，训练集和验证集差距没有继续扩大，说明没有明显过拟合。第四，badcase 主要变成“偏好排序、复杂推理、边界策略”问题，而不是基础指令理解和格式错误。

切入 RL 还需要额外条件：reward 能可靠计算，且能区分候选答案好坏；同一 prompt 多次采样有足够多样性；评测集能覆盖简单、困难、长尾和安全样本；KL 约束、长度约束、格式约束和停止条件设计清楚。否则 RL 会把模型推向 reward 漏洞，而不是业务目标。

一句话总结：SFT 结束的信号是“模型已经会做事，但还需要学会做得更好”；如果模型还不会稳定做事，先继续修数据和 SFT，不要急着上 RL。

<a id="meituan-llm-algo-20260706-q21"></a>
###### 6. 大模型训练为什么必须拆分SFT预微调+RL强化优化两个阶段，能否简化流程？

**回答：**

SFT 和 RL 优化的是不同问题。SFT 负责把模型拉到“会听指令、会按格式回答、具备基本领域能力”的区域；RL 负责在多个可行回答之间优化偏好、正确性、推理策略和长期收益。直接 RL 一个没经过 SFT 的模型，探索空间太大、输出不稳定、reward 难以计算，训练成本和风险都很高。

拆分的核心原因有三点。第一，稳定性。SFT 给模型一个可用初始策略，减少 RL 随机探索。第二，目标差异。SFT 学习专家示范，RL 学习 reward 最大化，二者互补。第三，工程可控。SFT 阶段可以先解决格式和基础能力，RL 阶段再解决“哪个答案更好、哪个推理路径更稳”。

能否简化？可以，但要看场景。简单业务可以只做 SFT 或 SFT + DPO，不一定上完整 RLHF/PPO/GRPO。偏好数据充足时，DPO/IPO/ORPO 这类方法能用监督式目标替代一部分 RL 流程。可验证任务如数学、代码、字段抽取，可以直接做 RLVR/GRPO，但通常仍需要冷启动数据或指令微调来保证可读性和格式稳定。

所以不是所有项目都必须重型 SFT + RL，但不能混淆两个阶段的作用。简化流程的前提是任务简单、reward 明确、风险可控；复杂推理和高价值业务场景仍然需要分阶段训练和严格评测。

##### 三、RL强化学习专项问题

<a id="meituan-llm-algo-20260706-q22"></a>
###### 1. 针对字段提取业务，需要设计哪些维度的Reward函数？

**回答：**

字段提取业务的 reward 要围绕“抽得准、抽得全、格式对、业务可用”来设计。不能只给最终字符串相等打分，因为真实业务里会有同义归一、边界差异、缺失字段、无效字段和格式错误。

第一类是字段准确性 reward。字段值完全匹配给高分，部分匹配按字符级 F1、token F1、实体边界 IoU 或归一化后匹配给分。对金额、时间、地址、类目等字段，要先做单位、格式、别名归一化。

第二类是字段完整性 reward。该抽的字段不能漏，多抽无关字段要扣分。可以拆成 precision、recall、F1，避免模型为了高准确率只抽最确定的字段，也避免为了高召回乱抽。

第三类是结构合法性 reward。JSON 可解析、schema 字段齐全、枚举值合法、类型正确、数组/嵌套结构符合规范。结构错了即使语义部分对，线上也不可用。

第四类是证据一致性 reward。抽取结果必须能在原文中找到证据，或能由原文合理推导。对于不可确定字段，应奖励空值/拒答，而不是鼓励编造。

第五类是业务权重 reward。核心字段如订单号、商家名、金额、时间、地址、风险标签可以更高权重。最终 reward 通常是多项加权：准确性 + 完整性 + 格式 + 证据 + 安全约束，并配合长度惩罚和无效输出惩罚，防止模型钻空子。

<a id="meituan-llm-algo-20260706-q23"></a>
###### 2. 项目中是否遇到过Reward Hacking问题，对应的识别与解决方案？

**回答：**

Reward Hacking 是模型找到了提高 reward 的捷径，但这个捷径不等于真实业务目标。字段提取里常见例子包括：为了提高召回把很多无关文本都塞进字段；为了格式 reward 输出漂亮但内容错；为了匹配标准答案过度模板化；为了长度惩罚输出过短导致漏字段；或者学会利用 reward 规则的边界漏洞。

识别方法要看“reward 涨了，真实指标是否也涨”。如果训练 reward 持续升高，但人工评审、线上回放集、字段级 F1、证据一致性没有提升甚至下降，就要怀疑 reward hacking。还可以按错误类型抽样：多抽、漏抽、编造、格式合法但语义错、无法追溯证据。另一个信号是输出分布异常，例如长度突然变短、某些默认值频繁出现、拒答率异常、模板句暴增。

解决方案第一是改 reward，把单一指标改成多目标约束。例如同时看 precision、recall、证据一致性和格式合法。第二是加入 hard negative 和对抗样本，让模型不能靠简单模式得分。第三是设置 reward clipping、KL penalty、长度约束和拒答约束，防止策略漂移。第四是 reward model 或规则本身要做验证集回归测试，不能只在训练样本上有效。第五是保留人工审查和线上 A/B，尤其对高风险字段。

本质上，reward 不是业务目标本身，只是业务目标的近似。工程上要持续检查这个近似有没有被模型利用。

<a id="meituan-llm-algo-20260706-q24"></a>
###### 3. RL训练数据集的构建思路，能否直接复用SFT阶段的训练数据？

**回答：**

RL 数据可以复用一部分 SFT prompt，但不能简单复用 SFT 的“输入-标准答案”格式。SFT 数据强调示范答案，RL 数据强调可采样、可比较、可打分。两者的数据对象不同。

构建 RL 数据集时，首先要选 prompt。prompt 应覆盖真实业务分布、困难样本、长尾样本、容易出错样本和高价值样本。其次要定义 reward 或偏好标准。字段抽取可以用规则和标准答案算 reward；开放问答可能需要人工偏好、reward model 或 LLM judge；推理任务最好有可验证最终答案。第三要准备评测集和隔离集，避免训练 reward 对评测泄漏。

SFT 数据可以复用的部分主要是 prompt、任务说明、部分高质量参考答案和冷启动样本。但 RL 阶段更需要同一 prompt 下的多候选答案、chosen/rejected 偏好、规则 reward、错误样本和边界样本。如果直接拿 SFT 标准答案做 RL，容易变成另一次监督学习，无法提供策略探索信号。

所以答案是：可以复用，但要重构。把 SFT 数据中的高质量 prompt 作为 RL 初始池，补充模型采样结果、reward 标注、困难样本和线上 badcase。SFT 教模型“怎么做”，RL 数据要让模型知道“做成什么样更值钱”。

<a id="meituan-llm-algo-20260706-q25"></a>
###### 4. 如何解释RL训练初期Reward数值先降后升的现象？

**回答：**

RL 初期 reward 先降后升是常见现象，不一定代表训练失败。原因通常是策略从 SFT 初始分布开始探索，新策略会尝试不同回答长度、推理路径和格式，短期内偏离原来稳定但保守的输出，导致 reward 下降；当模型逐渐学到 reward 偏好的行为后，reward 再回升。

具体可以从四个角度解释。第一，探索成本。RL 需要采样多种动作，早期会产生更多低质量输出。第二，KL 约束调整。如果 KL penalty 太弱，策略偏离 SFT 过快，reward 可能先掉；如果太强，模型学不动。第三，reward 归一化和 advantage 估计在早期不稳定，尤其 group size 小、样本方差大时，GRPO 的相对优势会有噪声。第四，任务难度课程变化。如果训练逐步加入困难样本，平均 reward 也可能阶段性下降。

判断是否健康，要同时看 reward、KL、entropy、格式合法率、长度、验证集指标和人工样本。如果 reward 先降后升，同时 KL 可控、entropy 没塌、验证集后续提升，这是正常探索。如果 reward 下降后不回升、输出格式崩坏、长度异常、KL 爆炸，就要降低学习率、加强 KL、缩短生成长度、调 reward 权重或回滚 checkpoint。

一句话：reward 曲线不能孤立解读，要和策略漂移、输出分布和验证集效果一起看。

###### 5. RL全流程需要重点监控哪些核心训练指标，如何根据指标调优？

**回答：**

RL 训练要监控三类指标：优化指标、行为指标和业务指标。只看平均 reward 很危险，因为 reward 可能被 hacking，也可能掩盖不同场景的退化。

优化指标包括 reward mean/std、advantage 分布、policy loss、KL divergence、entropy、clip ratio、梯度范数、学习率、生成长度、拒答率。KL 太高说明策略偏离 reference 太快，要加大 KL penalty、降低学习率或缩短更新步数；KL 太低且 reward 不涨，说明模型学不动，可以降低 KL、提高采样多样性或增强 reward 区分度。Entropy 太低说明策略塌缩，可能需要提高探索、增加样本多样性；entropy 太高则说明输出不稳定。

行为指标包括格式合法率、字段准确率、证据命中率、多抽/漏抽率、幻觉率、长度分布、模板化比例和安全违规率。这些能发现 reward 没覆盖的问题。比如 reward 涨但格式合法率下降，说明 reward 权重不平衡；长度持续变长，可能是模型用冗长推理骗分；拒答率异常升高，可能是模型学会保守逃避。

业务指标包括线上 A/B、人工复核通过率、工单节省、人效提升、转化率、延迟和成本。调优时要按问题定位：数据问题补样本，reward 问题改规则，策略问题调 KL/学习率/group size，工程问题优化 rollout 和 batch。成熟 RL 不是一次跑完，而是“训练指标 - badcase - reward 修正 - 数据回流”的循环。

###### 6. RL训练中如何兼顾难易样本，提升模型对高难度业务任务的探索能力？

**回答：**

兼顾难易样本，本质是控制训练课程和探索分布。简单样本保证基础能力和稳定 reward，高难样本提升上限和业务价值。只训练简单样本，模型会保守；只训练难样本，reward 稀疏、方差大、训练容易崩。

第一，建立难度分层。可以用人工标签、历史模型错误率、字段数量、文本长度、歧义程度、规则复杂度、reward 方差、是否长尾场景来定义 difficulty score。训练集按 easy、medium、hard、adversarial 分桶。

第二，做课程学习。早期 easy/medium 比例高，让模型学会格式和基础策略；中期提高 hard 样本比例；后期加入线上 badcase 和对抗样本。也可以用自适应采样：模型在哪类样本错得多，就提高该类样本采样权重。

第三，提升探索质量。GRPO 场景可以对高难 prompt 增加 group size 或采样温度，让模型产生更多候选路径；对可验证任务用 outcome reward + format reward；对长链路任务加入过程检查或中间 reward，降低稀疏奖励难度。

第四，防止困难样本破坏基础能力。保留一部分简单样本 replay，使用 KL 约束和验证集分层监控。如果 hard set 提升但 easy set 退化，说明训练过度偏向困难分布。

最终目标不是让所有样本平均出现，而是让采样概率与业务价值、模型薄弱点和训练阶段匹配。高难任务的探索能力来自更好的课程设计，而不是简单把难题堆进训练集。
