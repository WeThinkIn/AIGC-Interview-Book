### 1. 月之暗面 Agent 开发岗一面面经

#### 面试问题汇总

<a id="moonshot-agent-dev-20260721-q1"></a>
##### 1. 简单讲讲你的Agent项目？

**回答：**

我会先用一句话定义项目：这是一个面向长链任务的 Agent 系统，以 ReAct 为基本决策循环，结合工具调用、RAG、短期工作记忆和长期记忆，让模型能够围绕目标持续执行，而不是只生成一次答案。

核心执行链路可以抽象为：

$$
s_t \xrightarrow{\pi_\theta} a_t
\xrightarrow{\text{tool/env}} o_{t+1}
\xrightarrow{\text{state update}} s_{t+1}
$$

其中 $s_t$ 是当前任务状态，$a_t$ 是模型选择的动作或工具调用，$o_{t+1}$ 是外部环境返回的 Observation。运行时不是简单写一个无限 `while`，而是受状态机约束：每一步都要做工具权限检查、参数校验、超时与重试控制，并在完成条件、失败条件、人工审批点或最大步数到达时终止。

项目中我把能力拆成四层。第一层是规划与路由，决定当前任务应直接回答、检索知识、读取记忆还是调用工具。第二层是执行层，对工具 schema 做约束解码和参数验证。第三层是状态与记忆层，保存当前目标、已完成步骤、中间产物以及需要跨会话复用的信息。第四层是可观测与评测层，记录每一步输入、动作、工具返回、耗时和失败原因，并用任务完成率、工具选择正确率、参数正确率、无效步骤数和失败恢复率评估。

需要纠正一个常见说法：Observation 来自真实工具，只能提供外部证据，不能自动“保证长线任务可靠”。工具可能超时、返回脏数据，模型也可能误读 Observation 或陷入循环。真正的可靠性来自工具反馈、状态约束、验证器、预算、回退策略和人工复核共同组成的 Harness。

面试中还应落到个人贡献和结果：具体负责了哪些模块、如何构建评测集、哪类失败最常见、上线后任务成功率或人工耗时改善了多少。没有真实指标时不要编造，可以诚实说明当前做到离线验证还是灰度阶段。

<a id="moonshot-agent-dev-20260721-q2"></a>
##### 2. 短期记忆的具体实现方式是什么？

**回答：**

短期记忆不是一个单独的神秘模块，它本质上是当前任务继续执行所必需的工作状态。它既包括送入模型的近期上下文，也包括不一定直接放进 Prompt 的结构化状态，例如当前目标、执行计划、已完成步骤、工具返回引用和待确认事项。

我会把短期记忆拆成三层：

| 层次 | 主要内容 | 处理方式 |
|---|---|---|
| 固定约束 | System 指令、权限、安全边界、输出契约 | 高优先级保留，版本化管理 |
| 工作集 | 最近对话、当前计划、未完成步骤、关键工具结果 | 每轮更新，按任务相关性裁剪 |
| 历史压缩 | 更早轮次的结构化摘要和事件引用 | 达到预算或阶段边界时生成 |

工程上可以使用“近期窗口 + 结构化摘要 + 外部原始事件存储”。近期窗口保留最后若干轮完整交互；摘要不是泛泛复述，而应固定字段，例如 `goal`、`constraints`、`completed_steps`、`open_questions`、`artifacts`、`decisions` 和对应的事件 ID。这样能减少摘要遗漏关键约束，也方便回溯原文。

短期状态可以存在 Workflow state、会话数据库或检查点存储中；Prompt 只是它的一种投影视图。每次模型调用前，由 Context Builder 根据 Token 预算选择哪些状态进入上下文。这样即使模型窗口变化，底层任务状态仍保持稳定。

最后要强调：KV Cache 是推理加速状态，不等于 Agent Memory；聊天记录也不等于短期记忆。短期记忆的标准是“它是否帮助当前任务正确继续”，而不是“它是否最近出现过”。

<a id="moonshot-agent-dev-20260721-q3"></a>
##### 3. 什么叫“快到上限了”？对话是怎么逐步叠加的？

**回答：**

“快到上限”不是简单看对话轮数，而是预测下一次请求会不会超过可用上下文预算。一次模型调用的 Token 预算至少包含：System 指令、工具 schema、历史消息、长期记忆/RAG 片段、当前用户输入，以及必须为模型输出预留的 Token。

可以写成：

$$
B_{\mathrm{used}}=B_{\mathrm{system}}+B_{\mathrm{tools}}+
B_{\mathrm{history}}+B_{\mathrm{retrieval}}+B_{\mathrm{query}}
$$

只有满足 $B_{\mathrm{used}}+B_{\mathrm{reserved\ output}}<B_{\mathrm{window}}$，请求才是安全的。工程上还要保留安全余量，避免 tokenizer 估算误差、工具返回突增或模型自动推理预算占满窗口。

在常见的无状态 Chat API 中，对话确实是不断追加消息，并在下一轮把所需历史重新发送给模型。模型不会因为“上一轮见过”就天然记住；服务端会话、Prompt caching 或 KV Cache 可以降低传输/计算成本，但不能改变模型本次只能基于可见上下文决策的事实。

增长最快的往往不是用户对话，而是工具 schema、代码文件、搜索结果和日志。因此系统应在调用工具前估算潜在返回大小，限制日志行数、分页读取大文件，并把完整产物存外部对象存储，只向模型提供摘要和可按需展开的引用。

所以“快到上限”的正确实现是动态 Token Budgeter，而不是写死“第 10 轮总结”或固定使用窗口的 70%。不同轮次的信息量差异很大，轮数只能作为辅助信号。

<a id="moonshot-agent-dev-20260721-q4"></a>
##### 4. 如果对话轮次过多，你怎么去做优化？

**回答：**

对话过长时，我不会只做一次粗暴截断，而是按“保真优先级 + 按需取回”管理上下文。

第一，永远保护系统约束、权限、当前目标和输出契约；这些内容一旦被摘要掉，Agent 可能改变行为边界。第二，保留最近的完整交互和当前未完成步骤，它们构成工作集。第三，把已完成阶段压缩成结构化状态，而不是自然语言流水账。第四，将原始对话、工具日志和大文件放在外部存储中，通过事件 ID、文件句柄或检索接口按需取回。

上下文组装时再做相关性选择：与当前子任务直接相关的历史优先；已经完成且没有后续依赖的细节只保留结论；失败轨迹只保留失败原因和不可重复的动作，避免模型再次踩坑；高风险决策要保留证据和审批记录，不能只剩一句摘要。

还可以把长任务切成多个有边界的阶段，每个阶段结束生成 checkpoint。子任务之间通过结构化产物传递，而不是把全部对话继续往后拖。对代码 Agent 来说，这种产物可以是修改文件列表、测试结果、剩余 TODO 和当前 Git diff；对业务 Agent，可以是订单 ID、审批状态和待补字段。

Rocky认为，长上下文优化的本质不是“把 Prompt 压短”，而是把对话历史升级为可管理的任务状态。只有状态可恢复、原文可回溯、信息可按需展开，压缩才不会变成不可逆的信息丢失。

<a id="moonshot-agent-dev-20260721-q5"></a>
##### 5. 什么时候去触发这个总结动作？

**回答：**

总结触发应由预算风险和任务语义共同决定，而不是每隔固定轮数机械执行。

预算触发上，Context Builder 在每次调用前预估总 Token，并为输出、工具返回和误差保留余量。当预计下一步会超预算时触发压缩。可以设置软水位和硬水位：软水位提前异步生成候选摘要；硬水位则阻塞新调用，确保上下文完成一致性压缩后再继续。但具体比例要通过真实请求分布校准，不能把 70%/90% 当成通用答案。

语义触发包括：一个子任务完成、用户目标发生切换、工具产生大体量结果、连续失败需要重规划、对话即将转入人工审批，或者当前状态已经稳定到可以形成 checkpoint。这些时刻比“第 N 轮”更适合总结，因为信息边界清晰。

并发场景中，总结必须基于带版本号的不可变快照。生成摘要后使用 compare-and-swap 或事务检查：如果原会话版本已经变化，就不能直接覆盖最新状态，而要合并新增事件或重新生成。摘要落库后还应记录覆盖的事件范围，例如 `events[1:42]`，确保不会重复压缩或漏掉消息。

总结前后要运行一致性检查：目标、硬约束、未完成事项、关键实体和工具产物引用是否都保留。高风险字段可以规则化提取并单独存储，不完全依赖 LLM 摘要。

<a id="moonshot-agent-dev-20260721-q6"></a>
##### 6. 是每一轮对话都要去做总结吗？

**回答：**

不是。每轮总结会增加一次模型调用、显著拉高延迟和费用，而且会制造“摘要的摘要”：信息每经过一次有损压缩，都可能丢失限定条件、时间关系和否定信息。

合理策略是事件驱动。平时只追加新事件并更新少量结构化状态；达到 Token 预算、阶段完成或目标切换时才总结。最近几轮通常保留原文，不需要立刻压缩。对工具返回，可以优先用确定性解析器提取字段，再决定是否需要 LLM 总结。

增量摘要虽然便宜，但不能无限滚动。系统应周期性地从原始事件或稳定 checkpoint 重建摘要，并用事实一致性检查比较新旧版本，防止错误不断累积。摘要中最好保存关键事实对应的原事件 ID，遇到低置信或高风险问题时回读原文。

是否总结最终看单位收益：减少了多少 Token、引入了多少延迟、保留了多少任务信息、是否影响任务成功率。不能只优化上下文长度，却让 Agent 忘记用户约束。

<a id="moonshot-agent-dev-20260721-q7"></a>
##### 7. 假设已进行 10 轮并做了总结，第 11 轮开始时，总结怎么处理？是重算前 11 轮还是叠加？

**回答：**

第 11 轮开始时，正常做法是使用已有摘要 $S_{1:10}$ 加上尚未被摘要的近期消息，而不是立刻重算前 11 轮，也不是每出现一条新消息就再压缩一次。

当后续再次达到触发条件时，可以基于旧摘要和增量事件生成新版本：

$$
S_{1:11}=\operatorname{Summarize}(S_{1:10},D_{11})
$$

这种增量更新成本低，但它是有损链路。若长期只在旧摘要上继续摘要，早期错误和遗漏会被固化。因此工程上应采用“增量更新 + 周期性重建”：日常基于上一版本和新事件更新；到达重要阶段、摘要质量下降或发生冲突时，从原始事件/最近稳定 checkpoint 全量重建。

全量重算是否是 $O(n^2)$ 取决于触发频率。若每一轮都重新处理从第 1 轮到当前轮，累计处理量确实呈二次增长；若只在少数阶段边界重建，则不能简单称为 $O(n^2)$。面试中应把复杂度前提讲清楚。

每个摘要版本要带 `version`、覆盖事件区间、生成模型/Prompt 版本和校验状态。这样第 11 轮到来时，系统能够准确判断它属于已压缩历史还是待合并增量，避免重复计算和事件丢失。

<a id="moonshot-agent-dev-20260721-q8"></a>
##### 8. 如果前 10 轮都变成了总结，那之前的原始上下文就不需要了吗？

**回答：**

不能说完全不需要。更准确的说法是：原始上下文通常不再每轮发送给模型，但仍应根据业务价值、隐私要求和合规期限做外部持久化。

保留原始事件有四个作用：第一，线上 bad case 可以回放，判断模型当时看到了什么；第二，摘要质量不佳时可以重建；第三，长期记忆或审计需要回到原始证据；第四，模型、Prompt 或压缩策略升级后可以离线重放评估。

但“全部永久保存”同样不正确。原始对话可能包含个人信息、商业秘密和工具返回凭证，需要加密、访问控制、脱敏、租户隔离、TTL 和用户删除能力。密钥、Token、支付信息等高敏内容原则上不进入通用记忆库。

摘要应保存原始事件引用，而不是切断证据链；模型需要核对某个关键约束时，通过引用精确读取相关片段。这样既控制上下文成本，又保留可追溯性。

本质上，摘要是运行时视图，原始事件是证据层。二者生命周期不同，不能把“当前不进 Prompt”误解成“系统可以删除”。

<a id="moonshot-agent-dev-20260721-q9"></a>
##### 9. 长期记忆可以按需检索召回，具体在什么情况下需要检索？

**回答：**

长期记忆检索的触发条件是：当前任务是否依赖跨会话的用户事实、历史决策或已沉淀经验。典型场景包括用户明确说“沿用上次方案”“继续昨天的任务”，当前输入出现历史项目/人物/文件实体，需要个性化偏好，存在未完成任务，或者当前信息与已知记忆可能冲突。

实现上可以使用多级门控。先用规则捕获显式历史指代和实体，再用轻量路由模型判断记忆依赖程度；对于高风险操作，即使用户没有明确提历史，也要查询授权状态和既有约束。路由结果还应带检索范围，例如只查当前项目、当前用户或某类记忆，避免全库搜索。

检索查询不应直接等于用户原句。可以结合当前目标、实体、时间范围和 memory type 做 query rewriting，再使用向量、关键词、结构化过滤或图关系检索。召回后按相关性、时间、置信度、来源权威性和作用域重排。

需要区分“检索”和“注入”。系统可以低成本检索候选记忆，但只有通过相关性和冲突校验的内容才进入模型上下文。若候选记忆互相矛盾，应标记冲突并询问用户，而不是静默选择一个版本。

知识图谱适合实体关系和多跳查询，但不是长期记忆的必选项。很多场景用关系数据库加向量/全文检索已足够；只有关系结构确实带来可验证收益时，才值得引入图数据库。

<a id="moonshot-agent-dev-20260721-q10"></a>
##### 10. 每轮对话都要注入记忆吗？长短期记忆是同时注入吗？

**回答：**

不是每轮都把所有记忆塞进 Prompt。短期工作状态通常每轮都需要，但也只应投影当前目标、最近交互和未完成步骤；长期记忆则按需检索，并在通过相关性、权限和冲突检查后注入。

组装顺序可以是：系统约束 -> 当前任务状态 -> 必要的长期记忆 -> RAG 证据 -> 最近对话 -> 当前问题。不同信息应带清晰标签、来源和时间，避免模型把历史偏好误当成当前指令，或把外部知识误当成用户事实。

当长短期记忆冲突时，不能简单按“长期更稳定”处理。当前用户明确表达通常优先于历史偏好；业务主库和工具实时状态优先于旧摘要；高风险冲突需要重新确认。模型推断出的记忆权重最低，不应覆盖用户明确事实。

还要给记忆设置独立 Token 预算和 Top-K/阈值。即使每条记忆相关，放得过多也会造成注意力竞争。可先提取结构化事实，再仅对关键记忆附带原文证据。

一句话总结：短期记忆维持当前任务连续性，长期记忆提供跨任务复用；可以同时存在，但不能不加选择地同时灌入。

<a id="moonshot-agent-dev-20260721-q11"></a>
##### 11. 如何减少工具过多带来的 Token 消耗？

**回答：**

工具过多时，最大浪费往往不是调用结果，而是每一轮都把所有工具的完整 JSON Schema 发送给模型。解决方向是让工具能力逐层披露，而不是让模型一次看完整个工具仓库。

第一层只提供工具或 Skill 的简短目录，包括名称、用途、风险级别和少量检索标签；路由器选出候选工具后，第二层才加载其完整 schema、示例和约束。复杂工具还可以在真正调用前再展开详细文档。这就是 progressive disclosure。

其他手段包括：按领域把工具分组并路由到子 Agent；去掉 schema 中重复描述和冗长示例；使用枚举、引用和共享类型减少重复字段；对稳定工具前缀使用 Prompt caching；限制工具返回体，采用分页、字段选择、摘要和对象句柄；将多个低层 API 封装成有业务语义的 Skill，减少模型规划步骤。

动态披露会带来路由漏选风险，因此必须有回退机制：候选工具不足或执行失败时允许扩大检索范围；高风险工具即使被召回，也要经过权限检查和人工审批。工具检索评测应关注 Recall@K，而不只是最终任务成功率。

工具越多，真正的挑战越不是 Token，而是能力发现、权限、版本和可观测性。压缩 schema 只能治表面，建立可检索、可测试、可治理的工具注册中心才是长期方案。

<a id="moonshot-agent-dev-20260721-q12"></a>
##### 12. 你的 RAG 是用什么技术实现的？

**回答：**

按这个项目的真实口径，RAG 使用 BGE-large-zh-v1.5 做中文 Embedding，向量维度为 1024；数据写入 Milvus，并使用 IVF_FLAT 索引；查询阶段先召回 Top-20，再用 BGE reranker 精排，最终选取 Top-5 进入生成上下文。

完整链路应从数据治理讲起。文档解析后保留标题层级、段落、表格和来源 metadata；切片优先遵循语义边界，并使用父子块或邻接扩展保留上下文。需要注意，LangChain `RecursiveCharacterTextSplitter` 默认的 `chunk_size=512` 通常按字符长度函数计算，不天然等于 512 Token；如果要控制模型预算，应使用 tokenizer-aware splitter。

Milvus 的索引和查询参数需要一起调优。IVF_FLAT 的 `nlist` 决定聚类桶数量，查询 `nprobe` 决定探测多少桶；召回率、延迟和内存要通过数据集评测确定。向量写入前是否归一化、Milvus 使用 COSINE/IP/L2 哪种 metric，也必须与 Embedding 训练和检索配置一致。

生产检索通常还会加入 BM25/关键词召回，解决型号、数字、专有名词和精确字符串问题；多路结果可用 RRF 融合，再交给 Cross-encoder reranker。最终上下文按来源去重、控制 token budget，并保留引用 ID。

RAG 不能只报技术栈，还要报评测：检索层看 Recall@K、MRR/nDCG 和命中率；生成层看答案正确性、忠实度、引用准确率和拒答质量。没有离线评测集，`Top-20 -> Top-5` 只是经验参数，不是可证明的最优配置。

<a id="moonshot-agent-dev-20260721-q13"></a>
##### 13. 如何判断向量的相似度？

**回答：**

如果使用余弦相似度，它衡量的是两个向量方向的一致程度：

$$
\cos(\mathbf{x},\mathbf{y})=
\frac{\mathbf{x}^{\top}\mathbf{y}}
{\|\mathbf{x}\|_2\,\|\mathbf{y}\|_2}
$$

值越接近 1，方向越相同；接近 0 表示近似正交；接近 -1 表示方向相反。很多文本 Embedding 训练目标会让语义相近样本方向更接近，因此余弦相似度常用于语义召回。

工程上不能只说公式，还要确认三件事：Embedding 模型是否建议归一化；向量库配置的 metric 是否一致；相似度阈值是否在本领域验证集上校准。不同模型的分数分布不同，不能把 0.8 当成通用“相似”标准。

检索效果也不能只看单条分数。应在标注查询集上计算 Recall@K、MRR 或 nDCG，并观察 hard negative 是否被排到前面。余弦只负责第一阶段几何近似，复杂语义关系通常还需要 reranker。

<a id="moonshot-agent-dev-20260721-q14"></a>
##### 14. 除了余弦相似度，还了解其他相似度算法吗？

**回答：**

常见方法至少包括点积、欧几里得距离、曼哈顿距离，以及针对集合或稀疏文本的 Jaccard、BM25；在检索系统中还会用 Cross-encoder 学习相关性，而不是只依赖固定距离公式。

欧几里得距离为：

$$
d_2(\mathbf{x},\mathbf{y})=\|\mathbf{x}-\mathbf{y}\|_2
$$

点积为 $\mathbf{x}^{\top}\mathbf{y}$，既受方向影响，也受模长影响。若向量已做 L2 归一化，点积就等于余弦相似度。曼哈顿距离计算各维绝对差之和，对某些稀疏或鲁棒场景有用，但在主流稠密向量 ANN 中不如 IP/L2/COSINE 常见。

BM25 不是向量距离，但对关键词、数字、产品型号和专有名词非常有效；Jaccard 适合比较集合重合度。Cross-encoder reranker 将 query 和 document 联合编码，能捕捉细粒度交互，精度通常高于双塔向量相似度，但计算成本更大，因此适合小规模候选精排。

选择算法看模型训练目标和业务需求，不是背一个列表。语义泛化用稠密向量，精确匹配用 BM25，多路召回后再重排，通常比只换一种距离函数更有效。

<a id="moonshot-agent-dev-20260721-q15"></a>
##### 15. 听不懂，说人话？

**回答：**

可以把每段文本想象成高维空间里的一支箭。余弦相似度主要看两支箭指向是否一致，不太关心箭本身有多长：方向越接近，模型认为它们表达的语义越接近。

例如“怎么重置登录密码”和“忘记密码如何找回”用词不完全相同，但 Embedding 可能把它们放到接近的方向；“密码”和“天气”则会指向不同方向。RAG 就是用当前问题这支箭，去找知识库里方向最接近的几支箭。

但这只是帮助理解的比喻。实际是否相似仍由 Embedding 模型的训练数据和目标决定，余弦公式本身不会理解语义；模型没学过的领域、歧义词或精确数字仍可能召回错误，所以还需要关键词检索、rerank 和真实测试集。

<a id="moonshot-agent-dev-20260721-q16"></a>
##### 16. 余弦相似度与欧几里得距离在工程应用中的具体区别是什么？

**回答：**

余弦关注方向，欧氏距离关注空间中的绝对位置；但工程上最关键的前提是向量是否归一化。

若 $\|\mathbf{x}\|_2=\|\mathbf{y}\|_2=1$，则：

$$
\|\mathbf{x}-\mathbf{y}\|_2^2=
2-2\mathbf{x}^{\top}\mathbf{y}
=2-2\cos(\mathbf{x},\mathbf{y})
$$

因此在单位向量上，按余弦从大到小、按点积从大到小、按欧氏距离从小到大，排序完全一致。此时选择哪种 metric 主要取决于向量库索引支持、实现效率和现有配置。

若没有归一化，两者就不等价。余弦消除模长影响，适合只关心方向的语义匹配；欧氏距离和点积会保留模长信息。如果模型训练时让模长表达置信度、流行度或其他信号，盲目归一化反而可能丢信息。

还要纠正两个常见误区。第一，不是所有 Embedding API 都保证返回单位向量，必须查模型说明或在代码中验证；BGE 常见用法会显式设置 `normalize_embeddings=True`。第二，欧氏检索不一定因为“要开方”而更慢，排序时可以直接比较平方距离，ANN 库也有专门优化。

所以工程结论是：遵循模型训练建议，入库与查询使用完全一致的归一化和 metric，并在同一评测集上比较召回率与延迟。

<a id="moonshot-agent-dev-20260721-q17"></a>
##### 17. 项目中一共使用了几个模型？分别是什么？

**回答：**

按原项目可以回答为四类模型，但要按职责说清楚，而不是只报名字。

第一类是 Embedding 模型 BGE-large-zh-v1.5，负责将文档和查询编码为 1024 维稠密向量。第二类是 BGE reranker，作为 Cross-encoder 对初召候选精排。第三类是轻量意图路由模型，例如微调后的 BERT-base，用于判断直接回答、查知识、查记忆或调用哪类工具。第四类是主 LLM，负责规划、工具参数生成、证据整合和最终回答；具体使用 Qwen-72B 或 Claude 3.5 要按项目真实部署口径说明，不能含糊地把两个模型都算作线上主模型。

这种拆分体现了分层选型：高频、确定、低延迟任务交给小模型；语义召回交给双塔；候选相关性判断交给 Cross-encoder；开放式推理和生成交给大模型。不是所有能力都堆在一个 LLM 上。

面试官通常会继续问为什么不用主 LLM 做路由、是否支持替换模型、四个模型如何统一监控。应准备回答延迟与成本、模型服务接口、灰度切换、版本绑定、超时回退和端到端评测。模型数量不是亮点，职责边界和替换成本才是。

<a id="moonshot-agent-dev-20260721-q18"></a>
##### 18. 如何控制模型的幻觉问题？

**回答：**

幻觉不能靠一个 Prompt 或调低温度解决，需要从“是否该回答、证据是否可靠、输出是否被验证”三层控制。

第一层是任务边界。对必须基于企业知识回答的问题，先判断是否有足够证据；检索分数低、证据冲突或超出权限时应该澄清、拒答或转人工。第二层是证据链。提升解析、切片、混合召回和 rerank 质量，去除过期/重复文档，并把最终使用的证据和来源 ID 明确提供给模型。

第三层是受约束生成和验证。要求答案中的关键事实能够映射到证据片段；对数字、实体、引用和工具执行结果使用规则或独立验证器核对；涉及写操作时采用“计划 -> 预览 -> 审批 -> 执行 -> 回读验证”，不能让模型直接宣称操作成功。

温度降低只能减少采样随机性，不能修复参数知识错误或错误证据；显式 Chain-of-Thought 也不等于事实验证，并可能产生貌似合理的错误解释。更稳的是输出简洁结论、引用和不确定性，再由验证器检查。类似“如果不知道就说不知道”的指令有帮助，但必须用评测集验证，不能脱离实验声称固定降幅。

评测至少包含有答案、无答案、证据冲突、诱导提问和越权查询。指标可看 groundedness/faithfulness、引用准确率、无答案拒答率、误拒答率和任务级事实错误率。Rocky认为，幻觉治理的本质不是让模型“更谨慎地说”，而是让系统只在证据和权限允许时行动。

<a id="moonshot-agent-dev-20260721-q19"></a>
##### 19. 如何观察模型召回了哪些块？

**回答：**

需要给检索链路建立可回放的 trace。每次请求至少记录：原始 query、改写后的检索 query、过滤条件、Embedding/索引版本、初召 chunk ID 与分数、融合前后的排名、rerank 分数、最终进入 Prompt 的片段、Token 数和引用 ID。

日志要能把问题拆成三层：相关文档是否进入初召候选，这是召回问题；进入候选后是否被 reranker 排除，这是排序问题；证据已进入上下文但回答仍错误，这是生成或上下文利用问题。没有这三层，所有 bad case 最后都会被笼统归因成“模型幻觉”。

线上可以做检索审计面板，展示 Top-K 文本高亮、分数、metadata、版本和最终引用；同时记录查询耗时、各阶段候选数和空召回率。对一条请求应有统一 trace ID，串联路由、检索、rerank、LLM 和工具调用。

可观测日志也有安全边界。正文可能含用户隐私和企业敏感信息，应脱敏、按租户授权，并设置保存期限。生产环境通常记录 chunk ID、摘要和哈希，只有受权排障时才读取完整原文。

<a id="moonshot-agent-dev-20260721-q20"></a>
##### 20. 简历别的项目随便问了问主要工作

**回答：**

这类问题虽然看似“随便问”，实际是在验证简历真实性。建议用 60 到 90 秒讲清楚：业务问题是什么、你负责的边界、关键技术决策、结果指标和一次失败复盘。

可以按 STAR 展开，但技术岗要把 Action 说具体。例如不要只说“负责模型优化”，而要说明自己处理了哪类数据、改了哪个模块、为什么不用替代方案、离线和线上指标如何变化、上线风险怎么控制。团队成果和个人贡献必须分开。

如果项目与 Agent 岗位不是直接相关，可以主动建立迁移关系：传统分类项目训练了数据治理和评测能力；后端项目训练了接口、并发和可观测性；搜索项目训练了召回与排序。这些能力如何迁移到 Agent 系统，比强行把所有经历包装成大模型项目更可信。

没有量化结果时可讲可验证交付物，例如上线模块、覆盖用户、故障率、测试集规模、节省的人工步骤，但不要编造百分比。面试官继续深挖时，应该能落到代码、日志和真实 bad case。

<a id="moonshot-agent-dev-20260721-q21"></a>
##### 21. 算法题：Leetcode 143. 重排链表

**回答：**

这题要求原地重排：

$$
L_0\rightarrow L_1\rightarrow\cdots\rightarrow L_n
$$

变为：

$$
L_0\rightarrow L_n\rightarrow L_1\rightarrow L_{n-1}\rightarrow\cdots
$$

标准解法分三步：快慢指针找到前半段尾节点；反转后半段；把两段链表交替合并。Python 实现如下：

```python
class ListNode:
    def __init__(self, val=0, next=None):
        self.val = val
        self.next = next


def reorder_list(head: ListNode | None) -> None:
    if head is None or head.next is None:
        return

    slow = fast = head
    while fast.next is not None and fast.next.next is not None:
        slow = slow.next
        fast = fast.next.next

    second = slow.next
    slow.next = None

    prev = None
    while second is not None:
        nxt = second.next
        second.next = prev
        prev = second
        second = nxt

    first, second = head, prev
    while second is not None:
        next_first = first.next
        next_second = second.next
        first.next = second
        second.next = next_first
        first = next_first
        second = next_second
```

找中点、反转和合并各遍历一次链表，时间复杂度为 $O(n)$，额外空间复杂度为 $O(1)$。使用数组加双指针也能做到 $O(n)$ 时间，但需要 $O(n)$ 额外空间，没有体现题目希望考察的链表断链、反转和交错合并能力。

边界测试至少包括空链表、单节点、两个节点、奇数长度和偶数长度。尤其要先执行 `slow.next = None` 断开前后半段，否则合并时可能形成环。

<a id="moonshot-agent-dev-20260721-q22"></a>
##### 22. 你平时调试代码怎么调试的？

**回答：**

我会按“稳定复现 -> 缩小范围 -> 增加观测 -> 验证假设 -> 修复回归”的顺序调试，而不是一上来把整段代码交给 AI。

算法题或单文件脚本，最有效的是构造最小输入，用 `print` 输出关键指针、循环变量和分支路径，再与手工推演对比。复杂业务代码会使用结构化日志、断点调试器、单元测试和 trace ID；并发/分布式问题还要看时间线、请求链路和资源指标。

具体工具取决于问题：Python 用 `pdb`/IDE debugger、pytest 和日志；服务端看请求日志、指标和分布式 tracing；模型/Agent 链路还要记录 Prompt 版本、模型参数、工具输入输出和状态 checkpoint，保证同一问题可回放。

AI 可以帮助解释堆栈、生成测试用例和提出排查假设，但它拿不到完整运行环境，也可能误判。把 AI 当副驾驶，自己用证据验证，才是可复现的调试方式。

<a id="moonshot-agent-dev-20260721-q23"></a>
##### 23. 不对吧，最简单的方式不是打印一下吗？

**回答：**

对，算法题里最简单有效的办法通常就是打印关键状态。比如重排链表可以在找中点、反转后半段和每次合并后打印链表，立刻判断错误发生在哪个阶段。

好的 print debugging 不是把所有变量全打出来，而是围绕假设设计输出：当前节点值、`next` 指向、循环次数和链表是否出现重复节点。输出要有标签，例如 `phase=reverse curr=3 next=4`，否则日志越多越难看。

但打印也有边界：并发时序、性能问题、内存破坏或生产环境敏感数据不适合随意 print，此时需要断点、结构化日志、指标和 trace。面试官这句追问考的不是工具高低，而是你是否会从最低成本的观测手段开始排查。

<a id="moonshot-agent-dev-20260721-q24"></a>
##### 24. 最近了解的 AI 内容？

**回答：**

最近我重点关注的是编码 Agent 从“模型会生成代码”转向“Harness 能否稳定交付软件”的变化。真正决定编码 Agent 可用性的，越来越不是一次补全的代码质量，而是它能否读取仓库上下文、检索代码、规划修改、调用测试、处理失败、遵守权限并留下可审计的变更。

这类系统通常包含 Context Builder、工具注册与检索、执行沙箱、权限策略、状态/checkpoint、错误恢复、可观测性和评测闭环。模型负责在当前状态下做判断，Harness 决定模型能看到什么、能做什么、失败后怎样继续。

这个方向值得关注，是因为基础模型能力会快速普及，但真实软件工程中的仓库约束、测试环境、安全边界和交付标准不会自动消失。未来编码 Agent 的竞争，不只是“谁的模型更强”，而是谁能把模型能力组织成可重复、可验证、可恢复的工程流程。

面试时讲“最近内容”不要堆新闻名词。最好选择一个主题，说明它的核心机制、与旧方案的差异、一个未解决问题，以及它如何影响自己的项目设计。

<a id="moonshot-agent-dev-20260721-q25"></a>
##### 25. 讲一下 Claude Code 架构

**回答：**

Claude Code 更适合被理解为一个编码 Agent Harness，而不是“模型外面套了一层终端”。从公开可观察的产品行为与文档看，其主线是一个迭代式 Agent loop：读取用户目标和项目上下文，模型决定下一步工具调用，权限层审核动作，工具执行后把结果作为 Observation 返回，直到完成、失败或达到预算。

可以按六个模块解释：

- **Context 层**：整合用户指令、项目级规则、相关代码、当前修改和压缩后的历史。
- **Agent loop**：模型基于当前状态决定检索、读取、编辑、执行命令、测试或委派子任务。
- **Tool 层**：文件读写、代码搜索、Shell、测试以及可扩展工具都通过明确输入输出契约调用。
- **Permission/Sandbox**：高风险命令、文件范围和外部访问受到权限策略约束，并支持人工确认。
- **State 与恢复**：保存任务进度、工具结果和修改状态，支持长任务压缩、检查点或回退。
- **扩展与自动化**：项目规则、Hooks、MCP/外部工具和子 Agent 用来连接仓库规范与企业系统。

不要声称自己“看过 Claude Code 全部源码”后给出精确代码行数、工具数量或“绝对无共享状态”等内部结论。产品分发包和公开文档能够支持行为与接口层分析，但不等于完整、可审计的内部源码。面试里最可信的是解释可验证架构和工程取舍。

Claude Code 的关键不在某个 ReAct 名词，而在于把上下文、权限、工具、状态和验证组织成闭环。模型更像决策内核，Harness 才是让它进入真实代码库的操作系统。

<a id="moonshot-agent-dev-20260721-q26"></a>
##### 26. 建议下去再补一点 Harness 的知识

**回答：**

这个反馈指向一个重要概念：Harness 不是某个单一框架，也不等于 `AGENTS.md`、`ARCHITECTURE.md` 和 `TASKS.md` 三份文件。它是围绕模型构建的整套运行与治理环境，决定 Agent 如何感知、行动、受约束、恢复和被评估。

一个完整 Harness 通常包括：上下文选择与压缩、工具发现和 schema 管理、执行沙箱、权限审批、状态机与 checkpoint、超时重试、模型路由、日志 tracing、评测器和人工接管。仓库级规则文件只是 Context/Policy 的一种载体，不同产品使用的文件名和优先级可能不同。

可以把它与模型分工概括为：模型给出“下一步做什么”的概率决策，Harness 负责“哪些动作可用、哪些证据可见、动作如何执行、结果怎样验证、失败如何恢复”。同一个模型放进不同 Harness，最终任务成功率可能差很多。

继续补知识时应围绕真实问题学习：工具过多如何检索；上下文超长如何管理；命令如何最小权限执行；长任务如何 checkpoint；测试失败如何回滚；怎样用轨迹数据评测工具选择与恢复能力。理解这些机制，比记住某几个配置文件更有跨周期价值。

<a id="moonshot-agent-dev-20260721-q27"></a>
##### 27. 反问业务

**回答：**

反问业务时要验证岗位真正解决的问题、失败成本和交付边界，而不是只问“团队做什么”。如果方向是安全风险相关 AI，可以围绕以下问题展开。

第一，当前主要场景是内容审核、违规行为识别、风险研判，还是给内部分析人员提供 Agent 辅助？模型输出是建议，还是会直接触发处置动作？这决定了自动化权限和人工复核要求。

第二，团队最看重哪些指标？安全场景不能只看总体准确率，还要区分误杀率、漏放率、严重风险召回、校准和人工复核成本。不同错误的业务代价可能完全不对称。

第三，数据和评测如何构建？是否有持续更新的对抗样本、灰产变体和线上回放集？LLM Judge 是否会由规则、专家标注或真实处置结果校准？

第四，岗位的工作比例如何分配：Agent 工程、模型训练、RAG/知识建设、业务规则和客户交付分别占多少？项目中的共性能力能否沉淀为平台，而不是长期做一次性需求。

第五，模型在高风险动作前有哪些权限、审计和 Human-in-the-loop 机制？出现误判后如何回放、归因和回滚？

这些问题既能帮助候选人判断岗位，也能向面试官证明：你理解安全 Agent 的核心不是“会调用工具”，而是让每个判断有证据、每个动作有权限、每次错误能追溯。
