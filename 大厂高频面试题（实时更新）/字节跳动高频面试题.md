### 问题1：如何解决类别极度不平衡的问题？
- 在机器学习中，类别不平衡是指数据集中各类样本的数量差异较大，这可能导致模型对多数类过度拟合，而对少数类泛化能力差。
![](./imgs/sample_imbalance.png)

### 问题2：说下 Transformer 模型
- **Transformer 模型** 是一种基于自注意力机制的神经网络架构，广泛应用于序列到序列的任务中。Transformer 本身是一个典型的encoder-decoder模型，
Encoder端和Decoder端均有6个Block， Encoder 端的Block 包括两个模块，多头self-attention模块以及一个前馈神经网络模块；Decoder端的
Block 包括三个模块，多头self-attention模块，多头Encoder-Decoder attention交互模块，以及一个前 馈神经网络模块； 
需要注意：Encoder端和Decoder端中的每个模块都有残差层和LayerNormalization层。

### 问题3：说下 Focal Loss
- **Focal loss 是目标检测中解决正负样本严重不平衡的方法**，在标准交叉熵损失基础上修改得到的。这个函数可以通过减少易分类样本的权重，
使得模型在训练时更专注于稀疏的难分类的样本；防止大量易分类负样本在loss 中占主导地位

### 问题4：介绍下深度可分离卷积和传统卷积的区别
- **深度可分离卷积首先在每个通道上独立进行空间卷积，然后使用 $1 \times 1$ 的卷积来组合通道特征，与传统卷积相比，它大大减少了参数数量和计算量。**
- 传统的卷积是各个通道上采用不同的卷积核，然后不同的卷积核用于提取不同方面的特征。 
- 深度可分离卷积先在各个通道上采用不同的卷积核提取不同的特征，但是这样对于某个通道来说，就只提取了一方面的特征，因此在此基础上加入点卷积，
用1*1的卷积对提取特征后的特征图再次提取不同方面的 特征，最终产生和普通卷积相同的输出特征图。

### 问题5：如何防止过拟合
- 防止过拟合的方法包括：获取更多数据、数据增强、正则化（如L1、L2正则化）、Dropout、Early Stopping等。

### 问题6：BN 在训练和测试的时候的区别？可以防止过拟合吗？
- **Batch Normalization (BN) 在训练时对每个mini-batch的数据进行归一化，而在测试时使用整个训练集的统计量。**
- **BN有助于防止过拟合，因为它使得模型对于输入数据的分布变化更加鲁棒。** BN算法防止过拟合：在网络的训练中，BN的使用使得一个minibatch中所有样本都被关联在了一起，
因此网络不会从某一个训练样本中生成确定的结果，即同样一个样本的输出不再仅仅取决于样本的本身，也取 决于跟这个样本同属一个batch的其他样本，
而每次网络都是随机取batch，这样就会使得整个网络不会朝这一个方向使劲学习。一定程度上避免了过拟合。

### 问题7：什么是 AUC？
- **AUC（Area Under the Curve）是ROC曲线下的面积，用来衡量分类器的排序能力。** AUC可以解读为从所有正例中随机选取一个样本A，
再从所有负例中随机选取一个样本B，分类器将A判为正例的概率比将B判为正例的概率大的可能性。AUC反映的是分类器对样本的排序能力。
AUC越大，自然排序能力越好，即分类器将越多的正例排在负例之前。

### 问题8：卷积核计算公式
- 卷积层的输出特征图可以通过以下公式计算： 

$$Output = (W - K + 2P) / S + 1$$ 

其中 $W$ 是输入特征图的大小， $K$ 是卷积核的大小， 
$P$ 是padding的大小， $S$ 是步长。
- 如果想保持卷积前后的特征图大小相同，通常会设定padding为：

$$P = \frac{K - 1}{2}$$

<a id="bytedance-ai-agent-hard-20260602-q1"></a>
##### 为什么在 RAG 流程中引入父子索引（Parent-Document Retrieval）？

**回答：**

父子索引的核心动机是同时解决“召回精度”和“上下文完整性”的矛盾。RAG 检索时，如果直接用大段文档建向量，段落语义会被稀释，embedding 很难精准命中用户问题；如果直接用很小的 chunk 建索引，召回会更准，但返回给模型的上下文可能被切碎，缺少定义、前后约束、表格标题、代码上下文或业务背景。

Parent-Document Retrieval 的做法是：用较小的 child chunk 做向量检索或关键词检索，保证召回粒度足够细；命中 child 后，不直接只把 child 塞给模型，而是回溯到它所属的 parent document 或 parent chunk，把更完整的上下文返回给生成阶段。也就是“用小块检索，用大块回答”。

它的价值主要有四点。第一，提升召回精度，小 chunk 更容易和 query 在局部语义上匹配。第二，保留上下文完整性，parent 能补足小 chunk 周围的定义、约束和例子。第三，减少答案断章取义，尤其适合法规、技术文档、产品说明、客服知识库、代码文档等上下文依赖强的资料。第四，便于做引用和溯源，可以从 child 命中点定位到 parent 文档的具体位置。

工程上要注意 parent 不能过大，否则会重新引入噪声和 token 成本；child 也不能过小，否则语义不完整。通常会让 child 用 200 到 500 tokens 左右，parent 用 800 到 2000 tokens 或按标题/章节切分，再根据模型上下文窗口和业务评测调参。

<a id="bytedance-ai-agent-hard-20260602-q2"></a>
##### 为什么引入 BM25？向量检索和 BM25 的融合比例是怎样的？

**回答：**

引入 BM25 是因为向量检索和关键词检索解决的是不同问题。向量检索擅长语义相似，比如“如何降低幻觉”和“怎么防止模型乱编”能被认为相近；但它对精确实体、代码符号、错误码、产品型号、专有名词、数字、缩写、接口名不一定敏感。BM25 基于词项匹配和逆文档频率，擅长精确命中关键词，能弥补向量检索在精确匹配上的短板。

两者融合不应该拍脑袋固定比例。常见做法有两类。

第一类是分数融合。分别做 dense retrieval 和 BM25 retrieval，对两路分数做归一化，然后按权重合并，例如 `score = alpha * dense_score + (1 - alpha) * bm25_score`。alpha 可以从 0.5、0.6、0.7 等值开始网格搜索。偏语义问答时 dense 权重大一些，偏文档编号、错误码、函数名、政策条款时 BM25 权重大一些。

第二类是排名融合，最常见是 RRF。RRF 不要求不同检索器的分数可比，只根据排名融合，公式类似 `sum(1 / (k + rank_i))`。在工程上 RRF 更稳，因为向量相似度和 BM25 分数分布不同，直接线性加权容易失真。

如果面试官追问具体比例，可以回答：初始可以用 dense:BM25 = 0.6:0.4 或 0.7:0.3 做 baseline，但最终一定要通过验证集调优，指标看 recall@k、MRR、nDCG、答案正确率、引用命中率和延迟。真正上线时还可以按 query 类型动态调权：包含编号、专有名词、代码符号时提高 BM25 权重；自然语言描述较多时提高向量检索权重。

<a id="bytedance-ai-agent-hard-20260602-q3"></a>
##### 检索融合的具体流程是什么？召回后有没有做 Rerank？

**回答：**

一个比较完整的检索融合流程可以分成六步。

第一步是 query 预处理。包括去噪、分词、大小写归一、同义词扩展、实体识别、时间和权限条件解析。复杂问题还可以做 query rewrite 或 query decomposition，把一个大问题拆成多个子问题。

第二步是多路召回。一路走向量检索，用 embedding 从向量库召回 topK dense chunks；一路走 BM25 或倒排索引，召回 topK sparse chunks；必要时还可以加入标题召回、FAQ 精确匹配、知识图谱、数据库过滤、历史对话记忆等通道。

第三步是归一化与融合。如果用分数融合，需要把 cosine similarity、BM25 score 等不同分布的分数归一化后加权；如果用 RRF，就按各路排名进行融合。融合后做去重，尤其父子索引场景下要把同一 parent 下高度重复的 child 合并。

第四步是过滤。按权限、业务线、文档类型、时间有效性、语言、版本号等 metadata 过滤，避免召回到用户无权访问或已经过期的内容。

第五步是 Rerank。召回阶段追求高召回，排序往往不够精确，所以通常会加 reranker。Reranker 可以是 cross-encoder、bge-reranker 这类相关性模型，也可以是轻量 LLM rerank。它会把 query 和候选 chunk 成对输入，重新判断相关性。

第六步是上下文组装。根据 rerank 分数、token budget、parent 合并、去重、多样性和引用需求，选择最终进入 prompt 的 chunk。

所以召回后通常要做 Rerank，特别是在混合检索和父子索引场景中。原因是召回阶段更像粗筛，Rerank 能修正向量检索“语义相关但不回答问题”、BM25“词命中但语义不对”、多路融合分数不可比等问题。

<a id="bytedance-ai-agent-hard-20260602-q4"></a>
##### Rerank 后返回几个块（Chunk）？有没有针对这个返回数量做过验证？

**回答：**

Rerank 后返回几个 chunk 不能只凭经验固定，应该由三个约束共同决定：答案所需证据量、模型上下文窗口和噪声控制。实际项目里常见的最终入模数量是 3 到 8 个 chunk。如果 chunk 较长或 parent 较大，可能只返回 3 到 5 个；如果 chunk 较短、问题需要多证据综合，可能返回 6 到 10 个。

验证方式应该是离线评测加线上观察。离线可以构建一批 query-answer-evidence 标注集，分别测试 top3、top5、top8、top10 的效果，看 recall@k、faithfulness、answer accuracy、引用命中率、无答案拒答率和平均 token 成本。线上可以看用户满意度、追问率、人工质检通过率、延迟和成本。

通常会出现一个拐点：topK 从 1 增加到 3 或 5 时，答案质量明显提升；继续增加到 10 或 15 后，召回证据更多，但噪声也更多，模型更容易被无关内容干扰，延迟和成本也上升。因此返回数量要通过实验找到质量、成本和延迟的平衡点。

面试里可以给一个具体但不绝对的说法：我会先用 rerank top5 作为 baseline，然后通过评测比较 top3、top5、top8。若问题是事实型或单文档问答，top3 到 top5 往往足够；若是综合型问题，可能增加到 top8，并配合去重、parent 合并和引用约束。

<a id="bytedance-ai-agent-hard-20260602-q5"></a>
##### Rerank 后的 TopK 截断是怎么做的？为什么是这个值？有没有其他截断方案？

**回答：**

最简单的 TopK 截断是按 rerank 分数从高到低排序，取前 K 个候选进入上下文。例如粗召回 50 个，rerank 后取 top5。这个 K 的选择要由验证集、token budget、延迟和噪声共同决定，而不是只靠经验。

为什么可能选择 top5？因为在很多知识库问答场景中，一个问题通常只需要 1 到 3 个核心证据，再加少量补充上下文。top5 能覆盖大部分必要证据，同时不会引入过多噪声。更大的 K 会增加 token 成本和上下文干扰，模型可能在多个相似但不一致的片段之间摇摆。

除了固定 topK，还有几种更稳的截断方案。第一是分数阈值截断，只保留 rerank score 高于阈值的 chunk，适合无答案拒答，但阈值要校准。第二是相对阈值截断，例如只保留分数不低于 top1 分数一定比例的候选。第三是 token budget 截断，按分数从高到低加入，直到达到上下文预算。第四是 MMR 多样性截断，在相关性和多样性之间折中，避免返回 5 个几乎重复的 chunk。第五是按 parent 聚合截断，先合并同一父文档下的 child，再限制每个 parent 的候选数量。第六是动态 K，根据 query 类型决定，简单事实题少取，复杂综合题多取。

工程上更推荐“固定 topK baseline + 阈值 + token budget + 去重”的组合。这样既有稳定上限，又能在低置信度问题上减少噪声。

<a id="bytedance-ai-agent-hard-20260602-q6"></a>
##### 讲一下上下文工程（Context Engineering），Agent 的记忆（Memory）是怎么做的？

**回答：**

上下文工程是指围绕模型上下文窗口进行信息选择、组织、压缩、注入和验证的工程体系。它不只是 prompt engineering。Prompt engineering 更关注指令怎么写；context engineering 更关注在每一轮模型调用时，应该把哪些任务目标、用户状态、历史对话、检索证据、工具结果、约束条件和输出格式放进去，以及如何控制成本和噪声。

一个 Agent 的上下文通常包括几类信息：系统指令、当前用户输入、短期对话历史、长期记忆、RAG 检索证据、工具调用结果、任务计划、执行状态、权限和安全约束。上下文工程要解决的问题是：哪些信息必须保留，哪些可以摘要，哪些应该检索，哪些应该丢弃，哪些需要结构化。

Agent Memory 可以分为短期记忆和长期记忆。短期记忆通常是当前会话内的历史消息、任务计划、已调用工具结果和中间状态。由于上下文窗口有限，短期记忆一般需要滑动窗口、摘要压缩、重要事件提取。长期记忆则是跨会话保存的用户偏好、稳定事实、项目配置、历史任务结果、失败经验等，通常存到数据库、向量库或结构化 profile 中。

从类型上还可以分为 semantic memory、episodic memory 和 procedural memory。Semantic memory 存稳定知识和用户偏好；episodic memory 存某次交互或任务经历；procedural memory 存执行流程、工具使用方法、skill 或工作流。

实现时通常有四个动作：写入、检索、压缩和遗忘。写入时要判断哪些信息值得长期保存，不能把所有对话都写入；检索时要根据当前任务从记忆库召回相关内容；压缩时把长历史变成结构化摘要；遗忘时处理过期、冲突、敏感或低价值记忆。

关键风险是记忆污染和隐私问题。错误记忆会长期影响 Agent 行为，敏感信息也不能随意持久化。因此需要置信度、来源、时间戳、权限、可删除机制和人工反馈。面试总结可以说：Agent Memory 不是把聊天记录塞回 prompt，而是一个有读写策略、检索策略、压缩策略、冲突解决和安全边界的状态管理系统。

<a id="bytedance-ai-agent-hard-20260602-q7"></a>
##### 请详细讲解分布式令牌桶限流的实现。

**回答：**

令牌桶算法的核心是：系统以固定速率向桶中放入令牌，桶有最大容量；每个请求到来时需要消耗一定数量的令牌，拿到令牌就允许通过，拿不到就拒绝或排队。它允许一定突发流量，因为桶里可以提前积累令牌。

分布式令牌桶的难点是多台应用实例要共享同一个限流状态，因此状态不能只放在单机内存里，通常放在 Redis 中。一个典型 Redis key 会保存两个字段：当前令牌数 tokens 和上次补充时间 last_refill_ms。还需要配置参数：桶容量 capacity、补充速率 rate、每次请求消耗 requested。

请求到来时执行逻辑如下：第一，读取当前时间 now。第二，读取 Redis 中的 tokens 和 last_refill。第三，根据时间差计算应补充的令牌数 `delta = (now - last_refill) * rate`，新令牌数为 `min(capacity, tokens + delta)`。第四，如果新令牌数大于等于 requested，则扣减令牌并允许请求；否则拒绝或返回需要等待的时间。第五，把新的 tokens 和 last_refill 写回 Redis。

这个过程必须保证原子性，否则并发请求会超卖令牌。实际实现通常用 Redis Lua 脚本，把读、计算、扣减、写回放在一个脚本里执行。Lua 在 Redis 单线程模型下原子执行，可以避免分布式锁开销。

还要注意几个工程细节。第一，时间源最好统一，可以使用 Redis TIME，避免不同机器时钟漂移。第二，key 要设置 TTL，避免长期不用的限流对象占用内存。第三，限流维度可以是用户、IP、接口、租户、模型名称或组合 key。第四，热点 key 可能成为瓶颈，可以按业务维度分片，但全局限流会更复杂。第五，失败策略要明确：Redis 故障时是 fail-open 还是 fail-closed，金融和安全场景通常更保守。第六，需要返回剩余令牌和 retry-after，方便上游做退避。

令牌桶适合需要允许短时突发、但长期平均速率受控的场景，例如大模型 API 调用、短信发送、用户请求限流、任务投递限速。

<a id="bytedance-ai-agent-hard-20260602-q8"></a>
##### 漏桶算法（Leaky Bucket）的原理讲一下。

**回答：**

漏桶算法把请求看成水，把系统处理能力看成桶底以固定速率漏水。请求进入桶中排队，桶以恒定速率处理请求；如果桶满了，新请求就被拒绝或丢弃。

它的核心特点是输出平滑。无论入口流量多么突发，漏桶都会按固定速率向下游放行，因此适合保护下游系统，避免瞬时流量冲击。例如日志写入、消息发送、第三方 API 调用、固定吞吐的后端服务。

漏桶和令牌桶的区别在于突发能力。令牌桶允许桶中积累令牌，所以短时间内可以处理超过平均速率的突发请求；漏桶强调恒定速率流出，不鼓励突发。如果业务希望用户偶尔突发访问但长期受控，令牌桶更合适；如果业务希望请求严格平滑进入下游，漏桶更合适。

实现上可以用队列加定时消费，也可以用时间戳计算理论流出进度。队列实现直观但需要存请求，内存压力更大；时间计算实现更轻量，适合只判断是否允许通过的限流器。

<a id="bytedance-ai-agent-hard-20260602-q9"></a>
##### 滑动窗口算法限流讲一下。如果用代码实现，滑动窗口的结构体会包含哪些字段？

**回答：**

滑动窗口限流用于统计“最近一段时间内”的请求数量，并判断是否超过阈值。相比固定窗口，它不会在窗口边界产生严重突刺。例如限制 1 分钟 100 次，固定窗口可能在 00:59 和 01:00 各打 100 次，瞬间形成 200 次；滑动窗口会按当前时间向前滚动统计最近 60 秒，边界更平滑。

实现方式有两类。第一类是滑动日志，记录每个请求的时间戳。请求到来时删除窗口外的时间戳，再判断剩余数量是否超过 limit。它精度最高，但内存随请求量增长，适合中低 QPS 或严格限流。

第二类是滑动窗口计数器，把大窗口切成多个小桶，例如 60 秒切成 60 个 1 秒桶或 6 个 10 秒桶。请求到来时落到当前桶，统计最近 N 个桶的总数。它内存固定，效率高，但精度取决于桶粒度。

如果用代码实现滑动窗口结构体，通常包含这些字段：

```go
type SlidingWindowLimiter struct {
    Limit       int           // 窗口内允许的最大请求数
    Window      time.Duration // 总窗口大小
    BucketSize  time.Duration // 每个小桶的时间粒度
    Buckets     []int64       // 每个桶的计数
    BucketStart []int64       // 每个桶对应的起始时间戳，避免复用旧桶
    Cursor      int           // 当前桶位置
    Total       int64         // 当前窗口内总请求数，可选，用于避免每次求和
    Mutex       sync.Mutex    // 并发安全
}
```

如果是滑动日志结构，则核心字段会是 limit、window、timestamps 队列和锁。分布式场景则需要把这些状态放到 Redis，并用 Lua 保证原子性。

<a id="bytedance-ai-agent-hard-20260602-q10"></a>
##### 滑动窗口和令牌桶相比有什么缺点？

**回答：**

滑动窗口的优点是统计语义直观，能严格表达“最近 N 秒最多 M 次”。但和令牌桶相比，它有几个缺点。

第一，突发友好性较差。令牌桶允许在桶中积累令牌，因此可以支持短时合理突发；滑动窗口更强调窗口内总量，可能对突发流量更敏感。

第二，存储和计算成本更高。滑动日志要保存窗口内每次请求的时间戳，高 QPS 下内存压力大；滑动计数器虽然内存固定，但要维护多个桶，还要处理桶过期、求和和并发安全。令牌桶只需要 tokens 和 last_refill 两个核心状态。

第三，分布式实现更复杂。滑动窗口在 Redis 中通常要用 ZSET 或多个桶计数，需要删除过期数据、统计数量、插入当前请求；令牌桶只需原子更新令牌数和时间戳。

第四，平滑程度取决于实现。滑动计数器如果桶粒度较粗，仍会有局部边界误差；桶粒度越细，成本越高。

因此，如果需求是精确限制最近时间窗口内的请求数，滑动窗口更合适；如果需求是控制平均速率并允许突发，令牌桶更合适。

<a id="bytedance-ai-agent-hard-20260602-q11"></a>
##### 在 Redis 中，你会用什么数据结构来实现滑动窗口限流？

**回答：**

如果要实现精确滑动窗口，我会用 Redis ZSET。ZSET 的 score 存请求时间戳，member 可以用请求 ID 或时间戳加随机数。每次请求执行一个 Lua 脚本：删除窗口外的数据 `ZREMRANGEBYSCORE key 0 now-window`，统计当前窗口内请求数 `ZCARD key`，如果小于 limit，就 `ZADD key now requestId` 并设置过期时间；否则拒绝。整个过程用 Lua 保证原子性。

ZSET 的优点是精确，能记录每一次请求；缺点是高 QPS 下内存和删除成本较高，适合用户维度、接口维度等单 key 请求量可控的场景。

如果 QPS 很高，可以用 HASH 做滑动计数器。HASH 的 field 是时间桶，例如秒级时间戳，value 是该桶计数。请求到来时对当前桶 `HINCRBY`，同时清理过期桶或设置 key TTL。统计时求最近 N 个桶的计数之和。这种方式内存更可控，但精度取决于桶粒度。

也可以用 Redis Stream 或 List 做日志，但 ZSET 更适合按时间范围删除和统计。面试里回答 ZSET + Lua 是最标准的精确滑动窗口实现，HASH buckets 是高吞吐近似实现。

<a id="bytedance-ai-agent-hard-20260602-q12"></a>
##### LRU 缓存的实现原理讲一下。

**回答：**

LRU 表示 Least Recently Used，淘汰最近最少使用的数据。它基于一个假设：最近被访问的数据未来更可能再次被访问。

经典实现是 HashMap + 双向链表。HashMap 负责 O(1) 查找 key 对应的节点；双向链表维护访问顺序，链表头表示最近访问，链表尾表示最久未访问。

get 操作时，如果 key 不存在，返回空；如果存在，通过 HashMap 找到节点，把节点移动到链表头，然后返回 value。put 操作时，如果 key 已存在，更新 value 并移动到链表头；如果 key 不存在，创建新节点插入链表头，并写入 HashMap。如果容量超过限制，就删除链表尾节点，并从 HashMap 中移除。

双向链表的作用是支持 O(1) 删除任意节点和 O(1) 插入头部；HashMap 的作用是 O(1) 定位节点。两者结合才能让 get 和 put 都是 O(1)。

工程上要注意并发安全、容量为 0、更新已有 key、淘汰回调、过期时间、内存估算等问题。Redis 的近似 LRU 不是严格 HashMap + 双链表，而是采样淘汰；本地缓存如 Java LinkedHashMap 或 Go list+map 可以实现严格 LRU。

<a id="bytedance-ai-agent-hard-20260602-q13"></a>
##### 布隆过滤器（Bloom Filter）的底层原理和适用场景讲一下。

**回答：**

布隆过滤器是一种空间效率很高的概率型集合结构，用于判断一个元素是否可能存在。它由一个 bit 数组和 k 个哈希函数组成。

插入元素时，使用 k 个哈希函数把元素映射到 bit 数组的 k 个位置，并把这些位置置为 1。查询元素时，也计算 k 个哈希位置，如果这些位置都为 1，则判断元素可能存在；如果任意一个位置为 0，则元素一定不存在。

它的特点是：不会出现假阴性，也就是如果判断不存在，那一定不存在；但可能出现假阳性，也就是判断存在时，可能只是多个元素的哈希位碰巧都为 1。误判率和 bit 数组大小、哈希函数个数、插入元素数量有关。元素越多，bit 数组越满，误判率越高。

适用场景包括缓存穿透防护、黑名单过滤、URL 去重、爬虫去重、推荐曝光去重、大规模集合预判。例如缓存穿透中，请求一个数据库根本不存在的 key，如果每次都打到数据库，会造成压力。可以先用布隆过滤器判断 key 是否可能存在，不存在就直接拒绝。

缺点是普通布隆过滤器不支持删除，因为多个元素可能共享同一个 bit。若需要删除，可以用 Counting Bloom Filter，用计数器代替 bit，但空间成本更高。它也不适合要求 100% 精确存在判断的场景。

<a id="bytedance-ai-agent-hard-20260602-q14"></a>
##### MySQL 索引会在哪些情况下失效？

**回答：**

MySQL 索引失效通常不是索引真的坏了，而是优化器无法有效利用索引，或者认为走索引成本比全表扫描更高。

常见情况包括：

第一，不满足联合索引最左前缀。例如联合索引 `(a, b, c)`，查询只用 `b` 或 `c`，通常无法有效使用该联合索引。

第二，对索引列使用函数、表达式或计算。例如 `where date(create_time) = ...`、`where age + 1 = 18`，会让索引列无法按原值有序匹配。

第三，发生隐式类型转换。例如字符串字段 `phone` 用数字查询 `where phone = 123`，可能导致索引失效或效果变差。

第四，LIKE 以通配符开头，例如 `like '%abc'` 或 `like '%abc%'`，B+ 树无法从左到右定位前缀。

第五，范围查询后面的联合索引列无法继续充分利用。例如 `(a,b,c)` 中 `where a=1 and b>10 and c=3`，`c` 往往不能继续用于索引定位。

第六，使用 `or` 时，如果两边条件不是都能用索引，优化器可能放弃索引。

第七，负向条件如 `!=`、`not in`、`not like` 通常选择性较差，可能不走索引。

第八，索引选择性太低，例如性别字段，优化器可能认为全表扫描更划算。

第九，排序或分组字段和索引顺序不一致，可能无法利用索引避免 filesort。

第十，字符集或排序规则不一致，也可能导致无法正确使用索引。

判断索引是否使用应看 `EXPLAIN`，重点关注 type、key、rows、Extra，而不是只凭语法判断。

<a id="bytedance-ai-agent-hard-20260602-q15"></a>
##### 在使用 LIKE 进行模糊查询时，索引什么情况下会失效？

**回答：**

LIKE 能否使用 B+ 树索引，关键看模式是否有固定左前缀。

`like 'abc%'` 通常可以使用索引，因为 B+ 树可以定位到以 `abc` 开头的范围，再向后扫描。它本质上等价于在有序索引中查找 `[abc, abd)` 这样的范围。

`like '%abc'` 和 `like '%abc%'` 通常不能有效使用普通 B+ 树索引，因为匹配条件不是从字符串左侧开始，索引无法定位起始范围，只能扫描大量数据。

`like '_abc%'` 这类以下划线或通配符开头的模式，也无法利用固定左前缀。对索引列做函数后再 LIKE，例如 `where lower(name) like 'abc%'`，如果没有函数索引，也会影响索引使用。

如果确实需要中缀匹配，可以考虑全文索引、倒排索引、ngram 分词、Elasticsearch/OpenSearch，或者额外维护反转字段来支持后缀匹配。例如要查 `'%abc'`，可以维护 reverse_name，转成反向前缀查询。

<a id="bytedance-ai-agent-hard-20260602-q16"></a>
##### MySQL 的事务隔离级别有哪些？如何保证一致性？

**回答：**

MySQL 标准事务隔离级别有四个：读未提交、读已提交、可重复读、串行化。

读未提交允许读取其他事务未提交的数据，会出现脏读，实际很少使用。读已提交只能读到已提交数据，避免脏读，但同一事务中两次读取同一行可能结果不同，会出现不可重复读。可重复读保证同一事务内多次一致性读看到相同快照，是 InnoDB 默认隔离级别，依靠 MVCC 和必要的锁避免很多并发异常。串行化隔离级别最强，事务近似串行执行，通常通过加锁实现，性能最低。

一致性可以从 ACID 理解。原子性由 undo log 保证，事务失败可以回滚。持久性由 redo log 保证，提交后的修改即使宕机也能恢复。一致性是目标，依赖原子性、隔离性、持久性以及约束、触发器、业务逻辑共同保证。隔离性由锁和 MVCC 保证。

在 InnoDB 中，普通一致性读主要靠 MVCC 读快照，写操作和当前读依赖行锁、间隙锁、next-key lock 等控制并发。除此之外，binlog 和 redo log 的两阶段提交保证主从复制和崩溃恢复之间的一致性。

面试中可以总结：事务一致性不是单个机制保证的，而是 undo log、redo log、锁、MVCC、约束和两阶段提交共同保证的。

<a id="bytedance-ai-agent-hard-20260602-q17"></a>
##### 详细说说 MVCC 的实现，ReadView 的生成时机是怎样的？

**回答：**

MVCC 是多版本并发控制，目标是在不加读锁的情况下实现一致性读，提高读写并发。InnoDB 的 MVCC 依赖隐藏字段、undo log 和 ReadView。

InnoDB 每行记录有隐藏字段，重要的是 `trx_id` 和 `roll_pointer`。`trx_id` 表示最后修改这行记录的事务 ID；`roll_pointer` 指向 undo log 中的旧版本。每次更新并不是简单覆盖旧值，而是生成新版本，并通过 undo log 保留旧版本链。

ReadView 是一致性读时生成的快照，里面通常包含：创建 ReadView 时活跃事务 ID 列表 `m_ids`，最小活跃事务 ID `up_limit_id`，下一个将分配的事务 ID `low_limit_id`，以及当前事务 ID。判断某个版本是否可见时，大致规则是：如果版本的 trx_id 小于最小活跃事务 ID，说明在快照前已提交，可见；如果 trx_id 大于等于 low_limit_id，说明是快照后才出现的事务，不可见；如果 trx_id 在活跃事务列表中，说明快照创建时还没提交，不可见；否则可见。

如果当前记录版本不可见，就沿着 undo log 版本链向前找，直到找到对当前 ReadView 可见的版本，或者找不到。

ReadView 的生成时机和隔离级别有关。在读已提交下，每次执行一致性读语句都会生成新的 ReadView，所以同一事务里两次 SELECT 可能看到不同已提交结果。在可重复读下，事务中第一次一致性读生成 ReadView，后续一致性读复用这个 ReadView，因此能保证同一事务内快照稳定。

注意，MVCC 主要针对普通 SELECT 的一致性读。`select ... for update`、`lock in share mode`、update、delete 属于当前读，会读取最新已提交版本并加锁。

<a id="bytedance-ai-agent-hard-20260602-q18"></a>
##### 在不同的隔离级别下，一个事务分别会创造几个 ReadView？

**回答：**

读未提交一般不需要 ReadView，因为它允许读取未提交数据，不提供一致性快照语义。

读已提交下，一个事务内每次执行一致性读都会创建新的 ReadView。因此同一事务中多次 SELECT 可能创建多个 ReadView，每次都能看到语句开始前已经提交的数据。

可重复读下，一个事务通常只在第一次一致性读时创建一个 ReadView，后续一致性读复用它。因此同一事务内多次 SELECT 能看到相同快照，这也是可重复读的核心。

串行化下，InnoDB 通常通过加锁把并发事务串行化，普通 SELECT 也可能变成加锁读，因此重点不在 ReadView，而在锁控制。不过具体执行还和语句类型、引擎实现有关。

还要区分一致性读和当前读。即使在可重复读下，普通 SELECT 复用 ReadView；但 UPDATE、DELETE、SELECT FOR UPDATE 读取的是当前最新可见版本，并会加锁，不按普通快照读理解。

<a id="bytedance-ai-agent-hard-20260602-q19"></a>
##### MySQL 都有哪些锁？它们的作用分别是什么？

**回答：**

MySQL 锁可以按层次和用途分类。

全局锁会锁住整个数据库实例，典型命令是 `Flush tables with read lock`，常用于全库逻辑备份，但会影响写入。

表级锁锁住整张表，例如 MyISAM 常用表锁，InnoDB 也有表级意向锁和元数据锁。元数据锁 MDL 用于保护表结构，防止查询过程中表结构被修改。意向锁包括 IS 和 IX，用来表示事务准备在表中某些行上加共享锁或排他锁，方便表锁和行锁之间做冲突判断。

行级锁是 InnoDB 并发控制的核心。记录锁锁住索引记录本身；间隙锁锁住索引记录之间的间隙，防止其他事务插入造成幻读；next-key lock 是记录锁加间隙锁，锁住一个左开右闭区间；插入意向锁表示事务想在某个 gap 中插入记录，多个插入不同位置的事务可以并发。

按读写关系还可以分为共享锁和排他锁。共享锁允许多个事务同时读，但阻塞写；排他锁用于写操作，会阻塞其他读写锁请求。

还有自增锁，用于处理 auto_increment 分配；在不同配置下可以有不同的锁粒度。对于外键约束、唯一索引检查，也会引入相应的锁行为。

面试中要强调：InnoDB 行锁是加在索引上的。如果查询条件没有命中索引，可能扫描更多记录并加更多锁，甚至表现得像锁表。锁的目的不是单纯互斥，而是配合隔离级别解决脏写、不可重复读、幻读和并发一致性问题。

<a id="bytedance-ai-agent-hard-20260602-q20"></a>
##### 为什么选择 Python 和 Go 作为技术栈？在处理并发时有什么区别？

**回答：**

Python 和 Go 的定位不同，组合使用很常见。Python 生态强，适合大模型应用、RAG、数据处理、模型推理调用、实验验证和 AI 工具链集成。很多 LLM、embedding、rerank、向量库、数据处理框架都优先支持 Python。开发效率高，适合快速迭代算法和应用逻辑。

Go 更适合高并发后端服务、网关、任务调度、RPC 服务和基础设施。它编译部署简单，运行时开销低，goroutine 和 channel 原生支持并发，服务稳定性和工程可维护性好。

并发方面，Python 的主要限制是 GIL。对于 CPU 密集型多线程，Python 线程不能真正并行执行 Python 字节码，通常要用 multiprocessing、C 扩展、向量化计算或把 CPU 密集任务交给模型服务。对于 IO 密集型任务，Python 可以用 asyncio、异步 HTTP 客户端、事件循环来高效处理，例如并发调用大模型 API、数据库、对象存储等。

Go 的 goroutine 是轻量级协程，由 Go runtime 调度到多个 OS 线程上，可以利用多核并行。Go 的 channel、context、select、sync 包让并发控制比较自然，适合大量连接、限流、超时控制、取消传播和服务端并发处理。

所以选型可以是：Python 负责 AI 能力层和快速实验，例如 RAG pipeline、模型调用、评测脚本；Go 负责高并发服务层，例如 API 网关、限流、任务队列、流式转发和服务治理。两者通过 HTTP/gRPC/消息队列衔接。

<a id="bytedance-ai-agent-hard-20260602-q21"></a>
##### 手撕代码：实现反转链表。

**回答：**

反转链表的核心是逐步改变每个节点的 `next` 指针。需要维护三个指针：`prev` 表示已经反转部分的头节点，`cur` 表示当前处理节点，`nxt` 暂存原链表的下一个节点，避免断链。

Python 实现如下：

```python
class ListNode:
    def __init__(self, val=0, next=None):
        self.val = val
        self.next = next


def reverse_list(head: ListNode | None) -> ListNode | None:
    prev = None
    cur = head
    while cur:
        nxt = cur.next
        cur.next = prev
        prev = cur
        cur = nxt
    return prev
```

Go 实现如下：

```go
type ListNode struct {
    Val  int
    Next *ListNode
}

func reverseList(head *ListNode) *ListNode {
    var prev *ListNode
    cur := head
    for cur != nil {
        next := cur.Next
        cur.Next = prev
        prev = cur
        cur = next
    }
    return prev
}
```

时间复杂度是 O(n)，每个节点只遍历一次；空间复杂度是 O(1)，只使用常数个指针。边界情况包括空链表、单节点链表、两节点链表，都可以被这套逻辑自然覆盖。

<a id="bytedance-ai-fde-20260712-q4"></a>
##### 2.项目中如何定位问题，有没有使用过Linux或者抓包等工具？

**回答：**

定位问题要体现方法论，而不是只说“看日志”。我一般会按“复现 -> 分层 -> 证据 -> 缩小范围 -> 修复验证”走。

第一步复现问题，明确输入、环境、版本、时间点和期望行为。第二步分层定位：前端交互、网关、后端服务、模型服务、工具服务、数据库、向量库、客户本地网络都要分开看。第三步收集证据：应用日志、trace id、接口状态码、请求耗时、模型输入输出、工具调用参数、数据库慢查询和系统资源。

Linux 工具上，常用 `top/htop` 看 CPU 和内存，`df/du` 看磁盘，`ps/jstack` 看进程和线程，`journalctl` 或服务日志看异常，`curl` 验接口，`netstat/ss/lsof` 看端口，`grep/awk/sed` 快速过滤日志。网络问题可以用 `ping`、`telnet/nc`、`traceroute`、`dig/nslookup`。抓包可以用 `tcpdump` 或 Wireshark，重点看 DNS、TLS 握手、TCP 重传、HTTP 状态码、请求体是否异常。

对 AI 应用还要额外看模型链路：prompt 是否被裁剪、RAG 召回是否为空、rerank 是否误排、工具参数是否被模型生成错、上下文是否超长、供应商 API 是否限流。定位的核心不是工具清单，而是每一步都能用证据排除一层。

<a id="bytedance-ai-fde-20260712-q5"></a>
##### 3.部署在客户本地的模型报错了怎么排查？

**回答：**

客户本地部署的问题要先区分是模型本身错误、环境问题、资源问题、网络问题，还是业务数据问题。不要一上来重启服务。

第一层看部署环境：GPU 型号、驱动版本、CUDA/cuDNN、Python 依赖、模型推理框架、容器镜像、模型权重 checksum 是否一致。很多问题来自客户环境和研发环境不一致。第二层看资源：显存是否 OOM、KV cache 是否过大、并发是否超过限额、磁盘是否满、CPU/内存是否瓶颈。第三层看服务：模型服务是否正常启动，健康检查是否通过，端口是否监听，日志里是加载失败、推理失败还是请求超时。

第四层看请求链路：客户系统到模型服务之间是否有代理、鉴权、网关、证书、DNS、跨域或防火墙问题。可以用 `curl` 在本机和客户服务机分别打接口，用 trace id 串起调用。第五层看输入数据：prompt 是否过长、图片/文件格式是否不合法、工具参数是否缺字段、RAG 索引是否为空、权限是否不允许访问某些数据。

排查时要建立最小可复现 case：用固定输入直接调用模型服务，如果可用，再逐层接回业务系统。客户现场最忌讳“猜”，要用日志、指标和复现实验把锅从模型、环境、网络、数据、业务代码中拆开。

##### 4.工作项目中最有挑战性的部分是什么？

**回答：**

这个问题要回答“难在哪里”和“你怎么拆解”，不要只说项目复杂。

AI FDE/Agent 项目最大的挑战通常不是把模型接上，而是把开放式需求变成可交付、可评测、可运维的系统。客户会说“帮我自动处理这个流程”，但真实问题里有权限、数据质量、历史系统接口、人工审批、异常分支、合规边界和交付期限。模型输出只是其中一环。

结果可以说成：从原来人工反复复制粘贴、容易漏字段，变成了有校验、有日志、有复核的半自动流程。这个回答的关键是让面试官看到你能从一句需求拆成流程、系统、指标和交付物。

<a id="bytedance-ai-fde-20260712-q8"></a>
##### 6.笔试题：找出文件中top k的字串，简单说了下用哈希以及用哈希的缺点和优化点

**回答：**

这题要先澄清“字串”含义：如果指按分隔符切出来的词/字符串，问题是大文件 Top K 频次；如果指所有连续子串，复杂度会爆炸，需要限制长度或使用后缀结构。面试里通常考的是大文件 Top K 高频字符串。

基础做法是哈希计数 + 小根堆。逐行读取文件，分词后用哈希表统计频次；统计完成后维护大小为 K 的小根堆，堆顶是当前 Top K 中频次最小的字符串。时间复杂度约为 $O(n + m\log k)$，其中 n 是总 token 数，m 是不同字符串数；空间复杂度是 $O(m)$。

哈希的缺点也要讲清楚。第一，大文件下不同字符串太多，哈希表可能放不进内存。第二，哈希冲突虽然平均可控，但极端情况下会退化。第三，分布倾斜时，热点 key 更新频繁。第四，如果文件远大于内存，单机一次性哈希不现实。

优化思路有几种：内存够时用哈希 + 小根堆；内存不够时按 hash 分桶，把字符串写入多个临时文件，每个桶内统计局部 Top K，再合并全局 Top K；近似统计可以用 Count-Min Sketch + heap；分布式场景可以用 MapReduce/Spark，map 做局部计数，reduce 汇总。回答时要体现：简单数据用哈希，海量数据要外部排序/分桶/分布式，精度可放松时才用近似算法。

<a id="bytedance-ai-fde-20260712-q9"></a>
##### 1.介绍工作项目，深挖项目细节

**回答：**

项目介绍要按“业务问题 -> 方案架构 -> 个人贡献 -> 指标结果 -> 复盘”讲，不要一上来堆框架名。

如果面 AI FDE，可以选一个最贴近客户交付或 Agent 落地的项目。比如：项目目标是把客户内部重复流程自动化，涉及需求解析、知识库检索、工具调用、审批流和结果回写。我负责需求拆解、Agent 工作流设计、工具接口封装、日志与评测体系，以及部分客户现场问题排查。

深挖时面试官通常会问四类细节。第一，需求是否真实：客户痛点是什么，原流程耗时多少，为什么需要 AI。第二，技术是否可落地：用 RAG、workflow、tool calling、权限控制、fallback 的原因是什么。第三，效果是否可证明：通过率、人工节省、平均耗时、失败率、客户满意度怎么衡量。第四，你是否真做过：具体写了哪些模块，遇到什么 bug，怎么定位，如何上线。一个成熟回答要能从产品语言切到代码、日志、指标和故障排查。

<a id="bytedance-ai-fde-20260712-q10"></a>
##### 2.有什么经历能说明你有对客户的沟通能力

**回答：**

客户沟通能力不是“会聊天”，而是能把客户的模糊诉求变成技术可执行的需求，并在分歧、故障和交付压力下推进共识。

可以举一个场景：客户说“模型效果不好”，但没有明确指标。我会先把问题拆开：是召回不到、回答不准、格式不稳、响应慢、权限不通，还是不符合业务话术。然后和客户一起定义样本集、验收口径和优先级。例如先选 50 到 100 条真实案例，标注期望输出和不可接受输出，再按错误类型分类。

沟通过程中要做到三件事。第一，不让客户陷入模型术语，而是用业务语言解释问题。第二，不承诺不可控结果，比如“模型一定百分百正确”，而是承诺可验证的改进路径。第三，把会议结论沉淀成文档、责任人、时间点和验收标准。

最后讲结果：通过沟通把“效果不好”转成可测试任务，明确哪些由数据补齐、哪些由 prompt/工具修复、哪些需要人工复核，最终减少反复拉扯。这就是 FDE 最核心的能力：既懂客户语言，也能落回工程闭环。

<a id="bytedance-ai-fde-20260712-q11"></a>
##### 3.工作中有什么流程可以拆解成提效的智能体？

**回答：**

适合拆成智能体的流程，通常具备三个特征：重复高频、输入输出相对明确、需要跨系统或跨文档处理。不是所有流程都适合 Agent，纯规则流程用脚本更稳，强审批流程需要人工兜底。

在 AI FDE 场景里，一个典型例子是“智能体/插件/技能上架流程”。它经常包含需求材料收集、能力说明编写、工具 schema 校验、权限申请、测试用例生成、发布检查、灰度验证和问题回滚。这个流程重复、细碎、容易漏项，很适合做成 workflow + Agent。

Agent 可以负责材料读取、缺失项识别、上架说明生成、测试用例生成、风险点提示、发布清单检查；确定性工具负责 schema 校验、接口连通性测试、权限检查、日志写入和发布调用；人工负责业务语义确认、权限批准、高风险发布确认。

判断标准是：自动化后能否降低人工复制粘贴，能否提高一次通过率，能否减少漏项，能否让每次上架有 trace。智能体提效不是让模型接管一切，而是让模型处理非结构化理解，工具处理确定性执行，人处理责任和风险。

<a id="bytedance-ai-fde-20260712-q12"></a>
##### 4.第3点我说了智能体上架的流程每次都重复单一，可以作为切入点，面试官询问：怎么设计这个智能体？需要有哪些skill和tool，共享一下屏幕写一下提示词，这个智能体验收的标准是什么？哪些步骤需要人工来复核？通过率和不通过率测试集怎么构建？

**回答：**

这个问题是二面的核心，回答要像真实方案设计，而不是泛泛说“用 LangChain 调工具”。

可以把智能体设计成“上架 Copilot + 发布前检查器”。输入是智能体说明、工具 schema、权限说明、测试样例、业务方需求和发布环境；输出是上架材料、测试报告、风险清单、缺失项提示和发布建议。主流程用 workflow 固定住：读取材料 -> 规范检查 -> 工具连通性测试 -> 生成测试集 -> 执行回归 -> 产出报告 -> 人工复核 -> 发布。

Skill 可以包括：需求摘要 skill、上架文档生成 skill、工具 schema 校验 skill、prompt 规范检查 skill、测试用例生成 skill、风险识别 skill、发布报告生成 skill。Tool 可以包括：文档读取、代码仓库读取、API schema validator、接口 ping/curl、权限查询、测试执行器、日志查询、发布平台 API、消息通知。

提示词要明确边界，例如：你是智能体上架审核助手，只能基于提供材料生成结论；缺字段必须列出，不允许编造；高风险操作必须标记人工复核；输出必须包含缺失项、测试项、风险项、建议结论和引用材料片段。验收标准包括：材料完整率、schema 校验通过率、测试用例覆盖率、一次上架通过率、人工修改次数、误放行率、误拦截率、平均上架耗时。

人工复核点包括：业务语义是否正确、权限是否合规、高风险工具是否允许调用、发布窗口和灰度策略、最终上线确认。测试集要分通过集和不通过集：通过集覆盖标准上架样例；不通过集覆盖缺字段、schema 错误、权限缺失、危险工具、接口不可达、prompt 越权、测试失败、文档与代码不一致等。真正成熟的设计，是让模型负责理解和生成，让工具负责验证，让人负责风险决策。

<a id="bytedance-ai-fde-20260712-q13"></a>
##### 5.笔试：ai coding一个简单网页

**回答：**

这类 AI Coding 题首先要明确需求，哪怕题目说“简单网页”，也要主动补齐验收标准：页面展示什么数据、是否要交互、是否要响应式、是否要本地状态、是否需要接口。

如果没有更多要求，可以实现一个“智能体上架检查清单”小网页：左侧输入智能体名称、负责人、工具列表、权限说明；右侧展示检查结果，包括必填项、风险项和通过状态。实现重点不是炫技，而是结构清晰、状态可维护、交互可用。

面试时可以按这个思路写：用 HTML 定义表单和结果区，用 CSS 做基础布局和移动端适配，用 JavaScript 维护状态和校验规则。核心校验包括名称不能为空、至少一个工具、权限说明不能为空、包含高风险关键词时提示人工复核。提交后生成检查报告。

评价点通常包括：需求澄清、组件拆分、状态管理、异常处理、可读性、可测试性和是否能快速交付。AI Coding 不是把代码交给模型就结束，而是你能不能审查模型输出，补边界条件，修交互问题，并解释为什么这样设计。

#### 三面【第六天】

<a id="bytedance-ai-fde-20260712-q14"></a>
##### 1.介绍自己和项目

**回答：**

三面自我介绍要比一面更聚焦“岗位匹配”和“成长曲线”。不要再流水账讲所有项目，而要把自己定位成能做 AI FDE 的人：懂技术、能交付、会和客户沟通、能把 Agent 做成可用系统。

可以说：我过去主要做 AI 应用开发和模型应用工程，经历过从需求沟通、方案设计、RAG/Agent 工作流、工具调用、部署排障到验收交付的完整链路。我的优势不是只会写某个模块，而是能把客户问题拆成工程对象：哪些用规则，哪些用模型，哪些必须人工复核，最后怎么评估是否真的提效。

项目部分重点讲一个最完整的闭环：客户场景是什么，为什么原流程低效，Agent 架构如何设计，接了哪些 tool/skill，如何处理上下文和记忆，如何做测试集、灰度、日志和回滚，最终指标如何改善。三面更看重你有没有全局意识，所以结尾要强调复盘：我对 AI FDE 的理解是，用工程纪律把模型能力翻译成客户可用的生产力。

<a id="bytedance-ai-fde-20260712-q15"></a>
##### 2.讲一下工作中有没有让人超出预期的东西，独立拆解需求并完成的流程

**回答：**

“超出预期”不要讲成加班很多，而要讲你在没有明确指令的情况下，发现了更本质的问题，并交付了比原需求更完整的结果。

可以举例：原需求只是让我接一个模型接口或做一个流程自动化，但我在梳理过程中发现真正影响交付的是缺少标准化验收和问题定位。于是我不仅完成接口接入，还补了三件事：第一，整理流程状态机，明确每一步输入输出和异常分支；第二，建设测试集和检查清单，区分通过样本、不通过样本和边界样本；第三，加上 trace 日志和失败原因分类，让后续问题能快速定位。

结果是，原来只解决“功能能跑”的问题，最后变成了“能测试、能交付、能复盘”的小系统。回答时要强调自己的思考路径：从需求表面往下挖，找到影响客户体验或团队效率的根因，再把方案做成可复用资产。这比单纯说“我独立完成某模块”更有说服力。

<a id="bytedance-ai-fde-20260712-q16"></a>
##### 3.遇到难沟通的客户怎么处理？

**回答：**

难沟通客户通常不是“人难”，而是诉求不清、预期不一致、风险压力大、或者过去被技术方案伤过。处理方式要专业，而不是情绪化。

第一步是对齐事实。客户说“系统不好用”，要追问具体场景、输入、输出、错误截图、发生频率、影响范围。第二步是对齐目标。客户真正要的是准确率、稳定性、响应速度、可解释性，还是流程提效。第三步是对齐边界。哪些能短期修，哪些需要数据或系统改造，哪些属于模型天然不确定性，必须讲清楚。

沟通策略上，我会用业务语言解释技术限制，用样例而不是抽象概念讨论效果；把争议变成可验证实验，比如选一批样本跑新旧方案对比；把会议结论沉淀成待办、负责人、截止时间和验收标准。如果客户情绪强烈，先承认影响和优先级，再给排查时间表，避免在没有证据时争辩。

FDE 的价值就在这里：既不无原则答应，也不把问题推给研发或客户，而是用结构化沟通把不确定性收敛成可执行计划。

<a id="bytedance-ai-fde-20260712-q17"></a>
##### 4.你的智能体上下文是怎么分层？怎么设计的？上下文存储在哪里？长短期记忆怎么划分？RAG数据库都有哪些，它们各自的区别？

**回答：**

智能体上下文要分成“运行时上下文、任务状态、外部知识、长期记忆”几层看。底层模型看到的仍然是一段 token 序列，所谓分层是工程侧为了控制信息优先级、可追溯性和成本做的组织。

第一层是系统层：角色、权限、安全边界、输出格式、工具调用规则。第二层是任务层：用户当前目标、约束、已完成步骤、待办事项、错误信息。第三层是工具层：工具 schema、工具返回、执行日志和环境状态。第四层是知识层：RAG 召回的文档片段、引用、版本和置信度。第五层是记忆层：用户偏好、历史项目、常用配置、客户环境差异等。

存储上，短期上下文通常存在请求上下文、会话状态或 workflow state 里，生命周期是一次任务或一次会话；长期记忆不能只放 prompt，而要结构化存储在数据库、向量库或画像系统中，并有写入策略、权限、过期和冲突处理。KV cache 只是推理加速缓存，不等于长期记忆。

长短期记忆的划分看生命周期和复用价值：当前任务中间结果是短期记忆，跨任务复用的客户偏好、环境配置、历史问题和解决方案是长期记忆。RAG 数据库常见有向量数据库如 Milvus/Qdrant/Chroma，适合语义检索；全文检索如 Elasticsearch/OpenSearch，适合关键词和精确匹配；关系型数据库如 PostgreSQL/pgvector，适合结构化数据和轻量向量；图数据库如 Neo4j，适合实体关系和路径推理。工程上常用 hybrid retrieval：BM25 保精确，向量保语义，rerank 做精排。

<a id="bytedance-ai-fde-20260712-q18"></a>
##### 7.反问工作内容和侧重点

**回答：**

反问要围绕岗位真实性和成长空间，不要只问薪资福利。AI FDE 岗位尤其要判断它到底是技术交付岗、解决方案岗、售前岗，还是披着 AI 外衣的传统支持岗。

可以问：团队目前的 FDE 更偏客户现场交付、Agent 应用开发，还是售前方案支持？日常工作里写代码、调模型、做工具链和客户沟通的比例大概是多少？现在客户最常见的 Agent 落地场景是什么，卡点更多在模型效果、数据接入、权限合规，还是业务流程改造？团队如何衡量 FDE 的绩效，是客户上线数、项目收入、技术资产沉淀、问题解决效率，还是研发贡献？

还可以追问成长：FDE 和产品研发、算法、解决方案之间怎么协作？是否有机会把客户共性问题沉淀成平台能力？有没有技术评审和代码质量要求？这些问题能帮你判断岗位是不是“能做出可用 Agent 的技术岗”，而不是只有包装和协调。

#### HR面【第九天】

<a id="bytedance-ai-fde-20260712-q19"></a>
##### 1.当前职级绩效薪资，期望职级薪资

**回答：**

末轮 HR 再问一次，通常是在做 offer 校准。回答口径要和第一轮保持一致，避免前后矛盾。

可以简洁表达：当前职级、绩效和薪资我可以按公司要求如实提供，期望职级希望结合面试评价和岗位职责判断。我更关注的是职责是否匹配：如果岗位需要独立面对客户、设计 Agent 方案、协调研发、处理部署排障并对交付结果负责，那么我希望职级能反映这种复合能力。

薪资上不要只报一个死数，可以说：我希望总包在合理涨幅和岗位带宽内沟通，最终看职级、绩效预期、发展空间和团队匹配度。这样既表达了预期，也给 HR 留出操作空间。

<a id="bytedance-ai-fde-20260712-q20"></a>
##### 2.为什么看机会？

**回答：**

末轮 HR 更关心稳定性和真实动机。回答要避免变成“原公司不行”，也不要突然换说法。

可以说：我看机会主要是因为职业方向更想往 AI Agent 和大模型应用落地走。过去的工作让我具备工程和交付基础，但下一阶段我希望在更强业务密度、更快迭代、更真实客户场景里成长。AI FDE 这个岗位比较吸引我，因为它要求把模型能力、工具链、客户需求和工程交付串起来，这和我想做的方向一致。

同时补一句稳定性：我不是短期试试，而是希望找一个能长期投入的方向。如果团队确实在做技术含量较高的 Agent 交付，而不是传统支持岗换名，我会很认真考虑。

<a id="bytedance-ai-fde-20260712-q21"></a>
##### 3.什么时候能够入职？

**回答：**

这个问题要明确、可执行，不要模糊说“都可以”。HR 需要判断招聘节奏和 offer 风险。

如果还在职，可以说：按照当前公司的离职流程，我通常需要提前 N 周或一个月交接。如果双方确认 offer，我会尽快启动流程，并保证项目交接、文档整理和权限移交。预计最早入职时间是某个日期区间。

如果有竞业、年假、项目节点，也要提前说明。对社招来说，诚信比“我明天就能来”更重要。一个成熟回答是：我可以配合团队紧急程度，但也会把现有工作交接好，避免给双方造成风险。

<a id="bytedance-ai-fde-20260712-q22"></a>
##### 4.沟通能力怎么样？

**回答：**

沟通能力要用场景证明，而不是自评“还不错”。对 AI FDE 来说，沟通能力包括需求澄清、技术解释、冲突处理和结果同步。

可以说：我比较擅长把复杂技术问题翻译成业务能理解的语言。比如客户反馈“Agent 不好用”，我不会直接解释模型原理，而是先把问题拆成召回、生成、工具调用、权限、响应时间、验收口径几类，再用真实样本和日志说明原因。对于研发侧，我会把客户需求整理成接口、数据、流程、优先级和验收标准，而不是转述一句模糊诉求。

如果遇到争议，我会尽量用样本、指标和实验对齐，而不是靠主观判断。沟通的目标不是让别人觉得我会说话，而是让事情推进，责任清楚，风险透明。

<a id="bytedance-ai-fde-20260712-q23"></a>
##### 5.怎么理解FDE这个岗位？

**回答：**

如果这里的 FDE 指偏 Forward/Field Deployment Engineer 一类岗位，我理解它是“面向客户场景的技术落地工程师”。它不是纯售前，也不是纯后端，更不是只会讲方案的角色。

FDE 的核心价值在于把客户的业务问题翻译成可交付的技术系统。它需要懂客户流程，能识别哪些问题适合 Agent，哪些只能做规则或传统系统；也要懂工程实现，能做 RAG、workflow、tool calling、权限、日志、部署和排障；还要懂交付节奏，能管理预期、构建验收指标、推动上线。

AI 时代的 FDE 更像“应用层的全栈 Agent 工程师 + 技术顾问”。它连接客户、产品、研发和算法，把客户个性化问题沉淀成可复用能力。判断一个 FDE 是否合格，不是看 PPT 做得多漂亮，而是看能不能让客户真实使用、能不能定位问题、能不能把一次性交付沉淀成平台资产。

<a id="bytedance-ai-fde-20260712-q24"></a>
##### 6.反问绩效考核方式团队组成等

**回答：**

末轮 HR 反问要问清楚岗位预期，尤其是 AI FDE 这种边界容易模糊的岗位。

可以问三类问题。第一，绩效考核：这个岗位主要看客户项目上线、客户满意度、技术方案质量、代码贡献、交付周期，还是收入结果？试用期和正式绩效分别看什么？第二，团队组成：FDE 团队和产品、算法、后端、解决方案、销售的协作模式是什么？遇到客户现场问题时，研发支持机制如何？第三，岗位成长：FDE 是否有机会沉淀通用工具、Agent 模板、评测集和平台能力，还是主要做项目制交付？

这些问题能帮你判断岗位是否值得接。好的 FDE 岗位会有清晰技术边界、研发协作和资产沉淀机制；风险较大的岗位往往只有“客户沟通”和“现场救火”，却没有工程成长路径。
