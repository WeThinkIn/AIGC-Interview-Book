<h2 id="p5-1-1">面试问题：PyTorch动态图与静态图区别有哪些，二者各自的适用场景与优势分别是什么？</h2>

**难度评分：⭐⭐ \(2/5\) \| 考察频率：⭐⭐⭐⭐⭐ \(5/5\)**

### 核心区别

|维度|动态图（PyTorch 原生）|静态图（TensorFlow 1\.x/ONNX/TorchScript）|
|---|---|---|
|**定义时机**|运行时逐行定义计算图，边执行边构建|编译期预先定义完整计算图，再执行|
|**灵活性**|极高，支持 Python 控制流（if/for/while）、动态输入形状、实时调试|较低，控制流需特殊语法转换，输入形状需预先指定|
|**调试体验**|原生 Python 调试器支持，可断点查看中间变量|调试困难，需特殊工具查看计算图内部|
|**执行效率**|原生执行较慢，存在 Python 解释器开销|编译优化后执行更快，可进行算子融合、内存复用等全局优化|
|**部署难度**|较高，依赖 Python 环境|较低，可导出为独立模型文件，部署到多种硬件平台|

### 各自优势

- **动态图优势**：

    1. 开发效率高，代码直观易读

    2. 支持复杂动态模型（如 RNN、Transformer 的动态长度输入）

    3. 便于快速原型验证和学术研究

    4. 支持即时错误反馈

- **静态图优势**：

    1. 执行性能高，适合大规模生产部署

    2. 内存占用更低，优化更充分

    3. 跨平台兼容性好，支持移动端、嵌入式设备

    4. 可进行量化、剪枝等模型压缩操作

### 适用场景

- **动态图**：学术研究、模型开发、快速迭代、需要复杂控制流的场景

- **静态图**：生产环境部署、高性能推理、移动端 / 边缘设备部署、大规模分布式训练

### 补充：PyTorch 的混合方案

PyTorch 2\.0 引入了`torch.compile`，可以将动态图代码自动编译为静态图执行，兼顾开发效率和运行性能。

<h2 id="p5-1-2">面试问题：PyTorch中Tensor的核心特性有哪些？Tensor与Numpy数组的本质差异是什么？</h2>

**难度评分：⭐⭐ \(2/5\) \| 考察频率：⭐⭐⭐⭐⭐ \(5/5\)**

### Tensor 的核心特性

1. **多维数组结构**：支持任意维度的数值数组，是 PyTorch 中所有数据的基本表示形式

2. **GPU 加速支持**：可无缝在 CPU 和 GPU 之间转移，利用 CUDA 进行并行计算

3. **自动求导支持**：通过`requires_grad=True`开启梯度追踪，自动构建计算图

4. **广播机制**：支持不同形状张量之间的算术运算，自动扩展维度

5. **丰富的操作集**：提供数百种张量操作，包括数学运算、线性代数、索引切片等

6. **内存共享**：与 Numpy 数组可相互转换并共享底层内存（CPU 上）

7. **数据类型支持**：支持多种数值类型（float32/float64/int32/int64 等）和布尔类型

### Tensor 与 Numpy 数组的本质差异

1. **硬件支持**：

    - Numpy 数组仅支持 CPU 计算

    - Tensor 原生支持 CPU 和 GPU 计算，可通过`.to(device)`轻松切换

2. **自动求导**：

    - Numpy 数组无自动求导功能，需手动实现反向传播

    - Tensor 内置 Autograd 系统，自动追踪计算过程并计算梯度

3. **计算图集成**：

    - Numpy 操作不参与计算图构建

    - Tensor 操作会被记录到计算图中，用于反向传播

4. **分布式支持**：

    - Numpy 无原生分布式计算支持

    - Tensor 支持分布式训练，可在多 GPU / 多节点间通信

5. **部署能力**：

    - Numpy 数组难以直接部署到生产环境

    - Tensor 可导出为 TorchScript/ONNX 格式，部署到多种平台

### 补充：转换注意事项

- CPU 上的 Tensor 与 Numpy 数组共享内存，修改其中一个会影响另一个

- GPU 上的 Tensor 转换为 Numpy 数组时，必须先转移到 CPU

<h2 id="p5-1-3">面试问题：介绍一下PyTorch数据加载流程，Dataset与DataLoader的核心原理、使用方法及配合逻辑是什么？</h2>

**难度评分：⭐⭐⭐ \(3/5\) \| 考察频率：⭐⭐⭐⭐⭐ \(5/5\)**

### PyTorch 数据加载整体流程

```Plain Text
原始数据 → Dataset封装 → DataLoader调度 → 批量数据 → 模型输入
```

### Dataset 核心原理与使用方法

1. **核心作用**：定义如何读取单个样本及其标签，是数据加载的基础

2. **两种主要类型**：

    - `torch.utils.data.Dataset`：抽象基类，需自定义实现`__len__`和`__getitem__`方法

    - `torch.utils.data.IterableDataset`：适用于流式数据或无法预先确定长度的数据集

3. **自定义 Dataset 标准实现**：

    ```python
    class CustomDataset(Dataset):
        def __init__(self, data_path, transform=None):
            # 初始化数据路径、标签列表、数据变换等
            self.data = ...
            self.labels = ...
            self.transform = transform
        
        def __len__(self):
            # 返回数据集总样本数
            return len(self.data)
        
        def __getitem__(self, idx):
            # 根据索引读取单个样本和标签
            sample = self.data[idx]
            label = self.labels[idx]
            
            if self.transform:
                sample = self.transform(sample)
            
            return sample, label
    ```

### DataLoader 核心原理与使用方法

1. **核心作用**：对 Dataset 进行封装，提供批量加载、多进程加速、数据打乱、预取等功能

2. **关键参数**：

    - `dataset`：要加载的 Dataset 对象

    - `batch_size`：每个批次的样本数量

    - `shuffle`：是否在每个 epoch 开始时打乱数据

    - `num_workers`：用于数据加载的子进程数量

    - `pin_memory`：是否将数据加载到 CUDA 固定内存，加速 GPU 传输

    - `drop_last`：是否丢弃最后一个不完整的批次

    - `collate_fn`：自定义如何将多个样本组合成一个批次

3. **核心工作流程**：

    1. 根据`shuffle`参数生成样本索引序列

    2. 将索引划分为多个批次

    3. 启动`num_workers`个子进程，并行从 Dataset 中读取样本

    4. 使用`collate_fn`将读取的样本组合成批次

    5. 将批次数据传递给模型

### Dataset 与 DataLoader 的配合逻辑

1. Dataset 负责 \\*\\*"如何读单个样本"**，DataLoader 负责**"如何批量高效读取"\\*\\*

2. DataLoader 不直接读取数据，而是通过调用 Dataset 的`__getitem__`方法获取单个样本

3. DataLoader 通过多进程并行化数据加载，掩盖 I/O 延迟，提高训练效率

4. DataLoader 提供的批量数据可以直接输入到模型中进行前向传播

### 工程最佳实践

- 对于图像数据，使用`torchvision.datasets`中的预定义 Dataset

- 数据预处理（如归一化、数据增强）应放在 Dataset 的`transform`中

- `num_workers`通常设置为 CPU 核心数的一半或与 GPU 数量相等

- 训练时开启`shuffle=True`，验证和测试时设置为`False`

- 使用 GPU 时开启`pin_memory=True`，加速数据从 CPU 到 GPU 的传输

<h2 id="p5-1-4">面试问题：PyTorch自动求导（Autograd）的核心原理是什么？详解计算图构建与反向传播的完整过程？</h2>

**难度评分：⭐⭐⭐⭐ \(4/5\) \| 考察频率：⭐⭐⭐⭐⭐ \(5/5\)**

### Autograd 核心原理

PyTorch Autograd 系统基于**动态计算图**实现自动微分。它会记录所有对张量的操作，构建一个有向无环图 \(DAG\)，其中：

- **节点**：表示张量（Tensor）

- **边**：表示张量之间的函数关系（操作）

- **叶子节点**：用户创建的、不依赖其他张量的张量（如模型参数、输入数据）

- **根节点**：最终的输出张量（如损失函数）

当调用`loss.backward()`时，Autograd 会从根节点开始，沿着计算图反向传播，应用链式法则计算每个叶子节点的梯度，并将结果存储在张量的`.grad`属性中。

### 计算图构建过程（前向传播）

1. 当对一个`requires_grad=True`的张量执行操作时，PyTorch 会：

    - 创建一个`Function`对象，表示该操作

    - 将输入张量的`grad_fn`属性设置为该 Function 对象

    - 执行操作，生成输出张量

    - 将输出张量的`grad_fn`属性设置为该 Function 对象

2. 所有操作都会被记录下来，形成一个完整的计算图

3. 计算图是动态的，每次前向传播都会重新构建一个新的计算图

### 反向传播完整过程

1. **启动反向传播**：调用`loss.backward()`，其中`loss`是一个标量张量

2. **初始化梯度**：根节点（loss）的梯度初始化为 1\.0

3. **梯度反向传递**：

    - 从根节点开始，遍历计算图的所有节点

    - 对于每个节点，调用其`grad_fn`的`backward`方法

    - 该方法接收上游梯度作为输入，计算当前节点对输入的梯度

    - 将计算得到的梯度传递给输入节点

4. **累积梯度**：对于叶子节点，将计算得到的梯度累加到其`.grad`属性中

5. **计算图释放**：默认情况下，反向传播完成后计算图会被释放，以节省内存

### 关键概念

1. **`requires_grad`**：布尔值，指示是否需要计算该张量的梯度

2. **`grad_fn`**：指向创建该张量的 Function 对象，用于反向传播

3. **`grad`**：存储该张量的梯度值，初始为 None

4. **`retain_graph=True`**：在`backward()`中设置此参数，可保留计算图，允许多次反向传播

5. **`torch.no_grad()`**：上下文管理器，在其内部执行的操作不会被记录到计算图中

### 数学表示

对于函数 $y = f(x)$，链式法则表示为：
$\frac{\partial L}{\partial x} = \frac{\partial L}{\partial y} \cdot \frac{\partial y}{\partial x}$

其中 $L$ 是损失函数，$\frac{\partial L}{\partial y}$ 是上游梯度，$\frac{\partial y}{\partial x}$ 是当前操作的梯度。

### 工程注意事项

- 只有标量张量才能调用`backward()`，向量或矩阵需要指定`gradient`参数

- 每次反向传播前需要手动清零梯度（`optimizer.zero_grad()`），否则梯度会累积

- 对于不需要梯度的张量，设置`requires_grad=False`可以节省内存和计算资源

<h2 id="p5-1-5">面试问题：model.train、model.eval 与 torch.no_grad 区别？</h2>

**难度评分：⭐⭐ \(2/5\) \| 考察频率：⭐⭐⭐⭐⭐ \(5/5\)**

### 核心区别总览

|特性|`model.train()`|`model.eval()`|`torch.no_grad()`|
|---|---|---|---|
|**作用对象**|整个模型及其所有子模块|整个模型及其所有子模块|上下文内的所有张量操作|
|**影响的层**|Dropout、BatchNorm、LayerNorm 等|Dropout、BatchNorm、LayerNorm 等|所有层|
|**梯度计算**|开启梯度计算|开启梯度计算|关闭梯度计算|
|**计算图构建**|构建完整计算图|构建完整计算图|不构建计算图|
|**内存占用**|高（存储中间变量用于反向传播）|高|低（不存储中间变量）|
|**执行速度**|正常|正常|更快|

### 详细说明

1. **`model.train()`**：

    - 将模型设置为训练模式

    - Dropout 层会随机丢弃部分神经元

    - BatchNorm 层会使用当前批次的均值和方差进行归一化，并更新运行均值和方差

    - LayerNorm 等其他归一化层也会使用训练模式的行为

    - 必须在训练阶段调用，否则模型无法正确学习

2. **`model.eval()`**：

    - 将模型设置为评估模式

    - Dropout 层不会丢弃任何神经元，所有神经元都参与计算

    - BatchNorm 层会使用训练过程中累积的运行均值和方差进行归一化，不再更新

    - 其他归一化层也会使用评估模式的行为

    - 必须在验证和测试阶段调用，否则会导致评估结果不准确

3. **`torch.no_grad()`**：

    - 上下文管理器，在其内部执行的所有操作都不会计算梯度

    - 不会构建计算图，节省内存和计算资源

    - 不影响模型的训练 / 评估模式（即不会改变 Dropout 和 BatchNorm 的行为）

    - 主要用于推理阶段，加速模型执行并减少内存占用

### 正确使用范式

```python
# 训练阶段
model.train()
for batch in train_loader:
    optimizer.zero_grad()
    outputs = model(inputs)
    loss = criterion(outputs, labels)
    loss.backward()
    optimizer.step()

# 验证/测试阶段
model.eval()
with torch.no_grad():
    for batch in val_loader:
        outputs = model(inputs)
        loss = criterion(outputs, labels)
        # 计算评估指标
```

### 常见误区

- 只使用`torch.no_grad()`而不调用`model.eval()`：会导致 Dropout 和 BatchNorm 仍处于训练模式，评估结果不准确

- 只调用`model.eval()`而不使用`torch.no_grad()`：会浪费内存和计算资源，因为仍然会构建计算图

- 验证结束后忘记调用`model.train()`：会导致下一轮训练时模型仍处于评估模式，无法正确学习

<h2 id="p5-1-6">面试问题：PyTorch分布式训练的主流方式有哪些，请详细介绍一下。</h2>

**难度评分：⭐⭐⭐⭐ \(4/5\) \| 考察频率：⭐⭐⭐⭐ \(4/5\)**

### PyTorch 分布式训练主流方式总览

PyTorch 提供了多种分布式训练方案，主要分为：

1. **数据并行（Data Parallelism）**：最常用，将数据划分到多个 GPU / 节点上

2. **模型并行（Model Parallelism）**：将模型划分到多个 GPU / 节点上

3. **混合并行（Hybrid Parallelism）**：结合数据并行和模型并行

### 1\. 数据并行

数据并行是最常用的分布式训练方式，适用于模型可以完全放入单个 GPU 的情况。

#### \(1\) `nn.DataParallel`（DP）

- **原理**：单进程多线程，在主进程中复制模型到多个 GPU，将输入数据划分到各个 GPU，分别进行前向传播，然后在主 GPU 上汇总梯度并更新参数

- **优点**：使用简单，只需一行代码修改

- **缺点**：

    - 单进程瓶颈，主 GPU 负载过重

    - 不支持多节点训练

    - 效率较低，GPU 利用率不高

- **使用方法**：

    ```python
    model = nn.DataParallel(model, device_ids=[0, 1, 2, 3])
    ```

#### \(2\) `DistributedDataParallel`（DDP）

- **原理**：多进程多 GPU，每个 GPU 对应一个独立的进程，每个进程都有完整的模型副本和自己的优化器。数据被划分到各个进程，分别进行前向和反向传播，然后通过 Ring Allreduce 算法在所有进程间同步梯度，最后每个进程独立更新参数

- **优点**：

    - 效率高，GPU 利用率接近线性

    - 支持多节点训练

    - 没有单进程瓶颈

- **缺点**：使用相对复杂，需要启动多个进程

- **核心组件**：

    - `torch.distributed`：分布式通信后端，支持 NCCL（GPU）、Gloo（CPU）、MPI

    - `torch.nn.parallel.DistributedDataParallel`：分布式模型包装器

- **使用流程**：

    1. 初始化进程组：`dist.init_process_group(backend='nccl')`

    2. 设置当前进程使用的 GPU：`torch.cuda.set_device(local_rank)`

    3. 包装模型：`model = DDP(model, device_ids=[local_rank])`

    4. 使用`DistributedSampler`划分数据

    5. 启动训练：`torchrun --nproc_per_node=4 train.py`

### 2\. 模型并行

模型并行适用于模型太大，无法放入单个 GPU 的情况。

#### \(1\) 手动模型并行

- **原理**：手动将模型的不同层分配到不同的 GPU 上

- **优点**：灵活，可以根据模型结构进行优化

- **缺点**：需要手动编写代码，容易出错，效率较低

- **使用示例**：

    ```python
    class ModelParallelModel(nn.Module):
        def __init__(self):
            super().__init__()
            self.layer1 = nn.Linear(1000, 500).to('cuda:0')
            self.layer2 = nn.Linear(500, 100).to('cuda:1')
        
        def forward(self, x):
            x = self.layer1(x.to('cuda:0'))
            x = self.layer2(x.to('cuda:1'))
            return x
    ```

#### \(2\) `torch.distributed.pipeline.sync.Pipe`

- **原理**：流水线并行，将模型划分为多个阶段，每个阶段放在不同的 GPU 上，输入数据也被划分为多个微批次，通过流水线方式执行

- **优点**：提高 GPU 利用率，减少空闲时间

- **缺点**：存在流水线气泡，需要调整微批次大小

- **适用场景**：超大规模模型训练（如 GPT、LLaMA）

### 3\. 混合并行

混合并行结合了数据并行和模型并行，适用于超大规模模型训练。

#### \(1\) 3D 并行

- **数据并行**：在数据维度上划分

- **张量并行**：在张量维度上划分（如将线性层的权重矩阵按行或列划分）

- **流水线并行**：在层维度上划分

- **代表框架**：Megatron\-LM、DeepSpeed

#### \(2\) ZeRO 优化

- **原理**：零冗余优化，将模型参数、梯度和优化器状态划分到不同的 GPU 上，每个 GPU 只存储一部分

- **三个级别**：

    - ZeRO\-1：划分优化器状态

    - ZeRO\-2：划分优化器状态和梯度

    - ZeRO\-3：划分优化器状态、梯度和模型参数

- **优点**：大幅减少内存占用，可以训练更大的模型

- **代表框架**：DeepSpeed、FairScale

### 各方式对比与适用场景

|方式|适用场景|优点|缺点|
|---|---|---|---|
|DP|单节点、少量 GPU、快速原型|使用简单|效率低、不支持多节点|
|DDP|单节点 / 多节点、模型可放入单个 GPU|效率高、支持多节点|使用相对复杂|
|手动模型并行|模型略大于单个 GPU|灵活|效率低、易出错|
|流水线并行|超大规模模型|提高 GPU 利用率|存在流水线气泡|
|3D 并行 \+ ZeRO|千亿 / 万亿参数大模型|可训练超大规模模型|复杂度高、调试困难|

### 工程最佳实践

- 优先使用 DDP 进行数据并行训练

- 对于超大规模模型，使用 DeepSpeed 或 Megatron\-LM

- 使用 NCCL 作为 GPU 通信后端，Gloo 作为 CPU 通信后端

- 合理设置批次大小和学习率，分布式训练时学习率应与 GPU 数量成正比

- 使用`torchrun`启动分布式训练，替代旧的`distributed.launch`
