<h1 id="41.如何构建大语言模型评估体系？">41.如何构建大语言模型评估体系？</h1>

<h2 id="面试问题：为什么要评估大语言模型？评估最终要回答哪些问题？">面试问题：为什么要评估大语言模型？评估最终要回答哪些问题？</h2>

**难度评分：⭐⭐ (2/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

##### 1. 为什么大语言模型必须被系统评估？

评估大语言模型的主要目的，不是证明模型在某个榜单上分数高，而是判断它在目标场景中是否 **有用、可靠、安全、稳定、可部署**。大语言模型通常具备开放生成、多任务泛化和工具调用能力，输出空间远比传统分类模型更大，因此训练 loss、reward score 或单一 benchmark 分数都不足以说明真实可用性。

一套评估要回答五类问题：

- **能力问题**：模型是否具备语言理解、知识问答、推理、代码、数学、长上下文、工具调用等目标能力。
- **质量问题**：回答是否相关、准确、完整、连贯、有帮助，是否能遵循格式、风格和约束。
- **风险问题**：是否存在幻觉、越狱、隐私泄露、偏见、有害输出、版权和合规风险。
- **工程问题**：延迟、吞吐、显存、并发能力、可用性和 token 成本是否可接受。
- **业务问题**：模型是否提升解决率、用户满意度、工作效率，是否降低人工接管率和运营成本。

##### 2. 评估结果应该用于什么决策？

大模型评估不仅用于“打分”，更重要的是服务研发和上线决策：

- 比较不同模型、不同 checkpoint、不同微调方案和不同提示方案；
- 判断微调、后训练或数据配比是否真正提升目标能力；
- 发现能力退化、格式不稳定、安全边界破坏和事实性问题；
- 决定模型是否可以进入灰度（先让少量真实用户试用新版本，确认没问题后再逐步扩大流量）、A/B 测试（50% 用户使用旧模型 A；50% 用户使用新模型 B）或正式上线；
- 为下一轮数据构造、训练策略、RAG 设计和安全策略提供依据。

因此，评估体系可以看作模型研发、产品上线和风险治理之间的反馈链路，而不是一次性的排行榜比较。

> “大模型为什么要评估”应按 **能力验证、风险识别、工程部署、业务反馈链路** 四层展开，并说明评估结果最终服务于模型选型、训练迭代和上线决策。

<h2 id="面试问题：如何设计多维评估矩阵？不同类型模型的评估重点有什么差异？">面试问题：如何设计多维评估矩阵？不同类型模型的评估重点有什么差异？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

##### 1. 什么是多维评估矩阵？

多维评估矩阵是指围绕模型定位，把能力、风险、成本和业务效果拆成互相独立的评估维度。这样可以避免一个分数掩盖关键短板。例如，一个模型知识问答分数高，不代表数学推理强；生成流畅，不代表事实准确；指令遵循好，也不代表长上下文可靠。

常见维度包括：

- **基础能力**：语言理解、文本生成、知识问答、推理、数学、代码。
- **交互能力**：指令遵循、多轮对话、结构化输出、函数调用、工具使用。
- **长上下文能力**：长文档理解、信息定位、跨段落整合、长对话一致性。
- **事实与可靠性**：事实准确性、幻觉率、引用准确性、不确定性表达。
- **安全与合规**：有害性、越狱、偏见、隐私泄露、版权和行业合规。
- **工程性能**：延迟、吞吐、显存、并发、可用性、推理成本。
- **业务效果**：任务完成率、问题解决率、转人工率、用户满意度、人工节省。

##### 2. 不同类型模型的评估重点如何变化？

| 模型 / 系统类型 | 评估重点 |
|---|---|
| 通用基座模型 | 语言理解、知识、推理、数学、代码、长上下文、通用 benchmark |
| 对话助手模型 | 指令遵循、多轮对话、帮助性、自然度、安全对齐、拒答边界 |
| 推理模型 | 数学、代码、逻辑推理、pass@1/pass@k、推理稳定性、reasoning token 成本 |
| 代码模型 | 代码生成、补全、修复、单元测试通过率、编译成功率、真实代码修复任务 |
| 行业模型 | 领域知识、事实准确性、合规性、专家评估、真实业务任务完成率 |
| 长上下文模型 | 长文档检索、跨段落推理、needle test、lost in the middle、有效上下文长度 |
| RAG 系统 | 检索召回、上下文相关性、答案忠实性、引用准确性、拒答边界 |
| 部署模型 | 延迟、吞吐、显存、并发、稳定性、成本和监控能力 |

##### 3. 为什么评估维度之间不能互相替代？

不同维度观察的是不同风险面。通用能力高但安全性差，不能直接上线；安全拒答强但过度拒答，业务体验会变差；推理准确率高但 token 成本过高，生产成本可能不可接受；RAG 答案看似正确但引用不忠实，会带来合规和信任风险。

成熟评估体系应该同时看 **目标能力是否提升** 和 **非目标能力是否退化**。这也是微调、后训练和模型升级后必须做回归评估的原因。

<h2 id="面试问题：离线基准、自动评估、人工评估、LLM-as-a-Judge 和线上 A/B 分别解决什么问题？">面试问题：离线基准、自动评估、人工评估、LLM-as-a-Judge 和线上 A/B 分别解决什么问题？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

##### 1. 不同评估方式分别适合解决什么问题？

| 评估方式 | 主要解决的问题 | 优点 | 局限 |
|---|---|---|---|
| 离线 benchmark | 快速比较模型在标准任务上的能力，如知识、数学、代码、推理 | 成本低、可重复、便于横向比较 | 可能被污染，和真实业务分布不一致 |
| 自动评估 | 用程序化指标大规模评测分类、抽取、代码、数学、结构化输出等任务 | 快速、稳定、适合回归测试 | 难以完整判断开放式回答的帮助性和细微质量 |
| 人工评估 | 判断开放式回答是否有用、准确、自然、安全，尤其适合高风险任务 | 可信度高，能发现细节问题 | 成本高、速度慢、主观性强 |
| LLM-as-a-Judge | 批量评估开放式回答、偏好比较、质量初筛 | 效率高、维度灵活、成本较低 | 可能有位置偏差、长度偏好、同源偏置和评审模型偏见 |
| 线上 A/B 测试 | 验证模型在真实用户和业务流程中的效果 | 最接近真实业务价值 | 成本高、有风险，必须先通过质量和安全门槛 |

##### 2. 为什么要组合多种评估方式？

单一评估方式容易产生盲区。自动评估可以高频运行，但很难判断开放回答是否真正有帮助；人工评估质量高，但不适合每天大规模回归；LLM-as-a-Judge 效率高，但需要校准；线上 A/B 最接近真实应用效果，但不能在未通过安全评估时直接放量。

典型工程流程是：

```text
离线 benchmark / 自动评估
→ LLM-as-a-Judge 批量初筛
→ 人工偏好评估和专家抽检
→ 安全红队测试
→ 小流量线上 A/B
→ 线上反馈回流
```

##### 3. 线上 A/B 能否替代离线评估？

不能。线上 A/B 只能在模型已经通过离线质量、安全和业务门槛后进行。对于医疗、法律、金融、政务、企业客服等场景，直接把未充分测试的模型放到线上，会放大安全、合规和声誉风险。正确做法是先用离线集和红队测试排除明显问题，再做小流量灰度，并设置人工接管、报警和回滚机制。

<h1 id="42.如何评估大语言模型的生成质量与评估指标？">42.如何评估大语言模型的生成质量与评估指标？</h1>

<h2 id="面试问题：为什么生成质量不能只看流畅度？如何评估相关性、事实性、连贯性和有用性？">面试问题：为什么生成质量不能只看流畅度？如何评估相关性、事实性、连贯性和有用性？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

##### 1. 为什么不能只看流畅度？

大语言模型很擅长生成自然、顺滑、结构清楚的文本，但流畅并不等于正确。一个回答可能语言表达自然，却存在答非所问、事实错误、逻辑矛盾、遗漏关键信息、引用编造或对用户没有实际帮助等问题。

因此，生成质量要从多个维度综合评估：

- **流畅性**：语言是否自然、通顺、没有明显语病。
- **相关性**：是否真正回应用户问题，而不是空泛展开。
- **事实性**：内容是否准确，是否存在幻觉、编造或错误引用。
- **忠实性 / Groundedness**：回答是否基于给定资料、检索结果或上下文。
- **完整性**：是否覆盖问题的关键约束、边界条件和必要步骤。
- **连贯性 / 一致性**：前后是否矛盾，逻辑是否自洽。
- **信息量**：是否有实质内容，是否空泛、重复或套话。
- **有用性**：是否能帮助用户做决策、完成任务或理解问题。
- **可读性**：结构是否清晰，层次是否便于阅读。
- **创造性**：开放创作任务中是否新颖、有启发。
- **可控性**：是否遵守风格、格式、长度、角色和安全约束。

##### 2. 不同任务的质量权重有什么差异？

| 任务类型 | 评估重点 |
|---|---|
| 客服问答 | 相关性、事实性、有用性、拒答边界、话术一致性 |
| 创意写作 | 流畅性、创造性、风格一致性、表达多样性 |
| 摘要任务 | 忠实性、完整性、压缩质量、信息覆盖、无新增事实 |
| RAG 问答 | 引用正确性、groundedness、事实性、上下文相关性、拒答能力 |
| 法律 / 医疗 / 金融 | 事实性、合规性、边界说明、审慎表达、专家审核 |
| 结构化输出 | JSON / SQL / 表格可解析率、schema 通过率、字段准确率 |

核心判断标准是：生成内容是否 **自然、切题、准确、连贯、有用，并符合用户给出的约束和业务场景**。

<h2 id="面试问题：开放式任务和有参考答案任务有什么区别？自动指标应该如何选择？">面试问题：开放式任务和有参考答案任务有什么区别？自动指标应该如何选择？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

##### 1. 开放式任务和有参考答案任务的本质区别是什么？

有参考答案任务通常存在明确标准答案或可程序化验证的结果，适合用自动指标进行大规模评估。开放式生成任务没有唯一正确答案，更依赖人工偏好、成对比较、LLM-as-a-Judge 和业务指标。

| 任务类型 | 典型任务 | 推荐评估方式 |
|---|---|---|
| 有参考答案任务 | 翻译、摘要、抽取、标准问答、分类 | BLEU、ROUGE、BERTScore、EM、F1、Accuracy，并结合人工评估 |
| 可执行 / 可验证任务 | 代码生成、数学题、结构化输出 | 单元测试、编译检查、答案校验器、schema 校验、Pass@k |
| 开放式生成任务 | 对话、写作、头脑风暴、开放问答 | 人工评估、成对比较、LLM-as-a-Judge、业务指标 |
| RAG / 基于资料问答 | 企业知识库问答、文档问答 | 检索召回、引用准确性、groundedness、人工核查 |
| 高风险专业任务 | 医疗、法律、金融、政务 | 专家评审、事实核查、安全合规评估 |

##### 2. 自动指标如何选择？

| 任务类型 | 推荐指标 / 方法 | 注意事项 |
|---|---|---|
| 分类任务 | Accuracy、Precision、Recall、F1 | 类别不均衡时不能只看 Accuracy |
| 抽取任务 | EM、token-level F1、字段准确率 | 边界错误和等价表达要单独分析 |
| 选择题 / 考试题 | Accuracy | 要防止 benchmark contamination |
| 翻译任务 | BLEU、chrF、COMET、人工评估 | BLEU 不足以反映语义和流畅性 |
| 摘要任务 | ROUGE、BERTScore、事实一致性检查 | ROUGE 高不代表摘要事实正确 |
| 代码生成 | Pass@1、Pass@k、单元测试通过率 | 测试用例质量决定指标可信度 |
| 数学推理 | Accuracy、Pass@k、答案校验器 | 要区分最终答案正确和过程正确 |
| RAG 问答 | Recall@k、Citation Accuracy、Faithfulness | 必须判断答案是否基于检索内容 |
| 对话助手 | Win Rate、人工偏好、LLM-as-a-Judge、安全评估 | 需要多轮、边界和安全测试 |
| 线上业务 | A/B、满意度、解决率、转人工率、成本 | 离线指标不一定等于线上效果 |

<h2 id="面试问题：为什么 BLEU、ROUGE、准确率、困惑度和 Win Rate 都不能单独代表真实能力？">面试问题：为什么 BLEU、ROUGE、准确率、困惑度和 Win Rate 都不能单独代表真实能力？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

##### 1. 大模型时代常用指标有什么局限？

Exact Match、F1、Accuracy、Pass@k、Win Rate、Elo Rating、LLM-as-a-Judge 分数同样不能单独代表真实能力。

- **Accuracy** 适合选择题，但无法体现生成质量、拒答能力和成本。
- **Pass@k** 适合代码和数学多采样，但会受采样次数、温度、测试用例质量影响。
- **Win Rate** 适合偏好比较，但可能受评审偏好、答案长度和展示顺序影响。
- **Elo Rating** 能做模型相对排名，但不直接说明模型在哪些任务可靠。
- **LLM-as-a-Judge** 可扩展性强，但需要校准评审偏差。

真实能力还包括事实性、推理能力、指令遵循、长上下文、工具调用、安全性、鲁棒性、生成质量、推理成本和真实用户满意度。因此，大模型评估必须使用 **多指标、多维度、多方法结合** 的体系。

<h2 id="面试问题：人工评估和 LLM-as-a-Judge 应该如何设计？如何控制主观性和评审偏差？">面试问题：人工评估和 LLM-as-a-Judge 应该如何设计？如何控制主观性和评审偏差？</h2>

**难度评分：⭐⭐⭐⭐ (4/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

##### 1. 人工评估应该评什么？

人工评估常见维度包括：

- 正确性、事实性、相关性、完整性；
- 有用性、可执行性、清晰度；
- 流畅性、逻辑性、可读性；
- 安全性、礼貌性、合规性；
- 是否遵循指令、格式、长度和角色要求；
- 是否符合目标用户和业务场景的偏好。

常见形式包括打分制、成对比较、排序法和错误标注。成对比较通常比绝对打分更稳定，因为标注员只需要判断 A、B 哪个更好，而不是给出抽象分数。

##### 2. 如何提高人工评估可靠性？

重点在于把主观判断尽量结构化：

- 制定清晰评分 rubrics，给出正例、反例和边界案例；
- 对标注员进行一致性校准；
- 使用多标注员投票或仲裁机制；
- 隐藏模型名称，避免品牌偏见；
- 混入金标样本，监控标注质量；
- 计算标注一致性，例如一致率或 Cohen's kappa；
- 对争议样本做专家复核。

##### 3. LLM-as-a-Judge 如何控制偏差？

LLM-as-a-Judge 容易出现长度偏好、位置偏差、同源偏置、过度依赖提示词和评审模型自身偏见。常见控制手段包括：

- 随机交换 A/B 答案顺序，统计是否存在位置偏差；
- 要求 judge 按维度评分，而不是只给总体偏好；
- 明确惩罚幻觉、答非所问、冗长和不遵循格式；
- 对开放题使用参考资料或标准答案约束 judge；
- 使用多个 judge 模型或人工抽检校准；
- 报告置信区间和胜率差异，而不是只给单点分数。

LLM-as-a-Judge 适合作为批量初筛和辅助评估，但不应完全替代人工评估，尤其不能替代高风险场景的专家审核。

<h1 id="43.如何评估知识掌握、事实准确性与幻觉？">43.如何评估知识掌握、事实准确性与幻觉？</h1>

<h2 id="面试问题：如何评估模型的知识掌握程度和事实准确性？">面试问题：如何评估模型的知识掌握程度和事实准确性？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

##### 1. 如何评估知识掌握程度？

知识掌握评估关注模型是否知道某个领域的概念、事实、规则和关系。常见方法包括：

- **闭卷问答**：不提供外部资料，直接让模型回答，测试参数化知识。
- **多选题测试**：如 MMLU、C-Eval、CMMLU，用标准答案评估学科知识。
- **开放式问答**：考察模型能否生成准确、完整、可解释的答案。
- **专业领域问答**：医学、法律、金融、科学等需要专家评估。
- **时间敏感问题测试**：检查模型是否混淆过时信息和最新信息。

闭卷知识评估可以测试模型参数中的知识，但不能证明模型适合实时场景。涉及新闻、政策、产品版本、价格、论文和法规时，应优先结合检索或工具验证。

##### 2. 如何评估事实准确性？

事实准确性评估重点关注：

- 回答是否包含错误事实；
- 是否编造不存在的概念、论文、人物、数据、链接或法规；
- 是否能承认不知道；
- 是否给出可靠来源；
- 是否区分事实、观点和推测；
- 是否能保持与给定上下文一致。

常用方式包括人工事实核查、与权威知识库比对、检索增强评估、TruthfulQA 等事实性基准，以及 hallucination rate 统计。高风险领域还需要专家审核和可追溯证据。

<h2 id="面试问题：如何将幻觉问题转化为可评估的标签、指标和归因流程？">面试问题：如何将幻觉问题转化为可评估的标签、指标和归因流程？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

##### 1. 幻觉评估标签应该如何设计？

常见做法是把模型回答拆成原子事实声明，再逐条标注。标签可以包括：

- **Unsupported Claim**：答案中有断言，但给定上下文或权威来源无法支持。
- **Contradicted Claim**：答案与上下文、数据库或权威来源直接矛盾。
- **Fabricated Citation**：引用、链接、论文、法规或出处不存在。
- **Citation Mismatch**：引用存在，但不能支持对应结论。
- **Numeric / Date Error**：数值、日期、比例、版本号等关键事实错误。
- **Context Inconsistency**：回答与用户提供材料或对话历史不一致。
- **Overconfident Unknown**：证据不足时仍自信作答，没有澄清、限界或拒答。

每个标签还应记录严重程度、证据来源和错误位置。这样才能区分轻微表述问题、局部事实错误和高风险错误。

##### 3. 幻觉如何度量？

常见指标包括：

- **Hallucination Rate**：含幻觉回答占比。
- **Claim-level Error Rate**：原子事实声明中错误或无证据声明的比例。
- **Unsupported Claim Rate**：无法从证据中支持的断言比例。
- **Faithfulness**：答案是否忠实于给定上下文。
- **Factual Consistency**：回答事实是否与权威来源一致。
- **Citation Accuracy**：引用是否真实存在，是否支持回答内容。
- **Calibration / Refusal Accuracy**：证据不足时是否能表达不确定、澄清或拒答。

回答级幻觉率适合做整体趋势监控，claim-level 指标更适合定位细粒度问题。高风险业务中，引用准确率和专家核查通常比单一自动分数更可靠。

<h2 id="面试问题：如何验证幻觉治理方案是否真的有效？">面试问题：如何验证幻觉治理方案是否真的有效？</h2>

**难度评分：⭐⭐⭐⭐ (4/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

##### 1. 为什么不能只看“看起来更谨慎”？

幻觉治理方案可能让模型回答更保守，但这不一定代表系统更好。模型可能只是更频繁拒答、引用更多文本或输出更模板化，却没有真正提升事实正确性。

因此验证治理效果时，要同时看幻觉率、正确回答率、拒答边界、引用准确性、用户任务成功率和成本。

##### 2. 如何验证模型是否真的降低了幻觉？

应该用同一套固定评估集做治理前后对比：

- 幻觉率是否下降；
- 正确回答率是否保持或提升；
- 拒答是否只发生在缺证据场景；
- 引用是否真实且支持结论；
- 对长文档和多文档场景是否仍然忠实；
- 高风险场景中专家审核是否通过。

降低幻觉和提高帮助性之间存在权衡。模型如果过度拒答，幻觉率可能下降，但业务可用性也会下降。因此应同时监控正确回答率、拒答准确率、误拒率和用户满意度。

##### 3. 治理方案应该如何做消融实验？

常见治理手段包括 RAG、证据约束提示、引用绑定、工具调用、降低随机性、事实性微调和验证器后处理。评估时更重要的是做消融：

- 只加 RAG 是否提升 faithfulness；
- 加 reranker 后正确证据进入上下文的比例是否提高；
- 加引用约束后 Citation Accuracy 是否提高；
- 加拒答策略后漏拒率是否下降、误拒率是否上升；
- 加 verifier 后幻觉率是否下降，但延迟和成本是否可接受。

最终结论不能只说“幻觉少了”，而要说明：在哪类问题上下降、代价是什么、是否牺牲帮助性、是否通过专家抽检和线上 A/B。

<h2 id="面试问题：如何评估语言理解和问答能力？">面试问题：如何评估语言理解和问答能力？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

##### 1. 语言理解能力评估哪些任务？

常见语言理解任务包括：

- 文本分类；
- 情感分析；
- 阅读理解；
- 自然语言推理（NLI）；
- 信息抽取；
- 指代消解；
- 关键词识别；
- 语义匹配。

常见指标包括 Accuracy、Precision、Recall、F1、Exact Match。对于类别不均衡任务，要重点看 Precision、Recall 和 F1，而不能只看 Accuracy。

##### 2. 问答能力如何评估？

问答能力可分为多种类型：

- **开放域问答**：模型依赖参数化知识或检索回答。
- **阅读理解问答**：答案应来自给定文本。
- **多跳问答**：需要整合多个证据片段。
- **专业领域问答**：需要领域知识和专家核查。
- **多轮问答**：需要理解历史对话、指代和上下文。

常见数据集包括 SQuAD、Natural Questions、TriviaQA、HotpotQA、WebQuestions、DROP。评估时应区分答案正确性、证据支持、推理过程和拒答边界。对于开放域问答，EM/F1 只能辅助，事实核查更重要。

<h2 id="面试问题：Benchmark 评估有哪些局限？如何避免评测集污染和刷榜误导？">面试问题：Benchmark 评估有哪些局限？如何避免评测集污染和刷榜误导？</h2>

**难度评分：⭐⭐⭐⭐ (4/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

##### 1. Benchmark 的主要局限是什么？

Benchmark 常见问题包括：

- **数据污染**：训练数据中可能包含测试题，导致分数虚高。
- **刷榜问题**：模型或提示策略专门针对 benchmark 优化。
- **分布不一致**：公开评测和真实业务输入差异很大。
- **开放生成难以客观打分**：同一个问题可能有多个好答案。
- **Prompt 敏感**：提示模板、选项顺序、解码参数会明显影响结果。
- **语言和领域覆盖不足**：中文、低资源语言、垂直行业测试不足。
- **高分不等于体验好**：榜单分数无法完整反映安全、成本、稳定性和业务价值。

##### 2. 如何避免评测集污染？

常见做法包括：

- 对训练语料和测试集做 n-gram、embedding 或近重复检测；
- 使用动态构造的新题、私有评测集和时间切分测试集；
- 对公开 benchmark 分数保持谨慎，结合内部评估；
- 固定内部评测集版本，避免模型反复调参后过拟合；
- 记录 prompt、解码参数、模型版本和数据版本；
- 对异常高分样本做人工抽检和错误分析。

##### 3. 如何判断 benchmark 分数是否有实际价值？

应看分数是否能预测真实场景表现。如果 benchmark 提升没有带来用户满意度、任务完成率、安全性或成本改善，就说明该评测与业务目标相关性不足。成熟团队通常会维护公开 benchmark、内部回归集、业务测试集和线上监控四层评估，而不是只依赖公开榜单。

正式报告 benchmark 结果时，还应同时说明模型版本、评测集版本、prompt 模板、解码参数、采样次数、评分脚本和置信区间。否则分数难以复现，也难以判断差异是否来自模型能力本身。

<h1 id="45.如何评估推理、数学、代码与科学能力？">45.如何评估推理、数学、代码与科学能力？</h1>

<h2 id="面试问题：大语言模型的推理能力包括哪些类型？如何判断模型是真的会推理？">面试问题：大语言模型的推理能力包括哪些类型？如何判断模型是真的会推理？</h2>

**难度评分：⭐⭐⭐⭐ (4/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

##### 1. 推理能力包括哪些类型？

常见推理类型包括：

- **常识推理**：理解日常世界规律和隐含常识。
- **逻辑推理**：基于规则、条件和蕴含关系得出结论。
- **数学推理**：完成多步计算、代数、几何、概率等任务。
- **符号推理**：处理抽象符号、形式规则和变量关系。
- **多跳推理**：整合多个证据或中间结论。
- **因果推理**：区分相关性和因果关系。
- **规划推理**：把复杂任务拆成可执行步骤。
- **科学推理**：基于科学概念、证据和机制进行判断。

##### 2. 如何判断模型是真的会推理，而不是记住答案？

可以从以下角度验证：

- 使用新构造或动态生成题目，降低训练集记忆可能性；
- 改写题面、替换数值、改变实体，检查模型是否仍能求解；
- 要求模型解释关键推理步骤，并检查步骤是否可验证；
- 使用对抗样本和陷阱题，测试模型是否机械套模板；
- 比较最终答案正确率和过程正确率；
- 使用工具、单元测试、verifier 或人工审查验证推理链；
- 分析错误类型，区分计算错误、理解错误、逻辑跳步和幻觉。

真正可靠的推理评估不能只看最终答案是否正确，还要看模型是否能在分布变化、题面扰动和长链条任务中保持稳定。

<h2 id="面试问题：如何评估数学推理能力？GSM8K、MATH、AIME 和 Pass@k 分别意味着什么？">面试问题：如何评估数学推理能力？GSM8K、MATH、AIME 和 Pass@k 分别意味着什么？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

##### 1. 常见数学推理基准有哪些？

| Benchmark | 主要特点 |
|---|---|
| GSM8K | 小学数学应用题，重视自然语言理解和多步算术 |
| MATH | 竞赛级数学题，难度更高，覆盖代数、几何、数论等 |
| AIME | 美国数学邀请赛，常用于高难推理模型评估 |
| MathQA | 数学问答和选择题 |
| SVAMP | 数学应用题变体测试，考察鲁棒性 |
| CMATH | 中文数学题 |

##### 2. 数学推理应该看哪些指标？

常见指标包括：

- **最终答案准确率**：答案是否正确。
- **推理步骤正确性**：中间步骤是否成立。
- **解题过程一致性**：过程和最终答案是否匹配。
- **多步推理成功率**：长链条推理是否稳定。
- **Pass@1**：单次生成答案正确率。
- **Pass@k**：采样 k 次中至少一次正确的概率。若一次题目生成多个候选，常用无偏估计形式为 $1-\frac{C\binom{n-c}{k}}{C\binom{n}{k}}$，其中 $n$ 是采样数，$c$ 是正确样本数。

Pass@k 更适合评估模型的潜在解题能力和多采样收益，但如果上线只允许一次回答，Pass@1 更接近真实体验。推理模型还要同时看平均输出长度、reasoning token 成本和延迟。

<h2 id="面试问题：如何评估代码能力和科学推理能力？">面试问题：如何评估代码能力和科学推理能力？</h2>

**难度评分：⭐⭐⭐⭐ (4/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

##### 1. 代码能力如何评估？

常见代码评估基准包括：

| Benchmark | 主要评估内容 |
|---|---|
| HumanEval | Python 函数级代码生成 |
| MBPP | 基础 Python 编程题 |
| CodeContests | 竞赛编程题 |
| APPS | 程序设计和算法题 |
| LiveCodeBench | 更接近实时和真实代码场景 |
| SWE-bench | 基于真实 GitHub issue 的修复能力 |

常见指标包括 Pass@1、Pass@k、单元测试通过率、编译成功率、代码可读性、代码安全性、issue 修复成功率。代码评估的关键是测试用例质量。测试太弱会让错误代码通过，测试太窄又无法代表真实工程。

##### 2. 科学推理如何评估？

常见科学和专业推理基准包括 GPQA、ARC-Challenge、SciQ、PubMedQA、MedQA、BioASQ。

评估重点包括：

- 是否理解科学概念和术语；
- 是否能进行因果分析和机制解释；
- 是否能基于证据推理；
- 是否能识别不充分证据和不确定性；
- 是否避免伪科学、过度推断和错误解释。

科学、医学等专业任务不能只看选择题准确率。开放式回答还需要专家评审、证据核查、引用检查和安全边界评估。

<h1 id="46.如何评估长上下文能力？">46.如何评估长上下文能力？</h1>

<h2 id="面试问题：长上下文能力应该评估哪些任务、基准和指标？">面试问题：长上下文能力应该评估哪些任务、基准和指标？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

##### 1. 长上下文评估关注哪些任务？

长上下文能力不是“上下文窗口标称很长”，而是模型能否稳定利用长输入中的关键信息。常见任务包括：

- 长文档问答；
- 长文档摘要；
- 多文档综合；
- 信息定位；
- 跨段落推理；
- 长对话一致性；
- 前后文矛盾检查；
- 长代码或长表格理解。

##### 2. 常见长上下文基准有哪些？

常见基准包括 LongBench、L-Eval、Needle In A Haystack、RULER、NarrativeQA、Qasper、GovReport、Multi-Doc QA、MRCR 等。

评估指标通常包括：

- 长文档问答准确率；
- 关键信息召回率；
- 长距离依赖推理成功率；
- 摘要忠实性和覆盖率；
- 不同位置插入信息的召回差异；
- 上下文长度增加时的性能衰减；
- 延迟、显存和成本。

<h2 id="面试问题：什么是 Needle In A Haystack 和 lost in the middle？它们分别暴露什么问题？">面试问题：什么是 Needle In A Haystack 和 lost in the middle？它们分别暴露什么问题？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

##### 1. 什么是 Needle In A Haystack？

Needle In A Haystack 是一种长上下文测试方法：在很长的文本中插入一条关键信息，然后要求模型找出这条信息。它用于测试模型是否真的能利用长上下文，而不是只看开头、结尾或局部片段。

评估时通常会改变：

- 文档总长度；
- needle 插入位置；
- needle 的表述形式；
- 干扰信息数量；
- 问题是否需要简单检索或进一步推理。

##### 2. 什么是 lost in the middle？

lost in the middle 指模型对上下文中间位置的信息利用较差。很多模型在开头和结尾附近表现较好，但对中间段落的关键信息召回下降。这说明标称窗口长度不等于有效上下文能力。

长上下文评估应特别关注：

- 是否能记住前文信息；
- 是否能准确定位细节；
- 是否能综合多个位置的信息；
- 是否忽略中间或早期上下文；
- 上下文越长时性能是否下降；
- 长对话中角色、约束和事实是否保持一致。

<h2 id="面试问题：长上下文能力和 RAG 能力有什么区别？如何评估有效上下文长度？">面试问题：长上下文能力和 RAG 能力有什么区别？如何评估有效上下文长度？</h2>

**难度评分：⭐⭐⭐⭐ (4/5) | 考察频率：⭐⭐⭐ (3/5)**

##### 1. 长上下文和 RAG 的区别是什么？

长上下文能力关注模型在一次请求中处理和利用大量输入信息的能力；RAG 能力关注系统是否能从外部知识库检索出相关内容，并基于这些内容生成答案。两者相关但不等价。

- 长上下文强，不代表检索系统能召回正确文档；
- 检索强，不代表模型能忠实生成答案；
- RAG 可以降低上下文长度压力，但会引入召回、排序、切块、引用和拒答问题；
- 长上下文模型仍可能出现 lost in the middle、成本高和延迟高。

##### 2. 什么是有效上下文长度？

有效上下文长度是指模型在某个准确率或召回率门槛下，真正能够稳定利用的信息长度。它比厂商标称的最大窗口更有业务意义。

评估方式包括：

- 在不同上下文长度下测试信息召回和推理准确率；
- 在不同位置插入关键信息，观察位置敏感性；
- 对比短上下文能力是否因扩展窗口而下降；
- 同时记录延迟、显存、吞吐和成本；
- 在真实业务文档上评估，而不是只用合成 needle。

<h1 id="47.如何评估指令遵循、人类偏好与对齐效果？">47.如何评估指令遵循、人类偏好与对齐效果？</h1>

<h2 id="面试问题：如何评估模型是否遵循指令？IFEval、MT-Bench、AlpacaEval 分别评估什么？">面试问题：如何评估模型是否遵循指令？IFEval、MT-Bench、AlpacaEval 分别评估什么？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

##### 1. 指令遵循评估哪些维度？

指令遵循关注模型是否理解并执行用户要求。常见维度包括：

- 是否回答了用户真正的问题；
- 是否遵守输出格式；
- 是否遵守长度限制；
- 是否遵守角色设定；
- 是否执行多约束任务；
- 是否在需要时追问澄清；
- 是否拒绝不该回答的问题；
- 是否避免答非所问和偷换任务。

结构化输出、工具调用、表格、JSON、SQL 等任务应加入程序化校验，因为肉眼看起来正确并不代表系统可解析。

##### 2. 常见指令和偏好基准分别评估什么？

| Benchmark | 主要用途 |
|---|---|
| IFEval | 指令遵循，尤其是格式、长度、关键词、语言等可验证约束 |
| MT-Bench | 多轮对话能力和开放回答质量 |
| AlpacaEval | 开放式指令回答的偏好比较 |
| Arena-Hard | 更高难度的偏好比较和开放任务 |
| Chatbot Arena | 基于人类投票的模型竞技场和 Elo 排名 |

这些基准能反映助手体验，但仍要结合安全、事实性、成本和业务测试。偏好分数高的模型可能只是更会写长答案，不一定更准确。

<h2 id="面试问题：Helpful、Honest、Harmless 和 Fairness 如何落到评估指标上？">面试问题：Helpful、Honest、Harmless 和 Fairness 如何落到评估指标上？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

##### 1. HHH 分别意味着什么？

- **Helpful**：回答是否有帮助，是否解决用户问题，是否具体可执行。
- **Honest**：是否诚实表达不确定性，是否避免编造和过度自信。
- **Harmless**：是否避免有害指导、违法内容、隐私泄露和危险建议。
- **Fairness**：是否避免歧视、刻板印象和不公平差异对待。

##### 2. 如何评估这些目标？

| 目标 | 可观测指标 |
|---|---|
| Helpful | 任务完成率、用户满意度、人工偏好胜率、解决率 |
| Honest | 幻觉率、错误事实率、不确定性表达正确率、引用准确率 |
| Harmless | 有害输出率、越狱成功率、危险请求拒答率、隐私泄露率 |
| Fairness | 不同群体错误率差异、拒答率差异、情感倾向差异、刻板印象比例 |

对齐评估的难点在于这些目标会冲突。过度强调无害性可能导致过度拒答；过度强调帮助性可能导致模型在不知道时强行回答。因此需要同时监控安全性和可用性。

<h2 id="面试问题：RLHF、DPO、GRPO 后的模型应该如何评估？为什么不能只看 reward score？">面试问题：RLHF、DPO、GRPO 后的模型应该如何评估？为什么不能只看 reward score？</h2>

**难度评分：⭐⭐⭐⭐ (4/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

##### 1. 为什么不能只看 reward score？

后训练章节已经说明 reward score 在训练中可能被过度优化。评估章节更关心发布验收：reward score 只是代理指标，不等于真实人类偏好，也不等于真实任务成功。

发布级评估应同时覆盖：

- Win Rate 和人工偏好；
- Reward score 和 KL 是否受控；
- 通用能力 benchmark 是否退化；
- 事实性、幻觉率和引用准确性；
- 安全拒答率、误拒率、漏拒率；
- 生成质量和多轮一致性；
- 平均输出长度、推理 token 成本和延迟；
- 线上 A/B 和真实业务指标。

##### 2. 不同算法评估重点有什么差异？

| 方法 | 评估重点 |
|---|---|
| PPO / RLHF | win rate、KL 是否受控、通用能力和安全性是否退化 |
| DPO | 偏好胜率、人工偏好、偏好数据过拟合风险 |
| GRPO / DAPO 等长推理强化学习方法 | 数学、代码、长推理任务的 pass@1/pass@k、verifier 通过率、推理长度和 token 成本 |
| 安全对齐算法 | 拒答准确率、误拒率、漏拒率、越狱成功率、有害输出率 |
| 领域偏好优化 | 业务成功率、事实性、引用一致性、专家评估、用户满意度 |

评估时必须设置基线，例如 base model、SFT model、上一版线上模型或 prompt-only 方案。没有 baseline，就难以判断对齐是否真的带来收益。

<h2 id="面试问题：如何评估拒答边界和过度拒答？">面试问题：如何评估拒答边界和过度拒答？</h2>

**难度评分：⭐⭐⭐⭐ (4/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

##### 1. 什么是拒答边界？

拒答边界是模型区分“应该回答”和“不应该回答”的能力。安全模型既要拒绝危险、违法、越权、隐私泄露等请求，也要避免对正常教育、科普、合规咨询和安全防御问题过度拒答（over-refusal）。边界样本还要关注不确定性校准（calibration）：模型应能在信息不足或政策边界不清时澄清、限界回答或安全转化，而不是自信误答。

##### 2. 常见拒答指标有哪些？

- **危险请求拒答率**：有害请求中被正确拒绝的比例。
- **漏拒率**：危险请求被错误回答的比例。
- **误拒率**：正常请求被错误拒绝的比例。
- **过度拒答率（over-refusal rate）**：模型过于保守导致可回答问题被拒的比例。
- **安全转化质量**：拒答后是否提供安全替代建议。

好的安全对齐不是“拒绝越多越好”，而是边界清晰、拒答理由合理、校准良好，并且对正常用户保持帮助性。

<h1 id="48.如何评估安全性、偏见与对抗鲁棒性？">48.如何评估安全性、偏见与对抗鲁棒性？</h1>

<h2 id="面试问题：如何系统评估模型的有害性？如何做红队测试？">面试问题：如何系统评估模型的有害性？如何做红队测试？</h2>

**难度评分：⭐⭐⭐⭐ (4/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

##### 1. 有害性评估覆盖哪些风险？

常见安全风险包括：

- 暴力、自残、仇恨、色情内容；
- 违法指导、网络攻击、欺诈和绕过安全机制；
- 生物、化学、武器等危险信息；
- 隐私泄露和个人信息推断；
- 虚假信息生成和操纵；
- 版权、合规和行业监管风险；
- 工具调用场景中的越权操作和数据泄露。

常见基准和测试集包括 RealToxicityPrompts、ToxiGen、AdvBench、HarmBench、SafetyBench、XSTest 等。BBQ、StereoSet、CrowS-Pairs 等更偏向偏见与公平性评估，应单独分析。

##### 2. 红队测试怎么做？

红队测试是主动构造高风险和对抗输入，寻找模型安全策略的薄弱点。流程通常包括：

```text
定义风险分类
→ 构造正常、安全边界和恶意样本
→ 使用多轮诱导、角色扮演、编码、多语言、上下文污染等攻击
→ 记录模型是否拒答、是否泄露、是否给出危险步骤
→ 分析失败案例
→ 回流安全数据、规则和对齐策略
→ 做回归测试
```

红队测试不能只统计“拒答了多少”，还要检查拒答是否准确、解释是否安全、是否给出替代建议，以及是否影响正常请求。

<h2 id="面试问题：如何评估模型偏见与公平性？BBQ、StereoSet、CrowS-Pairs 分别关注什么？">面试问题：如何评估模型偏见与公平性？BBQ、StereoSet、CrowS-Pairs 分别关注什么？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐ (3/5)**

##### 1. 偏见评估关注哪些类型？

常见偏见类型包括：

- 性别偏见；
- 种族或族群偏见；
- 地域偏见；
- 职业偏见；
- 年龄偏见；
- 宗教偏见；
- 政治立场偏见；
- 语言和文化偏见。

常见指标包括不同群体上的错误率差异、有害刻板印象比例、拒答率差异、情感倾向差异、公平性指标等。

##### 2. 常见偏见基准分别关注什么？

| Benchmark | 主要关注点 |
|---|---|
| BBQ | 在含歧义和非歧义场景中测试社会偏见 |
| StereoSet | 语言模型是否偏向刻板印象表达 |
| CrowS-Pairs | 成对句子比较，检测模型对刻板句的偏好 |
| WinoBias | 职业和性别相关的指代消解偏见 |
| BOLD | 生成文本中的群体描述和情感倾向 |

偏见评估需要结合目标地区、语言和业务场景。单一英文基准不能完全代表中文或行业应用中的公平性风险。

<h2 id="面试问题：Jailbreak 和 Prompt Injection 有什么区别？如何评估对抗鲁棒性？">面试问题：Jailbreak 和 Prompt Injection 有什么区别？如何评估对抗鲁棒性？</h2>

**难度评分：⭐⭐⭐⭐ (4/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

##### 1. Jailbreak 和 Prompt Injection 的区别是什么？

**Jailbreak（jailbreak，越狱攻击）** 通常指诱导模型绕过安全策略，输出本应拒绝的有害内容。例如角色扮演、多轮诱导、编码绕过、假设情景等。

**Prompt Injection** 通常发生在模型读取外部内容或接入工具时，攻击者在网页、文档、邮件等外部文本中植入恶意指令，试图覆盖系统指令、窃取信息或诱导错误工具调用。

可概括为：

- Jailbreak 主要攻击模型的安全对齐；
- Prompt Injection 主要攻击模型对指令层级和外部内容的隔离能力；
- 工具增强系统和 RAG 系统中 Prompt Injection 风险更高，因为模型会读取不可信外部内容。

##### 2. 对抗鲁棒性如何评估？

常见攻击方式包括：

- 角色扮演绕过；
- 多语言绕过；
- 编码、拼写和同音扰动；
- 多轮上下文污染；
- 恶意工具调用诱导；
- 系统提示词泄露诱导；
- 检索文档中的 prompt injection。

评估重点包括：

- 模型是否输出有害内容；
- 是否泄露系统提示词、隐私或内部数据；
- 是否忽略安全策略；
- 是否错误调用工具；
- 是否能识别不可信指令；
- 攻击失败后是否仍保持有帮助的安全回答。

<h1 id="49.如何评估 RAG 场景下的大语言模型系统？">49.如何评估 RAG 场景下的大语言模型系统？</h1>

<h2 id="面试问题：RAG 系统需要评估哪些部分？为什么不能只看最终答案？">面试问题：RAG 系统需要评估哪些部分？为什么不能只看最终答案？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

##### 1. RAG 评估为什么要拆层？

RAG 系统通常包含文档处理、检索、排序、上下文构造、生成、引用和拒答等多个环节。最终答案错误可能来自检索没召回、排序错误、切块不好、上下文太长、生成不忠实、引用错误或拒答边界不清。如果只看最终答案，无法定位问题来源。

RAG 评估至少包括：

- **文档与索引评估**：文档是否被正确解析、清洗、切分，metadata 和权限信息是否保留。
- **检索评估**：相关文档是否被召回，排序是否合理，chunk 是否包含关键证据。
- **生成评估**：答案是否正确、相关、完整，并忠实于检索内容。
- **引用评估**：引用是否真实存在，是否支持对应结论。
- **拒答评估**：知识库没有答案或权限不足时，模型是否拒答、澄清或说明不确定。
- **端到端业务评估**：用户问题是否被解决，延迟、成本、满意度和人工接管率是否可接受。

##### 2. RAG 常见失败模式有哪些？

- 解析失败：表格、公式、代码、页眉页脚或 OCR 文本处理错误；
- 切块不合理：关键证据被切散，或者 chunk 过大混入多个主题；
- 召回缺失：正确文档没有被检索出来；
- 排序错误：正确文档排在后面，被截断或未进入上下文；
- 噪声过多：无关上下文干扰生成；
- 忠实性不足：模型加入检索内容没有的信息；
- 引用错误：引用位置不支持答案；
- 过度回答：资料不足时仍然编造结论。

##### 3. 如何构造 RAG 测试集？

RAG 测试集应包含问题、标准答案、支撑文档或支撑片段、无答案样本、边界样本和权限样本。问题来源可以是用户日志、专家标注、文档反向生成和人工设计。测试集要版本化，记录文档版本、索引版本、检索参数、提示模板和评估标准，避免每次优化只凭少数案例判断。

<h2 id="面试问题：RAG 检索指标、排序指标和上下文指标如何区分？">面试问题：RAG 检索指标、排序指标和上下文指标如何区分？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

##### 1. 检索与排序阶段常见指标如何理解？

| 指标 | 含义 | 关注点 |
|---|---|---|
| Hit Rate | top-k 中是否至少命中一个相关文档 | 是否命中关键证据 |
| Recall@k | top-k 中召回了多少应召回的相关文档 | 正确证据是否被找出来 |
| Precision@k | top-k 中有多少是相关文档 | 检索结果是否噪声少 |
| MRR | 第一个相关结果排名的倒数平均值 | 相关文档是否排得靠前 |
| nDCG / NDCG | 考虑相关性等级和排序位置的指标 | 排序质量和相关性强弱 |
| Context Recall | 答案所需证据是否被检索上下文覆盖 | 关键证据是否缺失 |
| Context Precision | 检索上下文中有多少内容真正相关 | 上下文噪声是否过高 |

##### 2. 指标应该如何选择？

如果任务只要命中一个关键文档，Hit Rate 和 Recall@k 很重要；如果上下文窗口有限，需要把最相关证据放前面，MRR 和 nDCG 更重要；如果模型容易被噪声干扰，Precision@k 和 Context Precision 也很关键。

RAG 评估不能只追求 Recall@k。召回更多文档可能提高覆盖率，但也会增加噪声、成本和延迟。检索指标必须和生成质量、引用准确性及业务效果一起看。

<h2 id="面试问题：Faithfulness、Answer Relevance、Context Relevance 和 Citation Accuracy 有什么区别？">面试问题：Faithfulness、Answer Relevance、Context Relevance 和 Citation Accuracy 有什么区别？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

##### 1. 生成阶段常见指标如何区分？

| 指标 | 含义 | 典型问题 |
|---|---|---|
| Faithfulness | 答案是否忠实于检索内容 | 是否加入了上下文没有的信息 |
| Answer Relevance | 答案是否回答了用户问题 | 是否答非所问 |
| Context Relevance | 检索上下文是否与问题相关 | 给模型的资料是否有用 |
| Citation Accuracy | 引用是否准确且支持结论 | 引用是否错链、错段或不支持 |
| Hallucination Rate | 答案中不被上下文支持或事实错误内容的比例 | 是否存在无依据编造 |
| Completeness | 答案是否覆盖问题要求的关键点 | 是否漏掉必要信息 |

##### 2. 人工评估和自动评估如何结合？

自动评估适合大规模回归测试和快速对比方案；人工评估适合建立标准、审查边界问题和高风险样本。LLM-as-a-Judge 可以辅助评估，但要设计评分 rubric、抽样复核、控制偏差，并避免评审模型与被评模型同源导致偏差。

高风险企业知识库、合同、医疗、法律等场景中，引用和事实核查不能完全交给自动分数，必须保留专家抽检或人工复核流程。

<h2 id="面试问题：RAGAS、TruLens、DeepEval、LangSmith 和 BEIR 分别能帮助评估什么？">面试问题：RAGAS、TruLens、DeepEval、LangSmith 和 BEIR 分别能帮助评估什么？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐ (3/5)**

##### 1. RAGAS、TruLens 和 DeepEval 主要做什么？

这些工具常用于 RAG 或大模型应用评估：

- **RAGAS**：关注 answer relevance、faithfulness、context precision、context recall 等 RAG 指标。
- **TruLens**：强调应用追踪、反馈函数和可观测评估，可分析 RAG 链路中的中间结果。
- **DeepEval**：提供大模型应用测试框架，可做单元式评估、指标评估和回归测试。

##### 2. LangSmith、LlamaIndex 评估模块和 BEIR 有什么作用？

LangSmith 更偏向链路追踪、数据集管理、实验比较和线上观测，适合 LangChain 生态下的大模型应用调试。LlamaIndex 的评估模块常用于检索、响应和 faithfulness 等评估，适合围绕 LlamaIndex 构建的 RAG 项目。

BEIR 是检索基准集合，主要用于评估检索模型在多任务上的泛化能力。它不能替代业务 RAG 评估，因为业务文档、权限、问题分布和答案标准不同，但可作为检索模型选型参考。

##### 3. 如何看待评估工具？

工具只是框架，真正重要的是测试集和指标设计。没有高质量业务测试集，工具只能给出形式化分数，无法证明系统满足真实需求。

<h2 id="面试问题：如何定位 RAG 系统问题发生在解析、分块、检索、重排序还是生成阶段？">面试问题：如何定位 RAG 系统问题发生在解析、分块、检索、重排序还是生成阶段？</h2>

**难度评分：⭐⭐⭐⭐⭐ (5/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

##### 1. 如何区分“没检到”“检到了但排序不好”“上下文足够但模型答错”？

可以按链路逐步排查：

1. 查看知识库中是否存在正确答案；
2. 查看正确文档是否被解析入库；
3. 查看正确片段是否被合理分块；
4. 查看初始召回 top-k 是否包含正确片段；
5. 查看重排序后正确片段是否进入最终上下文；
6. 查看最终 prompt 是否清楚要求基于上下文回答；
7. 查看模型是否忠实引用了上下文。

若正确片段不在初始召回中，是召回问题；若在候选中但没进入上下文，是排序或截断问题；若在上下文中但答错，是生成、提示或模型能力问题。

##### 2. 如何判断问题出在文档解析或分块阶段？

检查原文、解析文本和入库文本是否一致。常见信号包括表格错位、页码缺失、标题丢失、乱码、重复页眉页脚、扫描 PDF OCR 错误、代码缩进丢失。如果正确信息在原文中存在但索引文本中不存在，问题在解析或清洗阶段。

分块阶段要检查相关信息是否被切散、chunk 是否过大混入多个主题、overlap 是否不足、metadata 是否丢失。若正确答案跨多个 chunk，而系统只召回其中一部分，说明分块策略需要调整。

##### 3. 如何通过日志和可观测性工具分析链路？

每次请求应记录原始问题、改写 query、检索参数、召回结果、reranker 分数、最终上下文、最终 prompt、模型输出、引用、延迟和成本。没有日志的 RAG 系统很难优化，只能凭感觉调 prompt。

<h2 id="面试问题：一个效果不好的 RAG 系统应该如何系统优化？">面试问题：一个效果不好的 RAG 系统应该如何系统优化？</h2>

**难度评分：⭐⭐⭐⭐⭐ (5/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

##### 1. 如果召回率低，优先检查什么？

优先检查知识库是否覆盖答案、文档是否成功解析入库、chunk 是否包含完整语义、embedding 模型是否适合领域、query 是否需要改写、是否需要关键词检索或混合检索、metadata 过滤是否过严。

优化手段包括改进清洗和分块、增加同义词扩展、多查询召回、混合检索、调整 top-k、替换 embedding 模型、使用 HyDE 或查询拆解。

##### 2. 如果召回内容相关但答案错误，应该怎么办？

这通常是生成阶段或上下文构造问题。可检查是否把最相关片段放在前面、上下文是否过长或冲突、提示是否要求只基于上下文回答、是否要求引用、是否需要更强模型、是否需要答案后校验。

也可以引入 faithfulness 检查，让模型逐条判断答案中的结论是否被引用片段支持。

##### 3. 如果延迟或成本太高，如何优化？

可从链路拆解：减少检索 top-k、优化向量库索引、减少 reranker 候选数、缓存热门 query、使用并行检索、压缩上下文、选择更快模型、流式输出、减少多轮迭代检索。

成本优化还包括用小模型处理路由和 query 改写，只对复杂问题调用强模型，缓存 embedding 和检索结果，控制自我一致性和多查询数量，对无答案或低价值请求提前拒答。

##### 4. 如何通过 A/B 测试比较不同 RAG 方案？

应固定测试集和业务指标，比较不同方案在答案正确率、faithfulness、引用准确率、延迟、成本和用户满意度上的表现。A/B 测试不能只看平均分，还要分析失败案例：哪些问题类型提升，哪些下降，是否引入新的安全或权限风险。

> RAG 优化要沿链路定位，不要只调 prompt。先判断问题是文档、分块、召回、排序、上下文还是生成，再选择对应手段，并用评估集和线上指标验证。

<h1 id="50.如何设计完整的大模型评估方案与业务闭环？">52.如何设计完整的大模型评估方案与业务闭环？</h1>

<h2 id="面试问题：如何从零设计一个合理的大模型评估方案？">面试问题：如何从零设计一个合理的大模型评估方案？</h2>

**难度评分：⭐⭐⭐⭐ (4/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

##### 1. 通用设计流程是什么？

一个合理的大模型评估方案通常包括：

```text
明确评估目标
→ 选择基线模型
→ 构建评估数据
→ 确定指标和评分标准
→ 固定 prompt 与推理参数
→ 执行自动评估
→ 执行人工评估 / LLM-as-a-Judge
→ 做安全红队和回归测试
→ 分析错误案例
→ 小流量线上 A/B
→ 反馈回流并迭代
```

##### 2. 评估目标如何定义？

评估目标必须和模型用途绑定。例如：

- 通用模型选型：看综合能力、成本、部署方式和安全性；
- 微调效果验证：看目标任务提升和通用能力是否退化；
- RAG 系统验收：看检索、忠实性、引用和拒答边界；
- 生产替换：看线上解决率、转人工率、用户满意度、延迟和成本。

如果目标不清晰，指标就会互相冲突，评估结果也无法指导决策。

##### 3. 评估集应该如何构成？

出版级或工程级评估集通常不应只有“正常样本”，而应覆盖：

- **常规样本**：代表主要用户需求和高频任务；
- **困难样本**：覆盖长上下文、多约束、复杂推理和领域专业问题；
- **边界样本**：测试拒答、不确定性表达、格式约束和权限边界；
- **对抗样本**：测试越狱、提示注入、干扰信息和鲁棒性；
- **回归样本**：固定旧版本已知能力，防止模型升级后退化；
- **业务样本**：来自真实应用分布，用于判断线上价值。

评估集应同时保留“可自动打分样本”和“需要人工评审样本”。前者保证高频回归效率，后者保证开放式回答和高风险场景的质量判断。

<h2 id="面试问题：评估集版本管理、统计显著性、回归测试和错误分析为什么重要？">面试问题：评估集版本管理、统计显著性、回归测试和错误分析为什么重要？</h2>

**难度评分：⭐⭐⭐⭐ (4/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

##### 1. 为什么要做评估集版本管理？

内部评估集需要固定版本，否则不同模型、checkpoint 和微调方案无法公平比较。版本管理应记录：

- 样本来源；
- 数据清洗规则；
- 标注版本；
- prompt 模板；
- 解码参数；
- 评估脚本；
- judge 模型版本；
- 指标计算方式。

没有版本管理，评估分数变化可能来自数据或脚本变化，而不是模型能力变化。

##### 2. 为什么要看统计显著性？

模型差异较小时，单次评估分数可能受样本抽样、生成随机性和评审偏差影响。需要使用足够样本量、置信区间、bootstrap 或显著性检验来判断差异是否可靠。尤其是 Win Rate 和人工偏好评估，最好报告胜率、平局率、样本数和置信区间。

##### 3. 错误分析为什么比总分更有价值？

总分只能告诉你模型好不好，错误分析能告诉你为什么不好。常见错误分类包括：

- 答非所问；
- 事实错误；
- 引用错误；
- 格式错误；
- 推理错误；
- 拒答错误；
- 工具参数错误；
- 长上下文遗漏；
- 安全边界失败。

错误分类可以直接指导数据补充、提示词调整、微调策略、RAG 切块、工具 schema 和安全规则优化。

<h2 id="面试问题：为什么业务场景评估比通用榜单更重要？上线后应该监控什么？">面试问题：为什么业务场景评估比通用榜单更重要？上线后应该监控什么？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

##### 1. 为什么业务评估更重要？

通用 benchmark 不能完全代表真实使用效果。业务评估关注的是模型是否解决真实用户问题、是否降低人工成本、是否提高效率、是否满足安全和合规要求、是否能稳定处理边界情况。

业务指标通常包括：

- 任务完成率；
- 问题解决率；
- 用户满意度；
- 转人工率；
- 人工处理时长；
- 首响时间和总响应时间；
- 工单重开率；
- 成本和调用量；
- 安全事件和投诉率。

##### 2. 上线后应该持续监控什么？

上线后评估不是结束，而是持续监控：

- 输入分布是否变化；
- 幻觉率和事实错误是否上升；
- 安全拦截和越狱是否异常；
- 延迟、错误率、超时率是否异常；
- 成本是否超预算；
- 用户反馈是否恶化；
- 新版本是否引入回归；
- 外部知识库是否过期。

成熟系统需要灰度发布、报警、回滚、人工接管和反馈回流机制。大模型评估的最终目标，是把离线能力、线上体验和风险治理连接成可迭代的反馈链路。
