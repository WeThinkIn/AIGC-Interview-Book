<h1 id="28.什么是模型微调？">28.什么是模型微调？</h1>

<h2 id="面试问题：什么是微调？与预训练、后训练、提示工程有什么关系？">面试问题：什么是微调？与预训练、后训练、提示工程有什么关系？</h2>

**难度评分：⭐⭐ (2/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**


##### 1. 在大语言模型中，什么是微调？

**微调（Fine-tuning）** 是指在预训练大语言模型的基础上，使用相对较小规模的任务数据、领域数据、指令数据或对话数据继续训练模型，使模型更适配特定任务、领域表达、输出格式、工具调用方式或回答风格。

具体训练方式包括全参数微调、参数高效微调和监督微调等，后续问题会分别展开；本题只需要先把微调理解为“在预训练之后继续更新参数以改变模型行为”。

##### 2. 微调分别和预训练、后训练、提示工程有什么关系？

它和预训练、提示工程的区别如下：

| 方法 | 是否更新参数 | 数据类型 | 主要目标 | 典型作用 |
| --- | --- | --- | --- | --- |
| 预训练（Pre-training） | 更新参数 | 海量无标注文本、代码、数学、多语种数据 | 学习通用语言规律、知识表示和基础推理能力 | 构建基座模型能力 |
| 微调（Fine-tuning） | 更新全部或部分参数 | 标注数据、指令数据、领域数据、对话数据、工具调用数据 | 让模型适配特定任务、领域表达、格式或行为模式 | 任务适配、领域适配、格式适配、工具调用适配 |
| 提示工程（Prompt Engineering） | 不更新参数 | Prompt、示例、上下文、系统指令 | 通过输入引导模型临时改变行为 | 快速适配任务、降低开发成本 |

> **要点**：预训练学习通用能力，微调学习任务行为和输出模式，提示工程通过上下文临时引导模型。

微调和后训练不是完全割裂的概念。微调更像是一类“继续训练模型参数”的技术动作；后训练更像是预训练之后的一整套训练阶段或流程。SFT 既可以被称为微调，也经常被视为后训练的第一阶段。

##### 3. 微调到底改变了什么？

微调的本质是在特定数据分布上继续优化模型参数，改变模型在给定输入条件下的输出概率分布，让模型更倾向于生成符合任务要求、业务规范或格式约束的答案。

例如：

- 在客服场景中，微调会让模型更倾向于生成符合业务话术、服务流程和知识边界的回答；
- 在法律辅助场景中，微调会让模型更熟悉法律术语、引用方式和审慎表达；
- 在医疗辅助场景中，微调会让模型更符合问询流程、风险提示和专业表达规范；
- 在工具调用场景中，微调会让模型学会在合适时机输出函数名、参数和结构化调用格式。

微调不能简单理解为“往模型里塞知识”。它更适合让模型学习 **行为模式、输出格式、任务流程和领域表达**。

<h2 id="面试问题：大语言模型微调的目标有哪些？">面试问题：大语言模型微调的目标有哪些？</h2>

**难度评分：⭐⭐ (2/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

##### 1. 大语言模型微调的目标有哪些？

前一问已经说明微调是什么。本题关注“到底要让模型学会什么行为”。常见目标可以分为几类：

- **领域适配**：让模型熟悉医疗、法律、金融、工业等专业领域的术语、表达规范、业务流程和任务边界。

- **指令跟随**：让模型更好地理解并执行用户指令，包括多约束指令、复杂步骤、输出格式和角色要求。

- **风格迁移**：让模型输出符合特定语气、品牌风格、写作规范或沟通方式。

- **任务优化**：提升分类、抽取、问答、摘要、翻译、代码生成、审核、质检等具体任务表现。

- **工具调用**：让模型学会在合适时机调用函数、API、数据库或外部工具，并生成合法参数。

- **多轮对话**：增强上下文理解、指代消解、追问澄清、角色一致性和连续交互能力。

- **结构化输出**：稳定生成 JSON、SQL、Markdown、表格、XML、函数参数等固定格式，便于系统解析和自动化处理。

- **安全拒答**：学习对高风险、不合规或越权请求进行拒绝、安全转化或给出替代建议。

- **小模型蒸馏**：用大模型生成的高质量指令数据、推理数据或业务数据训练小模型，使小模型在特定场景中接近大模型效果。

- **推理能力增强**：通过数学、代码、逻辑推理、分步解题和可验证任务数据，提升模型在复杂问题上的分析和求解能力。

> **注**：安全拒答、人类偏好优化和推理能力增强也常被归入后训练或对齐阶段。微调和后训练不是完全割裂的，SFT 既可以被称为微调，也经常是后训练的第一步。


##### 2. 不同微调目标对应的数据要求是什么？

| 微调目标 | 数据应包含什么 | 关键要求 |
|---|---|---|
| 领域适配 | 专业问答、术语解释、业务流程、领域案例、规范表达样例 | 事实准确、术语规范、边界清晰 |
| 指令跟随 | Instruction-Input-Output 样本、多约束任务样本 | 指令明确、输出符合意图和约束 |
| 风格迁移 | 特定语气、品牌文案、写作样例、改写对照数据 | 风格一致，避免内容空泛 |
| 任务优化 | 分类、抽取、摘要、翻译、质检等标注数据 | 标签准确，任务定义稳定 |
| 工具调用 | Tool Schema / 工具定义、调用时机、参数样例、调用结果样例 | 参数合法、触发条件清晰 |
| 多轮对话 | 多轮对话历史、追问、澄清、上下文引用、角色信息 | 轮次连贯，角色清晰 |
| 结构化输出 | JSON、SQL、表格、模板化答案、格式校验样例 | 格式合法，可解析 |
| 安全拒答 | 风险请求、安全回答、替代建议、边界案例 | 边界明确，拒答不过度 |
| 小模型蒸馏 | 大模型生成的高质量问答、推理过程、任务样本 | 质量高、覆盖广、无明显幻觉 |
| 推理能力增强 | 数学、代码、逻辑题、分步推理、可验证答案 | 过程可靠，答案可验证 |

微调的重点不是“有数据就训”，而是 **目标决定数据，数据塑造行为**。如果数据行为和目标行为不一致，模型就会学到错误的输出模式。

<h2 id="面试问题：全量参数微调与参数高效微调分别是什么？有什么区别？">面试问题：全量参数微调与参数高效微调分别是什么？有什么区别？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

##### 1.大语言模型中全量参数微调和参数高效微调有什么区别？

**全量参数微调（Full Fine-tuning）** 是指在微调过程中更新模型的全部参数；**参数高效微调（Parameter-Efficient Fine-tuning, PEFT）** 是指冻结大部分预训练参数，只训练少量适配参数或只更新一小部分原有参数。具体 PEFT 方法会在后续小节分别展开，这里只比较两类训练策略。

| 方法 | 更新参数 | 训练成本 | 表达能力 | 遗忘风险 | 适用场景 |
|---|---|---|---|---|---|
| 全量微调 | 更新全部参数 | 高 | 参数更新自由度最高 | 较高 | 强领域迁移、底座能力不足、数据和算力充足 |
| PEFT | 只更新少量参数 | 低 | 依赖底座能力和任务难度 | 较低 | 低成本定制、多任务适配、资源受限场景 |


##### 2. 全量微调有什么特点？

全量微调会更新模型所有权重，因此参数更新自由度最高，理论表达能力最强，可以让模型整体表示空间发生较大变化。

它适合：

- 领域差异很大的任务；
- 底座模型能力明显不足；
- 数据量足够大且质量高；
- 有充足训练算力和存储资源；
- 需要深度改变模型行为或能力边界的场景。

但全量微调成本高，主要体现在：

- 需要保存全部参数的梯度；
- 需要保存优化器状态；
- 显存和存储成本高；
- 训练时间更长；
- 每个任务通常需要保存一套完整模型权重；
- 更容易破坏预训练模型原有能力，产生灾难性遗忘。

因此，全量微调适合高资源、高质量数据、强领域迁移的场景，不是大多数业务定制的首选。

##### 3. 参数高效微调有什么特点？

PEFT 的主要价值在于用较低显存、较低训练成本和较小任务权重完成模型适配，适合多数业务定制、多任务版本管理和资源受限场景。由于底座参数大多不变，它通常比全量微调更不容易破坏原有通用能力。

但 PEFT 不是万能方案。它依赖底座模型原有能力，表达能力有上限；如果任务需要注入大量新知识、跨越很大的领域差异，或显著改变模型能力边界，继续预训练、全量微调、RAG、偏好优化或更换底座可能更合适。

<h2 id="面试问题：大语言模型微调训练中的关键超参数有哪些？它们如何影响效果、成本和稳定性？">面试问题：大语言模型微调训练中的关键超参数有哪些？它们如何影响效果、成本和稳定性？</h2>

**难度评分：⭐⭐⭐⭐ (4/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

##### 1. 大语言模型微调训练中的关键超参数有哪些？

微调训练中的关键超参数包括 **学习率、micro batch size、global batch size、epoch、warmup ratio、learning rate scheduler、weight decay、dropout、max length、gradient accumulation、gradient clipping、optimizer、precision** 等。如果使用 LoRA/QLoRA，还包括 **LoRA rank、alpha、dropout、target modules** 等。

这些超参数会一起影响：

- 收敛速度；
- 训练稳定性；
- 目标任务效果；
- 过拟合风险；
- 灾难性遗忘风险；
- 显存占用和训练成本。

##### 2. 是否有推荐的调参顺序？

实际微调中，通常不建议一开始同时调所有参数，可以按优先级逐步调整：

1. **先确定数据和目标**：确认数据质量、chat template、loss mask 和验证集。
2. **确定 max length**：保证关键指令、上下文和答案不被截断。
3. **根据显存确定 micro batch size**：尽量避免 OOM。
4. **用 gradient accumulation 设置 global batch size**：兼顾稳定性和显存。
5. **优先调 learning rate 和 epoch**：这是影响效果和遗忘风险最大的组合。
6. **再调 warmup、scheduler、weight decay 和 gradient clipping**：提升稳定性。
7. **最后微调结构参数**：调 LoRA rank、alpha、dropout 和 target modules等参数，在效果和成本之间做权衡。

<h2 id="面试问题：大语言模型微调后如何评估效果？">面试问题：大语言模型微调后如何评估效果？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

##### 1. 大语言模型微调后如何评估效果？

微调后的评估通常采用 **分层评估** 的方式，不能只看目标任务是否提升，还要同时评估 **目标任务效果、通用能力回归、安全性、事实性、格式稳定性、生成质量、人工偏好和线上业务指标**。

常见评估方式包括：

- **目标任务指标**：accuracy、F1、EM、BLEU、ROUGE、pass@k 等。
- **通用能力回归**：评估通用问答、推理、代码、数学、长文本等是否退化。
- **安全性评估**：检查越狱、有害输出、隐私泄露和拒答边界。
- **人工评估**：评价回答正确性、有用性、完整性、风格和可读性。
- **LLM-as-judge**：用强模型辅助评估开放式回答。
- **线上 A/B 测试**：用真实用户反馈和业务指标验证最终效果。

微调评估一定要做 **微调前后对比**。常见 baseline 包括 base model、prompt-only 方案、RAG-only 方案、上一版微调模型和人工规则系统。只有和 baseline 对比，才能判断微调是否真正带来收益，而不是只在某个测试集上看起来变好。

##### 2. 为什么微调后必须分层评估？

微调可能提升目标任务，但同时带来副作用。例如：

- 领域问答更专业，但通用对话能力下降；
- 输出格式更稳定，但回答变得模板化；
- 目标任务准确率提升，但安全拒答能力退化；
- 模型更符合业务话术，但事实性变差；
- 离线测试表现好，但线上真实用户不满意。

因此，评估不能只看单一指标，而要同时判断：

1. 目标任务是否变好；
2. 原有通用能力是否退化；
3. 安全和合规是否受到影响；
4. 真实用户场景是否真正有效。

<h1 id="29.什么是 LoRA 微调？它有什么变种？">29.什么是 LoRA 微调？它有什么变种？</h1>

<h2 id="面试问题：LoRA 的原理是什么？为什么适合大语言模型参数高效微调？">面试问题：LoRA 的原理是什么？为什么适合大语言模型参数高效微调？</h2>

**难度评分：⭐⭐⭐⭐ (4/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**


##### 1. 为什么LoRA适合大语言模型参数高效微调？

**低秩适配（Low-Rank Adaptation, LoRA）** 是一种参数高效微调方法。它的基本思路是：**冻结原始预训练权重** $W$，不直接更新 $W$，而是在部分线性层旁边增加一个可训练的低秩增量：

$$
\Delta W = BA
$$

训练时只更新 $A$ 和 $B$ 两个低秩矩阵，原始模型权重保持冻结。推理时可以将 LoRA 增量合并到原始权重中：

$$
W' = W + BA
$$

因此，LoRA 可以明显减少 **可训练参数量、梯度显存、优化器状态显存和任务权重存储成本**。如果推理时将 LoRA 权重合并进原始权重，通常几乎不增加推理延迟；如果推理时动态加载 LoRA Adapter，则可能带来少量额外计算和管理开销。

##### 2. LoRA 的数学形式

LoRA 不直接更新 $W^{d \times k}$，而是将权重更新写成低秩分解：

$$
\Delta W = BA
$$

其中：

$$
A \in \mathbb{R}^{r \times k}, \quad B \in \mathbb{R}^{d \times r}, \quad r \ll \min(d,k)
$$

加入 LoRA 后，线性层输出变为：

$$
h = Wx + BAx
$$

实际中常使用缩放形式：

$$
h = Wx + \frac{\alpha}{r}BAx
$$

其中：

- $r$ 是 LoRA 的秩，决定低秩增量的表达能力；
- $\alpha$ 是缩放系数，用于控制 LoRA 分支对原始权重的影响强度。

##### 3. LoRA 如何减少显存开销？

LoRA 节省显存的关键是：（1）**可训练参数少**。LoRA 的可训练参数量为：

$$
r \times k + d \times r = r(d+k)
$$

当 $r$ 远小于 $d$ 和 $k$ 时，训练参数量会大幅下降。

（2）**底座模型权重被冻结**。在全量微调中，需要为全部模型参数保存模型权重，梯度， 优化器状态（例如 Adam 的一阶动量和二阶动量）以及训练过程中的激活值。而在 LoRA 中，底座模型参数不更新，因此通常不需要为全部底座参数保存梯度和优化器状态，主要只需要为 LoRA 参数保存梯度和优化器状态。

但 LoRA 并不会让训练成本按参数减少比例等比例下降。因为训练时仍然要经过底座模型做前向计算和反向传播，激活显存也仍然存在。因此，LoRA 对显存和存储的节省非常明显，但对训练计算量的节省通常没有参数量下降那么夸张。

##### 4. 为什么低秩矩阵有效？

LoRA 的核心假设是：对于很多下游任务，模型需要学习的参数变化并不需要覆盖完整的高维参数空间，而是集中在一个较低维的子空间中。

预训练大模型已经具备较强的通用语言能力、知识表示能力和一定推理能力。很多业务微调并不是重新训练一个模型，而只是调整模型在特定输入下的输出倾向（例如：回答风格，输出格式，领域术语使用等）。因此，用低秩矩阵近似权重增量，往往就能在许多任务上达到接近全量微调的效果。

当然，如果任务和底座模型差异很大，或者需要大幅改变模型能力边界，LoRA 的表达能力也可能不足，此时可能需要继续预训练、全量微调，或更换更强的底座模型。

<h2 id="面试问题：LoRA 的低秩矩阵 Rank 和 alpha 参数有什么作用？还有哪些关键参数？">面试问题：LoRA 的低秩矩阵 Rank 和 alpha 参数有什么作用？还有哪些关键参数？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

| 超参数 | 含义 | 影响 |
|---|---|---|
| rank $r$ | 低秩矩阵维度 | 越大表达能力越强，但参数量、显存和过拟合风险增加 |
| alpha | LoRA 缩放系数 | 控制 LoRA 增量对原权重的影响强度 |
| dropout | LoRA 分支 dropout | 缓解小数据过拟合 |
| target modules | LoRA 插入哪些层 | 决定适配能力、参数量和训练成本 |
| learning rate | LoRA 参数学习率 | 过大易不稳定，过小适配不足 |

实践中常见初始化方式是：

- $A$ 随机初始化；
- $B$ 初始化为 0。

这样训练开始时：

$$
\Delta W = BA \approx 0
$$

模型初始行为接近原始底座模型，有利于提升训练稳定性。

<h2 id="面试问题：LoRA 通常插入哪些位置？不同 target modules 有什么影响？">面试问题：LoRA 通常插入哪些位置？不同 target modules 有什么影响？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

LoRA 一般插入 Transformer 中的线性层，常见位置包括：

- **q_proj**：影响 Query 表示；
- **k_proj**：影响 Key 表示；
- **v_proj**：影响 Value 表示；
- **o_proj**：影响 Attention 输出；
- **up_proj / down_proj / gate_proj**：影响 MLP 表示。

早期和轻量场景中，常见做法是在 Attention 的 **q_proj 和 v_proj** 上使用 LoRA；如果任务更复杂，也也可以用于 **q_proj、k_proj、v_proj、o_proj**，甚至 MLP 的 **up_proj、down_proj、gate_proj**。插入位置越多，表达能力通常越强，但可训练参数、显存占用、训练时间和过拟合风险也会增加。

<h2 id="面试问题：QLoRA 如何在 LoRA 基础上进一步节省显存？">面试问题：QLoRA 如何在 LoRA 基础上进一步节省显存？</h2>

**难度评分：⭐⭐⭐⭐ (4/5) | 考察频率：⭐⭐⭐⭐ (4/5)**


##### 1.QLoRA是什么？

**QLoRA（Quantized LoRA）** 在 LoRA “冻结底座模型、只训练低秩适配器”的基础上，将冻结的底座模型权重量化到 **4-bit** 存储，从而进一步降低显存占用。

需要注意的是，QLoRA 通常不是把所有参数都用 4-bit 训练，而是：

- **冻结的底座模型权重** 以 4-bit 形式存储；
- **LoRA 参数** 通常保持 FP16/BF16 等较高精度训练；
- 前向和反向传播时，4-bit 权重会按需反量化到 FP16/BF16 等计算精度参与矩阵运算；
- 反向传播会经过冻结的量化底座模型，但梯度和优化器状态主要只为 LoRA 参数保存；
- 训练完成后通常只保存 LoRA Adapter 权重。

QLoRA 的核心技术包括：

- **4-bit 量化**：将冻结底座模型权重从 FP16/BF16 压缩到 4-bit；
- **NF4（NormalFloat4）**：适合神经网络权重近似正态分布的 4-bit 数据格式；
- **Double Quantization**：对量化 scale / 常数再次量化，进一步节省量化元数据显存；
- **Paged Optimizer**：利用分页机制缓解训练过程中的显存峰值，降低 OOM 风险。

QLoRA 训练可以概括为：

```text
加载预训练模型 → 将底座模型权重量化为 4-bit → 冻结底座模型参数 → 插入 LoRA 低秩适配器 → 前向时按需反量化底座权重参与计算 → 反向时梯度传过底座模型，但只更新 LoRA 参数 → 保存 LoRA Adapter 权重
```

训练完成后，通常保存的是 LoRA Adapter，而不是完整模型权重。

推理时常见方式包括：

- 加载 4-bit 量化底座模型 + LoRA Adapter；
- 或将 LoRA 权重合并到反量化后的权重中，再根据部署框架决定是否重新量化。

需要注意的是，QLoRA 的合并和部署通常比普通 LoRA 更复杂，因为涉及量化格式、反量化、重新量化和推理引擎兼容性。

##### 5. QLoRA 的优缺点有哪些？

**优点：**

- **显存占用低**：底座模型 4-bit 存储，明显降低权重显存；
- **硬件门槛低**：可以在较小显存 GPU 上微调更大的模型；
- **训练成本低**：相比全参微调和普通 LoRA，更适合个人、中小团队和低成本实验；
- **效果较好**：在很多指令微调、领域适配和业务任务上，可以接近普通 LoRA 或全量微调效果；
- **Adapter 管理灵活**：可以为不同任务保存不同 LoRA Adapter，共享同一个量化底座模型。

**缺点：**

- **训练速度可能变慢**：量化和反量化会带来额外计算开销；
- **存在量化误差**：对高精度数学、代码、强推理或细粒度任务可能有影响；
- **实现依赖较强**：依赖 bitsandbytes、Transformers、PEFT、CUDA 内核和硬件支持；
- **部署复杂度更高**：量化底座与 LoRA Adapter 的加载、合并、转换需要考虑框架兼容；
- **表达能力仍有限**：本质上仍是 PEFT，如果任务和底座模型差异很大，可能不如继续预训练或全量微调。

##### 6. QLoRA 和 LoRA 的核心区别是什么？

| 对比项           | LoRA                | QLoRA                  |
| ---------------- | ------------------- | ---------------------- |
| 底座模型权重     | 通常 FP16/BF16 加载 | 4-bit 量化加载         |
| 可训练参数       | LoRA 参数           | LoRA 参数              |
| 底座模型是否冻结 | 冻结                | 冻结                   |
| LoRA 参数精度    | 通常 FP16/BF16      | 通常 FP16/BF16         |
| 显存占用         | 低于全量微调        | 进一步低于 LoRA        |
| 训练速度         | 通常较快            | 可能因反量化略慢       |
| 部署复杂度       | 较低                | 较高                   |
| 适用场景         | 常规低成本微调      | 显存更紧张时微调大模型 |

<h2 id="面试问题：LoRA 有哪些常见改进方案？如 PLoRA、LoRA+、AdaLoRA、DoRA 等分别解决什么问题？">面试问题：LoRA 有哪些常见改进方案？如 PLoRA、LoRA+、AdaLoRA、DoRA 等分别解决什么问题？</h2>

**难度评分：⭐⭐⭐⭐ (4/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

##### 1. 为什么LoRA需要改进？

标准 **低秩适配（Low-Rank Adaptation, LoRA）** 的基本思路是：冻结原模型权重 $W$，只训练低秩增量矩阵。但标准 LoRA 仍存在一些局限：

- **rank 固定**：不同层、不同模块统一使用相同 rank，可能造成参数浪费；
- **表达能力受限**：低秩增量可能不足以表示复杂任务所需的权重变化；
- **训练稳定性依赖超参数**：rank、alpha、学习率、target modules 等都会影响效果；
- **参数分配不灵活**：无法自动判断哪些层更需要适配；
- **量化、多任务、长上下文场景适配不足**：标准 LoRA 本身不专门解决这些工程问题。

因此，后续 LoRA 改进方法主要围绕 **显存占用、表达能力、训练稳定性、参数分配、多任务适配和量化兼容性** 进行优化。

##### 2. LoRA+：解决 A/B 矩阵学习率不匹配问题

**LoRA+** 主要改进标准 LoRA 中 $A$、$B$ 两个低秩矩阵使用相同学习率的问题。

$A$ 和 $B$ 在低秩分解中承担的作用不同，梯度尺度和收敛速度也可能不同。如果二者使用完全相同的学习率，可能导致一个矩阵更新不足，另一个矩阵更新过快，从而影响收敛效率。

LoRA+ 的核心做法是：**为 $A$ 和 $B$ 设置不同学习率**，让两个低秩矩阵以更合适的速度更新。

它的优势包括：

- 不明显增加参数量；
- 实现成本低；
- 可提升收敛速度；
- 适合在已有 LoRA 流程上轻量改造。

局限是：它主要改善优化效率和收敛速度，并不从根本上改变 LoRA 的表达能力上限。

> **标准 LoRA 让 A、B 用同样步长更新，LoRA+ 让它们按不同节奏更新。**

##### 3. AdaLoRA：解决 rank 固定和参数分配不灵活问题

**AdaLoRA（Adaptive LoRA）** 主要解决标准 LoRA 中所有层使用固定 rank 的问题。

标准 LoRA 通常给不同模块统一设置 rank，例如所有 `q_proj`、`v_proj` 都用 `r=8`。但不同层、不同模块对任务适配的重要性不同，统一 rank 可能造成参数浪费。

AdaLoRA 的基本思路是：**动态评估不同 LoRA 参数或秩方向的重要性，把有限的 rank 预算分配给更关键的层或方向**。

它通常将增量矩阵表示为类似奇异值分解的形式：

$$
\Delta W = P \Lambda Q
$$

其中，$\Lambda$ 中的不同方向可以根据重要性进行保留或裁剪，从而实现自适应 rank 分配。

AdaLoRA 的优势包括：

- 在相同参数预算下提升效果；
- 自动为不同层分配不同 rank；
- 减少人工调参；
- 适合层间差异明显、任务复杂的场景。

局限是：实现和训练流程比标准 LoRA 更复杂，动态 rank 分配也会增加调试成本。

> **AdaLoRA 不是平均分配 rank，而是把参数预算分给更重要的位置。**

##### 4. DoRA：解决标准 LoRA 表达能力不足问题

**DoRA（Weight-Decomposed Low-Rank Adaptation）** 主要解决标准 LoRA 表达能力受限的问题。

标准 LoRA 只学习低秩增量，而 DoRA 的思想是将权重适配拆成两个部分：

- **方向（direction）**
- **幅度（magnitude）**

可以简化理解为：

$$
W = m \cdot \frac{V}{\|V\|}
$$

其中：

- $V$ 表示权重方向；
- $m$ 表示权重幅度。

DoRA 通常使用 LoRA 学习方向变化，同时额外学习幅度参数，从而增强模型的适配能力。

DoRA 的优势包括：

- 表达能力通常强于标准 LoRA；
- 在部分任务上更接近全量微调；
- 适合标准 LoRA 效果不足的场景；
- 可以和量化微调结合使用。

局限是：相比标准 LoRA, DoRA 会带来一定额外参数和实现复杂度，训练和部署也需要框架支持。

> **LoRA 主要学习“往哪个方向改”，DoRA 同时学习“往哪个方向改”和“改多大”。**

##### 5. 其他常见 LoRA 改进方法有哪些？

| 方法       | 中文说明              | 核心思想                                 | 主要解决的问题                 |
| ---------- | --------------------- | ---------------------------------------- | ------------------------------ |
| rsLoRA     | Rank-stabilized LoRA  | 调整 LoRA scaling 方式                   | 大 rank 训练不稳定             |
| PiSSA      | 主奇异方向初始化 LoRA | 用主奇异方向初始化低秩矩阵               | 随机初始化收敛慢、量化误差较大 |
| OLoRA      | 正交 LoRA             | 使用正交初始化或权重平移                 | 初始化和优化稳定性问题         |
| LoftQ      | 量化感知 LoRA 初始化  | 联合考虑量化误差和 LoRA 初始化           | 量化微调初始化误差             |
| LoHa       | Hadamard LoRA         | 用 Hadamard product 增强表达             | 低秩表达受限                   |
| LoKr       | Kronecker LoRA        | 用 Kronecker product 提高参数效率        | 参数效率与表达能力平衡         |
| MoLoRA     | 混合专家 LoRA         | 多个 LoRA expert 加路由选择              | 多任务适配和任务干扰           |
| LongLoRA   | 长上下文 LoRA         | 结合长上下文训练策略                     | 长上下文扩展成本高             |
| Delta-LoRA | 增量增强 LoRA         | 让 LoRA 增量更充分影响原权重             | 缩小与全量微调差距             |

##### 6. 如何选择这些 LoRA 改进方案？

实际业务中，通常优先从 **LoRA / QLoRA** 开始。如果效果不足，再根据问题选择改进方案：

- 显存不够：优先 QLoRA；
- 收敛慢：考虑 LoRA+ 或更好初始化；
- rank 分配不合理：考虑 AdaLoRA；
- 表达能力不足：考虑 DoRA；
- 多任务干扰：考虑 MoLoRA；
- 量化误差明显：考虑 LoftQ、PiSSA 或 QLoRA 配套优化。


<h1 id="30.什么是Adapter微调？它有什么变体？">30.什么是Adapter微调？它有什么变体？</h1>

<h2 id="面试问题：Adapter 的原理是什么？它适合什么场景？">面试问题：Adapter 的原理是什么？它适合什么场景？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

##### 1. Adapter 的基本原理是什么？

**Adapter** 是一种典型的参数高效微调方法。它的基本思路是：**冻结预训练模型的大部分参数，只在 Transformer 层中插入少量可训练的小型模块**，通过训练这些新增模块来适配下游任务。

Adapter 通常插入在 Transformer 的注意力层或前馈网络层之后，其结构一般是一个瓶颈网络：

```text
hidden → down projection → non-linear activation → up projection → residual
```

也就是说，Adapter 会先把 hidden state 从高维压缩到低维，再经过非线性变换，最后升维回原来的 hidden size，并通过残差连接加回原表示。

> Adapter 不直接改原模型的大部分权重，而是在模型内部插入一些“小插件”，让这些插件学习特定任务或领域的适配能力。

##### 2. Adapter 的优点是什么？

Adapter 的主要优点包括：

- **训练参数少**：只训练新增 Adapter 模块，不需要更新全部模型参数。
- **模块化强**：不同任务可以保存不同 Adapter，推理时按需加载。
- **适合多任务切换**：同一个底座模型可以搭配多个 Adapter，用于不同任务或领域。
- **底座模型稳定**：原始预训练模型参数被冻结，降低灾难性遗忘风险。
- **表达能力较好**：相比 Prompt Tuning 这类只在输入侧添加软提示的方法，Adapter 插入到模型中间层，通常具有更强的适配能力。

##### 3. Adapter 的缺点是什么？

Adapter 的局限主要包括：

- **会改变模型结构**：需要在 Transformer 层中插入额外模块。
- **推理有额外开销**：每层都要经过 Adapter 模块，可能增加推理延迟。
- **部署复杂度较高**：线上服务需要支持 Adapter 模块加载和切换。
- **现代大语言模型微调中使用频率低于 LoRA**：LoRA 可以合并到原权重中，部署更方便，因此在大语言模型业务微调中更常见。

##### 4. Adapter 适合什么场景？

Adapter 比较适合以下场景：

| 场景 | 原因 |
| --- | --- |
| 多任务微调 | 不同任务可以保存不同 Adapter，便于切换 |
| 领域适配 | 可以为医疗、法律、金融等领域训练独立 Adapter |
| 模块化管理 | 底座模型不变，只管理任务相关 Adapter 参数 |
| 希望降低灾难性遗忘风险 | 冻结原模型，只训练新增模块 |
| 任务数量多但单个任务数据有限 | 每个任务只训练小模块，成本较低 |

> Adapter 是一种参数高效微调方法，它会在 Transformer 层中插入小型瓶颈网络，只训练这些新增模块而冻结原模型参数。它的优势是参数量少、模块化强、适合多任务切换和领域适配；缺点是会改变模型结构，推理时有额外计算开销，部署复杂度通常高于 LoRA。因此 Adapter 更适合需要任务插件化管理、多任务切换或领域模块化适配的场景。

<h2 id="面试问题：Prefix Tuning 的原理是什么？它适合什么场景？">面试问题：Prefix Tuning 的原理是什么？它适合什么场景？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

##### 1. Prefix Tuning 的基本原理是什么？

**Prefix Tuning** 是一种参数高效微调方法。它的基本思路是：**冻结原模型参数，在每层 Transformer Attention 中加入一组可训练的 Key / Value 前缀向量，用这些前缀向量引导模型生成结果**。

在标准 Transformer Attention 中，模型会根据输入计算 Query、Key、Value。Prefix Tuning 不直接修改原模型权重，而是在每一层 Attention 的 Key / Value 前面拼接一段可训练的前缀：

```text
K' = [K_prefix; K]
V' = [V_prefix; V]
```

其中，`K_prefix` 和 `V_prefix` 是可训练参数，原模型参数保持冻结。

可以简单理解为：

> Prefix Tuning 不是改模型权重，而是在每层注意力机制中加入一段“可训练的上下文条件”，让模型在生成时受到这些条件的引导。

##### 2. Prefix Tuning 和普通 Prompt 有什么区别？

普通 Prompt 是人工写在输入文本前面的离散文本，例如：

```text
请你作为法律助手回答下面的问题：
```

而 Prefix Tuning 学到的是连续向量，不是人类可读的自然语言文本。它通常作用在每层 Attention 的 Key / Value 上，而不只是输入层。

| 方法 | 作用位置 | 是否可训练 | 表示形式 |
| --- | --- | --- | --- |
| 人工 Prompt | 输入文本 | 否 | 离散自然语言 |
| Prompt Tuning | 输入 Embedding 前 | 是 | 连续向量 |
| Prefix Tuning | 每层 Attention 的 K/V 前缀 | 是 | 连续向量 |

因此，Prefix Tuning 的表达能力通常强于只在输入侧添加软 Prompt 的方法。

##### 3. Prefix Tuning 的优点是什么？

Prefix Tuning 的优点包括：

- **训练参数较少**：只训练前缀向量，不更新原模型参数。
- **对生成任务友好**：适合摘要、对话、文本生成等任务。
- **不修改原模型权重**：底座模型保持稳定。
- **可以引导模型生成风格和内容**：通过学习前缀向量影响 Attention 过程。
- **适合低资源任务**：在数据较少时可以用较低成本适配模型。

##### 4. Prefix Tuning 的缺点是什么？

Prefix Tuning 的局限包括：

- **会增加 KV Cache 或 Attention 开销**：因为每层都引入额外前缀。
- **对前缀长度敏感**：Prefix 太短表达能力不足，太长会增加开销。
- **对初始化较敏感**：初始化不好可能影响收敛和最终效果。
- **表达能力通常不如 LoRA**：尤其是在复杂业务微调或格式适配中。
- **部署时需要额外管理 Prefix 参数**：推理服务需要支持加载和拼接前缀向量。

##### 5. Prefix Tuning 适合什么场景？

Prefix Tuning 比较适合以下场景：

| 场景 | 原因 |
| --- | --- |
| 文本生成 | Prefix 直接影响生成过程 |
| 摘要任务 | 可以学习摘要风格和任务条件 |
| 对话生成 | 可以引导回复风格、角色和任务边界 |
| 低资源生成任务 | 参数量少，训练成本低 |
| 不希望修改模型权重的场景 | 只训练 Prefix 参数，底座模型冻结 |

> Prefix Tuning 是一种参数高效微调方法，它冻结原模型参数，在每层 Attention 的 Key / Value 前拼接可训练的前缀向量，从而引导模型生成。它的优点是参数量少、对生成任务友好、不修改原模型权重；缺点是会增加 KV Cache 或 Attention 开销，对前缀长度和初始化敏感，表达能力通常不如 LoRA。它更适合摘要、对话生成、文本生成和低资源生成任务。

<h2 id="面试问题：Prompt Tuning 和 P-Tuning / P-Tuning v2 有什么区别？">面试问题：Prompt Tuning 和 P-Tuning / P-Tuning v2 有什么区别？</h2>

**难度评分：⭐⭐⭐⭐ (4/5) | 考察频率：⭐⭐⭐ (3/5)**

##### 1. Prompt Tuning 的原理是什么？

**Prompt Tuning** 是一种参数高效微调方法。它的基本思路是：**冻结原模型参数，在输入 Embedding 前添加一组可训练的软 Prompt 向量，只训练这些 Prompt Embedding 来适配任务**。

它和人工 Prompt 的区别在于：人工 Prompt 是离散文本，而 Prompt Tuning 学到的是连续向量。

例如，原始输入可以看作：

```text
[input tokens]
```

Prompt Tuning 后变成：

```text
[soft prompt embeddings] + [input token embeddings]
```

其中 soft prompt embeddings 是可训练参数。

##### 2. Prompt Tuning 的优点和缺点是什么？

Prompt Tuning 的优点包括：

- **参数量极少**：通常只训练少量 Prompt Embedding。
- **训练成本低**：显存和计算开销都比较小。
- **不修改模型结构**：只在输入侧添加连续向量。
- **适合轻量任务适配**：尤其适合简单分类或格式较固定的任务。

Prompt Tuning 的缺点包括：

- **表达能力有限**：只在输入侧加入提示，难以深度影响模型内部表示。
- **通常对大模型更有效**：小模型上效果往往不稳定。
- **会占用上下文长度**：软 Prompt 会占用输入 token 空间。
- **对 Prompt 长度和初始化敏感**：不同设置可能造成明显效果差异。
- **复杂任务上通常不如 LoRA**：尤其是需要稳定格式输出、风格适配或业务知识适配时。

##### 3. P-Tuning 的原理是什么？

**P-Tuning** 可以看作 Prompt Tuning 的增强版本。它同样使用连续 Prompt，但通常会引入 **Prompt Encoder** 来生成更强的提示表示。

Prompt Encoder 可以是：

- MLP；
- LSTM；
- Transformer；
- 其他轻量网络。

也就是说，P-Tuning 不只是直接训练一组 Prompt Embedding，而是通过一个可训练网络生成或优化 Prompt 表示，从而提升表达能力。

##### 4. P-Tuning v2 的原理是什么？

**P-Tuning v2** 进一步增强了 P-Tuning。它不再只在输入层添加连续 Prompt，而是将 Prompt 扩展到多层 Transformer 中，也就是常说的 **Deep Prompt**。

可以简单理解为：

- Prompt Tuning：只在输入层加软 Prompt；
- P-Tuning：使用 Prompt Encoder 增强软 Prompt；
- P-Tuning v2：在多层加入连续 Prompt，增强对模型内部表示的影响。

P-Tuning v2 的表达能力通常比普通 Prompt Tuning 更强，也更适合 NLU、分类、抽取等理解类任务。

##### 5. Prompt Tuning、P-Tuning、P-Tuning v2 有什么区别？

| 方法 | 核心思想 | 作用位置 | 表达能力 | 参数量 | 适合场景 |
| --- | --- | --- | --- | --- | --- |
| Prompt Tuning | 训练输入前的软 Prompt | 输入 Embedding 前 | 较弱 | 极少 | 简单分类、轻量任务适配 |
| P-Tuning | 使用 Prompt Encoder 生成连续 Prompt | 通常在输入侧 | 中等 | 较少 | NLU、分类、抽取、小样本任务 |
| P-Tuning v2 | 多层深度 Prompt | 多层 Transformer | 较强 | 较少 | NLU、抽取、复杂理解任务 |

##### 6. P-Tuning / P-Tuning v2 的优点和缺点是什么？

P-Tuning / P-Tuning v2 的优点包括：

- **比 Prompt Tuning 表达能力更强**；
- **参数量仍然较少**；
- **适合分类、抽取、NLU 等理解任务**；
- **可以在不全量更新模型的情况下增强任务适配能力**；
- **小样本任务中可能比普通 Prompt Tuning 更稳定**。

它们的缺点包括：

- **实现和调参更复杂**；
- **可能占用上下文或隐状态空间**；
- **对任务形式和数据质量敏感**；
- **在现代大语言模型 通用业务微调中，通用性通常不如 LoRA**。

> Prompt Tuning 是在输入 Embedding 前加入可训练的软 Prompt，参数量极少，但表达能力有限，通常更适合简单任务和大模型。P-Tuning 是 Prompt Tuning 的增强版，会用 Prompt Encoder 生成更强的连续提示表示。P-Tuning v2 进一步把 Prompt 扩展到多层 Transformer 中，表达能力更强，更适合分类、抽取、NLU 等理解任务。总体来说，它们都属于参数高效微调方法，但在现代大语言模型业务微调中，LoRA / QLoRA 通常更通用。

<h2 id="面试问题：Adapter、Prefix Tuning、Prompt Tuning、P-Tuning 和 LoRA 的核心区别是什么？">面试问题：Adapter、Prefix Tuning、Prompt Tuning、P-Tuning 和 LoRA 的核心区别是什么？</h2>

**难度评分：⭐⭐⭐⭐ (4/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

##### 1. 这些方法有什么共同点？

Adapter、Prefix Tuning、Prompt Tuning、P-Tuning 和 LoRA 都属于 **参数高效微调（PEFT, Parameter-Efficient Fine-Tuning）** 方法。

它们的共同目标是：

> 尽量冻结大部分预训练模型参数，只训练少量新增参数或增量参数，从而降低训练成本、显存开销和模型存储成本。

相比全参数微调，PEFT 方法通常具有以下特点：

- 可训练参数少；
- 显存占用低；
- 训练成本低；
- 更适合中小规模数据；
- 便于为不同任务保存不同适配参数；
- 降低灾难性遗忘风险。

##### 2. 它们的核心区别是什么？

这些方法的主要区别在于：

- **作用位置不同**；
- **可训练参数量不同**；
- **表达能力不同**；
- **推理开销不同**；
- **部署复杂度不同**。

具体对比如下：

| 方法 | 作用位置 | 训练参数 | 推理影响 | 主要特点 | 适合场景 |
| --- | --- | --- | --- | --- | --- |
| Adapter | Transformer 层间插入小型瓶颈模块 | 较少 | 增加额外计算 | 模块化强，表达能力较好，但会改变模型结构 | 多任务切换、领域适配、模块化管理 |
| Prefix Tuning | 每层 Attention 的 K/V 前缀 | 较少 | 增加 KV Cache 或 Attention 开销 | 通过可训练前缀影响生成过程 | 文本生成、摘要、对话生成、低资源生成任务 |
| Prompt Tuning | 输入 Embedding 前添加软 Prompt | 极少 | 占用上下文长度 | 参数最少，但表达能力有限，通常对大模型更有效 | 轻量任务适配、分类、简单指令适配 |
| P-Tuning / P-Tuning v2 | 连续 Prompt、Prompt Encoder，或多层深度 Prompt | 较少 | 占用输入或隐状态空间 | 比普通 Prompt Tuning 表达能力更强 | NLU、分类、抽取、提示增强 |
| LoRA | Attention / MLP 线性层的低秩权重增量 | 较少 | 可 merge，合并后几乎无额外延迟 | 效果、效率和部署最均衡 | 大多数 LLM 业务微调、工具调用、风格适配、格式适配 |
| QLoRA | 4-bit 量化底座模型 + LoRA 增量 | 较少 | 依赖量化推理支持 | 在 LoRA 基础上进一步降低显存 | 显存有限时微调较大模型 |

> Adapter、Prefix Tuning、Prompt Tuning、P-Tuning 和 LoRA 都属于 PEFT 方法，目标都是冻结大部分模型参数，只训练少量适配参数。区别在于作用位置不同：Adapter 是在 Transformer 层中插入小模块，Prefix Tuning 是在每层 Attention 中加入 K/V 前缀，Prompt Tuning 是在输入侧加入软 Prompt, P-Tuning 是用 Prompt Encoder 或多层 Prompt 增强提示表示，LoRA 是在 Attention 或 MLP 线性层上学习低秩权重增量。实际业务中，LoRA / QLoRA 因为效果、成本和部署便利性更均衡，通常是默认优先选择。

<h2 id="面试问题：实际项目中如何选择不同的 PEFT 方法？">面试问题：实际项目中如何选择不同的 PEFT 方法？</h2>

**难度评分：⭐⭐⭐⭐ (4/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

##### 1. 为什么 PEFT 方法需要选型？

不同 PEFT 方法虽然都能降低训练成本，但它们在表达能力、显存占用、推理开销、部署复杂度和任务适配能力上并不相同。

实际项目中不能只看“参数量少”，还要综合考虑：

- 任务类型；
- 数据规模；
- 模型大小；
- 显存资源；
- 效果要求；
- 推理延迟；
- 部署框架是否支持；
- 是否需要多任务切换；
- 是否需要合并权重；
- 是否需要长期维护多个适配版本。

##### 2. 不同场景下如何选择 PEFT 方法？

前面已经分别解释过 LoRA、QLoRA、Adapter、Prefix Tuning 和 Prompt Tuning 的原理，这里只保留选型判断：

- **默认业务微调**：优先 LoRA；显存紧张时用 QLoRA。
- **需要低延迟部署**：优先可 merge 的 LoRA，避免引入额外推理结构。
- **多任务多版本管理**：优先多个 LoRA adapter 或 Adapter，便于独立保存和切换。
- **只做极轻量分类 / 抽取适配**：可考虑 Prompt Tuning 或 P-Tuning，但要警惕表达能力不足。
- **生成任务需要更强条件控制**：可考虑 Prefix Tuning，但要评估 KV Cache 和上下文开销。
- **底座能力明显不足或领域差异很大**：不要只依赖 PEFT，应考虑继续预训练、全量微调、RAG 或更换底座。

##### 3. 为什么大多数 LLM 业务微调优先选择 LoRA / QLoRA？

因为它们在效果、训练成本、生态支持和部署便利性之间最均衡。面试中不要把 LoRA 说成“永远最好”，而要说明：LoRA 是默认强基线；QLoRA 是显存受限时的成本优化；最终仍要以验证集、人工评估和线上指标决定。

##### 4. 什么时候不适合只用 PEFT？

虽然 PEFT 成本低，但并不是所有场景都适合只用 PEFT。

以下情况中，PEFT 可能不够：

- 任务与底座模型差异很大；
- 底座模型本身能力不足；
- 需要注入大量新领域知识；
- 训练数据规模很大，且希望充分改变模型能力；
- 需要明显提升基础推理、代码、数学或多语言能力；
- PEFT 后模型仍然无法稳定满足任务要求。

这时可以考虑全参数微调、继续预训练、领域自适应预训练、更换更强底座、结合 RAG 或采用 SFT + 偏好优化组合训练。判断标准不是“PEFT 参数少”，而是目标能力是否真的达到上线要求。


<h1 id="31.什么是指令微调 / 监督微调？">31.什么是指令微调 / 监督微调？</h1>

<h2 id="面试问题：大语言模型监督微调数据如何构造？高质量 SFT 数据需要满足哪些要求？">面试问题：大语言模型监督微调数据如何构造？高质量 SFT 数据需要满足哪些要求？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

##### 1. SFT 数据如何构造？

**SFT（Supervised Fine-tuning，监督微调）** 数据通常由高质量的 **指令-回答样本** 构成，常见格式包括 `system`、`user`、`assistant` 三类角色，并应与目标模型实际推理时的 **chat template** 保持一致。训练时通常将 `system` 和 `user` 作为条件输入，只对 `assistant` 输出计算 loss。

典型样本格式：

```text
<|system|>
你是一个专业、可靠的 AI 助手。

<|user|>
请解释什么是 LoRA。

<|assistant|>
LoRA 是一种参数高效微调方法，它冻结原始模型权重，只训练低秩增量矩阵，从而降低训练成本。
```

##### 2. 高质量 SFT 数据包含什么？

一条完整的 SFT 数据通常包括：

| 组成部分 | 作用 |
|---|---|
| System prompt | 定义模型身份、行为边界、回答风格和安全规则 |
| User instruction | 用户问题、任务指令或业务输入 |
| Assistant response | 符合预期的高质量回答 |
| 多轮上下文 | 历史对话、追问、澄清和上下文引用 |
| 特殊任务字段 | 工具定义、函数参数、引用文档、输出 schema 等 |
| Loss mask | 通常只让 assistant 输出参与 loss |

具体任务类型前文已在“微调目标”中列过。SFT 数据构造更关注这些样本是否被组织成模型真实可学习的对话格式：角色边界清晰、assistant 目标明确、特殊字段可解析、loss mask 正确。

##### 3. 为什么 SFT 格式要和推理一致？

SFT 的目标是让模型学习：

```text
P(assistant response | system, user, history)
```

也就是在给定系统指令、用户输入和上下文时，生成正确的 assistant 回复。

如果训练格式和推理格式不一致，会造成训练分布和推理分布不匹配，例如：

- 训练时没有 `system prompt`，推理时加入系统约束，模型可能无法正确理解规则；
- 训练和推理使用不同的角色标记，模型可能无法识别 user/assistant 边界；
- 训练中没有工具调用格式，推理时要求函数参数，模型容易生成非法格式；
- `loss mask` 错误时，模型可能学习复述用户输入，而不是学习如何回答。

因此，SFT 数据必须遵循实际使用的 chat template，并正确设置 loss mask。

##### 4. 高质量 SFT 数据有哪些核心原则？

| 原则 | 说明 |
|---|---|
| 指令明确 | 用户输入目标清楚，避免无效歧义 |
| 回答正确 | 输出事实准确、逻辑清晰 |
| 格式稳定 | 约定字段、schema 和边界值必须一致且可解析 |
| 场景真实 | 覆盖真实业务或真实用户问题 |
| 多样性充足 | 覆盖不同表达、难度、领域和边界条件 |
| 避免低质样本 | 删除错误、空泛、幻觉和自相矛盾回答 |
| 避免过度模板化 | 防止模型学到机械化回答 |
| 保留拒答样本 | 对高风险请求给出安全回应 |
| 拒答不过度 | 避免正常问题被频繁拒绝 |
| 多轮关系清晰 | 上下文、指代和追问关系要连贯 |
| Loss mask 正确 | 通常只训练 assistant 输出部分 |

核心结论：

> **SFT 数据质量通常比数量更重要。少量高质量、行为一致的数据，往往优于大量低质量、模板化或错误样本。**

##### 5. 不同任务的数据构造重点有哪些？

不同任务的共同要求是：训练样本必须体现推理时真实输入、真实约束和真实输出形态。例如：

- 工具调用数据不能只包含自然语言回答，还应包含函数名、参数、调用时机和结果处理；
- 法律辅助问答不能只给结论，还应包含依据、限制条件、审慎表达和风险提示。

##### 6. SFT 数据构造流程是什么？

实际工程中可以按以下流程构造：

```text
明确目标能力 → 设计 chat template 和数据 schema → 收集真实任务样本 → 人工标注或模型辅助生成 → 质量审核与事实校验 → 去重清洗 → 安全与合规检查 → 格式校验 → 设置 loss mask → 划分训练集和验证集
```

其中，**格式校验和质量审核最关键**：

- 结构化任务要检查 JSON、SQL、函数参数是否可解析；
- 领域问答要检查事实依据、适用条件和风险提示；
- 安全拒答要检查拒答边界，避免漏拒或过度拒答。

<h2 id="面试问题：大语言模型 SFT 中为什么通常只对助手回复计算损失？损失掩码应该如何设置？">面试问题：大语言模型 SFT 中为什么通常只对助手回复计算损失？损失掩码应该如何设置？</h2>

**难度评分：⭐⭐⭐⭐ (4/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

##### 1. 什么是损失掩码？

**损失掩码（Loss Mask）** 用来控制哪些 token 参与损失计算，哪些 token 不参与损失计算。它直接决定模型在 SFT 中到底学习什么。在对话 SFT 中，通常只让 **assistant 回复部分** 参与 loss，这也常被称为 **assistant-only loss** 或 completion-only loss；而 **system、user、tool observation、padding** 等部分作为条件上下文，不参与 loss。

实际实现中，常见做法是：

- 不参与训练目标的 token，其 `labels` 设置为 `-100`；
- 参与训练目标的 assistant token，其 `labels` 保留为真实 token id；
- 交叉熵损失函数会忽略 `labels = -100` 的位置。

> **注**：**loss mask 不等于 attention mask**。  被 loss mask 掉的 user/system token 虽然不参与损失，但模型仍然可以通过 attention 看到它们，并把它们作为生成 assistant 回复的条件。

##### 2. 为什么通常只对 assistant 回复计算损失？

对话 SFT 的目标不是让模型复述用户输入或系统提示，而是让模型学会在给定上下文后生成合适的助手回复。

其训练目标可以看作：

$$
P_\theta(\text{assistant answer} \mid \text{system}, \text{user}, \text{history})
$$

也就是说：

- **system**：提供角色、人设、安全规则和行为约束；
- **user**：提供任务需求；
- **history**：提供多轮对话上下文；
- **assistant answer**：才是模型需要学习生成的目标。

如果把 user/system 也纳入 loss，模型会学习预测或复述用户输入、系统提示和角色标记，而不是专注学习如何回答问题。

##### 3. 损失掩码应该如何设置？

| 场景 | 哪些 token 参与 loss |
|---|---|
| 预训练 | 通常所有有效文本 token 参与自回归预测 |
| 单轮对话 SFT | assistant 回复参与，system/user 不参与 |
| 多轮对话 SFT | 可以让每一轮 assistant 回复参与，也可以只训练最后一轮 assistant 回复 |
| 指令微调 | instruction/input 不参与，output/response 参与 |
| 工具调用训练 | assistant 生成的 tool call、函数名、参数 JSON 参与 |
| 工具返回结果 | 通常作为上下文输入，不参与 loss |
| 结构化输出训练 | 目标 JSON、SQL、Markdown、表格等参与 |
| 安全拒答训练 | assistant 的拒答、安全转化、替代建议参与 |
| Padding | 不参与 loss，通常也不参与 attention |

##### 4. 多轮对话中如何处理 loss mask？

多轮对话 SFT 常见两种策略：

- **训练所有 assistant 回复**：每一轮 assistant 回复都参与 loss，适合学习完整多轮对话模式，数据利用率更高。

- **只训练最后一轮 assistant 回复**：历史对话全部作为上下文，只让最后一轮 assistant 回复参与 loss，适合偏向单样本任务监督或避免历史回复被重复学习。

如果目标是训练完整对话能力，通常可以让所有 assistant turn 参与 loss；如果目标是训练“给定历史后回答最后问题”，可以只让最后一轮 assistant 参与 loss。

##### 5. 特殊 token 和 Chat Template 如何处理？

SFT 通常会使用模型对应的 **Chat Template**，例如：

```text
<|system|>...
<|user|>...
<|assistant|>...
```

loss mask 设置时要特别注意边界：

- **user/system 内容**：通常不参与 loss；
- **assistant 内容**：通常参与 loss；
- **assistant 结束符 EOS**：通常建议参与 loss，让模型学会在合适位置停止；
- **assistant 起始标记/header**：是否参与 loss 取决于模板和训练目标，常见做法是不训练 role header，只训练 assistant 内容和结束符；
- **role token 边界**：必须和 tokenizer 后的 token 边界一致，不能按字符串粗暴截断；
- **packing 多样本拼接**：必须防止不同样本之间互相 attention 或 loss 边界错乱。

##### 6. 工具调用场景下如何设置 loss mask？

工具调用 SFT 中，要区分 **assistant 生成的工具调用** 和 **工具返回结果**。

通常：

- assistant 生成的 function call / tool call 参与 loss；
- function name、arguments、JSON 参数参与 loss；
- tool 返回结果作为上下文，不参与 loss；
- assistant 基于 tool result 生成的最终回答参与 loss。

例如：

```text
user: 查询今天北京天气
assistant: {"tool": "weather", "arguments": {"city": "北京"}}   loss: yes
tool: {"temperature": "28℃", "condition": "晴"}                loss: no
assistant: 今天北京天气晴，约 28℃。                            loss: yes
```

这样模型才能学会：

- 什么时候调用工具；
- 调用哪个工具；
- 如何生成合法参数；
- 如何基于工具返回结果回答用户。

##### 7. Loss mask 设置错误会有什么后果？

常见错误包括：

- **user 参与 loss**：模型可能学习复述用户输入，而不是回答问题。

- **system 参与 loss**：模型可能泄露或复述系统提示，影响安全性和对话格式。

- **assistant 没有完整参与 loss**：模型回答能力学习不足，输出容易截断或格式不稳定。

- **EOS 被 mask 掉**：模型可能学不会在合适位置停止，导致输出过长或停不下来。

- **工具参数被 mask 掉**：模型无法稳定生成函数名、参数 JSON 或工具调用格式。

- **tool response 参与 loss**：模型可能学习伪造工具返回结果，而不是调用工具获取真实结果。

- **padding 参与 loss**：模型会学习无意义 padding token，影响训练稳定性。

- **chat template 边界错误**：模型可能混淆 user、assistant、tool 等角色。

- **packing 边界错误**：不同样本之间可能串扰，导致模型学习到错误上下文关系。

<h2 id="面试问题：大语言模型微调中如何避免过拟合和灾难性遗忘？">面试问题：大语言模型微调中如何避免过拟合和灾难性遗忘？</h2>

**难度评分：⭐⭐⭐⭐ (4/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

##### 1.大语言模型微调为什么容易发生过拟合和灾难性遗忘？

微调数据通常远小于预训练数据，而且任务分布更窄，参数容易被小规模任务数据牵引。如果学习率过大、训练轮次过多、样本重复率高或数据质量不稳定，模型参数会向特定领域、格式或输出风格过度偏移，模型容易过度贴合微调样本。大语言模型微调中常见两个风险：**过拟合** 和 **灾难性遗忘**。

- **过拟合**：模型过度记住微调数据，在训练集表现好，但验证集和真实场景效果差。

- **灾难性遗忘**：模型在新任务上变好，但破坏了预训练或后训练阶段学到的通用能力、安全能力、基础对话能力或其他任务能力。

二者的区别是：

- 过拟合主要关注 **新任务泛化能力差**；
- 灾难性遗忘主要关注 **原有能力被破坏**。

实际微调中两者可能同时出现。例如，模型在领域训练集上表现很好，但通用问答、安全拒答、多轮对话和工具调用能力明显下降。

##### 2. 如何判断是否发生过拟合或遗忘？

| 现象 | 可能问题 |
|---|---|
| train loss 下降，valid loss 上升 | 过拟合 |
| 训练集回答很好，真实问题效果差 | 过拟合或数据分布不匹配 |
| 对问题改写非常敏感 | 过拟合 |
| 输出高度模板化 | 过拟合或数据多样性不足 |
| 领域任务提升，但通用问答下降 | 灾难性遗忘 |
| 安全拒答变弱或过度拒答 | 安全能力遗忘或对齐边界偏移 |
| 多轮对话变差 | 通用对话能力被破坏 |
| JSON、工具调用等格式变差 | 原有格式能力被干扰 |
| 生成风格突然僵硬或单一 | 数据分布过窄或过拟合 |

##### 3.如何缓解过拟合？

- **Early Stopping**：验证集不再提升时停止训练

- **降低学习率**：减小参数更新幅度，避免模型快速贴合训练集

- **Learning Rate Warmup**：训练初期平滑升高学习率，减少不稳定更新

- **减少 Epoch**：避免反复记忆小数据集

- **Dropout**：增加随机扰动，提升泛化

- **权重衰减**：限制参数过大，提高稳定性

- **梯度裁剪**：防止梯度爆炸和训练不稳定

- **数据增强**：增加表达多样性和覆盖面

- **去重和质量过滤**：减少重复样本导致的记忆化

- **增加真实场景样本**：提高训练分布和线上分布的一致性

- **控制 LoRA Rank**：避免适配器容量过大导致过拟合

- **验证集监控**：及时发现泛化下降

微调时不能只看训练 loss。如果训练 loss 持续下降但验证集效果变差，通常说明模型已经开始过拟合。

##### 4. 如何缓解灾难性遗忘？

- **使用 LoRA/QLoRA**：限制可训练参数规模，减少对底座能力的破坏。

- **降低学习率和训练轮次**：避免模型参数或 adapter 过度偏向窄领域数据。

- **混合通用数据**：在领域数据中加入少量通用指令、对话、格式和安全数据。

- **Replay 数据**：保留部分旧任务或通用能力样本，用于维持原有能力。

- **正则约束**：限制新模型参数或输出分布偏离原模型过远。

- **分阶段训练**：领域差异较大时，可以先做领域继续预训练，再做 SFT，减少直接迁移冲击。

- **保留安全数据**：在微调集中混入安全拒答、合规边界和风险场景样本，避免安全能力退化。

- **多维回归评估**：同时评估目标任务、通用问答、多轮对话、安全能力、工具调用和结构化输出。

- **保留 checkpoint 和回滚机制**：对比不同训练步数和超参数下的模型效果，选择综合表现最优的版本。

其中，数据配比非常关键。如果领域数据占比过高，模型可能变得“过度领域化”；如果通用数据占比过高，领域适配效果又可能不足。实际工程中通常需要在目标任务效果和原有能力保留之间做权衡。

##### 5. 实践中的选择原则有哪些？

- **先评估是否真的需要微调**：如果只是补充企业知识或实时信息，优先使用 RAG、数据库或工具调用，而不是用微调硬塞知识。

- **优先使用 PEFT**：大多数业务场景优先尝试 LoRA/QLoRA，再考虑全量微调。

- **数据质量优先于数据数量**：低质量、重复、冲突数据会放大过拟合和遗忘风险。

- **小步试验，逐步放大**：先用小规模数据、少量 epoch 和多个 checkpoint 对比，再扩大训练。

- **多维评估而非单指标优化**：不能只看训练 loss 或目标任务分数，要同时评估通用能力、安全性和线上真实表现。

核心原则是：**微调不能只追求目标任务训练集效果，而要在任务适配、泛化能力、通用能力保留和安全边界之间取得平衡**。

<h1 id="32.如何让大语言模型与特定用户的习惯和偏好对齐？">32.如何让大语言模型与特定用户的习惯和偏好对齐？</h1>

<h2 id="面试问题：为什么大语言模型对齐训练会从 PPO 发展到 DPO 和 GRPO？三者在奖励模型、训练稳定性和成本上有什么区别？">面试问题：为什么大语言模型对齐训练会从 PPO 发展到 DPO 和 GRPO？三者在奖励模型、训练稳定性和成本上有什么区别？</h2>

**难度评分：⭐⭐⭐⭐⭐ (5/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

##### 1. 为什么会从 PPO 发展到 DPO 和 GRPO？

PPO、DPO、GRPO 都服务于后训练中的偏好对齐或能力增强，但目标场景不同，不能简单理解为替代关系。

- **近端策略优化（Proximal Policy Optimization, PPO）**：经典 RLHF 路线，需要 rollout、reward model、critic/value model 和 KL 约束，探索能力强，但链路复杂、成本高、训练不稳定。
- **直接偏好优化（Direct Preference Optimization, DPO）**：直接使用 chosen / rejected 偏好对训练，不需要在线 rollout、reward model 调用和 critic，工程上更接近监督学习，稳定且低成本。
- **组相对策略优化（Group Relative Policy Optimization, GRPO）**：对同一 prompt 采样多个回答，用组内相对奖励估计 advantage，去掉 critic，常用于数学、代码、逻辑推理等可验证任务。

一句话概括：**PPO 是标准 RLHF 强化学习路线，DPO 是离线偏好对上的直接优化，GRPO 是面向可验证推理任务的组内相对强化学习。**

##### 2. 三者的核心差别是什么？

| 方法 | 奖励信号 | Critic / Value | 数据形式 | 复杂度 | 稳定性 | 典型场景 |
|---|---|---|---|---|---|---|
| PPO | 通常需要 reward model | 需要 | prompt + rollout + reward / advantage | 高 | 较难调 | 通用 RLHF、复杂偏好优化 |
| DPO | 偏好对隐含奖励，可由人工、AI 或 RM 产生 | 不需要 | prompt + chosen / rejected | 低 | 较稳定 | 指令模型、对话偏好、安全偏好 |
| GRPO | verifier、规则、测试器或 RM | 不需要 | 同题多个采样回答 + 奖励 | 中 | 通常比 PPO 稳定 | 数学、代码、可验证推理 |

面试时要强调：DPO 不是“没有奖励”，而是把偏好奖励隐式写进损失；GRPO 不是“没有 baseline”，而是用同组回答的平均奖励作为相对 baseline。

##### 3. PPO 的优势和问题是什么？

PPO 把语言模型看作策略 $\pi_\theta(y|x)$，通过采样回答、奖励模型打分、critic 估计价值、计算 advantage，再用带 clipping 和 KL penalty 的目标更新策略。它的优势是能在线探索，理论上可以直接优化复杂 reward；问题是 reward scale、KL、advantage、学习率和 rollout 分布都难调，且 reward model 不准时容易出现 reward hacking、模式坍缩或通用能力退化。

##### 4. DPO 为什么更稳定？

DPO 使用同一 prompt 下的 chosen / rejected 偏好对，让模型提高 chosen 相对 rejected 的概率。它从带 KL 约束的偏好优化推导出可直接训练的损失，不需要训练过程中的 reward model 调用、critic 或在线 RL rollout，因此实现简单、显存和算力成本更低、训练曲线更接近 SFT。

局限是：DPO 依赖高质量偏好对，探索能力弱于在线 RL；如果偏好数据覆盖不足、标注噪声高或存在长度偏差，模型会把这些偏差学进去。

##### 5. GRPO 为什么适合推理能力训练？

GRPO 对同一道题采样多个回答，并用规则、verifier、单元测试或奖励模型打分。它用组内平均奖励替代 critic 作为 baseline，学习“同题下哪些推理轨迹更好”。这特别适合数学、代码和逻辑推理，因为这些任务常有可验证奖励。

GRPO 的优势是保留采样探索，同时省掉 value / critic，工程成本低于 PPO；风险是奖励稀疏、组内样本差异过小或 verifier 有偏时，advantage 信号会变弱，长推理还可能存在 token-level credit assignment 不够精细的问题。

##### 6. 实际项目如何选择？

- 有高质量 chosen / rejected 偏好对，优先考虑 **DPO**。
- 需要稳定、低成本地做指令和安全偏好优化，优先考虑 **DPO**。
- 有可自动验证的数学、代码、工具调用或格式任务，可考虑 **GRPO**。
- 有成熟 RM、RLHF 工程能力，并且确实需要在线探索复杂偏好，才考虑 **PPO**。
- 无论选择哪种方法，都要评估 win rate、通用能力回归、安全边界、reward over-optimization、输出长度和推理成本，不能只看 reward score。

<h2 id="面试问题：大语言模型后训练中 GRPO 如何提升推理能力？为什么可以去掉 critic？">面试问题：大语言模型后训练中 GRPO 如何提升推理能力？为什么可以去掉 critic？</h2>

**难度评分：⭐⭐⭐⭐⭐ (5/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

##### 1. GRPO 为什么能提升推理能力？

推理任务通常存在多条生成路径。同一个问题下，模型可能生成：

- 正确推理链；
- 中间步骤错误但最终碰巧正确的回答；
- 逻辑不完整的回答；
- 完全错误的回答；
- 冗长但无效的推理过程。

如果只看单个样本，模型只能知道“这个回答得分是多少”。而 GRPO 对同一 prompt 采样多个回答，可以比较同题下不同推理路径的优劣，使模型更明确地学习：

> 哪些推理轨迹比同题的其他轨迹更值得强化。

这对数学、代码、逻辑推理尤其有效，因为这些任务通常可以用可验证奖励判断回答好坏，例如：

- 最终答案是否正确；
- 代码单元测试是否通过；
- 输出格式是否符合要求；
- 推理结论是否满足规则；
- verifier 是否判定为正确。

这种训练范式也常被称为 **RLVR（Reinforcement Learning with Verifiable Rewards）**，即使用可验证奖励进行强化学习。奖励信号越可靠，GRPO 对推理能力的提升通常越稳定。

##### 2. 为什么 GRPO 可以去掉 critic？

在 PPO 中，critic / value model 用来估计状态价值，帮助计算 advantage，降低 policy gradient 的方差。

而 GRPO 的做法是：对同一个 prompt 生成多个 completions，然后用组内奖励均值作为 baseline。对于第 $i$ 个回答：

$$
A_i = \frac{r_i - \mu_r}{\sigma_r+\epsilon}
$$

其中：

- $\mu_r$ 是同组回答奖励均值；
- $\sigma_r$ 是同组奖励标准差；
- $\epsilon$ 用于避免标准差过小时数值不稳定。

这样，模型不再依赖单独的 critic 去估计 value，而是直接用组内相对表现衡量回答好坏：

- 高于组内平均奖励的回答被强化；
- 低于组内平均奖励的回答被抑制；
- 奖励尺度被归一化，训练更稳定；
- 不同题目之间的 reward 尺度差异被削弱。

不过，GRPO 去掉 critic 不代表完全没有方差问题。如果同组样本数量太少、奖励过于稀疏、同组回答奖励差异太小，advantage 信号仍然可能较弱或不稳定。

##### 3. GRPO 的训练目标通常包含什么？

GRPO 仍然属于策略优化方法。实际训练中，它通常会包含：

- **策略比率项**：比较新策略和旧策略生成当前回答的概率（例：如果这个回答是高分回答，新模型有没有适当提高它的概率？）；
- **裁剪项**：限制策略更新幅度，避免一次更新过大；
- **组内 advantage**：用组内相对奖励衡量回答好坏；
- **KL 正则**：约束当前模型不要偏离 reference model 太远。

可以简化理解为：

```text
同一 prompt 多采样
→ verifier / reward model 打分
→ 组内归一化得到 advantage
→ 用策略优化目标强化高 advantage 回答
→ 用 KL 约束控制模型偏移
```

KL 约束很重要，因为如果只追求 reward，模型可能出现 reward hacking、格式异常、语言质量下降或推理过程变得不自然。

##### 5. 为什么说 credit assignment 仍然是问题？

GRPO 通常使用最终答案或整体回答的 reward 来更新整段输出。但长推理链中，最终正确或错误可能由某几个关键步骤决定。（例：一个数学推理回答有 30 步，最终答案错误，可能只是第 18 步计算错了。如果只用序列级 reward（整个回答统一打分），模型很难知道具体是哪一步导致失败。）

因此，GRPO 虽然去掉 critic 降低了训练复杂度，但仍可能存在 **token-level 或 step-level credit assignment 不够精细** 的问题。

缓解方式包括：

- 使用过程奖励模型（PRM, Process Reward Model）；
- 使用 step-level verifier；
- 对中间步骤进行标注或自动校验；
- 引入更细粒度的推理过程评估；
- 结合反思、搜索、多轮采样等推理时策略。

这也是后续 DAPO、VAPO 等方法尝试改进的方向之一。

<h2 id="面试问题：为什么 GRPO 要对同一问题采样多个回答并做组内奖励归一化？这对大语言模型推理能力训练有什么帮助？">面试问题：为什么 GRPO 要对同一问题采样多个回答并做组内奖励归一化？这对大语言模型推理能力训练有什么帮助？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

##### 1. 为什么需要组内归一化？

不同 prompt 的难度不同，reward 分布也不同。

例如：

- **简单题**：大多数回答可能都接近满分；
- **难题**：大多数回答可能都得分很低；
- **代码题**：通过测试和未通过测试的 reward 差异可能很大；
- **数学题**：最终答案正确与否可能导致 reward 非常稀疏。

如果直接使用绝对 reward，模型可能受到题目难度和 reward 尺度影响。组内归一化会在同一 prompt 内部比较回答优劣，使训练信号更加稳定。

它的主要作用包括：

- **减少不同题目之间 reward 尺度差异**；
- **用组内均值作为 baseline，替代 critic 的部分作用**；
- **让模型关注同题候选回答之间的相对好坏**；
- **改善 advantage 尺度，提升训练稳定性**；
- **缓解不同任务或题型之间 reward 不可比的问题**。

需要注意，组内归一化不是万能的。如果同一组回答 reward 全部相同或差异极小，advantage 信号会很弱，样本对训练几乎没有贡献；如果 reward 本身不可靠，归一化也无法修复错误的奖励方向。

##### 2. 为什么要对同一问题采样多个回答？

多采样的价值在于 **探索** 和 **比较**。

对于推理任务，同一道题可能存在多条生成路径。单次采样只能得到一个结果，无法判断它在同题候选答案中是好是坏。多采样可以得到多个候选回答：

```text
prompt x
→ response 1：错误推理
→ response 2：中间步骤正确但答案错误
→ response 3：正确推理和正确答案
→ response 4：冗长但无效
```
<h2 id="面试问题：为什么 DAPO 被认为是对 GRPO 的改进？它在 clip、采样、loss 和训练稳定性上做了哪些优化？">面试问题：为什么 DAPO 被认为是对 GRPO 的改进？它在 clip、采样、loss 和训练稳定性上做了哪些优化？</h2>

**难度评分：⭐⭐⭐⭐⭐ (5/5) | 考察频率：⭐⭐⭐ (3/5)**

##### 1. 为什么 GRPO 还需要改进？

GRPO 通过同题多采样和组内相对奖励去掉 critic，训练链路比 PPO 更简单，但在长推理任务中仍然存在一些问题：

- **奖励信号较粗**：很多任务只在最终答案处给 reward，很难判断中间哪些步骤真正有贡献。

- **有效样本比例不高**：同一 prompt 多采样时，如果所有回答都对或都错，组内相对 advantage 接近无效，训练信号很弱。

- **长度偏置明显**：模型可能倾向生成更长、更啰嗦或重复的推理链，以提高表面 reward 或规避错误。

- **策略更新可能过度受限**：传统 PPO/GRPO 式 clip 可能限制高质量样本的学习强度，使模型对优秀推理轨迹学习不足。

- **长序列训练更不稳定**：长推理回答 token 多、reward 方差大、输出长度差异大，训练更容易出现波动。

DAPO 的核心目标就是让长推理 RL 的训练信号更有效、更稳定，同时减少无效采样和无效长输出。

##### 2. DAPO 为什么更适合长推理？相较于GRPO有什么核心改进？

长推理任务中，一个回答往往包含大量中间步骤。最终 reward 只能告诉模型“整段回答好不好”，但不能清楚告诉模型“哪一步好、哪一步错”。

DAPO 通过几类改进提升长推理 RL 的有效性：

**DAPO（Decoupled Clip and Dynamic sAmpling Policy Optimization）** 可以看作是 GRPO 之后面向长推理强化学习的改进方法。它主要针对 GRPO 在长推理训练中的 **训练稳定性、样本效率、长度偏置、策略更新受限** 等问题进行优化。

DAPO 常见的关键改进包括：

- **Clip-Higher / 解耦 clip 策略**：放宽正向优势样本的更新上界，让高质量推理轨迹可以被更充分学习，同时缓解策略更新过度保守和熵过快下降问题。

- **Dynamic Sampling**：动态筛选有效 prompt 或调整采样样本，减少全对/全错这类组内没有区分度的样本，提高有效训练样本比例。

- **Token-level Policy Gradient Loss**：将 loss 聚合从样本级或序列级调整为更细的 token 级统计方式，使长回答中的 token 对训练目标的贡献更合理。但这不等于引入 token-level reward, advantage 通常仍来自序列级或回答级奖励。

- **Overlong Reward Shaping**：对过长输出进行奖励塑形或惩罚，缓解模型生成冗长、重复、低效推理链的问题。

> **GRPO 解决了 PPO 依赖 critic 的问题，DAPO 进一步解决 GRPO 在长推理训练中的采样效率、clip 约束、loss 聚合和长度控制问题。**

不过，Token-level Policy Gradient Loss 不等于完全解决 credit assignment。若 reward 仍然只来自最终答案，模型仍可能不知道具体哪一步推理导致成功或失败。要进一步改善这个问题，通常还需要过程奖励模型、step-level verifier 或更细粒度的推理过程监督。

<h2 id="面试问题：VAPO 等新方法主要解决什么问题？它们如何改进长推理强化学习训练？">面试问题：VAPO 等新方法主要解决什么问题？它们如何改进长推理强化学习训练？</h2>

**难度评分：⭐⭐⭐⭐⭐ (5/5) | 考察频率：⭐⭐⭐ (3/5)**

##### 1. VAPO 类方法主要改进什么？

**VAPO 等新方法**主要面向 **长推理强化学习（Long-chain Reasoning RL / RLVR）** 中的 value / advantage 估计、credit assignment、训练稳定性和样本效率问题。

它们的核心动机是：GRPO 用同一 prompt 下多个回答的组内平均奖励作为 baseline，去掉了独立 critic，降低了训练复杂度；但这种 baseline 相对粗糙，尤其在长推理任务中，难以提供稳定、细致的训练信号。

> **GRPO 用组内相对奖励替代 critic，训练更轻；VAPO 等方法则尝试改进 value / advantage 估计，让长推理 RL 的训练信号更稳定、更有效。**

VAPO 等方法对长推理 RL 的改进，可以从几个角度理解：

- **更稳定的 baseline**：相比只用组内均值，改进后的 value / advantage 估计可以降低方差，让策略更新更稳定。

- **更好的长序列优化信号**：长推理回答包含大量 token 和中间步骤，改进后的训练目标可以减少长序列中的无效梯度和噪声更新。

- **更高样本效率**：通过筛选更有学习价值的样本、改进 advantage 估计或结合 verifier，减少无效 rollout 带来的训练浪费。

- **更适合可验证任务**：数学、代码、逻辑题可以用答案校验、单元测试、规则奖励或 verifier 判断结果质量，因此更适合这类 RLVR 方法。

- **缓解 credit assignment 问题**：如果结合过程奖励模型、step-level verifier 或更细粒度价值估计，模型可以更清楚地学习哪些推理步骤有价值。

<h2 id="面试问题：为什么不能只用 reward score 评估大语言模型偏好优化算法？PPO、DPO、GRPO 等方法应如何评估？">面试问题：为什么不能只用 reward score 评估大语言模型偏好优化算法？PPO、DPO、GRPO 等方法应如何评估？</h2>

**难度评分：⭐⭐⭐⭐ (4/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

##### 1. 为什么不能只用 reward score 评估大语言模型偏好优化算法？

偏好优化算法的训练阶段不能只盯 reward score。这里关注的是**训练过程中的选择与风险控制**；完整发布评估会在“模型评估”章节展开。

Reward score 只是奖励模型或规则奖励给出的代理指标，不一定等于真实人类偏好或真实任务成功。模型可能通过 reward hacking 获得更高分，也可能牺牲事实性、安全性、简洁性或通用能力来迎合奖励模型。若继续训练只推高代理奖励、真实质量反而下降，这通常称为 **reward over-optimization / over-optimization**。

训练中最需要同步监控：

- **Win rate**：模型输出相对基线模型被偏好的比例。
- **KL / policy drift**：策略是否偏离 reference model 过大。
- **Reward score 与人工抽检是否背离**：用于发现 reward hacking。
- **通用能力小回归集**：防止对齐训练破坏基础问答、格式、工具调用和安全边界。
- **长度和成本**：防止模型通过变长、模板化或冗余回答迎合奖励模型。

评估时必须和基线模型比较，例如 base model、SFT model、上一版对齐模型、prompt-only 方案或现有线上模型。没有 baseline，就很难判断偏好优化是否真正带来收益。

##### 2. 不同算法训练时应重点看什么？

- **PPO / RLHF**：重点看 reward、win rate、KL、advantage 稳定性、rollout 质量和是否模式坍缩。
- **DPO**：重点看偏好对胜率、偏好数据过拟合、长度偏差和通用能力回归。
- **GRPO / DAPO / VAPO**：重点看 verifier 通过率、pass@k、平均推理长度、样本利用率和 token 成本。

训练 checkpoint 选择不能只选 reward 最高点，而要选目标能力提升、KL 可控、回归集稳定、输出长度合理的折中点。

##### 3. 工程实践中如何闭环？

```text
选择基线模型
→ 训练中记录 reward / KL / win rate / 长度
→ 固定间隔保存 checkpoint
→ 对 checkpoint 跑小型回归集和人工抽检
→ 定位 reward hacking、过拟合和能力退化样本
→ 调整奖励、数据、KL 或采样策略
→ 再进入完整离线评估和上线验证
```
