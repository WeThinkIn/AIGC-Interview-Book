<h1 id="1.大语言模型的基础概念与概述有哪些？">1.大语言模型的基础概念与概述有哪些？</h1>

<h2 id="面试问题：什么是大语言模型？其核心特征和功能是什么？">面试问题：什么是大语言模型？其核心特征和功能是什么？</h2>

**难度评分：⭐ (1/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

![](./imgs/1-1-1.png)

##### 1. 什么是大语言模型？

**大语言模型**，简称 **LLM，Large Language Model**，是指基于大规模神经网络，通常以 **Transformer 架构**为核心，在海量文本、代码等数据上进行自监督预训练，从而具备自然语言理解、生成、推理和多任务泛化能力的模型。

**核心思想**是：将文本切分为一个个 **Token**，通过学习大量语料中的语言规律，预测上下文中最可能出现的内容。（例： GPT 系列这类生成式模型的主要训练目标通常是根据前文预测下一个 token，从而实现文本生成、问答、翻译、摘要、代码生成等能力。）

##### 2. 核心特征和功能有哪些？

a. **核心特征**主要包括：

- **大规模参数和数据**：LLM 通常拥有数十亿甚至上千亿级参数，并使用海量语料训练，因此具备较强的语言建模能力和知识表示能力。

- **基于 Transformer 架构**：Transformer 通过自注意力机制建模上下文关系，能够较好地处理长距离依赖，是当前主流大语言模型的基础架构。

- **自监督预训练**：LLM 通常不依赖人工逐条标注，而是在大规模文本中通过预测 token、补全文本等方式学习语言规律和知识表示。

- **后训练与对齐能力**：现代 LLM 通常还会经过指令微调、RLHF、DPO 等后训练阶段，从而提升问答、对话、安全性和指令遵循能力。

- **通用多任务能力**：LLM 可以通过 Prompt 或指令完成问答、摘要、翻译、改写、分类、代码生成等多种任务，不需要为每个任务单独训练一个模型。

- **上下文学习能力**：LLM 可以根据用户提供的上下文、示例或任务说明临时适配任务，具备一定的零样本和少样本泛化能力。

> **注**：Transformer 的典型组成包括自注意力机制、多头注意力、位置编码、前馈网络、残差连接和 LayerNorm 等。指令微调、RLHF、DPO 属于模型后训练或对齐阶段的常见技术。

b. **核心功能**主要包括：

- **自然语言理解**：理解用户输入的语义、意图、上下文关系和问题需求。

- **自然语言生成**：生成连贯、流畅、符合上下文的文本，例如回答问题、写文章、写邮件、写总结等。

- **推理与分析**：进行一定程度的逻辑推理、常识推理、信息归纳和步骤规划，但结果仍可能出现幻觉，需要结合校验、检索或外部工具提升可靠性。

- **知识问答与信息整合**：模型参数中压缩了大量知识，可以回答通用知识问题；如果结合 RAG，还可以接入外部知识库进行检索增强。

- **代码与工具能力**：可以生成、解释、修改代码，也可以结合插件、API、搜索、数据库等工具完成更复杂的任务。

- **多模态扩展能力**：在多模态大模型中，LLM 还可以与图像、音频、视频等模态结合，实现图文理解、语音交互、视觉问答等能力。

<h2 id="面试问题：词元（Token）是什么？为什么是大语言模型的基石概念？">面试问题：词元（Token）是什么？为什么是大语言模型的基石概念？</h2>

**难度评分：⭐ (1/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

![](./imgs/1-2-1.png)

##### 1. 什么是词元（Token）？

**词元（Token）** 是大语言模型处理文本的基本离散单位。模型通常不能直接处理原始字符串，而是先通过 **Tokenizer（分词器）** 将文本切分成 token，再将 token 映射为 **Token ID**，最后转换为 **Embedding 向量** 参与模型计算。

Token 的切分方式取决于具体分词器，并没有固定规则。常见分词方法包括 **BPE、WordPiece、SentencePiece、tiktoken** 等。一个 token 可能是一个字、一个词、一个子词、一个符号，也可能包含空格或标点。

例如：

- “人工智能”可能被切分为“人工”和“智能”两个 token；
- 英文中的词缀、空格、标点也可能成为 token 的一部分。

##### 2. 为什么词元是大语言模型的基石概念？

Token 是大语言模型输入、计算、上下文管理和商业计费的基础单位。

- **模型运算基础**：文本需要先被转换为 token id，再经过 embedding 转成向量，模型才能进行注意力计算、语义建模和生成预测。

- **资源管控核心**：上下文窗口长度、训练数据规模、推理显存占用、计算量和延迟，通常都与 token 数量密切相关。

- **产业计费标准**：主流大模型 API 服务通常按输入 token 和输出 token 计费，例如以每百万 token 为单位设定价格、调用限额和套餐配额。

##### 3. 什么是上下文窗口？与词元有什么关联？

**上下文窗口（Context Window）** 是指大语言模型在一次请求中最多能够接收和处理的 token 数量。它决定了模型一次性能“看到”的信息上限。

- **影响可处理信息量**：窗口越大，模型一次可容纳的文本、代码、表格、多轮对话或文档内容越多。

- **影响应用场景**：短窗口，如 4K-8K，适合简单问答和短文本处理；中长窗口，如 32K-128K，适合长文档分析、多轮对话和代码理解；超长窗口，如 1M+，可用于超长文档、大规模代码库或长转写内容分析。

- **影响成本和延迟**：更长上下文通常意味着更高的计算量、显存占用、响应延迟和调用费用。

- **不等于必然理解更好**：上下文窗口越大，只代表可输入的信息更多；模型能否准确利用长距离信息，还取决于模型架构、训练方式、位置编码、注意力优化和检索增强等能力。

> **注**：上下文窗口长度正在持续提升，但实际应用中仍需要结合 RAG、分块、摘要、缓存等方法，提高长文本处理的准确性和成本效率。

<h2 id="面试问题：为什么称为大模型？大体现在哪些维度？">面试问题：为什么称为“大”模型？“大”体现在哪些维度？</h2>

**难度评分：⭐ (1/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

之所以称为 **大模型**，是因为它在参数规模、训练数据、计算资源、上下文长度和能力范围等多个维度上，都显著大于传统机器学习模型或传统 NLP 模型。

![](./imgs/1-3-1.png)

##### “大”主要体现在以下几个维度：

- **参数量大**：大模型通常包含数十亿、数百亿、千亿甚至更高规模的可学习参数。对于 MoE 模型，还需要区分 **总参数量** 和每次推理实际使用的 **激活参数量**。

- **数据量大**：大模型通常基于海量文本、代码、网页、书籍、论文、多模态数据等进行训练，训练语料规模常以万亿级 Token 计量。

- **计算量大**：大模型的训练需要大规模 GPU/TPU 集群和复杂的分布式训练系统；推理阶段也需要较高的显存、算力和工程优化能力。

- **上下文窗口大**：现代大模型通常支持更长的上下文窗口，可以处理长文档、多轮对话、复杂代码库、表格和任务流程等更大规模的输入信息。

- **能力范围大**：大模型不再只解决单一任务，而是可以通过 Prompt 或指令完成问答、摘要、翻译、写作、代码生成、推理分析、工具调用等多种任务。

这些维度共同推动了大模型能力的提升，使其具备传统小模型通常不具备的 **泛化能力、上下文学习能力和多任务适配能力**。

<h2 id="面试问题：大语言模型与传统自然语言处理（NLP）模型的核心区别是什么？">面试问题：大语言模型与传统自然语言处理（NLP）模型的核心区别是什么？</h2>

**难度评分：⭐ (1/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

大语言模型与传统自然语言处理（NLP）模型的核心区别，主要体现在 **架构、训练范式、任务适配方式和能力边界** 上。

- **架构差异**：传统 NLP 模型包括规则系统、统计模型、机器学习模型以及早期深度学习模型；大语言模型通常基于 Transformer 架构，参数规模更大，上下文建模能力更强。

- **训练范式不同**：传统 NLP 模型通常依赖特定任务的人工标注数据进行监督训练；大语言模型通常先在海量无标注数据上进行自监督预训练，再通过指令微调、RLHF、DPO 等后训练方式提升对话能力、指令遵循能力和安全性。

- **任务适配方式不同**：传统 NLP 模型往往是“一个任务训练一个模型”，例如分类、分词、命名实体识别、情感分析等任务需要分别建模；大语言模型可以通过 Prompt、指令或少量示例适配多种任务。

- **泛化能力不同**：传统 NLP 模型在特定任务上可以表现稳定，但跨任务迁移能力较弱；大语言模型具备更强的零样本、少样本和上下文学习能力，可以在未专门训练的任务上完成一定程度的泛化。

- **输出形式不同**：传统 NLP 模型多输出标签、类别、序列标注等结构化结果；大语言模型可以生成自然语言、代码、推理过程、工具调用参数等更灵活的内容。

- **应用边界不同**：传统 NLP 模型主要面向文本分类、分词、命名实体识别、机器翻译等单点任务；大语言模型可以扩展到问答、摘要、写作、代码生成、RAG、智能体、工具调用和多模态理解等复杂场景。

<h2 id="面试问题：当前主流的大语言模型有哪些？">面试问题：当前主流的大语言模型有哪些？</h2>

**难度评分：⭐ (1/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

当前主流大语言模型可以按 **海外闭源模型、国内主流模型、开源/开放权重模型** 三类理解。

- **海外主流闭源模型**：GPT 系列（OpenAI）, Claude 系列（Anthropic）, Gemini 系列（Google DeepMind）。
- **国内主流模型**：Qwen 系列（阿里巴巴），DeepSeek 系列（DeepSeek），Doubao 系列（字节跳动），Kimi 系列（月之暗面）

- ** 开源或开放权重模型**：Llama 系列（Meta），Mistral / Mixtral 系列（Mistral AI），Qwen、DeepSeek、GLM 等国产开放模型。

- **不同模型侧重点不同**，需要结合通用能力、推理、代码、长上下文、多模态、成本和部署方式综合选择。

其他模型和实时排名可参考 Artificial Analysis、LMArena、OpenCompass 等评测榜单。

<h2 id="面试问题：什么是闭源/开源大模型？什么是通用/垂直大模型？">面试问题：什么是闭源/开源大模型？什么是通用/垂直大模型？</h2>

**难度评分：⭐ (1/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

##### 1. 什么是闭源/开源大模型？

**闭源/开源大模型** 是按照模型的开放程度进行分类。实际面试中更严谨的说法是：区分 **闭源模型、开放权重模型和真正开源模型**。

| 分类 | 核心定义 | 代表模型 | 核心特点 |
| :--- | :--- | :--- | :--- |
| 闭源大模型 | 不开放模型权重，通常通过 API、在线平台或企业服务提供能力 | GPT、Claude、Gemini 等 | 开箱即用、服务成熟、综合能力强；但本地部署、深度定制和底层可控性较弱 |
| 开放权重模型 | 开放模型权重，允许用户下载、部署和微调，但许可证可能限制商用或再分发 | Llama、Qwen、DeepSeek、GLM、Mistral 等部分模型 | 可本地部署、可微调、数据可控性更强；但需要算力、工程能力和安全治理 |
| 完全开源模型 | 权重、代码、训练细节、数据处理流程等开放程度更高，并遵循较开放的许可证 | 具体需看模型许可证和发布内容 | 透明度更高，适合研究和深度改造；但完整复现成本通常很高 |

> **注**：很多被称为“开源大模型”的模型，本质上更准确地说是 **开放权重模型**。是否能商用、是否能再分发、是否能修改，需要查看具体许可证。

##### 2. 什么是通用/垂直大模型？

按照 **能力覆盖范围和应用场景**，大模型可以分为通用大模型和垂直大模型。

| 分类 | 核心定义 | 代表模型 | 核心特点 |
| :--- | :--- | :--- | :--- |
| 通用大模型 | 面向多领域、多任务训练，具备较强的通用理解、生成、推理和泛化能力 | GPT、Claude、Gemini、Qwen、GLM、DeepSeek 等通用系列 | 能力覆盖广，适合作为底座模型；但在高专业度场景中通常需要 RAG、微调或工具增强 |
| 垂直大模型 | 面向特定行业或任务优化，通常结合领域数据、微调、RAG、工具链和业务流程适配 | 代码模型、医疗模型、法律模型、金融模型、工业模型等 | 在特定领域更贴合业务需求；但跨领域泛化能力通常弱于通用模型 |

例如，**DeepSeek-Coder、Qwen-Coder** 更偏代码领域模型；医疗、法律、金融、客服等行业模型则属于典型垂直大模型或行业大模型。

##### 3. 其他常见的分类标准有哪些？

除了开源属性和能力范围，大模型还可以按照参数规模、模型架构、模态类型和部署方式进行分类。

- **按参数规模分类**：可以粗略分为小模型、中型模型、大模型和超大模型，例如 7B、14B、32B、70B、100B+ 等。但这些阈值不是绝对标准，不同机构口径可能不同。

- **按模型架构分类**：常见 Transformer 架构包括 Encoder-only、Decoder-only、Encoder-Decoder。其中主流生成式大语言模型多采用 **Decoder-only Transformer**。

- **MoE 架构**， **模态类型**，**部署方式**等等。

<h2 id="面试问题：什么是基座模型？什么是对话模型？什么是推理模型？">面试问题：什么是基座模型？什么是对话模型？什么是推理模型？</h2>

**难度评分：⭐ (1/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

基座模型、对话模型和推理模型，可以理解为大语言模型在不同训练阶段和能力优化方向上的不同产物。其中，**基座模型**通常是后续对话模型、指令模型、推理模型和垂直模型的基础。

- **基座模型**：也叫基础模型或 Base Model，是在海量通用语料上通过自监督学习完成预训练的模型。它的核心目标是**学习语言规律、语法结构、语义关系和世界知识，从而具备基础的理解与生成能力**。基座模型通常没有经过充分的人类意图对齐，指令遵循能力和对话体验较弱，更偏向文本续写或补全。 （例：DeepSeek-Base、Qwen-Base、Llama-Base 等。）

- **对话模型**：通常是在基座模型基础上，经过指令微调、偏好学习和安全对齐后得到的模型，目标是**更好地理解用户意图、遵循指令并进行多轮人机对话**。常见训练方式包括 SFT、RLHF、DPO 等。  （例：ChatGPT、Claude、Gemini、豆包、Qwen-Chat、GLM-Chat 等。）

- **推理模型**：是在通用模型基础上，针对数学、代码、逻辑推理、多步规划等能力进行专项增强的模型。它通常会使用高质量推理数据、代码和数学数据、思维链数据、可验证任务反馈，以及强化学习等方法提升复杂问题求解能力。现代推理模型还常通过更长的思考过程、搜索、采样和测试时计算扩展来提升答案质量。  （例：OpenAI o 系列、GPT-5 Thinking、DeepSeek-R1、Gemini Thinking、Claude Extended Thinking 等。）

这三类模型也对应了大语言模型研发中常见的训练阶段：**预训练、微调和对齐（偏好对齐，推理增强）**。

| 阶段 | 目标 | 数据 | 代表产物 |
| :--- | :--- | :--- | :--- |
| **预训练** | 学习语言规律、知识表示和基础生成能力 | 海量无标注文本、代码、多模态数据等 | 基座模型，如 DeepSeek-Base、Qwen-Base |
| **指令微调** | 适配指令格式、任务要求和对话场景 | 高质量指令数据、问答数据、任务数据 | 对话模型、指令模型、垂直模型 |
| **偏好对齐** | 提升人类偏好、安全性、可靠性和指令遵循能力 | 人类偏好数据、AI 反馈数据、拒答与安全数据 | 安全对齐后的商用对话模型 |
| **推理增强** | 提升数学、代码、逻辑、多步规划等复杂推理能力 | 推理数据、代码数据、数学数据、可验证任务反馈 | 推理模型、Thinking 模型 |

> **核心理解**：预训练决定模型的基础能力，对齐决定模型是否好用和安全，推理增强决定模型在复杂问题上的思考和求解能力。
> **注**：后续小节会详细拆分预训练、微调与对齐。

<h2 id="面试问题：大语言模型有哪些主流使用方式？">面试问题：大语言模型有哪些主流使用方式？</h2>

**难度评分：⭐ (1/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

根据使用主体、技术能力和应用场景，大语言模型的主流使用方式可以分为 4 类。

- **在线对话平台**（个人使用 / 零开发门槛）：直接使用大模型厂商提供的官方网页、App 或桌面端产品，无需代码开发和算力投入，适合办公辅助、内容创作、学习问答、资料总结等场景。 （例：使用 ChatGPT、Claude、Gemini、豆包、Kimi 等进行办公辅助和文档总结。）

- **API 接口调用**（产品集成 / 商用落地）：开发者通过 API 或 SDK 调用大模型能力，将模型集成到应用、系统和业务流程中，常用于智能客服、企业助手、内容生成、代码助手、数据分析等场景。  （例：使用 OpenAI API、Qwen API 或 DeepSeek API 构建智能客服、知识库问答或数据库查询助手。）

- **开源或开放权重模型本地部署**（私有化 / 数据敏感场景）：下载开源或开放权重模型，在本地服务器、私有云或企业内网中部署，适合对数据安全、合规、成本控制和系统可控性要求较高的场景。  （例：政企、金融、医疗等机构私有化部署 Qwen、DeepSeek、Llama、GLM 等模型。）

- **二次开发与定制优化**（垂直领域落地）：基于通用大模型，结合行业数据和业务流程进行增强，常见方式包括 Prompt Engineering、RAG、工具调用、Agent 编排、SFT、LoRA/QLoRA 微调等。对于极高要求的场景，才会考虑全参微调或继续预训练。  （例：基于医疗文献和病例知识库构建医疗问答助手，或基于企业内部文档构建 RAG 知识库系统。）

> **注**：API 调用代码应参考各模型厂商官方文档；本地部署工具和推理框架可参考 vLLM、Ollama、SGLang、TensorRT-LLM、llama.cpp 等部署方案。

<h2 id="面试问题：什么是大模型的“涌现能力”？">面试问题：什么是大模型的“涌现能力”？</h2>

**难度评分：⭐⭐ (2/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

**涌现能力**（Emergent Abilities）是指大语言模型在规模较小时表现很弱或几乎不可观察，但当模型规模、训练数据量和计算量提升到一定程度后，某些复杂能力会出现明显的非线性提升，表现为模型突然能够完成原来难以完成的任务。（ 《Emergent Abilities of Large Language Models》）**涌现能力描述的是大模型能力随规模增长出现非线性增强的现象**。

##### 1. 核心特征

- **非线性**：能力不是简单随参数量线性提升，而是在某些规模区间出现明显跃升。

- **规模相关**：通常与参数量、训练 token 数、计算量、数据质量和训练方式有关，而不只由参数量决定。

- **任务特异性**：不同任务的涌现阈值不同。简单语言任务可能较早出现，复杂推理、代码、数学和工具使用能力通常需要更大规模和更强训练。

- **难以精确预测**：很难仅根据小模型表现，准确预测某种能力会在多大规模、以何种程度出现。

##### 2. 典型能力

- **上下文学习能力**：模型可以根据 prompt 中给出的少量示例，临时理解任务规则并完成新样本预测。  （例：给出 3 个情感分类示例后，模型可以直接对新的句子进行情感判断。）

- **复杂指令遵循能力**：模型能够理解并执行多约束、多步骤的自然语言指令。  （例：将一段产品说明改写成小红书种草文案，提炼 3 个卖点，并翻译成英文。）

- **多步推理能力**：模型在数学、逻辑、代码和规划任务中，可以进行多步骤分析和中间过程组织。  （例：分步骤解数学题、分析因果关系、规划任务流程。）

- **代码与工具调用能力**：模型可以生成代码、理解工程问题，并调用搜索、数据库、API、解释器等工具完成复杂任务。  （例：根据需求生成脚本、修复代码问题、调用 API 完成数据分析。）

涌现能力说明，大模型并不是传统小模型的简单放大版。随着模型规模、数据规模和计算量提升，模型可能出现更强的上下文学习、指令遵循、推理、代码和工具使用能力。但**涌现能力不是魔法，也不是只靠参数量自动产生**。现代大模型的能力通常来自预训练规模、数据质量、模型架构、后训练对齐、推理增强和测试时计算等因素共同作用。

> **注**：不只来自规模，也依赖高质量推理数据、思维链训练、强化学习、工具使用等。

<h2 id="面试问题：为什么小模型也开始具备一定的涌现能力？">面试问题：为什么小模型也开始具备一定的涌现能力？</h2>

**难度评分：⭐⭐ (2/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

小模型开始具备一定的复杂能力，并不意味着涌现能力完全不再依赖规模，而是说明模型能力不只由参数量决定，还受到 **数据质量、训练策略、模型架构、后训练方法和推理增强方式** 的共同影响。

##### 1. 为什么小模型也能表现出较强能力？

- **高质量训练数据提升**：小模型如果使用经过筛选、去噪、高密度知识覆盖的数据训练，效果可能明显优于使用低质量海量数据训练的模型。

- **合成数据和蒸馏增强**：通过大模型生成高质量指令数据、推理数据和代码数据，小模型可以学习大模型的解题过程、表达方式和任务模式，这相当于把大模型的一部分能力迁移给小模型。

- **模型架构优化**：更高效的 Transformer 结构、GQA/MQA、RoPE 改进、长上下文优化、MoE、小型专家模型等方法，可以提升小模型的训练效率和推理效率。

- **后训练能力增强**：指令微调、偏好对齐、DPO、强化学习、可验证奖励训练等方法，可以显著提升小模型的指令遵循、数学推理、代码生成和对话表现。

- **任务和领域更聚焦**：小模型如果专门面向代码、数学、医疗、法律、客服等垂直领域优化，在特定任务上可能接近甚至超过通用大模型。

- **RAG 和工具调用补足能力**：小模型结合检索增强、数据库、代码解释器、搜索工具和业务 API，可以弥补自身参数知识和推理能力不足。

- **测试时计算扩展**：通过多轮思考、多样本采样、自我验证、搜索和反思等方式，小模型在推理阶段也能获得更好的任务表现。

##### 2. 和传统涌现能力有什么区别？

传统意义上的涌现能力强调：随着模型规模扩大，某些复杂能力出现非线性增强。

而现在小模型表现出较强能力，很多时候并不是单纯“参数规模带来的自然涌现”，而是来自：

- 大模型蒸馏；
- 高质量合成数据；
- 指令微调和偏好对齐；
- 领域数据专项训练；
- RAG、工具调用和推理时计算增强。

因此，小模型的能力提升更像是 **训练效率和能力迁移的结果**，不应简单理解为“小模型天然具备了和大模型一样的涌现能力”。

<h2 id="面试问题：什么是大模型的缩放定律？">面试问题：什么是大模型的缩放定律？</h2>

**难度评分：⭐⭐⭐ (3/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

**缩放定律**（Scaling Law）是指在模型架构、训练方法和数据分布相对稳定的前提下，大语言模型的预训练损失通常会随着 **参数量 N**、**训练数据量 D** 和 **计算量 C** 的增加，呈现较稳定的幂律下降关系。（当模型、数据和算力按合理比例扩大时，模型的验证集 Loss 通常会平滑、可预测地下降，语言建模能力和部分下游能力也会随之提升。）

缩放定律主要描述的是 **预训练 Loss 的变化规律**，不能简单等同于所有下游任务能力、推理能力或产品体验。实际能力还会受到数据质量、模型架构、后训练、对齐、推理策略和评测方式影响。

##### 1. OpenAI 原始缩放定律

OpenAI 在论文 **《Scaling Laws for Neural Language Models》** 中提出，当其他因素足够充足、不成为瓶颈时，模型损失与参数量、数据量、计算量分别呈幂律关系。

- **参数量缩放**：参数量越大，Loss 越低，但下降幅度会逐渐变小。

$$
L(N) \propto N^{-\alpha_N}
$$

- **数据量缩放**：训练 token 越多，Loss 越低，但同样存在边际收益递减。

$$
L(D) \propto D^{-\alpha_D}
$$

- **计算量缩放**：训练计算量越大，Loss 越低。

$$
L(C) \propto C^{-\alpha_C}
$$

其中：

- $N$ 表示模型参数量；
- $D$ 表示训练数据量，通常以 token 数计量；
- $C$ 表示训练计算量，通常以 FLOPs 计量；
- $L$ 表示验证集交叉熵损失，Loss 越低，语言建模效果通常越好；
- $\alpha$ 表示幂律指数，反映对应变量扩大后 Loss 下降的速度。

OpenAI 原始缩放定律的一个重要结论是：在固定计算预算下，更倾向于增加模型参数量，而数据量增长相对较慢，可近似表示为：

$$
N_{opt} \propto C^{0.73}, D_{opt} \propto C^{0.27}
$$

这个结论推动了早期大模型训练中的“增大参数量”趋势，但后续研究发现，许多大模型其实存在 **参数很大、训练 token 不足、欠训练** 的问题。

##### 2. DeepMind Chinchilla 修正缩放定律

DeepMind 在论文 **《Training Compute-Optimal Large Language Models》** 中提出了 Chinchilla Scaling Law，修正了早期对参数量和数据量配比的认识。

Chinchilla 的核心观点是：在固定算力预算下，不能只增大模型参数量，还要同步增加训练数据量。很多早期大模型参数量很大，但训练 token 不足，因此不是计算最优的。

其损失函数可简化理解为：

$$
L(N, D) = L_{\infty} + \frac{A}{N^{\alpha}} + \frac{B}{D^{\beta}}
$$

其中：

- $L_{\infty}$ 表示不可约损失，即理论上的最低损失；
- $N$ 表示模型参数量；
- $D$ 表示训练 token 数；
- $A$、$B$、$\alpha$、$\beta$ 是通过实验拟合得到的常数。

在计算量近似满足：

$$
C \approx 6ND
$$

的情况下，Chinchilla 给出的计算最优结论是：

$$
N_{opt} \propto C^{0.5}, D_{opt} \propto C^{0.5}
$$

也就是说，**模型参数量和训练数据量应该近似同步扩展**。一个常见经验是：计算最优训练中，训练 token 数大约是模型参数量的 20 倍左右。

例如：

| 参数量 | 计算最优训练 token 数量 |
| :--- | :--- |
| 1B | 约 20B tokens |
| 10B | 约 200B tokens |
| 70B | 约 1.4T tokens |
| 175B | 约 3.5T tokens |
| 1T | 约 20T tokens |

##### 3. 缩放定律的核心意义

- **指导模型训练配比**：帮助判断在固定算力下，应该增大模型参数量，还是增加训练数据量。

- **预测训练效果**：通过小规模实验外推大规模训练的 Loss，降低盲目训练超大模型的风险。

- **解释模型能力提升**：随着参数、数据和算力扩大，模型的语言建模能力、泛化能力和部分复杂能力通常会增强。

- **推动高质量数据的重要性**：现代大模型训练不再只追求参数量，也强调训练 token 数、数据质量、数据配比和去重清洗。

##### 4. 局限性

- **主要预测预训练 Loss**：Loss 下降不一定直接等于推理能力、对话体验或工具调用能力提升。

- **依赖训练条件**：架构、优化器、数据质量、训练策略变化后，原有缩放规律可能需要重新拟合。

- **存在边际收益递减**：参数、数据和算力继续扩大仍可能提升效果，但单位收益会下降。

<h2 id="面试问题：缩放定律对模型规模的扩展有何指导意义？">面试问题：缩放定律对模型规模的扩展有何指导意义？</h2>

**难度评分：⭐⭐  (2/5)  |  考察频率：⭐⭐⭐ (3/5)**

缩放定律的核心意义是：帮助我们在给定算力预算下，更合理地规划 **模型参数量、训练数据量和计算量** 的配比，从而降低盲目堆参数或盲目增加数据带来的训练浪费。

- **指导模型研发的资源规划**：缩放定律可以用于估算在固定算力预算下，模型参数量和训练 token 数应该如何分配。尤其是 Chinchilla 定律指出，参数量和训练数据量需要相对均衡扩展，避免出现“参数很大但训练 token 不足”的欠训练模型。

- **支持小规模实验外推大规模训练效果**：研发团队可以先在小模型、小数据和较低算力上做实验，拟合 loss 与参数量、数据量、计算量之间的关系，再外推到更大规模训练，降低大规模训练失败的风险。

- **帮助模型选型与微调决策**：在选择基座模型时，不能只看参数量大小，还要看模型是否训练充分、数据质量如何、训练 token 是否足够。同等条件下，训练充分的小模型可能比欠训练的大模型更适合微调和部署，而且推理成本更低。

- **解释部分能力提升现象**：随着参数量、数据量和计算量扩大，模型预训练 loss 通常会下降，部分下游任务能力也可能增强。某些复杂能力可能在模型达到一定规模后表现出非线性提升，这与涌现能力有关。但涌现能力不只由规模决定，还与数据质量、后训练、评测方式和推理策略有关。

- **明确规模扩展的边界和瓶颈**：缩放定律也说明，继续扩大规模会存在边际收益递减。模型能力提升不能只依赖堆参数，还会受到高质量数据供给、数据多样性、训练稳定性、推理成本、硬件效率和对齐方法的限制。

<h1 id="4.大语言模型存在哪些内在的局限性？">4.大语言模型存在哪些内在的局限性？</h1>

<h2 id="面试问题：什么是“幻觉”？其产生的主要原因有哪些？">面试问题：什么是“幻觉”？其产生的主要原因有哪些？</h2>

**难度评分：⭐⭐ (2/5)  |  考察频率：⭐⭐⭐⭐⭐ (5/5)**

##### 1. 什么是“幻觉”？

大语言模型的 **幻觉**（Hallucination）是指模型生成了 **看似流畅、语法正确、表达自信**，但实际上 **事实不准确、与上下文矛盾，或缺乏可靠证据支持** 的内容。（例：模型“说得像真的”，但内容可能是错的、编的、过时的，或者无法被证据支持。）

从技术分类看，幻觉常见可以分为 **内在幻觉** 和 **外在幻觉**：

- **内在幻觉**：模型输出与用户输入、给定上下文或提供资料相矛盾，可以直接通过上下文对照发现。  （例：资料中写“公司成立于 2019 年”，模型却回答“公司成立于 2016 年”。）

- **外在幻觉**：模型输出无法从输入上下文中得到支持，或与外部真实世界事实不一致，需要借助外部知识库、搜索引擎、数据库或人工核查验证。  （例：模型编造不存在的论文、法律条文、新闻事件、API 参数或参考文献。）

也可以从表现形式上分为：

- **事实不一致**：与真实世界事实矛盾；
- **上下文不一致**：与输入资料或对话历史矛盾；
- **逻辑不一致**：输出内部前后矛盾或推理链条不成立。

##### 2. 幻觉产生的主要原因有哪些？

大语言模型产生幻觉，主要可以从 **训练数据、模型机制、推理过程和系统设计** 四个层面理解。

- **训练数据问题**：训练数据可能存在错误、噪声、过时信息、互相矛盾的信息，或者对某些领域覆盖不足，导致模型学习到不完整或不准确的知识。

- **模型目标限制**：大语言模型的核心训练目标通常是预测下一个 token，而不是直接验证事实真伪。因此，模型更擅长生成“概率上合理”的文本，不天然等同于事实数据库或搜索引擎。

- **参数化知识的局限**：模型知识主要压缩在参数中，更新困难，也缺乏显式的出处和可追溯证据。当问题涉及冷门知识、实时信息或专业细节时，更容易出现编造。

- **不确定性校准不足**：模型有时无法准确判断自己是否知道答案，可能在缺乏证据时仍然给出自信回答。

- **上下文窗口限制**：当输入内容过长、上下文被截断或关键信息被淹没时，模型可能遗漏事实依据，导致回答与上下文脱节。

- **解码策略影响**：较高 temperature、top-p 采样等策略会增加输出多样性，但也可能提高生成不稳定或事实错误内容的概率。

- **对齐目标的副作用**：模型经过对话对齐后，往往倾向于“有帮助地回答问题”。如果没有良好的拒答和不确定性表达机制，可能会在不知道时仍然尝试生成答案。

##### 3. 幻觉的核心危害有哪些？

幻觉的危害主要体现在以下几个方面：

- **专业决策风险**：在医疗、法律、金融、政务等领域，错误答案可能误导关键判断。

- **信息可信度下降**：模型编造事实、引用或数据，会降低用户对系统和机构的信任。

- **企业合规与声誉风险**：错误内容可能引发版权、名誉、合规、合同和客户服务纠纷。

- **安全风险**：在代码生成、自动化执行、工具调用等场景中，幻觉可能导致错误操作、系统故障或安全漏洞。

##### 4. 幻觉的检测手段和缓解策略有哪些？

a. **幻觉的检测手段**

- **检索验证**：将模型输出与搜索结果、知识库、数据库、文档原文进行比对，验证关键事实是否有依据。

- **引用与溯源检查**：要求模型给出引用来源，并检查引用是否真实存在、是否支持模型结论。

- **一致性检测**：对同一问题多次生成，或使用多个模型交叉验证；如果答案差异很大，说明存在不确定性或幻觉风险。

- **NLI / 语义蕴含检测**：判断模型回答是否能被原始资料支持，是否与资料存在矛盾。

- **不确定性估计**：利用 logprob、置信度、校准模型或辅助分类器识别高风险回答。但需要注意，token 置信度不等于事实正确性，只能作为辅助信号。

- **人工审核**：在医疗、法律、金融、政务等高风险场景中，仍需要专业人员进行最终确认。

b. **幻觉的缓解策略**

- **RAG 检索增强生成**：先从外部知识库检索相关资料，再让模型基于资料回答，减少纯靠参数记忆生成带来的错误。

- **要求基于证据回答**：提示模型“只根据给定资料回答”“找不到依据就说明无法确定”，降低无依据编造。

- **引用来源与答案分离**：要求模型输出结论时附带来源、原文片段或证据编号，便于验证。

- **工具调用与实时检索**：对实时信息、数据库查询、计算、代码执行等任务，让模型调用搜索、数据库、计算器、代码解释器等工具。

- **降低随机性**：在事实问答、知识库问答等场景中，适当降低 temperature，减少不必要的创造性生成。

- **训练与对齐优化**：通过高质量事实数据、拒答数据、偏好数据、DPO/RLHF、事实性微调等方式，提升模型的真实性和不确定性表达能力。

- **验证器和后处理**：使用 verifier、规则校验、事实核查模型或 LLM-as-a-judge 对模型答案进行二次检查。

- **人机协同审核**：对高风险输出设置人工审核、灰度发布和追责机制，避免模型直接自动决策。

<h2 id="面试问题：大模型的知识为什么存在截止日期？如何获取实时信息？">面试问题：大模型的知识为什么存在截止日期？如何获取实时信息？</h2>

**难度评分：⭐⭐ (2/5)  |  考察频率：⭐⭐⭐⭐ (4/5)**

##### 1. 什么是知识截止日期？

**知识截止日期**（Knowledge Cutoff）是指模型训练数据覆盖到的时间点。对于这个时间点之后发生的新事件、新政策、新产品、新论文或实时数据，模型本体参数中通常并不包含相关知识。

##### 2. 为什么会产生知识截止日期？

知识截止日期来自大语言模型的训练机制。

大模型通过预训练学习大量文本、代码和其他数据中的模式与知识，而**训练数据本质上是某个时间段内的信息快照**。预训练完成后，知识主要以分布式方式压缩在模型权重中，**模型参数不会随着现实世界变化自动更新**。

因此，随着时间推移，新的新闻、政策、价格、论文、公司信息和技术版本不断出现，模型本体就可能出现知识过时或不知道最新信息的问题。

##### 3. 如何获取实时信息？

大模型获取实时信息主要有两类路线：**推理时接入外部信息** 和 **训练时更新模型知识**。

a. **推理时接入外部信息**，这是当前最常见的工程方案。

- **联网搜索**：调用搜索引擎或浏览器工具，获取新闻、网页、公告、论文等最新信息，再让模型基于搜索结果回答。

- **RAG 检索增强生成**：从企业知识库、向量数据库、文档库、官网资料、产品手册等外部知识源中检索相关内容，再提供给模型生成答案。

- **数据库/API 调用**：通过函数调用或工具调用访问实时数据库、业务系统、订单系统、库存系统、天气接口、金融行情接口等，获取结构化实时数据。

- **外部记忆或知识库同步**：定期更新知识库或索引，而不是频繁更新模型本身，使模型在推理时始终能查到最新资料。

- **人工审核与事实校验**：在医疗、法律、金融、政务等高风险场景中，对实时信息和模型答案进行人工确认。

b. **训练时更新模型知识**，适合需要长期固化能力或领域知识的场景。

- **继续预训练**：用新增数据继续训练模型，使模型吸收某个领域或时间段的新知识。

- **指令微调 / 领域微调**：用高质量领域问答、业务数据或标注数据，让模型更适配特定任务。

- **持续学习**：让模型不断学习新数据，但需要解决灾难性遗忘、数据污染、训练成本和安全评测问题。

<h2 id="面试问题：大语言模型在对齐和安全层面存在哪些无法彻底消除的内在局限？">面试问题：大语言模型在对齐和安全层面存在哪些无法彻底消除的内在局限？</h2>

**难度评分：⭐⭐⭐ (3/5)  |  考察频率：⭐⭐⭐ (3/5)**

大语言模型的对齐与安全问题很难被彻底消除，原因在于模型本身是从大规模人类语料中学习语言和行为模式，再通过后训练去约束输出。安全对齐可以显著降低风险，但很难保证在所有场景、所有用户、所有输入下都绝对安全。

- **数据偏见与分布不均**：LLM 的训练语料来自人类社会文本，而这些语料天然包含性别、地域、种族、职业、文化、阶层等偏见，也存在领域覆盖不均和数据质量差异。即使经过数据清洗和安全对齐，模型仍可能在某些群体、语言或场景上输出带偏见、不公平或刻板化的内容。

- **价值观冲突与对齐目标不唯一**：不同国家、文化、行业、组织和用户对“正确”“安全”“有帮助”的标准并不完全一致。模型对齐的目标不是单一数学真理，而是一组复杂的人类偏好和社会规范，因此很难做到对所有人、所有场景都完全一致且满意。

- **迎合性与真实目标冲突**：RLHF、DPO 等对齐方法通常基于人类偏好数据优化模型，使模型更有帮助、更自然、更符合用户期待。但这也可能导致模型过度迎合用户观点，即 sycophancy，甚至附和用户的错误判断。也就是说，“让用户满意”和“保持客观真实”之间可能存在冲突。

- **对齐税与过度拒答问题**：安全对齐可以降低有害输出，但如果约束过强或策略过粗，可能导致模型在正常医学、法律、网络安全、历史、创作等问题上过度拒答，影响可用性。反过来，如果约束过松，又会增加有害内容生成风险。因此需要在安全性、可用性和能力表现之间持续权衡。

- **越狱攻击与提示注入风险**：LLM 需要理解复杂自然语言，也就可能被攻击者通过角色扮演、指令混淆、编码绕过、多轮诱导等方式绕过安全策略。对于接入工具、浏览器、数据库或代码执行环境的模型，还存在 prompt injection、越权调用和数据泄露风险。

- **有害能力的组合泛化风险**：模型并不只是复述训练数据，还能组合已有知识生成新方案。即使某些危险文本被过滤，模型仍可能通过推理、类比和组合生成有害步骤，因此很难靠删除数据彻底解决风险。

- **不可解释性与验证困难**：LLM 的内部表示和决策过程并不完全可解释，安全测试也无法覆盖所有可能输入。红队测试、自动评测和安全基准只能发现一部分问题，无法证明模型在开放环境中绝对安全。

- **分布外输入与长尾场景风险**：现实用户输入非常复杂，可能包含罕见语言、隐喻、专业术语、恶意提示、多模态内容和上下文污染。模型在训练分布之外的场景中，安全策略更容易失效。

<h2 id="面试问题：大语言模型在哪些复杂任务上表现仍不理想？">面试问题：大语言模型在哪些复杂任务上表现仍不理想？</h2>

**难度评分：⭐⭐ (2/5)  |  考察频率：⭐⭐⭐⭐  (4/5)**

尽管大语言模型在问答、写作、摘要、代码和多模态任务上表现出色，但它仍然存在能力边界，尤其是在需要高可靠性、严格验证和长链条决策的复杂任务上表现不够稳定。

- **严谨数学与符号推理**：LLM 可以解决很多数学题，但在精确计算、形式证明、符号操作、边界条件和多步骤数学推导中仍可能出错。常见问题包括用错公式、漏掉约束、计算错误、过程看似合理但结论错误。

- **因果推断与反事实推理**：LLM 主要从数据中的语言模式和统计关联中学习知识，不天然具备严格的因果模型。因此在判断“因为什么导致什么”“如果条件改变会怎样”等因果和反事实问题时，容易把相关性误当因果性。

- **长链多步推理与规划**：在需要很多步骤的任务中，早期小错误可能影响后续推理，导致误差累积。若没有工具验证、搜索、反思或外部检查机制，模型很难稳定保证每一步都正确。

- **上下文学习的不稳定性**：LLM 对 prompt 表述、示例顺序、示例质量和上下文噪声比较敏感。在小样本、分布外或提示不清晰的场景中，可能学习到虚假规律，导致输出波动较大。

- **事实可靠性与实时知识**：模型参数中的知识存在截止日期，也可能记忆不完整或混淆相似事实。在涉及最新新闻、政策、价格、论文、API 文档等实时或细节型知识时，容易产生幻觉，需要结合检索或工具验证。

- **跨长文档一致性**：在长文档、多轮对话、复杂代码库或多源资料中，模型可能遗漏关键信息、混淆引用来源，或无法稳定保持全局一致性。

- **复杂工具调用与任务执行**：Agent 场景中，模型需要选择工具、构造参数、调用 API、读取结果并继续规划。它可能出现工具选择错误、参数编造、权限误用、执行顺序错误等问题。

- **空间、物理与真实世界约束推理**：在涉及空间关系、物理常识、工程约束、操作步骤和真实环境反馈的任务中，模型可能生成看似合理但不可执行的方案。

- **高风险专业决策**：医疗、法律、金融、政务等领域要求可解释、可追溯和高准确率。LLM 可以作为辅助工具，但不适合在缺乏验证和人工审核的情况下直接做最终决策。
