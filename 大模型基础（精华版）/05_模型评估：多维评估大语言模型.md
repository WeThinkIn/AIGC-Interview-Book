<h1 id="25.大语言模型评估的基本概念">25.大语言模型评估的基本概念</h1>

<h2 id="面试问题：为什么要评估大语言模型？需要评估哪些能力和风险？">面试问题：为什么要评估大语言模型？需要评估哪些能力和风险？</h2>

**难度评分：⭐⭐ (2/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

评估大语言模型的核心目的，是判断模型是否具备 **真实可用能力**，而不是只看训练 loss、reward score 或榜单分数。

完整评估不仅要看模型“会不会”，还要看它是否：

- **可靠**：事实是否准确，输出是否稳定；
- **安全**：是否会产生有害内容、越狱、隐私泄露；
- **可控**：是否遵循指令、格式和业务规则；
- **高效**：延迟、token 消耗和推理成本是否可接受；
- **适配场景**：是否满足具体业务需求和上线标准。

大模型评估通常用于：

- 判断模型是否具备通用语言理解、推理、代码、数学、长文本等能力；
- 比较不同模型、不同 checkpoint、不同微调方案之间的效果差异；
- 发现模型缺陷，例如幻觉、事实错误、偏见、不安全输出和指令不遵循；
- 指导预训练、微调、后训练和数据配比优化；
- 判断模型是否满足具体业务场景的上线要求。

<h2 id="面试问题：如何系统评估一个大语言模型？通常需要覆盖哪些能力维度和风险维度？">面试问题：如何系统评估一个大语言模型？通常需要覆盖哪些能力维度和风险维度？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

##### 1. 如何系统评估一个大语言模型？

系统评估大语言模型，不能只看某个榜单分数，而要建立覆盖 **能力、风险、稳定性、成本和业务效果** 的评估矩阵。

大语言模型评估通常覆盖：

- **基础能力**：语言理解、文本生成、知识问答、推理、数学、代码；
- **交互能力**：指令遵循、多轮对话、结构化输出、工具调用；
- **长上下文能力**：长文档理解、长程依赖、跨段落信息整合；
- **安全风险**：幻觉、越狱、隐私泄露、偏见、版权和合规风险；
- **工程性能**：延迟、吞吐、显存、并发、推理 token 成本；
- **业务效果**：任务完成率、用户满意度、问题解决率、人工接管率。

系统评估的目标不是证明模型“某个分数高”，而是判断模型在目标场景中是否 **有用、可靠、安全、稳定、可部署**。

##### 2. 为什么要多维度评估？

大语言模型是通用模型，不同能力之间并不完全同步提升。

例如：

- 模型知识能力强，不代表数学推理强；
- 文本生成流畅，不代表事实准确；
- benchmark 分数高，不代表安全性好；
- 指令跟随强，不代表长上下文能力好；
- 微调后业务能力提升，可能导致通用能力退化；
- 推理模型准确率提升，可能带来更高 reasoning token 成本；
- Agent 工具调用能力强，不代表端到端任务可靠。

因此，模型评估不能只看单一榜单，而要根据模型定位建立评估矩阵。

##### 3. 不同模型应重点评估什么？

| 模型类型 | 评估重点 |
|---|---|
| 通用基座模型 | 语言理解、知识、推理、代码、数学、长上下文 |
| 对话助手模型 | 指令遵循、多轮对话、生成质量、安全对齐 |
| 推理模型 | 数学、代码、逻辑推理、pass@1/pass@k、推理稳定性、推理 token 成本 |
| 代码模型 | 代码生成、补全、修复、单元测试通过率、SWE-bench 类任务 |
| 行业模型 | 领域知识、事实准确性、合规性、专家评估 |
| 长上下文模型 | 长文档检索、长程依赖、上下文一致性、needle test |
| RAG 系统 | 检索召回率、引用忠实性、答案 groundedness、拒答边界 |
| Agent 模型 | 任务完成率、规划能力、工具调用正确率、端到端稳定性 |
| 部署模型 | 延迟、吞吐、显存占用、并发能力和成本 |

##### 4. 系统评估还要注意什么？

- **评测集污染**：公开 benchmark 可能出现在训练数据中，导致分数虚高。

- **评测集版本管理**：内部测试集要固定版本，便于比较不同模型和 checkpoint。

- **统计显著性**：模型差异较小时，需要足够样本量和显著性检验，避免误判。

- **人工标注一致性**：人工评测需要明确评分标准，并检查标注员之间的一致性。

- **LLM-as-judge 校准**：需要控制答案顺序、长度偏好、judge 模型偏见，并进行人工抽检。

- **回归评估机制**：每次微调或模型升级后，都要检查是否破坏旧能力、安全边界和格式输出能力。

- **业务指标闭环**：最终还要看真实业务效果，例如解决率、转人工率、用户满意度、响应时延和成本。


<h2 id="面试问题：如何构建大语言模型评估体系？离线基准、人工偏好、LLM-as-a-judge和线上A/B测试分别解决什么问题？">面试问题：如何构建大语言模型评估体系？离线基准、人工偏好、LLM-as-a-judge 和线上 A/B 测试分别解决什么问题？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

##### 1. 如何构建大语言模型评估体系？

大语言模型评估可以按 **评估方式、评估阶段和任务类型** 三个角度划分。

- 按评估方式：自动评估、人工评估、LLM-as-a-judge。
- 按评估阶段：预训练模型评估、指令微调模型评估、后训练模型评估、上线后在线评估。
- 按任务类型：通用能力评估、专项能力评估、安全性评估、业务场景评估。

不同评估类型解决的问题不同，实际项目中通常会组合使用，而不是依赖单一评估方式。一个成熟评估体系通常会先用离线基准和自动评估做大规模筛选，再用 LLM-as-a-judge 和人工偏好评估开放式回答，最后在通过安全门槛后做小流量线上 A/B 测试。

##### 2. 离线基准、人工偏好、LLM-as-a-judge 和线上 A/B 分别解决什么问题？

| 评估方式 | 主要解决的问题 | 优点 | 局限 |
|---|---|---|---|
| 离线基准 / benchmark | 快速比较模型在标准任务上的能力，如知识、数学、代码、推理 | 成本低、可重复、便于横向比较 | 可能被污染，和真实业务分布不一致 |
| 人工偏好评估 | 判断开放式回答是否真正有用、正确、自然、安全 | 可信度高，能发现细节问题 | 成本高、速度慢、主观性强 |
| LLM-as-a-judge | 批量评估开放式回答、偏好比较和质量初筛 | 效率高、维度灵活、成本较低 | 可能有评审偏见、位置偏差和长度偏好 |
| 线上 A/B 测试 | 验证模型在真实用户场景中的效果 | 最接近真实业务价值 | 成本高，有风险，需要安全和质量门槛 |

##### 3. 为什么要组合多种评估方式？

单一评估方式容易产生盲区：

- 自动评估快，但无法完整判断回答是否有帮助；
- 人工评估准，但成本高，难以高频运行；
- LLM-as-a-judge 高效，但可能偏好长答案或受提示影响；
- 离线 benchmark 稳定，但可能和真实线上分布不一致；
- 线上 A/B 真实，但需要先通过安全和质量门槛。

因此，成熟评估体系通常是：

```text
离线 benchmark / 自动评估
→ LLM-as-a-judge 批量初筛
→ 人工偏好评估和专家抽检
→ 安全红队测试
→ 小流量线上 A/B
→ 线上反馈回流
```

线上 A/B 测试不能替代离线质量和安全评估。对于可能产生安全、合规或业务损失的模型，必须先通过离线回归、安全红队和人工验收，再进入小流量实验。
