<h1 id="补充：Qwen">补充：Qwen</h1>

> **范围**：本补充依据 `qwen-series-analysis/Qwen系列模型技术讲解.md` 中整理的公开技术报告和官方材料，重点介绍 Qwen 主线模型及重要衍生模型。未公开的训练数据明细、完整配方和内部实现不作推测。

<h1 id="1.Qwen 系列如何从第一代演化到 Qwen3？">1.Qwen 系列如何从第一代演化到 Qwen3？</h1>

<h2 id="面试问题：什么是 Qwen 系列？它包含哪些主要模型类型？">面试问题：什么是 Qwen 系列？它包含哪些主要模型类型？</h2>

**难度评分：⭐ (1/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

Qwen，也称通义千问，是阿里巴巴推出的大模型系列。它不是单个模型，而是一组覆盖不同规模、架构、模态和任务方向的模型家族。

主要类型包括：

- **主线语言模型**：提供 Base 和 Instruct/Chat 版本，用于通用问答、写作、代码和工具调用
- **Dense 与 MoE 模型**：Dense 每次使用大部分模型参数；MoE 根据 Token 选择少量专家参与计算
- **推理模型或推理模式**：面向数学、代码和复杂推理，并允许使用更多测试时计算
- **专项模型**：包括 Qwen-Coder、Qwen-Math、Qwen-VL 和 Qwen-Omni 等

理解 Qwen 系列时，需要同时看模型版本、参数规模、开放方式、上下文长度、是否支持 Thinking，以及它面向的任务和模态。

<h2 id="面试问题：Qwen、Qwen2、Qwen2.5 和 Qwen3 的核心差异是什么？">面试问题：Qwen、Qwen2、Qwen2.5 和 Qwen3 的核心差异是什么？</h2>

**难度评分：⭐⭐ (2/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

四代主线模型的演化重点不同：

| 版本 | 主要变化 | 解决的问题 |
| :--- | :--- | :--- |
| Qwen | 建立中英双语、代码、数学、ChatML 和工具调用基础 | 形成完整的通用模型与对话模型体系 |
| Qwen2 | 引入 GQA、MoE、DCA 和 YaRN，扩大多语言数据 | 提高推理效率、模型容量和长上下文能力 |
| Qwen2.5 | 扩大高质量数据、SFT 和多阶段强化学习 | 提升指令遵循、结构化输出、代码、数学和长文本能力 |
| Qwen3 | 统一 Thinking 与 Non-thinking，引入 Thinking Budget | 让同一个模型在推理质量、延迟和成本之间切换 |

Qwen 的主线并不是每一代都更换 Transformer 骨架，而是在相对稳定的架构上持续优化数据、效率、长上下文和后训练。

<h2 id="面试问题：Qwen 系列的训练数据和能力重点是如何演化的？">面试问题：Qwen 系列的训练数据和能力重点是如何演化的？</h2>

**难度评分：⭐⭐ (2/5) | 考察频率：⭐⭐⭐ (3/5)**

公开报告中的训练数据规模大致从第一代的最高约 3T Token，增长到 Qwen2 的约 7T、Qwen2.5 的 18T，Qwen3 达到约 36T Token。

数据规模增加只是表面变化，更重要的是数据治理和任务覆盖持续增强：

- 第一代重点建立中文、英文、代码、数学和工具调用能力
- Qwen2 扩展多语言覆盖，并增加合成数据和执行反馈
- Qwen2.5 强化高质量领域数据、结构化数据、长文本和可验证任务
- Qwen3 进一步扩大多语言、STEM、代码、推理和长上下文训练

因此，版本升级不能只理解为“使用更多 Token”，还包括数据配比、去重过滤、质量评估、合成数据验证和后训练方法的变化。

<h1 id="2.Qwen 系列采用了哪些核心架构与效率优化？">2.Qwen 系列采用了哪些核心架构与效率优化？</h1>

<h2 id="面试问题：Qwen 主线模型采用了哪些共同的基础架构？">面试问题：Qwen 主线模型采用了哪些共同的基础架构？</h2>

**难度评分：⭐⭐ (2/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

Qwen 主线模型采用自回归 Decoder-only Transformer，根据前文 Token 预测下一个 Token。问答、代码、工具参数和对话格式都可以统一为 Token 序列生成任务。

常见组件包括：

- **RoPE**：为注意力计算提供位置信息，并支持后续长度外推
- **RMSNorm 与 Pre-norm**：改善深层网络的训练稳定性
- **SwiGLU**：作为带门控的前馈网络，提高表达能力
- **GQA**：减少 Key/Value Head 数量，降低 KV Cache 成本
- **Dense 或 MoE FFN**：在计算密度、模型容量和部署成本之间取舍

不同版本主要调整这些组件的具体配置，而不是改变自回归语言建模的基本方式。

<h2 id="面试问题：GQA、QK-Norm 和 QKV bias 分别解决什么问题？">面试问题：GQA、QK-Norm 和 QKV bias 分别解决什么问题？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

三者作用在注意力计算的不同位置：

- **GQA（Grouped Query Attention）**：让多个 Query Head 共享较少的 Key/Value Head，减少 KV Cache 显存和读取开销，适合长上下文与高并发推理
- **QK-Norm**：对 Query 和 Key 做归一化，控制 Attention Logits 的数值尺度，降低大规模训练和长上下文下的不稳定风险
- **QKV bias**：是在 Q、K、V 投影中加入偏置项。早期 Qwen 和 Qwen2 使用该设计，Qwen3 去掉 QKV bias 并加入 QK-Norm

GQA 主要优化推理效率；QK-Norm 和 QKV bias 的取舍更偏训练稳定性与结构设计。

<h2 id="面试问题：Qwen 的 Dense 模型与 MoE 模型有什么区别？">面试问题：Qwen 的 Dense 模型与 MoE 模型有什么区别？</h2>

**难度评分：⭐⭐ (2/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

Dense 模型在一次前向计算中使用各层的主要参数，参数量增加通常会直接增加计算成本。MoE 模型把部分前馈网络替换为多个专家，并由 Router 为每个 Token 选择少量专家。

| 维度 | Dense 模型 | MoE 模型 |
| :--- | :--- | :--- |
| 参数使用 | 大部分参数参与每次计算 | 每个 Token 只激活少量专家 |
| 模型容量 | 与计算量联系较紧 | 总参数可明显大于激活参数 |
| 推理特点 | 结构简单、部署成熟 | 容量/计算比高，但通信更复杂 |
| 主要难点 | 大模型计算和显存成本高 | 路由、负载均衡、专家并行和 Kernel 优化 |

MoE 的价值不是“免费增加参数”，而是在计算量可控的前提下提高模型容量。专家参数仍需要存储，跨设备通信也会增加工程成本。

<h2 id="面试问题：Qwen3-235B-A22B 中的总参数和激活参数应该如何理解？">面试问题：Qwen3-235B-A22B 中的总参数和激活参数应该如何理解？</h2>

**难度评分：⭐⭐ (2/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

`235B-A22B` 表示模型总参数量约为 235B，但每个 Token 前向计算时只激活约 22B 参数。类似地，`30B-A3B` 表示总参数约 30B，每个 Token 激活约 3B。

这两个数字回答的是不同问题：

- **总参数量**反映模型需要保存的整体容量，影响权重存储、加载和分布式部署
- **激活参数量**更接近单个 Token 的实际计算规模，但不等于完整的延迟或显存成本

推理成本还受到上下文长度、KV Cache、批大小、专家通信、量化精度和推理框架影响。因此，不能只用激活参数量直接推断实际吞吐或硬件需求。

<h2 id="面试问题：Qwen2 和 Qwen3 的 MoE 设计有哪些变化？">面试问题：Qwen2 和 Qwen3 的 MoE 设计有哪些变化？</h2>

**难度评分：⭐⭐⭐⭐ (4/5) | 考察频率：⭐⭐⭐ (3/5)**

Qwen2 的代表性 MoE 模型是 57B-A14B，公开设计包含 Routed Experts 和 Shared Experts。Router 选择路由专家，共享专家则为所有 Token 提供较通用的计算路径。

Qwen3 的代表性 MoE 模型包括 30B-A3B 和 235B-A22B。公开报告披露其使用 128 个专家，每个 Token 激活 8 个专家，并取消 Qwen2.5-MoE 中的 Shared Experts。

这一变化说明 MoE 设计需要在三个目标之间权衡：专家分工是否充分、路由是否均衡，以及专家通信是否能够高效执行。专家数量更多不自动意味着效果更好，仍取决于训练数据、路由策略和系统实现。

<h1 id="3.Qwen 系列如何扩展长上下文能力？">3.Qwen 系列如何扩展长上下文能力？</h1>

<h2 id="面试问题：Qwen 系列的上下文长度是如何演化的？">面试问题：Qwen 系列的上下文长度是如何演化的？</h2>

**难度评分：⭐⭐ (2/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

Qwen 的上下文能力经历了从常规窗口到超长上下文专项模型的演化：

- 第一代 Qwen 主要是 2K/8K 量级，具体长度取决于模型版本
- Qwen2 通过 DCA 和 YaRN 扩展长上下文
- Qwen2.5 的部分主线模型支持 128K 上下文
- Qwen2.5-1M 将目标窗口扩展到 100 万 Token
- Qwen3 在长上下文数据训练的基础上，继续结合 RoPE 调整和长度外推

上下文长度增加后，主要问题从“输入是否放得下”转向“模型能否稳定利用信息，以及服务成本是否可接受”。

<h2 id="面试问题：RoPE、YaRN 和 DCA 在 Qwen 长上下文中分别起什么作用？">面试问题：RoPE、YaRN 和 DCA 在 Qwen 长上下文中分别起什么作用？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

- **RoPE** 把位置信息编码到 Query 和 Key 的旋转关系中，使注意力能够感知相对位置
- **YaRN** 对 RoPE 的频率或位置尺度进行调整，缓解推理长度超过训练长度时的位置失真
- **DCA（Dual Chunk Attention）** 将长序列划分为多个块，并分别处理块内、相邻块和跨块注意力，使模型能够复用短窗口中学到的注意力模式来扩展上下文

三者不能单独保证长上下文效果。位置外推还需要长文本训练数据、长上下文 SFT、注意力优化和真实长文任务评估配合。

<h2 id="面试问题：Qwen2.5-1M 如何把上下文扩展到 100 万 Token？">面试问题：Qwen2.5-1M 如何把上下文扩展到 100 万 Token？</h2>

**难度评分：⭐⭐⭐⭐ (4/5) | 考察频率：⭐⭐⭐ (3/5)**

Qwen2.5-1M 同时处理训练、位置编码和推理服务三个层面的问题。

##### 1. 训练侧

- 使用长数据合成和 Progressive Pretraining，让模型逐步适应更长序列
- 混合长短样本，避免模型只适应单一长度分布
- 加入长文档检索、跨段推理和长文本生成任务

##### 2. 位置与模型侧

- 调整 RoPE Base，并使用长度外推方法
- 通过多阶段 SFT，让模型学习如何使用远距离信息

##### 3. 推理侧

- 使用稀疏注意力减少不必要的注意力计算
- 使用 Chunked Prefill 分块处理超长输入，降低峰值显存
- 配合 Sparsity Refinement、Kernel 优化、并行和调度策略提高吞吐

因此，1M 上下文不是简单修改一个配置值，而是训练数据、位置编码、注意力计算和服务系统共同作用的结果。

<h2 id="面试问题：为什么支持 1M 上下文不等于能够稳定理解 1M Token？">面试问题：为什么支持 1M 上下文不等于能够稳定理解 1M Token？</h2>

**难度评分：⭐⭐ (2/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

标称上下文窗口说明模型可以接收多长的 Token 序列，但不代表它能同等准确地利用每个位置的信息。

长上下文仍可能出现：

- 中间位置的信息召回下降，即 Lost in the Middle
- 干扰信息增多后，模型选错证据或混淆来源
- 长距离检索成功，但跨段推理和全局一致性不足
- Prefill、KV Cache、延迟和调用成本快速增加

实际选型应关注有效上下文长度，而不是只看标称窗口。需要使用 Needle Test、长文档问答、跨段推理和真实业务材料验证模型表现。

<h1 id="4.Qwen 系列的后训练与推理能力是如何演化的？">4.Qwen 系列的后训练与推理能力是如何演化的？</h1>

<h2 id="面试问题：Qwen 从第一代到 Qwen3 的后训练路线发生了什么变化？">面试问题：Qwen 从第一代到 Qwen3 的后训练路线发生了什么变化？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

Qwen 的后训练目标从“让基座模型能够对话”逐步扩展到“让模型在复杂任务中稳定执行，并控制推理成本”。

| 版本 | 后训练重点 |
| :--- | :--- |
| Qwen | SFT、RLHF、ChatML、工具调用和对话对齐 |
| Qwen2 | SFT、DPO/RLHF，增加合成数据、执行反馈和约束检查 |
| Qwen2.5 | 超过百万级 SFT 样本与多阶段 RL，强化格式、长文本、结构化数据、数学和代码 |
| Qwen3 | 长 CoT Cold Start、Reasoning RL、Thinking/Non-thinking 融合和 General RL |

变化的核心是训练反馈越来越可验证，能力目标也从普通对话扩展到推理、代码、Agent 和多任务成本控制。

<h2 id="面试问题：SFT、RLHF、DPO 和 GRPO 在 Qwen 后训练中分别有什么作用？">面试问题：SFT、RLHF、DPO 和 GRPO 在 Qwen 后训练中分别有什么作用？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

- **SFT**：用高质量指令与回答数据教模型理解任务格式、回答方式和工具调用规范
- **RLHF**：根据人类偏好训练奖励模型，再通过强化学习优化回答偏好和安全行为
- **DPO**：直接利用偏好对调整模型，不需要显式执行完整的奖励模型强化学习流程
- **GRPO**：对同一问题采样多个回答，根据组内相对奖励优化模型，常用于可验证推理任务

这些方法不是互斥替代关系。常见流程是先用 SFT 建立稳定行为，再使用偏好优化或强化学习改善回答选择、推理和安全边界。

<h2 id="面试问题：Qwen3 的 Thinking、Non-thinking 和 Thinking Budget 分别是什么？">面试问题：Qwen3 的 Thinking、Non-thinking 和 Thinking Budget 分别是什么？</h2>

**难度评分：⭐⭐ (2/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

- **Thinking Mode**：允许模型使用更多推理 Token 处理数学、代码和多步骤问题，通常延迟和成本更高
- **Non-thinking Mode**：减少长推理过程，适合简单问答、摘要和低延迟任务
- **Thinking Budget**：限制 Thinking Mode 可以消耗的推理 Token 数量，用于控制质量、延迟和成本

Qwen3 的重点是让同一个模型覆盖快速回答和复杂推理，而不是为两类任务分别部署完全独立的模型。Thinking Token 更多通常能提供更充分的搜索空间，但不能保证答案一定正确，仍需要任务评估和外部验证。

<h2 id="面试问题：为什么 Qwen 的数学和代码训练重视可验证反馈？">面试问题：为什么 Qwen 的数学和代码训练重视可验证反馈？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

开放式文本的质量通常依赖人工偏好，而数学和代码任务往往具有更明确的验证信号：答案是否正确、程序能否编译、单元测试是否通过、执行结果是否满足约束。

Qwen 系列在数学和代码训练中使用的思路包括：

- Rejection Sampling：生成多个候选，再保留通过验证的样本
- Execution Feedback：运行代码或工具，根据执行结果判断输出质量
- Reward Model：学习候选答案的相对质量，并指导采样或强化学习
- Tool-Integrated Reasoning：在推理过程中调用计算器、解释器或其他工具

可验证反馈可以减少“语言上合理但结果错误”的样本，并为强化学习提供更稳定的奖励。不过，验证器本身的覆盖范围和正确性也需要评估。

<h1 id="5.Qwen 的衍生模型分别适合什么场景？">5.Qwen 的衍生模型分别适合什么场景？</h1>

<h2 id="面试问题：Qwen-Coder、Qwen-Math 和 Qwen-VL 分别解决什么问题？">面试问题：Qwen-Coder、Qwen-Math 和 Qwen-VL 分别解决什么问题？</h2>

**难度评分：⭐⭐ (2/5) | 考察频率：⭐⭐⭐⭐ (4/5)**

三类模型都基于 Qwen 语言底座，但训练数据和能力目标不同。

| 模型 | 主要能力 | 典型场景 |
| :--- | :--- | :--- |
| Qwen-Coder | 代码生成、补全、解释、修复和代码推理 | IDE 助手、代码审查、仓库问答、脚本生成 |
| Qwen-Math | 数学推理、CoT、TIR 和可验证求解 | 数学问答、竞赛题、计算辅助、推理研究 |
| Qwen-VL | 图像、文档、图表、OCR 和视觉问答 | 文档理解、图表分析、截图问答、视觉 Agent |

专项模型不一定在所有任务上都优于通用模型。选型时应比较目标任务效果、延迟、上下文、部署资源和工具生态。

<h2 id="面试问题：Qwen3-Coder-Next 为什么强调可执行环境和 Agentic Training？">面试问题：Qwen3-Coder-Next 为什么强调可执行环境和 Agentic Training？</h2>

**难度评分：⭐⭐⭐⭐ (4/5) | 考察频率：⭐⭐⭐ (3/5)**

传统代码生成主要判断文本是否像代码，而 Coding Agent 需要在仓库、终端和测试环境中完成任务。模型必须读取文件、选择工具、修改代码、运行测试，并根据错误信息继续迭代。

可执行环境提供了更强的训练反馈：

- 代码是否能够编译或运行
- 单元测试和集成测试是否通过
- 修改是否真正解决仓库问题
- 工具调用顺序和参数是否正确

Agentic Training 因此把训练目标从“生成一段合理代码”扩展为“在环境中完成任务”。这更接近 SWE-Bench、Terminal-Bench 等真实软件工程评测，但也增加了环境构建、权限隔离和奖励设计难度。

<h2 id="面试问题：Qwen-Omni 的 Thinker-Talker 架构如何支持全模态交互？">面试问题：Qwen-Omni 的 Thinker-Talker 架构如何支持全模态交互？</h2>

**难度评分：⭐⭐⭐ (3/5) | 考察频率：⭐⭐⭐ (3/5)**

Qwen-Omni 面向文本、图像、音频和视频的统一理解与生成。Thinker-Talker 架构把多模态理解推理与语音生成分成两个协作模块：

- **Thinker**：接收多模态输入，完成内容理解、推理和回答规划
- **Talker**：根据 Thinker 的结果生成语音单元，负责实时语音输出和交互体验

这种拆分允许理解和语音生成采用不同的模型结构、专家路由和时间节奏。对于流式语音，还需要处理文本 Token 与语音 Token 粒度不一致、首包延迟、韵律和长音视频上下文成本等问题。

<h2 id="面试问题：实际项目中应该如何选择 Qwen 模型？">面试问题：实际项目中应该如何选择 Qwen 模型？</h2>

**难度评分：⭐⭐ (2/5) | 考察频率：⭐⭐⭐⭐⭐ (5/5)**

选型应先确定任务、模态和部署约束，再比较模型大小与具体版本。

- **通用聊天、摘要和信息抽取**：优先选择 Qwen2.5 或 Qwen3 Instruct 类模型
- **复杂数学、代码和逻辑推理**：选择支持 Thinking 的 Qwen3，或 Qwen-Math 等专项模型
- **代码仓库修改和终端任务**：选择 Qwen-Coder 或面向 Coding Agent 的模型
- **长文档、日志和大规模代码上下文**：评估 Qwen2.5-1M 或其他长上下文版本的有效上下文能力
- **图像和文档理解**：选择 Qwen-VL
- **音频、视频和实时语音交互**：选择 Qwen-Omni 类模型

模型越大不一定越适合生产环境。还需要在业务评估集上比较准确率、稳定性、延迟、吞吐、显存、调用成本、许可证和部署方式。
